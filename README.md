# 【完全版】汎用・超自律型「AI日産1万行」実現プレイブック

本ドキュメントは、人間が「仕様書」や「設計書」を書くことを一切やめ、AIがプロンプトチェーンを通じて**「仕様策定」「基本設計」「実装計画」「実装」「テスト」「インフラ操作」**の全工程を自律実行するための、完全自動化マニュアルである。

---

## 0. 序章：AI駆動開発の「特異点」を越える

### 0.1 人間が書く設計書の「曖昧さ」がAIの生産性を殺す
AI駆動開発（AI-Driven Development）が失敗する最大の要因は、「人間が不完全な設計書を書き、それをAIに実装させようとする」ことにある。人間の自然言語による設計は必ずエッジケース（境界値、例外処理、非同期の競合）を見落とし、それがAIの「迷い」と「バグ」を生む。

### 0.2 要件（要求）だけを与え、設計とテストをAIに「丸投げ」する勇気
AIの圧倒的な計算量と静的解析能力を活かすには、**「要求（こんな機能が欲しい）」だけを人間が与え、以後はAI同士のフィードバックループの中で完結させる**必要がある。これが「Rail Coding（レール駆動）」の真髄である。

## 1. 2体のAI設定と役割分担 (Two-Agent Setup & Roles)

### 1.1 実装担当：Claude Code (The Doer)
- **役割**: プロンプト・仕様に基づくコードの生成、修正、リファクタリング。
- **設定**: `CLAUDE.md` に「実装の掟（例：抽象化レイヤーの強制、ピクセル直書き禁止）」を定義し、変更権限を与える。
- **コミット習慣**: 変更意図（Why）を抽出する `commit` スキルを使い、`main` ブランチへ直接コミットする。

### 1.2 監査担当：Gemini CLI (The Auditor / CAO)
- **役割**: 実装内容の論理的・構造的監査（CAOプロトコル）。コードの書き込みは禁止。
- **設定**: `GEMINI.md` に「3層監査命令（数学/性能、アーキテクチャ、自己批判）」を定義し、監査結果を `docs/10_ai_docs/` に永続化させる。
- **出力**: `Status: [PASS / WARNING / BLOCK]` を含む標準化された監査レポート。

### 1.3 Main-Branch Workflow (Atomic Commits via Prompts)
ブランチ管理のオーバーヘッドを排除し、`main` 上で「実装（Claude）→ 監査（Gemini）」の高速サイクルを回す。
- **プロンプトによる段階的コミットの徹底**: 各フェーズ（仕様、型定義、計画、実装、テスト）の完了ごとに、人間が**「コミットして」**と指示し、AIに `git commit` を実行させよ。
- **目的**: コミット履歴を「AIのための地図」として機能させる。これにより、監査担当（Gemini）は `git log` や `git diff` を参照するだけで、迷うことなく修正対象のファイルを特定できる。

## 2. AIの設定と環境構築 (Setup & System Prompting)

### 2.1 初期化プロンプト：AIに「怒られる環境」を作らせる
新規プロジェクト開始時、または既存プロジェクトへの導入時に以下のプロンプトを投下せよ。

> **【初期化プロンプト】**
> 「このプロジェクトを『AI駆動開発（日産1万行対応）』の基盤としてセットアップせよ。
> 1. TypeScript (`strict: true`) を導入せよ。
> 2. 超高速Linter `Biome` (または `oxlint`) を導入し、CIスクリプトを設定せよ。
> 3. `Husky` を導入し、型エラーがある状態での `git commit` を物理的にブロックせよ。
> 4. 人間の『コミットして』という指示で、実装の意図を自動付与してコミットする `commit` スキルを整備せよ。
> 実行完了後、`npm run typecheck` が通ることを確認し、最初のコミットを作成せよ。」

## 3. 超自律型・6段階開発ライフサイクル（段階的コミット形式）

各段階の完了後に、人間が「コミットして」と指示し、AIに履歴を刻ませることで、2体のAI間のコンテキスト同期と対象ファイルの特定を完璧にする。

### 第1段階：仕様の自律定義 (Autonomous Specification)
> **【プロンプト】**
> 「要求：『[要件内容]』から仕様書を `.ai_docs/specs/` に作成せよ。」
> （承認後）
> **「仕様定義が完了した。コミットして。」** → AIが `docs: [機能名]の仕様定義` として実行。

### 第2段階：基本設計と型駆動開発 (Architecture & Types)
> **【プロンプト】**
> 「仕様書に基づき、TypeScriptの型定義のみを作成せよ。」
> （型チェック後）
> **「型定義が完了した。コミットして。」** → AIが `feat: [機能名]の型定義とインターフェース` として実行。

### 第3段階：実装計画の自律立案 (Implementation Planning)
> **【プロンプト】**
> 「型定義に基づき、実装計画書を作成せよ。」
> （承認後）
> **「計画立案が完了した。コミットして。」** → AIが `docs: [機能名]の実装計画立案` として実行。

### 第4段階：自律的「実装」ループ (Autonomous Coding)
> **【プロンプト】**
> 「実装計画を順次実行せよ（テストは書かない）。」
> （完了後）
> **「実装が完了した。コミットして。」** → AIが `feat: [機能名]の実装完了（テスト未実装）` として実行。
> ※この履歴により、Geminiは「どのファイルが新規・修正されたか」を即座に把握できる。

### 第5段階：テスト仕様の策定とテスト実装 (Testing & Verification)
> **【プロンプト】**
> 「実装を読み込み、テスト仕様書を作成してテストを実装・実行せよ。」
> （パス後）
> **「テストが完了した。コミットして。」** → AIが `test: [機能名]のユニット/結合テスト実装` として実行。

### 第6段階：CAO監査と品質の永続化 (Final Audit & Report)
> **【プロンプト (Gemini向け)】**
> 「直近のコミット群を読み込み、CAOプロトコルで監査せよ。」
> （保存後）
> **「監査レポートの保存が完了した。コミットして。」** → AIが `docs: [機能名]の監査完了レポート` として実行。

### 第7段階：監査フィードバックと再実装ループ（人間による手動ブリッジ）
`main` ブランチ上での直接開発において、監査で `BLOCK` が出た場合は、人間がレポートを Claude に渡し、修正後に再度「コミットして」と命じて履歴を積み上げる。
> **【手動手順とプロンプト (Claude向け)】**
> 1. Gemini の出力した監査レポート `.ai_docs/reports/XX-REPORT.md` を Claude に読み込ませる。
> 2. 「監査レポートを読み込め。指摘された違反を修正するための実装計画を立て、修正とテストを実行せよ」と指示する。
> 3. 修正完了後、再び「実装が完了した。コミットして。」と指示し、Gemini に再監査を依頼する。

## 4. 第8段階：No-CLI 原則（完全プロンプト駆動オペレーション）

人間がターミナル操作を手放し、AIにすべてのCLI操作を委譲することが「完全自律」への最後の鍵である。

### 4.1 理由：コンテキストの完全同期
人間がコマンドを打つと、その結果ログはAIの記憶から消える。AIに打たせれば、AIはログを解析し、エラーを自律的に解決できる。段階ごとの「コミットして」というプロンプトによる指示も、AI自身が「自分が何をしたか」を履歴から正確に把握し、迷わずファイルを特定するために不可欠である。

## 5. 運用上のアンチパターンと回避策
- **【禁止】人間による中途半端なコード修正**: AIの設計思想を壊し、型エラーの連鎖を招く。
- **【禁止】Geminiの監査（BLOCK）を無視して開発を進める行為**: `main` 直コミット環境下でBLOCKを放置して進むと、設計が完全に崩壊する。必ず手動でClaudeにレポートを渡し、PASSになるまで修正ループを回せ。
- **【対策】無限ループ時のリセット**: 「一度立ち止まり、現在のディレクトリ構造とエラーログを整理して報告せよ」というプロンプトで復帰させる。
