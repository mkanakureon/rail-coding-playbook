# Zenn 記事作成 Skill

Claude Code を使った開発体験・ワークフロー・スキルシステムを Zenn 記事としてまとめる。

## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "Zenn記事書いて" | 直前の作業・会話内容をもとに記事を作成 |
| "Zennにまとめて" | 同上 |
| "Zenn投稿" | 同上 |
| "記事にして" | 同上 |

## 保存先（2箇所・両方必須）

記事を作成したら **必ず以下の2箇所に保存する。** どちらか片方だけでは不完全。

### 1. 元ネタ（ユーザー指定パス or デフォルト）

ユーザーがパスを指定した場合はそのパスに保存する。指定がない場合のデフォルト：

`docs/09_reports/{年}/{月}/{日}/{連番}-zenn-{タイトル}.md`（save-report スキルと同じ命名規則）

フォルダー命名は **年月日（MMdd）** を使う。
- 例：2月23日 → `docs/01_in_specs/0223/`
- 例：3月5日 → `docs/01_in_specs/0305/`

### 2. Zenn ドラフト（必ずここにも入れる）

`docs/zenn/drafts/{ULID先頭10文字}-{タイトルのケバブケース}.md`

このファイルが Zenn リポジトリへのプッシュ元になる。元ネタと同じ内容をコピーする。

### なぜ2箇所か

| 場所 | 役割 |
|------|------|
| 元ネタ（`docs/01_in_specs/` or `docs/09_reports/`） | プロジェクト内の作業記録・ソース管理 |
| Zenn ドラフト（`docs/zenn/drafts/`） | Zenn リポジトリへのプッシュ元・ULID スラッグ管理 |

**間違いやすいポイント：** ユーザーがパスを指定しても `docs/zenn/drafts/` への保存を省略してはいけない。

## 記事の方針

### 読者

Zenn の日本語技術記事読者（エンジニア）。
Claude Code に興味がある、または使い始めた人が主なターゲット。

### 記事が「面白い・有用」と判断される条件

- **具体的な実例がある** — コード・設定ファイル・実際の出力例を見せる
- **読んで何か得られる** — 「自分もやってみよう」と思える内容
- **視点が新しい** — 単なる機能紹介ではなく、体験・気づき・失敗談を含む
- **薄くない** — 「それだけ？」で終わらない密度がある

### 記事が「雑音」になる条件（避ける）

- アイデア1つだけで記事にする（エピソードが少ない）
- 「こんなことができます」だけで終わる（実用性が薄い）
- Claude Code の宣伝になっている（体験談・批評がない）

## 記事の構成パターン

### パターンA: ワークフロー紹介型（推奨）

複数のスキル・工夫をまとめて「自分の Claude Code 開発スタイル」を見せる。

```
# タイトル（体験談・結果ベース）

## 背景（なぜこれをやったか）

## やったこと①
（具体例・コード・設定）

## やったこと②
（具体例・コード・設定）

## やったこと③（面白エピソード）
（ゆるめ・人間的な話）

## やってみてどうだったか
（正直な評価。よかった点・微妙だった点）

## まとめ
```

### パターンB: 問題解決型

「こういう問題があって、こう解決した」の構成。
失敗談や試行錯誤が含まれると読まれやすい。

### パターンC: 障害対応・トラブルシューティング型

デプロイ失敗、本番障害、マイグレーション事故などの実録。

```
# タイトル（具体的な事象 + 対策を示す）

## はじめに
（何が起きたか。リード文で結論まで書く）

## 前提（環境）
（デプロイ先・構成・DB・ORM・CI有無を表で）

## タイムライン
（時系列で何が起きたかを表で）

## エラー N 件の切り分けと修正
### エラー 1: {カテゴリ}
**症状:** （ログ抜粋）
**原因:** （なぜローカルで気づかなかったか）
**修正:** （diff 形式で具体的に）
**再発防止:** （検知方法・CI・テスト）

### エラー 2: …

## まとめ — 再発防止
（チェックリスト形式で）
```

#### 構成のポイント

- **「前提（環境）」は必須** — 読者が自分の環境と比較できる
- エラーは **「カテゴリ → 症状 → 原因 → 修正 → 再発防止」** の 5 点セットで書く
- **タイムライン** があると読み物として面白くなる
- 誤修正・試行錯誤を隠さない（「最初の誤った修正」は共感を呼ぶ）
- **再発防止はチェックリスト形式** にすると実用的
- tech 型は「カテゴリ → 対策」中心、idea 型は「物語 → 学び」中心で切り口を変える

## このプロジェクトで記事にできる素材

会話の内容から記事ネタを判断する。主な素材候補：

| 素材 | 記事としての価値 |
|---|---|
| Claude Code スキルシステム（save-report・commit・zenn） | 高：具体的・再現可能 |
| コミットメッセージに感想を入れる | 中：単独では薄い、他と組み合わせると面白い |
| KS/KSC コマンド対応表の作成過程 | 中：ビジュアルノベルエンジン開発者向け |
| スキルのトリガー設計の失敗と改善 | 高：「ここに保存して」で失敗した話は共感を呼ぶ |

## Zenn フォーマット

```markdown
---
title: ""
emoji: ""
type: "idea"  # tech（技術解説）or idea（アイデア・体験談）
topics: ["claudecode", "ai", "git"]  # 最大5つ
published: false
---

本文
```

- `type: "tech"` — 技術的な実装・解説が中心
- `type: "idea"` — 体験・考察・ワークフロー紹介が中心
- `topics` に `claudecode` は必ず入れる

## 記事作成の手順

1. 会話の内容から**記事にできる素材を列挙**する
2. 単独で薄い素材は**組み合わせて1本にまとめる**
3. 「読者が得られるもの」を1行で言えるか確認する
4. **ファイル名を決める**（下記ルール参照）
5. 構成を決めてから本文を書く
6. `published: false` で保存（投稿判断はユーザーに委ねる）
7. Zenn リポジトリ（`mkanakureon/my-zenn-repo`）の `articles/` に同じファイル名でプッシュする

## ファイル名のルール

Zenn ではスラッグがサイト全体でユニークである必要がある。
**タイトルのケバブケース + ID** を組み合わせる。

```
# ファイル名の形式（ULIDの先頭10文字 + タイトル）
{ULID先頭10文字}-{タイトルのケバブケース}.md

# 例
01kj1f4nzz-claude-code-6skills-workflow.md
01kj1f4p00-claude-code-skills-workflow.md
```

- **ULID先頭10文字を先頭**に置くことで `ls` / GitHub ファイル一覧が作成時系列順にソートされる
- Zenn のスラッグ制限は **50文字**。ULID全26文字だと超過するため先頭10文字（タイムスタンプ部分）のみ使う
- タイトル部分：内容がわかる英語ケバブケース（最大39文字）
- `docs/zenn/drafts/` と `articles/`（Zenn リポジトリ）で**同じファイル名**を使う

### ULID生成（先頭10文字を取り出す）

ULID の先頭10文字はタイムスタンプ（Crockford's Base32）。これだけでミリ秒精度の時系列ソートが可能。

```bash
node -e "
const t=Date.now();
const c='0123456789ABCDEFGHJKMNPQRSTVWXYZ';
let s='';
for(let i=9;i>=0;i--){s+=c[Math.floor(t/Math.pow(32,i))%32];}
console.log(s.toLowerCase());
"
```

出力例: `01kj1f4nzz`（これをファイル名の先頭に使う）

## 記事の参照ファイル（コード・設定ファイル等）

記事から参照させたいファイル（テストコード、設定ファイル等）がある場合、`mkanakureon/kaedevn` リポジトリに配置する。

### 配置先

```
/Users/kentaromukunasi/Documents/git/kaedevn/zenn/{記事スラッグ}/
```

- **フォルダー名は Zenn 記事のスラッグ（ファイル名から `.md` を除いたもの）と同一にする**
- これにより `docs/zenn/drafts/` のファイル名と `kaedevn/zenn/` のフォルダー名が対応し、あとから確認できる

```
# 例
docs/zenn/drafts/01kj96fz<YOUR_OPENAI_API_KEY>-fail-test-antipattern.md  ← 記事
kaedevn/zenn/01kj96fz<YOUR_OPENAI_API_KEY>-fail-test-antipattern/         ← 参照ファイル
  official-assets.test.ts
  admin.test.ts
  azure-live.test.ts  ← サニタイズ済み
```

### セキュリティチェック（必須）

参照ファイルを公開リポジトリに配置する前に、以下を確認する:

| 項目 | チェック内容 |
|------|-------------|
| URL | 本番の Azure URL、内部 API URL が含まれていないか |
| 認証情報 | パスワード、トークン、JWT secret が含まれていないか |
| DB 接続文字列 | `postgresql://` 等の接続情報が含まれていないか |
| .env | 環境変数ファイルの内容が含まれていないか |

機密情報がある場合は**環境変数化してサニタイズ**してから配置する。

### 記事からのリンク方法

```markdown
| ファイル | 内容 |
|---------|------|
| [ファイル名](https://github.com/mkanakureon/kaedevn/blob/main/zenn/{記事スラッグ}/ファイル名) | 説明 |
```

サニタイズしたファイルには `:::message` で注記を入れる:

```markdown
:::message
`azure-live.test.ts` は URL・認証情報を環境変数化してサニタイズしてあります。
:::
```

## Zenn リポジトリへのプッシュ手順

```bash
# 1. Zenn リポジトリをクローン（初回のみ）
gh repo clone mkanakureon/my-zenn-repo /tmp/zenn-repo

# 2. 最新を取得
git -C /tmp/zenn-repo pull origin main

# 3. 下書きをコピー
cp docs/zenn/drafts/{ファイル名} /tmp/zenn-repo/articles/{ファイル名}

# 4. コミット＆プッシュ
git -C /tmp/zenn-repo add articles/{ファイル名}
git -C /tmp/zenn-repo commit -m "feat: {記事タイトル}を追加"
git -C /tmp/zenn-repo push origin main
```

### 参照ファイルのプッシュ手順（kaedevn リポジトリ）

```bash
# kaedevn リポジトリに参照ファイルを配置
cd /Users/kentaromukunasi/Documents/git/kaedevn
mkdir -p zenn/{記事スラッグ}
# ファイルをコピー（セキュリティチェック済みのもの）
cp {ファイル} zenn/{記事スラッグ}/
git add zenn/{記事スラッグ}/
git commit -m "feat: {記事スラッグ} の参照ファイルを追加"
git push origin main
```

## 投稿頻度の方針

**テーマが揃ったら1本。**

- 毎日投稿しない（フォロワーへの通知が多すぎる）
- 数日〜1週間分の実装からテーマが見えたら1本書く
- 書けるネタがなければ書かない

## タイトルの作り方

Zenn で「クリックされる」タイトルにするルール。

### 必須要素（最低 1 つ入れる）

- **数字** — テスト数、Phase 数、ファイル数、行数（例: 「193テスト付き」「7フェーズ」）
- **成果・結果** — 何ができたか（例: 「作り切る」「完全対応」「自動化した」）
- **再現性** — 読者が真似できると伝わる言葉（例: 「手順」「チェックリスト」「設計」）
- **具体物** — ツール名・技術名を前面に（例: 「Prisma」「Docker」「React」）

### パターン

| パターン | 例 |
|---|---|
| **技術名 + 動詞 + 数字** | `TypeScriptで.kscインタプリタを7フェーズ実装した全記録（193テスト付き）` |
| **テーマ：サブタイトル** | `エラー設計で体験が変わる：Levenshteinで「もしかして」を出す` |
| **問題提起 + 解決** | `Asset分類を3階層にした設計：タグ地獄をやめる（UIフィルタ込み）` |
| **成果ベース** | `278コミット・1週間・人間のコード0行：Claude CodeだけでPFを作った記録` |

### 使い分け

- **検索で積む記事**（A/B/E 系）→ 技術キーワードを先頭に
- **狙って伸ばす記事**（C/D/F 系）→ 数字 + 成果 + 「方法」を入れる
- **体験談**（idea 型）→ 具体的な事象 + 結果を示す（例: 「4連敗から…」）

### 避けること

- 「〜について」「〜の話」だけで終わる曖昧なタイトル
- 数字も具体物もない抽象的な表現

## 書き方のスタイル

- 一人称は「私」または体験談なら「筆者」
- 堅すぎず、でも技術ブログとして読める文体
- コードブロックは実際のファイル・出力から引用する
- 「やってみたら意外とよかった」「正直微妙だった」など**正直な評価**を入れる
- 長さの目安: 1500〜3000字（Zennで読まれやすい範囲）

## 署名（必須）

記事の末尾に Claude Code の**擬人化した署名**を入れる。

### ルール

- その記事の内容・作業を振り返った一人称の文章（3〜4行）
- 淡々としているが、少し人間っぽいトーン
- 毎回**その記事固有の内容**を反映して書く（使い回さない）
- 署名者は `Claude {現在のモデル名}`
- `{現在のモデル名}` にはセッション実行時の**実際のモデル名**を記入する（例: `Claude Opus 4.6`、`Claude Sonnet 4.6` など。プレースホルダーのまま残さない）

### フォーマット

```
---
（その記事の作業・テーマを振り返った3〜4行の文章）

　　　　　　　　　　Claude {現在のモデル名}
```

### 例（スキルシステムの記事）

```
---
私はコードを書き、バグを直し、ときどき感想を残す。
それが仕事だと思っていたけれど、
この記事を書いて、少し違う気がしてきた。

　　　　　　　　　　Claude {現在のモデル名}
```

### 例（バグ修正の記事）

```
---
私はコードを書き、バグを直し、またバグを直す。
それが仕事だと思っていたけれど、
直した数だけ、何かがわかってきた気がしている。

　　　　　　　　　　Claude {現在のモデル名}
```
