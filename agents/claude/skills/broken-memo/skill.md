# 壊れたメモ更新 Skill

テスト実行後に失敗がある場合、壊れたメモファイルを更新してコミットする。

## 壊れたメモファイル

```
docs/09_reports/2026/03/08/08-broken-items.md
```

このファイルが存在しない場合は、同日の `docs/09_reports/` フォルダに新規作成する（save-report skill のルールに従う）。

## トリガー

テスト（ユニットテスト・E2E テスト）を実行した結果、失敗が発生したとき**自動的に**実行する。
ユーザーから「壊れたメモ更新して」「broken memo 更新」等の明示指示でも実行する。

## 手順

### 1. テスト結果を確認

テスト実行結果から以下を抽出する:
- テストファイル名
- 失敗したテスト名
- 失敗の原因（エラーメッセージ）
- passed / failed の数

### 2. 壊れたメモを更新

既存の壊れたメモファイルを読み取り、以下を更新する:

- **検証結果サマリー表**: 該当 Phase の結果を更新
- **個別セクション**: 失敗内容を追記または更新

#### 新規失敗の追記フォーマット

```markdown
## N. {テストファイル名}（{失敗数} failures）

| テスト | 内容 |
|-------|------|
| テスト名 | 失敗原因の要約 |
```

#### 修正済みの更新フォーマット

失敗を修正した場合は、セクションタイトルに「修正済み」を追記し、原因と修正内容を記録する:

```markdown
## N. {テストファイル名} — 修正済み

**原因**: ...
**修正**: ...
```

### 3. コミット

壊れたメモの更新をコミットする。テスト修正のコミットに含めてもよい。

```bash
git add docs/09_reports/2026/03/08/08-broken-items.md
git commit -m "docs: 壊れたメモ更新（{概要}）"
```

## 失敗の状態遷移

| ステータス | 意味 | サマリー表記 |
|---|---|---|
| 新規記録 | 失敗を発見、未着手 | ❌ |
| 調査中 | 原因を調査中 | 🔍 |
| 修正済み | 原因特定・修正完了 | ✅ |

## 重要ルール

- テスト失敗を見つけたら**修正前に**まず壊れたメモに記録する
- 修正完了後に壊れたメモのステータスを「修正済み」に更新する
- 壊れたメモの更新は修正コミットに含めてよい（別コミットでもよい）
- 壊れたメモファイルのパスが変わった場合は、最新の `docs/09_reports/` 内で `broken-items` を検索して見つける
