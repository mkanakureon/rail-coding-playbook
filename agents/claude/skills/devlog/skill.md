# 開発日誌 Skill

1日のコミット履歴から「kaedevn開発日誌」を作成し、`docs/landing/devlog/{YYMM}/` に保存する。

## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "開発日誌書いて" | 今日のコミットから日誌を作成 |
| "devlog" | 同上 |
| "日誌" | 同上 |
| "今日の振り返り" | 同上 |
| "開発日誌書いて 2026-03-05" | 指定日のコミットから日誌を作成 |

## 保存先

```
docs/landing/devlog/{YYMM}/{YYYY-MM-DD}.md
```

- `{YYMM}`: 年月の短縮形（例: `2603` = 2026年3月）。1ヶ月分を同じフォルダに入れる
- 日付は対象日（デフォルトは今日）
- 1日1ファイル、同じ日に再実行したら上書き
- フォルダがなければ `mkdir -p` で作成

## 素材の収集手順

1. 対象日のコミットを時系列順に取得:
   ```bash
   git log --since="{日付}T00:00:00" --until="{翌日}T00:00:00" --all --reverse --format="%H %s"
   ```

2. 各コミットの変更統計を取得:
   ```bash
   git log --since="{日付}T00:00:00" --until="{翌日}T00:00:00" --all --reverse --stat --format="%n--- %s ---"
   ```

3. 合計行数を取得:
   ```bash
   git log --since="{日付}T00:00:00" --until="{翌日}T00:00:00" --all --numstat --format="" | awk '{ins+=$1; del+=$2} END {print ins, del}'
   ```

4. 必要に応じて個別コミットの diff を読む（大きな機能追加の詳細を書くとき）

## 文章のスタイル

### 人称

**あやふやにする。** 主語を省略し、誰がやったか明示しない。

- OK: 「取り掛かったのは〜」「追加した」「見つかった」「修正した」
- NG: 「私が〜」「開発者が〜」「Claude が〜」

### トーン

- 開発者日記っぽく、淡々と事実を語る
- 技術用語は使うが、理解してもらう必要はない（雰囲気が伝わればいい）
- 変更行数やファイル数を織り交ぜて具体性を出す
- 「地味だけど」「意外と」「結局」など、少し主観を混ぜる

### 読者ターゲット

ランディングページの訪問者。開発者とは限らない。
「このプロジェクトは活発に動いている」「中身のある開発をしている」と伝わることが目的。

## 記事の構成

```markdown
# kaedevn開発日誌 — {年}年{月}月{日}日

> {コミット数}コミット / +{追加行}行 / -{削除行}行

## {トピック1のタイトル}

（コミット内容を読み物風に展開。変更行数・ファイル数を含める）

## {トピック2のタイトル}

...

## {トピックNのタイトル}

...

――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――

（Claude Code の感想。10行以上。）

　　　　　　　　　　Claude {モデル名}
```

### 構成のルール

- **セクション見出し**: コミットメッセージそのままではなく、読み物として意味のあるタイトルにする
- **関連するコミットはまとめる**: 1コミット=1セクションではない。同じテーマのコミットは1セクションにまとめる
- **時系列を意識する**: 朝→昼→夕方の流れがあると読みやすい
- **コミット数が少ない日**: セクションを減らして1つ1つを深掘りする
- **コミット数が多い日**: 全コミットを網羅しつつ、メインの変更を厚く書く

### 本文の書き方

- 各セクションは 3〜8 行程度
- コードのファイル名やコマンド名はバッククォートで囲む
- 技術的な仕組み（「レクサーが〜」「API の 5 箇所で〜」）は説明しすぎず、事実として述べる
- バグの発見と修正は面白いので詳しめに書く
- 「追加して次のコミットで削除した」のような人間味のあるエピソードは残す

### 区切り線

本文と感想の間に長い罫線を入れる:

```
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
```

### Claude Code の感想（必須）

- **3〜5段落、各段落2〜3行**（計10行以上が目安）
- 1日の作業を振り返った Claude Code の一人称の文章
- 技術的な気づき、面白かったこと、印象に残ったバグ、全体の感触など
- 毎回その日固有の内容を反映して書く（使い回さない）
- 最終行に署名: `　　　　　　　　　　Claude {モデル名}`（全角スペースでインデント）

### 感想で触れるとよいテーマ

- その日一番インパクトの大きかった変更とその理由
- 予想外だったこと（バグの原因、影響範囲の広さなど）
- 「地味だけど大事」な変更への所感
- プロジェクト全体の進捗感・フェーズの変化
- 技術的な学び（「知っていたけど実際に踏むと違う」系）
- 1日の流れの振り返り（朝〜夜で何をやったか）

## コミットが少ない日・ない日

- **1〜3 コミット**: セクションは少なくし、1つ1つを深掘りする。感想も短くていい（5行程度）
- **0 コミット**: 日誌を作成しない。「今日はコミットがありませんでした」と伝える

## 過去の日誌を参照する

過去の日誌は `docs/landing/devlog/` にあるので、トーンの一貫性を保つために直近の日誌を読んでから書く。

```bash
ls docs/landing/devlog/
```
