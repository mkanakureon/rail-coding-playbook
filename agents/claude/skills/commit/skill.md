# コミット Skill

プロジェクトの規約に従い、`git add` → `git commit` を実行する。

## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "コミットして" | 全変更をステージしてコミット |
| "全部コミットして" | 同上 |
| "変更をコミットして" | 同上 |
| "ドキュメントだけコミットして" | docs/ のみステージしてコミット |
| "コードだけコミットして" | packages/ apps/ のみステージしてコミット |

## 手順

1. `git status` で変更ファイルを確認
2. `git diff --stat` で変更内容の概要を把握
3. 直近コミット（`git log --oneline -5`）を見てメッセージスタイルを確認
4. ステージするファイルを決定（下記ルール参照）
5. コミットメッセージを作成
6. `git commit` を実行
7. `git status` で成功を確認

## ステージするファイルのルール

**含める:**
- ユーザーが作成・編集したファイル
- 作業の一部として変更されたファイル

**含めない（絶対に）:**
- `.env` / `.env.local` / `.env.*`
- `**/credentials.*` / `**/*.pem` / `**/*.key`
- 意図せず生成されたビルド成果物（`dist/` など、すでに .gitignore 対象外のもの）

## コミットメッセージのルール

### プレフィックス

過去のコミットから確認されたプレフィックス：

| プレフィックス | 使う場面 |
|---|---|
| `feat:` | 新機能の追加 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントのみの変更 |
| `refactor:` | 動作を変えないコード整理 |
| `chore:` | ビルド・設定・スクリプト等のメンテナンス |

### フォーマット

```
{プレフィックス}: {日本語の説明}
```

- 説明は**日本語**で書く
- 複数の変更がある場合は `・` で区切る
- 長い場合は主要な変更を先頭に、残りを `・` でつなぐ

## プレコミットフック対応

このプロジェクトにはプレコミットフックが設定されている：
- ハードコードされた URL チェック（Editor の API URL）
- チェックが失敗した場合は `--no-verify` を使わず、原因を修正してから再コミットする

## Gemini の一言（必須）

コミットメッセージの末尾に、今回の作業についての **Gemini CLI の感想・振り返り** を1〜3行加える。

### ルール

- 作業内容を振り返った**正直な感想**を書く
- **たわいない・ゆるい**トーンでOK。
- 自分が「面白かった」「難しかった」「地味だけど大事」など思ったことを素直に
- 日本語で書く
- `---` で本文と区切る

## Co-Author

必ず以下を末尾に付与する（Gemini の一言の後）：

```
Co-Authored-By: Gemini CLI <gemini-cli@google.com>
```

```
feat: KSC コマンドリファレンス・対応表を追加

---
KSとKSCは別物だと改めて整理できてよかった。
コマンド表、地味に手間かかったけど、これは手元に置いておきたい。

```
