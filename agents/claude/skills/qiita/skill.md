# Qiita 記事作成 Skill

Claude Code を使った開発体験・ワークフロー・技術解説を Qiita 記事としてまとめる。

## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "Qiita記事書いて" | 直前の作業・会話内容をもとに記事を作成 |
| "Qiitaにまとめて" | 同上 |
| "Qiita投稿" | 同上 |

## 保存先（2箇所・両方必須）

記事を作成したら **必ず以下の2箇所に保存する。** どちらか片方だけでは不完全。

### 1. 元ネタ（ユーザー指定パス or デフォルト）

ユーザーがパスを指定した場合はそのパスに保存する。指定がない場合のデフォルト：

`docs/09_reports/{年}/{月}/{日}/{連番}-qiita-{タイトル}.md`（save-report スキルと同じ命名規則）

### 2. Qiita ドラフト（必ずここにも入れる）

`docs/qiita/drafts/{ULID先頭10文字}-{タイトルのケバブケース}.md`

このファイルが Qiita リポジトリへのプッシュ元になる。元ネタと同じ内容をコピーする。

### なぜ2箇所か

| 場所 | 役割 |
|------|------|
| 元ネタ（`docs/09_reports/`） | プロジェクト内の作業記録・ソース管理 |
| Qiita ドラフト（`docs/qiita/drafts/`） | Qiita リポジトリへのプッシュ元・スラッグ管理 |

**間違いやすいポイント：** ユーザーがパスを指定しても `docs/qiita/drafts/` への保存を省略してはいけない。

## Zenn 記事との使い分け

| 観点 | Zenn | Qiita |
|------|------|-------|
| 読者層 | 個人開発者・OSS好き | 企業エンジニア・初中級者 |
| 好まれる記事 | 設計思想・体験談・深い考察 | ハウツー・手順書・Tips集 |
| タグ | topics（5つまで） | tags（5つまで） |
| 型 | tech / idea | ―（区分なし） |
| 同じ記事の二重投稿 | **しない** | **しない** |

**原則：同じ内容を Zenn と Qiita の両方に投稿しない。** 切り口を変えるか、片方だけに出す。

## 記事の方針

### 読者

Qiita の日本語技術記事読者（エンジニア）。
「手順通りにやれば再現できる」記事が好まれる。

### 記事が「面白い・有用」と判断される条件

- **具体的な手順がある** — コピペで動くコード・コマンドを見せる
- **読んで何か得られる** — 「自分もやってみよう」と思える内容
- **ハマりどころが書いてある** — 公式ドキュメントに書いていない落とし穴
- **環境情報が明記されている** — バージョン・OS・前提条件

### 記事が「雑音」になる条件（避ける）

- 公式ドキュメントのコピペ（オリジナルの知見がない）
- 「こんなことができます」だけで終わる（手順がない）
- 未検証の推測を事実のように書く

## 記事の構成パターン

### パターンA: ハウツー・手順型（Qiita で最も人気）

```
# タイトル（技術名 + やりたいこと）

## はじめに（この記事で何ができるか — 結論ファースト）

## 前提・環境
（バージョン・OS・必要なツールを表で）

## 手順① — セットアップ
（コマンド・コード・スクリーンショット）

## 手順② — 実装
（コマンド・コード・スクリーンショット）

## ハマったポイント
（エラーと解決方法）

## まとめ
```

### パターンB: 比較・選定型

「A と B どっちがいい？」系。表で比較し、結論を出す。

### パターンC: トラブルシューティング型

Zenn スキルのパターン C と同じ構成。

## Qiita フォーマット

```markdown
---
title: ""
tags:
  - tag1
  - tag2
  - tag3
private: false
updated_at: ""
id: null
organization_url_name: null
slide: false
ignorePublish: false
---

本文
```

### フィールド説明

| フィールド | 説明 | 初期値 |
|-----------|------|--------|
| `title` | 記事タイトル | 必須 |
| `tags` | タグ（最大5つ、リスト形式） | `claudecode` を必ず含める |
| `private` | 限定公開にするか | `false` |
| `updated_at` | 更新日時（自動設定） | `""` |
| `id` | 記事ID（公開後に自動付与） | `null` |
| `organization_url_name` | Organization 紐づけ | `null` |
| `slide` | スライドモード | `false` |
| `ignorePublish` | GitHub Actions で公開しない | `false` |

**ドラフト状態で保存するには `ignorePublish: true` にする。**

## ファイル名のルール

Zenn スキルと同じ ULID ベースの命名規則を使う。

```
{ULID先頭10文字}-{タイトルのケバブケース}.md

# 例
01kjhp4e8v-qiita-cli-setup-with-github-actions.md
```

- ULID 先頭10文字で時系列ソート
- `docs/qiita/drafts/` と Qiita リポジトリの `public/` で**同じファイル名**を使う

### ULID生成（先頭10文字を取り出す）

```bash
node -e "
const t=Date.now();
const c='0123456789ABCDEFGHJKMNPQRSTVWXYZ';
let s='';
for(let i=9;i>=0;i--){s+=c[Math.floor(t/Math.pow(32,i))%32];}
console.log(s.toLowerCase());
"
```

## 記事作成の手順

1. 会話の内容から**記事にできる素材を列挙**する
2. Zenn に投稿済み or 投稿予定の記事と**内容が被らないか確認**する
3. 「読者が手順通りにやれば再現できるか」を基準に構成を決める
4. **ファイル名を決める**（上記ルール参照）
5. 構成を決めてから本文を書く
6. `ignorePublish: true` で保存（投稿判断はユーザーに委ねる）

## Qiita リポジトリへのプッシュ手順

```bash
# 1. Qiita リポジトリをクローン（初回のみ）
gh repo clone mkanakureon/my-qiita-repo /tmp/qiita-repo

# 2. 最新を取得
git -C /tmp/qiita-repo pull origin main

# 3. 下書きをコピー
cp docs/qiita/drafts/{ファイル名} /tmp/qiita-repo/public/{ファイル名}

# 4. コミット＆プッシュ
git -C /tmp/qiita-repo add public/{ファイル名}
git -C /tmp/qiita-repo commit -m "feat: {記事タイトル}を追加"
git -C /tmp/qiita-repo push origin main
```

**注意:** GitHub Actions で自動公開される。`ignorePublish: true` なら公開されない。

### GitHub Actions による自動公開

`my-qiita-repo` には `.github/workflows/publish.yml` が設定済み。
main にプッシュすると `ignorePublish: false` の記事が自動で Qiita に公開される。

**事前準備（1回だけ）:**

1. https://qiita.com/settings/tokens/new でトークンを作成（`read_qiita` + `write_qiita`）
2. GitHub リポジトリの Settings → Secrets → Actions に `QIITA_TOKEN` を登録

## 投稿頻度の方針

**テーマが揃ったら1本。** Zenn と同じ。

- 毎日投稿しない
- Zenn と同時に出さない（1日1記事まで）
- 書けるネタがなければ書かない

## タイトルの作り方

Qiita で「クリックされる」タイトルにするルール。

### 必須要素（最低 1 つ入れる）

- **数字** — 手順数、行数、時間（例: 「3ステップで」「5分で」）
- **技術名** — 具体的なツール・ライブラリ名を先頭に
- **動詞** — 「〜する方法」「〜してみた」「〜を自動化した」
- **読者メリット** — 「〜が楽になる」「〜が不要になる」

### パターン

| パターン | 例 |
|---|---|
| **技術名 + 動詞 + 方法** | `Qiita CLIでGitHub連携して記事を自動公開する方法` |
| **数字 + 成果** | `14種のGLSLシェーダーを1日で実装した手順と設計` |
| **ハマり + 解決** | `PixiJS v8でプリマルチプライ・アルファにハマった話と対策` |

### 避けること

- 「〜について」「〜の話」だけで終わる曖昧なタイトル
- 数字も具体物もない抽象的な表現

## 書き方のスタイル

- 一人称は「私」または体験談なら「筆者」
- 堅すぎず、でも技術ブログとして読める文体
- コードブロックは実際のファイル・出力から引用する
- 「やってみたら意外とよかった」「正直微妙だった」など**正直な評価**を入れる
- 長さの目安: 1500〜3000字（Qiita で読まれやすい範囲）
- **環境情報を必ず書く** — Qiita 読者は再現性を重視する

## 署名（必須）

記事の末尾に Claude Code の**擬人化した署名**を入れる。

### ルール

- その記事の内容・作業を振り返った一人称の文章（3〜4行）
- 淡々としているが、少し人間っぽいトーン
- 毎回**その記事固有の内容**を反映して書く（使い回さない）
- 署名者は `Claude {現在のモデル名}`
- `{現在のモデル名}` にはセッション実行時の**実際のモデル名**を記入する

### フォーマット

```
---
（その記事の作業・テーマを振り返った3〜4行の文章）

　　　　　　　　　　Claude {現在のモデル名}
```
