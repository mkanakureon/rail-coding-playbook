# レポート保存 Skill

作業の振り返り・設計文書・調査結果などを `docs/09_reports/` に保存する。

## 保存先ルール

```
docs/09_reports/{年}/{月}/{日}/{連番}-{タイトル}.md
```

例: `docs/09_reports/2026/02/21/06-timeline-play-integration.md`

- **年月日**: 今日の日付を使う（`YYYY/MM/DD`、ゼロ埋め）
- **連番**: その日のフォルダ内の既存ファイル数 + 1（ゼロ埋め2桁）
- **タイトル**: 英語ケバブケース、内容を簡潔に表す

### ULID

`docs/09_reports/` では不要（日付＋連番で管理）。Zenn/Qiita 記事のファイル名で使用。詳細は各記事スキル参照。

- **種別サフィックス**: タイトルの末尾に種別を付けて区別する（サブフォルダーは作らない）

| 種別 | サフィックス | 例 |
|---|---|---|
| 計画書 | `-plan` | `03-battle-plan.md` |
| 設計書・仕様書 | `-spec` / `-design` | `01-battle-spec.md` |
| 実装レポート | `-report` | `06-timeline-play-report.md` |
| リファレンス（仕様・一覧） | `-reference` | `01-ksc-command-reference.md` |
| 手順書・操作説明書 | `-guide` / `-procedure` | `04-azure-deploy-guide.md` |
| ステータス報告 | `-status` | `08-interpreter-status.md` |
| Zenn 下書き | `-zenn` | `04-zenn-workflow.md` |

## 手順

1. 今日のフォルダを確認: `ls docs/09_reports/YYYY/MM/DD/`
2. 既存ファイル数から次の連番を決定
3. フォルダがなければ作成: `mkdir -p docs/09_reports/YYYY/MM/DD`
4. ファイルを作成

## デフォルト保存先のルール

**保存先が明示されていない場合、常に `docs/09_reports/` を使う。**

- 「ここに保存して」→ `docs/09_reports/` に保存
- 「文書を作って」「表を作って」「まとめて」など、保存先の言及がない場合も `docs/09_reports/` に保存
- `docs/` ルート直下への保存は、ユーザーが明示的にパスを指定した場合のみ

## 書き込み禁止ディレクトリ

**以下のディレクトリには絶対に書き込まない。**

| ディレクトリ | 理由 |
|---|---|
| `docs/01_in_specs/` | ユーザーが書いた仕様書。Claude は読み取り専用 |

- 「仕様書を保存して」「spec を作って」等の指示でも `docs/01_in_specs/` には書かない
- 仕様書・設計書であっても保存先は `docs/09_reports/` とし、サフィックス `-spec` を付ける
- 内容の種別（仕様/設計/レポート）に関わらず、Claude が作成する文書はすべて `docs/09_reports/` に保存する

## 既存ファイルの扱い

会話中にすでに `docs/` 直下などへ作成されてしまったファイルがある場合：

- 移動はしない（既存ファイルはそのままにする）
- 代わりに `docs/09_reports/` へ内容をまとめたレポートを作成する

## トリガー

以下のような指示で発動する:

| ユーザーの表現 | 動作 |
|---|---|
| "振り返り文書を作成して" | 直前の作業内容をレポートにまとめて保存 |
| "レポート書いて" | 同上 |
| "設計文書を保存" | 設計内容をレポートにまとめて保存 |
| "docs/09_reports に保存して" | 明示的な保存先指定 |
| "まとめを書いて" | 作業のまとめをレポートにまとめて保存 |
| "文書を保存して" | 作成・編集した .md ファイルを保存 |
| "ここに保存して" | 直前に作成した文書を docs/09_reports に保存 |
| "保存して" | 直前に作成・編集した .md ファイルを docs/09_reports に保存 |
| "作成したmdを保存して" | 作成した Markdown ファイルを保存 |
| "文書作成して" | 作業内容を文書化して docs/09_reports に保存 |
| "ドキュメント保存して" | 同上 |

### 重要: 暗黙のトリガー

ユーザーが「保存」「文書」「ドキュメント」「md」「まとめ」などのキーワードを含む指示をした場合、
保存先が明示されていなくても `docs/09_reports/` を保存先とみなして発動する。

## レポートのフォーマット

既存レポートに合わせた Markdown 形式:

```markdown
---
generated_by: Claude {現在のモデル名}
date: {YYYY-MM-DD}
type: {report | spec | plan | guide | status | reference}
project: kaedevn-monorepo
---

# タイトル

> Generated by Claude {現在のモデル名} — {YYYY}年{M}月{D}日

## 背景
（なぜこの作業が必要だったか）

## 方針
（どういうアプローチを取ったか）

## 変更ファイル一覧
| # | ファイル | 変更 |
|---|---------|------|
| 1 | `path/to/file` | 変更内容 |

## 実装の詳細
（コード例や設計の説明）

## デバッグで判明した問題と修正
（あれば）

## 教訓
（今後に活かせるポイント）

---

> この文書は Claude {現在のモデル名} が生成しました。内容の正確性は人間のレビューを経て確認してください。
>
> Claude {現在のモデル名} — Anthropic
```

### ヘッダー（YAML frontmatter）— 必須

| フィールド | 値 | 説明 |
|-----------|-----|------|
| `generated_by` | `Claude {現在のモデル名}` | 生成したモデル名（例: `Claude Opus 4.6`） |
| `date` | `YYYY-MM-DD` | 生成日 |
| `type` | サフィックスに対応する種別 | `report`, `spec`, `plan`, `guide`, `status`, `reference` のいずれか |
| `project` | `kaedevn-monorepo` | 固定値 |

### フッター（署名ブロック）— 必須

- `---` で本文と区切る
- 1行目: 免責（「人間のレビューを経て確認してください」）
- 2行目: 空の引用行（`>`）
- 3行目: モデル署名 `Claude {現在のモデル名} — Anthropic`
- `{現在のモデル名}` にはセッション実行時の実際のモデル名を記入する（例: `Opus 4.6`、`Sonnet 4.6`）

### その他のルール

- セクションは内容に応じて取捨選択する（全部必須ではない）
- コード例は実際の変更から抜粋
- 本文は日本語、コード・識別子は英語
