# レポート保存 (Save Report) Skill

作業の振り返り・コードレビュー・アーキテクチャ分析・調査結果などを `docs/10_ai_docs/` に保存する。

## 保存先ルール (GEMINI.md 準拠)

**保存先が明示されていない場合、AIが作成する文書は常に `docs/10_ai_docs/` を使う。**
（`docs/09_reports/` は Claude Code 用であり、Gemini CLI は使用しないこと）

```
docs/10_ai_docs/{YYYY}/{MM}/{DD}/{連番}-{CATEGORY}.md
```

例: `docs/10_ai_docs/2026/02/21/01-CODE_REVIEW.md`

- **年月日**: 今日の日付を使う（`YYYY/MM/DD`、ゼロ埋め）
- **連番**: その日のフォルダ内の既存ファイル数 + 1（ゼロ埋め2桁）
- **カテゴリ**: 英語の**大文字**スネーク/ケバブケース、内容を簡潔に表す（`CODE_REVIEW`, `ARCH_ANALYSIS`, `BUG_REPORT` など）

## 手順

1. 今日のフォルダを確認: `list_directory` ツールなどで `docs/10_ai_docs/YYYY/MM/DD/` を確認する
2. 既存ファイル数から次の連番（XX）を決定する
3. フォルダがなければ自動作成されるため、そのまま `write_file` ツールでファイルを作成する

## トリガー

以下のような指示で発動する:

| ユーザーの表現 | 動作 |
|---|---|
| "振り返り文書を作成して" | 直前の作業内容をレポートにまとめて保存 |
| "レポート書いて" | 同上 |
| "設計文書を保存" | 設計内容をレポートにまとめて保存 |
| "まとめを書いて" | 作業のまとめをレポートにまとめて保存 |
| "文書を保存して" | 作成・編集した内容を保存 |
| "文書作成して" | 作業内容を文書化して docs/10_ai_docs に保存 |
| "ドキュメント保存して" | 同上 |

### 重要: 暗黙のトリガー

ユーザーが「保存」「文書」「ドキュメント」「レポート」「まとめ」などのキーワードを含む指示をした場合、保存先が明示されていなくても `docs/10_ai_docs/` を保存先とみなして発動する。

## レポートのフォーマット (Generative AI Standard)

Gemini が作成するすべての文書は、以下の標準フォーマットに従うこと。

```markdown
---
generated_by: generative-ai (Gemini CLI)
model: Gemini 2.0 Flash
date: {YYYY-MM-DD}
---

# タイトル

> **作成日**: {YYYY-MM-DD}
> **監査・執筆**: Gemini CLI (CAO Mode)

## 背景と目的
（なぜこの作業が必要だったか、何を調べたか）

## 作業・調査の内容
（どういうアプローチを取ったか、何を実装・修正したか）

## 重要な変更点・発見
（コード例や設計の説明、判明した事実）

## 結論・今後の課題
（今後に活かせるポイント）

---
*Generated by **Gemini 2.0 Flash***
*Authorized by Gemini CLI / Chief Audit Officer*
```

- **フロントマター**: `---` で囲まれたメタデータ（YAML）を冒頭に付与。
- **フッター署名**: 最後に水平線とモデル名 `Gemini 2.0 Flash` を明記。
- セクションは内容に応じて取捨選択する。
- 本文は日本語、コード・識別子は英語。
