## 認証 E2E テストの安定化

34 テストが 3 ワーカー並列で動く認証 E2E テスト群。問題は `apiLimiter`（200/min）が並列テストの API 呼び出しで枯渇し、`/api/auth/me` が 429 を返してトークンがクリアされるという Flaky test の根本原因だった。`rate-limit.ts` の apiLimiter を `API_RATE_LIMIT` 環境変数で設定可能にし、ローカル環境では 1000 に引き上げて解決。フォームログインの代わりにトークン注入パターン（`injectAuth(page)`）を採用して、loginLimiter の消費も回避した。

認証テストの Flaky 問題は技術的に面白かった。レートリミッターが並列テストで枯渇するという現象は、テスト環境と本番環境の非対称性が生む典型的な問題。本番では 200/min で十分でも、テストでは数十のリクエストが一瞬で飛ぶ。環境変数でリミットを切り替えられるようにしたのは正解だが、そもそもテスト用のバイパスを設けるべきか、リミッターのモック化が正しいかは議論の余地がある。今回は環境変数方式を選んだが、テスト専用ミドルウェアの差し替えもいずれ検討したい。

Static Web Apps のビルド順序問題は、モノレポならではの罠。ローカルでは `node_modules` にビルド済みパッケージが残っているから動くが、CI のクリーン環境では依存パッケージが未ビルドのまま参照される。Turborepo や Nx を使えば依存グラフに基づいたビルド順序を自動解決できるが、現時点ではシンプルに npm scripts の順序制御で対応している。

Malformed JSON の原因が zsh の `!` エスケープだったのは、盲点だった。`curl` のリクエストボディに `!` を含む文字列を渡すと、zsh がヒストリ展開として解釈してしまう。シングルクォートで囲めば防げるが、ダブルクォートの中に JSON を書くケースでは意外と見落とす。シェルスクリプトのデバッグは、言語仕様の問題ではなくシェルの挙動に起因することが多く、再現性の確保が難しい。
