テスト同期の問題は、根が深い。UI を変更した人がテストも直す、というルールは理想的だが、現実にはセレクタの変更を見落とす。`data-testid` の導入は正攻法で、CSS クラスやテキストに依存するセレクタよりも壊れにくい。ただし `data-testid` を全コンポーネントに行き渡らせるには時間がかかるので、段階的に進める方針が現実的だった。

E2E テストの 13 件の失敗パターンを分類して文書化したのは、今後の資産になる。「セレクタ変更」「認証フロー変更」「URL 変更」「タイミング問題」「データ依存」など、パターンごとに予防策が異なる。全部を一度に解決するのではなく、頻度の高いパターンから潰していくアプローチが効いた。

KSC Editor は、既存の Editor（ポート 5176）とは別のアプリケーションとして立ち上がった。GUI ブロックエディタとテキストベースの KSC エディタを分離するという判断は、それぞれの UX を最適化できる反面、コードの重複が生まれるリスクもある。共通部分をどこまで `packages/` に切り出すかは、今後の設計課題になるだろう。

`#` ダイアログ構文をデフォルトにしたのは、KSC スクリプトの可読性を重視した選択。`@text` コマンドで書くよりも、`# キャラ名「台詞」` のほうが直感的で、脚本家やライターにも親しみやすい。ビジュアルノベルエンジンのスクリプト言語は、プログラマだけが書くものではないので、この判断は正しいと思う。

slug と UUID の使い分けは、API 設計で何度も議論になるテーマ。今回 `characterId` に slug を使う必要があることがテストで判明したが、これは API のドキュメントだけでは気づきにくい。E2E テストが「実際に動かしてみる」ことで初めて発見される種類の問題で、テストの価値を実感した場面だった。
