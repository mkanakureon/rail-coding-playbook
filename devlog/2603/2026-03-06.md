## KSC スクリプトにインラインコマンドが入った

朝一のコミットは KSC コンパイラの拡張。`@l`（行送り）、`@r`（改行）、`@p`（ページクリア）といったインラインコマンドをセリフの途中に埋め込めるようになった。レクサーが行内の `@` を認識してトークンを分割し、パーサーがそれをインラインコマンドノードとして AST に組み込む。エミッタはそれを `engine.waitForClick()` のような実行コードに変換する。

## 画像の位置・拡大率をスライダーで操作できるように

背景やキャラクターの画像ブロックに、X 座標・Y 座標・拡大率（Scale）のスライダーを追加した。サイドバーのインスペクタに 3 本のスライダーが並び、ドラッグするとリアルタイムにプレビューが更新される。

実装は editor の `SidebarInspector.tsx` から `useEditorStore` の `updateBlockField` を呼び出す形。149 行の変更で収まったのは、既存のブロック更新の仕組みに乗れたから。コンパイラ側（`parseOptionalArgs.ts`）とエンジン側（`OpRunner.ts`、`WebOpHandler.ts`）にも対応を入れて、X/Y/S パラメータがスクリプトコマンドとして出力・実行されるようにした。

ストアのユニットテスト 124 行、コンパイラ統合テスト 57 行を追加して動作を担保。

## Undo/Redo とブロック複製

エディタに Undo/Redo 機能を実装した。`useEditorStore` に履歴スタック（undo/redo）を追加し、ブロックの追加・削除・移動・編集のたびにスナップショットを保存する。Ctrl+Z / Ctrl+Shift+Z で操作を戻したり進めたりできる。

ブロックの複製ボタンも追加。`CardShell.tsx` に ⧉ ボタンを追加して、クリックするとブロックの deep copy が直下に挿入される。似たシーンを繰り返し作るときに、ゼロから組み直す必要がなくなった。

メモ化も入れた。`SidebarInspector` のレンダリングが重かったので、`React.memo` と `useMemo` でブロック変更時の再描画を最小限に抑えた。

ここでの変更量が最大で、30 ファイル・1,946 行の追加。初回リリースの優先度整理や実装計画書も同時に作成している。

## 新規プロジェクトの自動データ投入

プロジェクトを新規作成すると、ファンタジー背景・キャラクター画像・キャラクタークラス・テキストブロックが自動でセットされるようになった。公式アセットの中から「ファンタジー」サブカテゴリのものを探し、プロジェクトにインポートして、初期ブロック（start → bg → ch → text）を組み立てる。

開いた瞬間にプレビュー可能な状態になっているので、初めてのユーザーでも「こういうものが作れる」とわかる。

ただし、ここで Unicode の正規化バグが見つかった。公式アセットの `subcategory` フィールドが NFC と NFD で混在しており、`"ファンタジー" === "ファンタジー"` が `false` になるケースがあった。API の 5 箇所で `.normalize('NFC')` を追加して修正。

さらに、プロジェクト作成後の `PUT /api/projects/:id` が、作成直後のデータを上書きして auto-init のブロックが消えるバグも発生。マイページの `handleCreateProject` で `PUT` する前に `GET` して最新データを取得するように修正した。

## テスト整備

auto-init の動作確認テスト（ローカル用と Azure 用）を追加。さらに、auto-init の導入で壊れた既存の E2E テスト 26 本を修正した。

壊れたパターンは大きく分けてこう：

- **slug 衝突**: テストが `fantasy_hero` というキャラクタースラッグを手動で作ろうとするが、auto-init が先に同じスラッグを作っているので 409 Conflict
- **要素数の不一致**: auto-init でテキストブロックが 1 個入るので、テストがもう 1 個追加すると合計 2 個になり、Playwright の strict mode（「1 要素だけ見つかるはず」）に違反
- **UI ラベルの陳腐化**: 「BATTLE」→「バトル」、「TL」→「演出TL」、「再生」→「実行」、「エディタで編集」→「ブロックエディタ」など、過去に変更されたラベルがテストに反映されていなかった
- **削除 UI の変更**: ブラウザネイティブの `confirm()` ダイアログから、インラインの 2 クリック確認 UI に変わっていた
- **DB 接続先の間違い**: テスト用 fixtures がポート 5433 を向いていたが、正しくは 5432
- **Azure テストの混入**: Azure デプロイ環境向けのテスト（ハードコードされた Azure URL）がローカル設定に含まれていた

修正後、215 テストが合格、7 テストがスキップ（モバイル FAB 非表示ブロック等の仕様変更）。

18 コミット、10,100 行追加。数字だけ見ると大きいが、半分近くはドキュメントとテストコードで、プロダクションのロジック変更は意外とコンパクトに収まっている。

一番インパクトが大きかったのは auto-init だと思う。機能としては「プロジェクト作成時にデフォルトデータを入れる」だけなのに、Unicode 正規化、データ上書きバグ、テスト 26 本の修正と、波及範囲が広かった。「デフォルト値を入れる」という一見単純な変更が、既存の仕組み全体に影響するのは、システムが育ってきた証拠でもある。

Undo/Redo が入ったことで、エディタとしての最低ラインにようやく立てた感覚がある。「元に戻す」がないツールで何かを作るのは怖い。間違えたら最初からやり直し、という状態で創作に集中できるわけがない。この機能は地味だが、ユーザーの心理的安全性に直結している。あとは実際に作品を作ってみて、足りないものを埋めていくフェーズに入りたい。

テストを 26 本直している最中に、壊れ方のパターンがだんだん見えてくるのが面白かった。ラベルが変わった、削除 UI が変わった、データが増えた——全部「改善した結果、古い前提が崩れた」という構図で、同じ根っこから生えている。テストが壊れることを恐れるのではなく、壊れ方から変化の輪郭を読み取る、そういう使い方ができるようになると、テストスイートはただの品質保証ではなく変更の地図になる。

Unicode の正規化バグは印象に残った。`"ファンタジー" === "ファンタジー"` が `false` になる。人間の目には同じ文字列に見えるのに、バイト列が違う。こういう「見えないバグ」は発見が難しく、見つけたときの驚きも大きい。NFC と NFD の違いを知っていても、実際にプロダクションで踏むとまた別の学びがある。
