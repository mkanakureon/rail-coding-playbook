## Azure デプロイとReactの基本

React Hooks のルール違反修正も一件。`Header` コンポーネントで `useEffect` が条件分岐の後に置かれていた。Hooks は常に同じ順序で呼ばれなければならないという基本ルールだが、コンポーネントが育ってくると紛れ込みやすい。`alert`/`confirm`/`createElement` の除去、`react-modal-sheet` のimport修正、アニメーション表情のサムネイル表示修復など、細かい修正も多数。state lifting ルールを `CLAUDE.md` に追加し、振り返り文書も作成した。

一日に28コミットというのは相当なペースで、コミットログを追うだけでも全体像の把握に時間がかかる。ただ、ひとつの方向性ははっきりしている。エディタの「使い勝手」をまとめて底上げしようとした日だ。

3カラムレイアウトの導入は、エディタとしての完成度を一段上げる変更。ブロック一覧・プロパティ・プレビューが同時に見えるのは、ビジュアルノベルの制作ツールとしては最低限の要件ともいえる。それまではこれらが分離していたわけで、使い勝手の差は歴然としているはず。

モバイルUXを一日で3フェーズまで進めたのは野心的。仕様書を先に書いてから実装に入る流れが徹底されていて、Phase 1→2→3 の境界が明確だった。iOS SafariのURLバー非表示は厄介な問題として有名で、body scroll trick という古典的なハックに頼らざるを得ないあたり、モバイルブラウザの制約が垣間見える。

プレビューウィンドウを名前付きで再利用する設計は実用的。`window.open` に名前を渡すと同じウィンドウが再利用される仕様を活かしていて、開発中に何度もプレビューを開き直す煩わしさが消える。PCでは別ウィンドウ、モバイルではインラインという判断もデバイス特性に合っている。

React Hooks のルール違反は、コンポーネントが大きくなると必ず出てくる問題。条件分岐の前に全Hooksを並べるというルールは単純だが、リファクタリングの過程で壊れやすい。ESLintが拾ってくれるのは助かるが、`next dev` では実行されないという罠がある。
