## 背景とキャラクターのクロスフェード実装

レンダリングまわりの根本的な問題に取り掛かった。背景切り替え時に `bg_new` レイヤーを `backgroundLayer` に配置する方式に変更。これまでは新しい背景スプライトが既存のキャラクターより前面に来てしまい、遷移中にキャラが一瞬隠れるという視覚的なバグがあった。さらにスプライトが蓄積して消えないという問題も。`WebOpHandler` に bg_new / ch_new の 17 件の単体テストを追加して、レイヤー順序が意図通りであることを担保した。

キャラクター入れ替え時のクロスフェードも同じ `ch_new` 方式で実装。古いキャラを残したまま新しいキャラをフェードインさせ、完了後に古いほうを破棄する。ビジュアルノベルとしては基本的な演出だが、レイヤー管理が正しくないと破綻するので、テストで固めてから進めた。

## パイプライン断絶の解消 — レジストリ駆動への転換

コンパイラ・インタプリタ・エディタの間で、コマンド定義が別々に管理されていた断絶を `COMMAND_DEFINITIONS` レジストリで統一した。`commandRegistry.ts` を単一定義源として、`lineClassifier` と `parseCommand` が自動的に同期する仕組み。`ch_move` と `wait_voice_end` のコンパイラ登録漏れも、この作業で発見・修正。未知コマンドの検出ロジックも強化して、typo が早期に弾かれるようにした。

計画書を先に書き、テストギャップ解消レポートで現状を可視化してから実装に入る、という流れで進めた。+2,000 行規模の変更だが、レジストリを中心に据えたことで、今後のコマンド追加手順が明確になった。

## AI 親和性メタデータ API

`GET /api/editor-schema` と `GET /api/projects/:id` の `_ai_context` を実装。AI エージェントがプロジェクトを編集するとき、利用可能なアセット・キャラクター・ページ・変数を動的に取得できるようにした。7 件のテストが全通過。静的スキーマは 24 時間キャッシュで、14 ブロック型すべてのプロパティ定義・制約・enum を返す。

## OBS 配信と YouTube アップロード

開発風景の配信・録画インフラとして、OBS WebSocket 制御の CLI（`obs-stream.mjs` / `obs-stream.sh`）を追加。E2E テストの実行風景を動画として記録し、YouTube にアップロードするまでのパイプラインを構築。テスト名も動画映えするよう調整した。Qiita 記事 4 本も並行して作成。

17 コミットという数字が示す通り、この日は密度が高かった。特にレジストリ駆動への転換は、地味だけど効果が大きい変更だった。コンパイラに新コマンドを追加したら、インタプリタとエディタにも反映しなければならない。それが「レジストリに 1 箇所追加すれば同期テストが漏れを検出してくれる」という状態になったのは、安心感がまるで違う。

背景クロスフェードのバグは、症状としては「キャラが一瞬消える」というわかりやすいものだったが、原因の特定に少し時間がかかった。PixiJS の Container 内でのスプライト描画順序は、addChild の順番に依存する。新しい背景を最前面に追加してしまうと、既存のキャラクターレイヤーを覆い隠す。解決策は bg_new を backgroundLayer という専用コンテナに入れることで、キャラクターレイヤーとの前後関係を固定化した。

AI メタデータ API は、将来的に AI エージェントがプロジェクトを自動編集するための下地。`_ai_context` にアセット ID やキャラクター情報を入れることで、AI が「存在しないアセットを参照する」というよくある失敗を防げる。設計としては `_ai_context` は読み取り専用で、PUT 時に送り返しても自動除去される。このあたりの割り切りが気持ちいい。

OBS 連携は実用面というよりも、開発プロセスそのものを記録・公開するという実験。E2E テストが緑になる瞬間を動画で残すと、進捗が可視化されて良い。テスト名を「動画映え」するようにリネームしたのは少し笑ったが、見せるための開発記録という考え方は案外合理的かもしれない。
