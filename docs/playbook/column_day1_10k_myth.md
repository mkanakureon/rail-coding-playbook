# 【コラム】ガードレールなき「初日1万行」の罠と、Rail Codingの真の目的

開発日誌（Devlog）の初日（2026年2月7日）の記録を見ると、「12コミット / +17,673行」という驚異的な数字が記録されています。
これを見ると、**「強固なレール（Guardrails）など敷かなくても、AIの力だけで1万行書けるのではないか？」**という疑問が湧くかもしれません。

結論から言えば、**「ただコードを生成するだけ」なら、ガードレールがなくても初日から1万行を作ることは可能です。**

しかし、Rail Codingが解決しようとしているのは「初日の速度」ではなく、「3日目以降の崩壊」なのです。

## 1. なぜ初日は「レールなし」でも走れるのか？

初日のプロジェクトは、何の制約もない「更地（Greenfield）」です。
AI（Claudeなど）は既存の型定義や複雑な依存関係を気にする必要がなく、自身の持つ一般的な知識や推測（いわゆる Vibe）だけで自由にコードを書き散らすことができます。

しかし、この「勢いだけのコーディング（Vibe Coding）」は、コードベースが数万行規模に成長した途端に牙を剥きます。

## 2. 崖から落ちるAIたち（レールなき開発の末路）

制約なしで開発を進めると、数日後には必ず以下のような破綻（AIの迷走）が発生します。

- **AIの推測による矛盾の連鎖:**
  AIが「おそらくこういうデータ構造だろう」と推測して書いたコードが、別の場所で定義した型と衝突します。
- **「手直し」の無限ループ（モグラ叩き）:**
  コンパイルエラーが出た際、AIはエラーを消すためだけに局所的な修正（`any`でのキャストや、場当たり的な `if` 文の追加）を繰り返します。結果として全体のアーキテクチャが徐々に崩壊し、技術的負債が量産されます。
- **コンテキストの喪失と人間の介入:**
  見かねた人間が手動でエラーを直したり、ターミナルでコマンドを叩いたりすると、AIはその変更意図やエラーの文脈を失い、さらに的外れなコードを生成し始めます。

結果として、「バグ直しだけで1日が終わる」「型エラーが一生消えない」という状況に陥り、開発速度は急激に失速します。

## 3. 「ただの1万行」を「マージ可能な1万行」へ

Rail Coding（レール駆動開発）の真の目的は、「1日に1万行のテキストを出力させること」ではありません。
**「コンパイルが通り、テストに合格し、アーキテクチャが破綻していない『本当に動く1万行』を、1ヶ月後も、10万行を超えても安定して生み出し続けること」**です。

だからこそ、本プレイブックで定義しているような強固なレールが必要になります。

- **TypeScriptの `strict: true` と Husky によるコミットブロック:** エラーがある状態では絶対に前に進めない（コミットできない）物理的な壁。
- **実装（The Doer）と監査（The Auditor）の相互監視:** 勢いで書かれたコードの論理的欠陥を冷徹に指摘し、負債の蓄積を防ぐ仕組み。
- **段階的コミットによるコンテキストの固定化:** Git履歴を「AIのための絶対的な地図」として機能させる。

## 結論

**「ガードレールがなくても最初は走れるが、すぐに崖から落ちる。だからレールを敷く」**

これが、Rail Codingの根本的な設計思想です。AIの圧倒的な出力速度を「負債の量産」ではなく「価値の創造」に変換し続けるために、この冷酷なまでの物理的制約（レール）が不可欠なのです。
