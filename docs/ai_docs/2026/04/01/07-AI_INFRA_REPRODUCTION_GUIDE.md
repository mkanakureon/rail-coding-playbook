# AI駆動開発基盤 構築ガイド: 成功例の再現手順 (ver 2.0)

本ガイドは、AI（Gemini CLI / Claude Code 等）が迷わず、日産1万行超の生産性を発揮するための基盤をゼロから構築し、**誰がプロンプトを打っても同じ高品質な成果を得るための「レール」**を定義する。

## 哲学: "Rail, not Vibe"
AIに「いい感じにやって（Vibe Coding）」と頼むのをやめ、**「この構造以外は失敗する（Rail Coding）」**状態を物理的・言語的に作る。

## 1. 物理的レール: 構造とスキルの定義

### 具体例 1: 最高監査責任者 (CAO) プロトコル
`GEMINI.md` に以下の「監査命令」を記述し、AIを単なる作業者から「品質の番人」に変える。
- **Layer 1 (数学/性能)**: 計算量 (O(n)) やメモリリークを監視せよ。
- **Layer 2 (アーキテクチャ)**: UI層からの直接 `fetch` や `localStorage` 使用を検知し、`IStorage` への移行を強制せよ。
- **Layer 3 (自己批判)**: 完了時に「自らのコードの欠陥」を3つ挙げよ。

### 具体例 2: 自己記録スキル (`save-report.skill`)
AIが自ら思考過程を残すためのパスを固定する。
- **Path**: `docs/10_ai_docs/{YYYY}/{MM}/{DD}/{XX}-{CATEGORY}.md`
- **Naming**: `01-CODE_AUDIT_REPORT.md`, `02-MISSING_TESTS.md`
- **Effect**: コンテキストが揮発せず、後続のAIや人間が「なぜこうなったか」を 0.1秒で把握できる。

## 2. 言語的レール: プロンプト・辞書 (Rail Dictionary)

再現性を高めるため、指示に使用する言葉遣いを以下の「制約語」に統一する。

### 指示語の変換表
| 曖昧な表現 (Vibe) | 推奨される表現 (Rail) | 効果 |
|---|---|---|
| 「いい感じに直して」 | **「CAOプロトコルで監査し、不整合を修正せよ」** | 構造的チェックが強制される |
| 「コードを書いて」 | **「テストコードで再現した後、実装を適用せよ」** | デグレ防止と動作保証 |
| 「これどう思う？」 | **「現状のアーキテクチャ上のリスクを3点挙げよ」** | 具体的な批判を引き出す |
| 「保存して」 | **「save-report スキルで docs/10_ai_docs に記録せよ」** | 保存場所のブレをなくす |

### 黄金のプロンプト・テンプレート

#### A. 新機能実装時

#### B. コード監査依頼時

## 3. 構築の 4 ステップ（再現手順）

1. **憲法（GEMINI.md）の制定**:
    - プロジェクトの「禁止事項（例: `any` 禁止、`fetch` 直接呼び出し禁止）」を明文化する。
2. **記憶の器（docs/10_ai_docs/）の作成**:
    - AIが「自分の成果をどこに置くか」迷う余地を物理的に消す。
3. **カスタムスキルの配備**:
    - `.gemini/skills/` に、特定のコマンド（`/commit`, `/report`）で動く自動化スクリプトを置く。
4. **自己修復ループの起動**:
    - AIがミスをした際、謝罪させるのではなく「次から防ぐための `GEMINI.md` の追記案」を出させる。
