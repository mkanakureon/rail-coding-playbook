## 0. 序章：AI駆動開発の「特異点」を越える

### 0.1 人間が書く設計書の「曖昧さ」がAIの生産性を殺す
AI駆動開発（AI-Driven Development）が失敗する最大の要因は、「人間が不完全な設計書を書き、それをAIに実装させようとする」ことにある。
人間の自然言語による設計は、本質的に「曖昧」である。境界値の処理、非同期の競合、エラーハンドリングの網羅性。これらを人間が完璧に定義しようとすると、その記述コストだけで日産1万行の速度は失われる。
逆に、不完全な設計をAIに渡せば、AIは「推測」を始め、それが技術的負債やバグの温床となる。

### 0.2 要件（要求）だけを与え、設計とテストをAIに「丸投げ」する勇気
AIの圧倒的な計算量と静的解析能力を活かすには、**「要求（こんな機能が欲しい）」という1行のDemand（需要）だけを人間が与え、以後はAI同士が「仕様書」「型定義」「計画書」を自動生成しながら自己完結させる**必要がある。
人間はAIが生成したドキュメントを読み、「意図と合っているか」をApprove（承認）するだけの存在へとシフトしなければならない。

### 0.3 本書のゴール：1日1万行をノーエラーで生み出す「AI専用ガードレール」の構築
本書で紹介する「Rail Coding（レール駆動）」は、AIを自由に泳がせるのではなく、型定義（TypeScript Strict）と超高速Linter、そして段階的プロンプトという「物理的なレール」の上だけを走らせる手法である。このレールがあるからこそ、AIは脱線することなく、人間の100倍の速度で「正しいコード」を書き続けることができる。

## 1. 2体のAI設定と役割分担 (Two-Agent Setup & Roles)

日産1万行を安定させるには、一人のAIにすべてを任せてはならない。「書く者」と「疑う者」を物理的に分けることで、ガバナンス（統治）を効かせる。

### 1.1 実装担当：Claude Code (The Doer)
- **役割**: 人間のプロンプトや自ら作成した仕様・計画に基づき、実際のコードを生成、修正、リファクタリングする。
- **行動原理**: 「コンパイルを通す」「要件を満たす」ことに全力を尽くす。
- **権限**: `src/`, `tests/` など、ドキュメント以外のすべてのファイルの書き込み権限を持つ。
- **コミット習慣**: 変更ごとに `/commit` スキルを用い、人間から「コミットして」というプロンプトを受けて履歴を刻む。

### 1.2 監査担当：Gemini CLI (The Auditor / CAO)
- **役割**: Claudeが書いたコードの論理的整合性、アーキテクチャの純粋性を監査する。
- **行動原理**: 「最高監査責任者（CAO）」として、AI自身のコードを一切信用せず、論理的な欠陥を指摘する。
- **権限**: `docs/10_ai_docs/` 以外への書き込みは**禁止**。読み取り専用。
- **出力形式**: 標準化された監査レポート。`Status: [PASS / WARNING / BLOCK]` を明記し、人間に最終判断を仰ぐ。

### 1.3 Main-Branch Workflow (Atomic Commits via Prompts)
本ワークフローではブランチ（Feature Branch）は使用しない。常に `main` 上で直接作業する。
- **理由**: ブランチの切り替え、マージ、コンフリクト解消といった「人間向けの儀式」は、AIの速度を阻害するノイズでしかない。
- **防衛策**: `main` を守るのは「小まめなコミット」と「Geminiによる即時監査」である。段階ごとにコミットを刻むことで、もし設計が歪んでも、1フェーズ分（数百行）戻るだけで済む。

## 2. AIの設定と環境構築 (Setup & System Prompting)

AIに「自律」を命じる前に、AIが「間違えた瞬間に即座にエラーが出る」環境をAI自身に作らせる。

### 2.1 初期化プロンプト：AIに「物理的防壁」を作らせる
プロジェクトの開始時（または導入時）、Claude Codeに対して以下のプロンプトを投下せよ。

> **【Claude向け：初期化プロンプト】**
> 「このプロジェクトを『AI駆動開発（日産1万行対応）』の基盤としてセットアップせよ。
> 
> 1. **TypeScript設定**: `tsconfig.json` を作成し、`strict: true`, `noImplicitAny: true` を設定せよ。
> 2. **Linter導入**: `Biome` (または `oxlint`) を導入せよ。設定は『推奨ルールをすべてON』とし、1秒以内に全ファイルをチェックできるようにせよ。
> 3. **Husky導入**: `.husky/pre-commit` を作成し、型エラーやLintエラーがある状態でのコミットを物理的にブロックせよ。
> 4. **コミットスキル設定**: 私が『コミットして』と言った際、変更意図を解析して Conventional Commits 形式で自動コミットする環境を整えよ。
> 
> 完了後、`npm run typecheck` が通ることを確認し、『環境構築完了。コミットして。』と私に伝えよ。」

### 2.2 システムプロンプト（憲法）の注入：`AI_RULES.md`
次に、AIの思考の「OS」となる憲法を定義させる。

> **【Claude向け：憲法制定プロンプト】**
> 「ルートに `AI_RULES.md` を作成せよ。以下の『鉄の掟』を刻め。
> 
> - **第1条 [Interface First]**: 実装を始める前に必ず `interface` を定義し、人間とGeminiの承認を得よ。
> - **第2条 [No Direct Access]**: UI層から直接データベースや外部APIを叩くことを禁ずる。必ず Repository インターフェースを経由せよ。
> - **第3条 [No Speculation]**: 存在しない関数や型を推測で使うな。必ず `grep` で存在を確認せよ。
> - **第4条 [No-CLI]**: 私（人間）がターミナルでコマンドを叩くことはない。Git, Docker, デプロイ操作はすべて私からのプロンプト指示を受けてお前が実行せよ。
> 
> 作成後、この掟を遵守することを誓え。」

## 3. 超自律型・6段階開発ライフサイクル

人間が1行の「要求」を投げてから、AIが完全に「実装・テスト・監査」を終えるまでの標準フローを定義する。

### 第1段階：仕様の自律定義 (Autonomous Specification)

AIは単なる実装者ではなく、仕様策定者である。

> **【人間からの指示例】**
> 「要求：ユーザーがログインした後、直近3件の作業履歴がダッシュボードに表示されるようにしたい。」
> 
> **【AI (Claude) へのプロンプト】**
> 「この要求を元に、`.ai_docs/specs/activity_log.md` を作成せよ。以下の項目を網羅すること。
> 1. ユーザー体験（何をどう見るか）
> 2. エッジケース（履歴が0件の場合、ログイン直後、異常なデータが入っている場合）
> 3. データの保持形式
> 4. 既存機能への副作用（DB負荷など）
> 実装はまだ行うな。仕様書が完成したら報告せよ。」

**【承認プロセス】**
人間は仕様書を読み、問題なければ「Approve（承認）。コミットして。」とだけ返す。Claudeはここで最初のコミットを行う。

### 第2段階：基本設計と型駆動開発 (Architecture & Types)

「実装」の前に「型」を固定する。これが日産1万行を崩壊させない最大のポイントである。

> **【AI (Claude) へのプロンプト】**
> 「仕様書に基づき、`src/types/` または `packages/core` に必要な TypeScript の `interface` と `type` 定義のみを作成せよ。
> **絶対にロジックやコンポーネントの中身を書いてはならない。** 
> 定義後、`npm run typecheck` を実行し、型不整合がないことを確認せよ。
> 完了後、『型定義完了。コミットして。』と報告せよ。」

**【重要】**
この段階で型チェックを通すことで、後の実装フェーズでの「名前の付け間違い」や「引数の不一致」といった初歩的なミスが物理的に発生しなくなる。

### 第3段階：実装計画の自律立案 (Implementation Planning)

AIが「何から手をつけるか」を自ら考え、人間と合意形成する。

> **【AI (Claude) へのプロンプト】**
> 「型定義に基づき、`.ai_docs/plans/activity_log_plan.md` を作成せよ。
> 以下の順序で『実装のみ』のフェーズを分割せよ（テストは後置する）。
> - Phase A: Repository層（Mockデータの返却まで）
> - Phase B: 状態管理（Store/Hooks）
> - Phase C: UIコンポーネント
> 各ファイル名と、変更内容を箇条書きで示せ。承認後、作業を開始する準備ができたら『計画立案完了。コミットして。』と報告せよ。」
