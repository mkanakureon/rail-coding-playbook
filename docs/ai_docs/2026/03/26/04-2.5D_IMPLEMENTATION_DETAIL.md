# 内部仕様案：2.5D 表現の実装詳細と技術的難易度分析

## 1. はじめに
本ドキュメントは、kaedevn における 2.5D 表現（奥行きのある 2D 演出）を、既存の `IRenderer` 抽象化層を活かしてどのように実装するか、その具体的な手法と開発工数・難易度を分析したものである。

---

## 2. 実装の根幹：Z 軸概念の導入

既存の 2D レンダリング命令に、オプションとして `z` 座標（深度）を追加する。

### A. Web 実装 (PixiJS)
*   **手法:** `sprite.zIndex` による重なり順制御に加え、`scale` を `1.0 / (1.0 + z * factor)` のような式で動的に変換する。
*   **難易度:** 低。既存の `WebOpHandler` の拡張で対応可能。

### B. Native 実装 (GLRenderer)
*   **手法:** 頂点シェーダーに `z` 値を転送し、内部の行列演算を **平行投影 (Orthographic)** から **透視投影 (Perspective)**、あるいはその中間的な疑似 3D 変換に切り替える。
*   **難易度:** 中〜高。射影行列の理解と、2D 座標系との整合性維持が必要。

## 3. 機能別難易度分析

| 機能 | 実装手法 | 難易度 | 工数目安 |
| :--- | :--- | :---: | :---: |
| **パララックス** | レイヤーごとの `scrollFactor` による位置オフセット。 | **低** | 2日 |
| **深度連動 Blur** | Z 値に基づき、フラグメントシェーダーでボカし強度を変更。 | **中** | 5日 |
| **台形変換 (パース)** | 頂点シェーダーで頂点座標を 3D 空間へ投影。 | **高** | 10日 |
| **動的ライティング** | 法線マップを用いたピクセルごとの光源計算。 | **極高** | 20日+ |

## 4. 推奨実装ステップ

### Phase 1: 疑似深度シミュレーション（即効性重視）
*   **内容:** Z 値に応じて自動でスケーリングと明度（奥ほど暗く）を調整する。
*   **効果:** 数学的な変更なしに「奥にいる」という視覚効果を即座に得られる。

### Phase 2: ポストプロセス連動 (DoF)
*   **内容:** `applyFilter` 命令に深度情報を統合し、特定の深度以外をボカす「被写界深度」を実現。
*   **効果:** メメントモリ級の「空気感」の核となる演出が可能になる。

### Phase 3: プロジェクション・マッピング
*   **内容:** 地面などの平面アセットを寝かせる（パースをつける）専用の `Op` を追加。
*   **効果:** RPG の広大なフィールド表現が飛躍的に向上する。

---

## 5. 結論と提言
2.5D 実装の最大の障壁は「数学的難易度」ではなく、**「制作者がいかに簡単に奥行きを設定できるか」という UI の設計**にある。

技術的には Phase 1 & 2 までの実装であれば、現在の kaedevn のアーキテクチャ上で AI を活用することで、**短期間（1〜2週間）で製品レベルの品質に到達可能**である。ライティング等の高コストな機能は、アセット制作側の負担も考慮し、将来の拡張要素として留めるべきである。
