# 考察：PixiJS と SDL2 の相乗効果 — 戦略的技術選定の勝利

## 1. はじめに
kaedevn モノレポにおいて、Web 版の PixiJS と Native 版の SDL2 が極めて良好なバランスで共存している。本レポートでは、なぜこの組み合わせが現在の「爆速開発」と「高品質な出口（Switch）」を両立できているのかを分析する。

---

## 2. 成功の 3 大要因

### ① 「作る」と「届ける」の役割分担
*   **Web (PixiJS):** React との親和性が高く、エディタ上での Hot Reload や瞬時のプレビューを可能にする「最強の制作環境」を提供している。
*   **Native (SDL2):** ブラウザのオーバーヘッドを排し、Switch や macOS 上でメモリ・CPU 効率を最大化する「最強の実行環境」を提供している。

### ② 抽象化層 (IOpHandler) による「双子のロジック」
両者は全く異なる言語（TypeScript / C++）でありながら、同じ `IOpHandler` インターフェースを介して命令を受け取る設計になっている。
*   **効果:** 演出ロジックを一度書けば、Web でも Native でも同じ名前の関数が呼ばれるため、プラットフォーム間の挙動の不一致が構造的に排除されている。

### ③ ライブラリの性格的類似性
PixiJS も SDL2 も「低レイヤー（WebGL / OpenGL）を扱いやすい API で包む」という共通の立ち位置にある。
*   **効果:** プログラミングモデルが似ているため、AI が Web 側のコードを Native 側へ翻訳する際の精度が極めて高く、デバッグコストが大幅に抑えられている。

## 3. 実績：macOS エクスポートの実現
このバランスの良さを最も象徴しているのが、ボタン一つで SDL2 ベースのスタンドアロンアプリ（.app）を書き出す機能である。「エディタの便利さ」と「配布の身軽さ」を理想的な形で繋ぎ込むことに成功している。

---

## 4. 今後の規律（Portability First）
このバランスを維持するためには、以下の規律が不可欠である。
*   **Web 固有機能の封印:** PixiJS 特有の便利すぎる機能をそのまま使わず、常に `IRenderer` 経由で「Native で再現可能か」を確認し続ける。
*   **インターフェース主権:** 描画の進化は PixiJS ではなく、共通の抽象化インターフェースの更新から始める。

---

## 5. 結論
PixiJS と SDL2 の組み合わせは、単なる技術選定ではなく、kaedevn の「Switch リリースへの執念」を具現化するための **「戦略的勝利」** である。この絶妙なバランスこそが、100万 MAU へのスケーラビリティを支える強固な基盤となっている。
