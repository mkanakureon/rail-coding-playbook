## 1. 現状の不安要素（調査結果）
- **環境変数化の不徹底**: 各アプリのソースコードに `localhost` や特定のポート番号がフォールバック値として残っており、本番環境での誤動作リスクがある。
- **TODO項目の残存**: プロジェクト削除機能など、UX 上不可欠な機能が一部 TODO のままとなっている。
- **アセット参照の不透明性**: 大量に自動生成したプロジェクト群において、一部のアセット ID が物理ファイルや DB と乖離している可能性がある。

## 2. 安定化アクションプラン

### フェーズ 1: 環境変数の完全移行
- **全 localhost 参照の摘出と置換**: `grep` で特定した全ての URL 文字列を `process.env` または `import.meta.env` 経由に統一。
- **本番環境用テンプレートの作成**: Azure 等へのデプロイ時に必要な最小限の環境変数リスト（.env.production.example）を整備。

### フェーズ 2: エラー耐性の強化
- **AI パースエラーの堅牢化**: `parser.ts` および `schemas.ts` の最終検証。想定外の LLM 出力が発生しても、アプリがクラッシュせずにユーザーへ再生成を促す挙動を徹底。
- **MVP TODO の解消**: `HamburgerMenu.tsx` のプロジェクト削除機能など、リリースに不可欠な項目のみを限定実装。

### フェーズ 3: データの完全性検証
- **アセット整合性チェック**: `projects/*/output/` 以下の JSON を自動走査し、DB 上に存在しない `assetId` を特定・報告する。
- **デフォルト・フォールバックの実装**: アセットが 404 になる場合、自動的にプレースホルダー画像を表示するガードレールをフロントエンド側で強化。

### フェーズ 4: 最終ビルド確認
- **モノレポ一括ビルド**: 全パッケージでの `npm run build` および `npm run typecheck` を実施し、デプロイ可能なバイナリの整合性を確認。
