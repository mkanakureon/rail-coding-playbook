# レビュー報告：命令形（禁止・厳禁）からリクエスト形式/Few-shotへの移行評価

## 1. 実施内容
プロンプトエンジニアリングの手法を、従来の「負の制約（禁止・厳禁）」ベースから、以下の 2 つの柱に転換した。
1. **リクエストベースへの緩和**: 「禁止する」「厳禁」といった強い表現を、「避けてください」「しないでください」等の丁寧なリクエスト表現に置換。
2. **Few-shot（出力例）の導入**: テキストによる長大な禁止事項を削除し、代わりに「理想的な JSON 形式」の実例（名前プレフィックスが含まれていない等）を 1 つ提示。

## 2. 評価：挙動の劇的な改善

### A. 名前プレフィックス混入の完全解消 (Stage 3)
- **以前の挙動**: 「名前を含めるな」という強い禁止指示があったにも関わらず、台本形式（ルカ：「...」）が頻繁に混入していた。
- **改善後の挙動**: 命令を削除し、Few-shot（出力例）による誘導のみにした結果、**名前の混入が完全に消失した。**
- **洞察**: AI（特に Gemini）にとって、抽象的な「禁止」指示よりも、具体的な「実例（型）」の提示の方が、出力形式を決定する上での強制力が極めて高いことが実証された。

### B. ドラマ性とトーンの高度なバランス
- **以前の挙動**: 「絶望」「冷たい」等のワードを物理的に禁止した結果、描写が「水彩画のように綺麗だが、危機感に欠ける」状態に陥っていた。
- **改善後の挙動**: 特定のワード禁止を解き、世界観リファレンス（ドラクエIV、風ノ旅ビト）と正のキーワード（温もり、小さな勇気）による誘導のみにした結果、**「王道トーン」を維持したまま、物語の「葛藤（危機感）」をより豊かに描けるようになった。**

### C. アドリブとバリデーションの関係 (Stage 0)
- **副作用**: 指示を「リクエスト」に緩和したことで、AIが良かれと思ってキーワードを規定数（3つ）を超えて出力する（4つ出力）等のアドリブが発生。
- **対応**: この「創造的なはみ出し」を許容するため、Zod スキーマ側の制約を緩和。

## 3. 結論：AIエージェントへの「接し方」の再定義

今回の実験により、高度な LLM（Gemini 2.5 系列）においては **「禁止によって自由度を奪う（負の制御）」よりも、「実例によって理想を示す（正の誘導）」** 方が、望ましい出力（特に構造化データ）を得る上で圧倒的に効果的であることが判明した。

## 4. 今後の推奨方針
- プロンプト内から「禁止・厳禁・NG」という言葉を排し、常に「〜してください（リクエスト）」と「Few-shot（実例）」のセットで指示を構成する。
- 数値の厳密な制約が必要な場合は、プロンプトでの強調よりも、Zod スキーマ等のバリデーション側で担保する。
