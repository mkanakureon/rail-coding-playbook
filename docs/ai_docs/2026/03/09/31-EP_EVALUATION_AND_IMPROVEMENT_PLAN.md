# 評価報告：エモーションポイント (EP) の適用結果と改善案

## 1. 評価：第1章 第1話 における EP 反映状況

修正後の「王道冒険トーン」において、指定された EP が文章の質感（テクスチャ）にどう作用したかを分析する。

### シーン 1: EP 20 (tension / 低熱量)
- **設計意図**: 静寂、予兆、客観的な描写。
- **文章の結果**: 
    - 「真夏の残暑が朝から肌にまとわりつく」「蝉の最後の声だけが、静寂を際立たせる」
    - 過度な形容詞を避け、視覚と聴覚、触覚（熱）という物理的な事実にフォーカスしている。
- **評価**: **極めて良好**。低 EP 設定により、読者を静かに世界観へ引き込むことに成功している。

### シーン 2: EP 45 (shock / 標準熱量)
- **設計意図**: 事実の判明と、それに伴う標準的な驚き。
- **文章の結果**:
    - 「冷たい空気が流れ込んだ」「ルカは息を呑んだ」
    - 描写の密度が上がり、比喩（冷たい空気の流入）が効果的に使われ始めている。
- **評価**: **良好**。日常から非日常への転換点が、標準的な筆致で違和感なく描かれている。

### シーン 3: EP 60 (relief / 高熱量への入り口)
- **設計意図**: 不安から決意へ。主観的な熱量の高まり。
- **文章の結果**:
    - 「ルカの決意がそこにあった」「小さな勇気の灯火をともした」
    - 内面描写が増え、「灯火」という王道ファンタジーらしい情緒的な比喩が採用された。
- **評価**: **成功**。EP 60 という「やや高め」の設定が、旅立ちの「ワクワク感」を演出する適度なスパイスとなっている。

## 2. 課題：データの「不連続性」と「消失」

分析の過程で、技術的な欠陥が 2 点特定された。

### A. メタデータの消失
現状の `apps/hono/src/routes/assist.ts` では、LLM が出力した `appliedEp`（実際に適用されたEP）と `stagingIntent`（演出意図）を、最終的な JSON ファイルに保存する過程で切り捨ててしまっている。
- **影響**: 後から「どのシーンがどの程度の熱量で書かれたか」をデータとして追跡できない。

### B. シーン内 EP の固定
現在は「1シーン = 1つの EP 値」として扱っている。
- **影響**: 長いシーンの場合、最初から最後まで同じ「熱量」で書かれてしまい、シーン内の緩急（例：静かな対話から急な告白へ）が生まれにくい。

---

## 3. 改善案 (Next Action)

### ステップ 1: メタデータ保存の完全化
`SceneText` 型を拡張し、`appliedEp` と `stagingIntent` を JSON に含める。これにより「熱量の可視化（ヒートマップ作成）」を可能にする。

### ステップ 2: EP カーブの実装 (Stage 2 & 3)
Stage 2（話プロット）において、シーンごとに `startEp` と `endEp` を定義できるようにする。
- **例**: `startEp: 30` → `endEp: 70` と指定することで、シーンの中で徐々に盛り上がる文章を生成させる。

### ステップ 3: ジャンル別 EP 解釈の最適化
「王道ファンタジー」における高 EP は「希望・熱血」とし、「ホラー」における高 EP は「恐怖・狂気」とする等、ジャンルルール (`fantasy.yaml` 等) に EP の解釈指針を明記する。
