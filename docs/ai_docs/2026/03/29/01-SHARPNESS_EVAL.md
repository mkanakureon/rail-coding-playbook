# kaedevn 尖り設計評価レポート

> **作成日**: 2026-03-29
> **担当**: Gemini CLI

## 背景と目的
「個人作品を Nintendo Switch に直接パブリッシュできる唯一のプラットフォーム」という kaedevn の想定される「A（尖り）」が、現在の設計・実装においてどの程度実現に近いか、また「透明化（均等仕上げによる凡庸化）」のリスクがないかを評価する。

## 評価内容

### 1. 現状のA（Switch直接パブリッシュ）の評価： **[A-]**
- **技術的根拠**: `packages/native-engine` における **C++/SDL2 による Dual-Engine 戦略** は、単なる Web ラッパーではなく、Switch のネイティブ SDK に直接マッピング可能な極小の抽象レイヤーを構築している。
- **優先度**: `docs/10_ai_docs` における競合比較でも「Switch への移植性」が最大の差別化要因として定義されている。
- **課題**: C++ 側でのスクリプトインタープリタの完全同期が途上であり、現在は「理論上の移植性」に留まっている。

### 2. 透明化リスクの確認： **[中]**
- **エディタの多機能化**: 直近の「RPG 的な UI テンプレート」やエディタの使い勝手向上は、プロダクトの完成度を上げるが、Switch パブリッシュという「A」を直接強化するものではない。
- **リソース配分**: `apps/editor` の開発に工数が割かれ、`native-engine` 側が後回しになっている兆候がある。

## 結論・今後の課題
kaedevn は現在、「A」を狙える鋭い武器（Dual-Engine）を手にしているが、エディタの利便性という「透明な正義」にリソースを吸い取られ始めている。

### 推奨アクション
1.  **C++ エンジンの「機能パリティ」を最優先する**: Web で実装した新機能を遅滞なく `native-engine` にも移植する。
2.  **「Switch プレビューモード」の導入**: エディタ上で Switch の解像度・セーフエリア・ボタン入力のみで操作を強制するモードを実装する。
3.  **ドキュメントの再定義**: 「Web でも動く」ではなく、「Switch で出すための開発環境が Web でも提供されている」という主客逆転の表現に統一する。
