## 2. 「ポータビリティ・ファースト」の 3 つの具現化

### ① 抽象化層による「Web 固有機能」の封印
通常の Web エンジン開発では PixiJS の便利な API（イベント、スプライト制御、タイマー等）を直接呼び出しがちだが、本プロジェクトでは初期段階でこれを厳禁とした。
*   **IRenderer / IInput / IAudio:** これらインターフェースを挟むことで、Web 版は「PixiJS を使って IRenderer のふりをしている」だけに過ぎない状態を作った。
*   **効果:** 移植時に「PixiJS 特有のバグ」に悩まされることがなく、ロジックの修正なしで C++ 側への移行が可能となった。

### ② 命令（Op）の純粋データ化
すべての演出指示を「純粋な JSON データ（Op）」に変換し、それを Handler が受け取るというアーキテクチャを徹底した。
*   **意義:** ロジック層は「どう描くか」を知る必要がなく、ただ「命令」を投げるだけ。
*   **効果:** TypeScript と C++ という全く異なる言語間において、ロジックの「翻訳精度」が 100% に近くなり、マルチプラットフォーム間での挙動の不一致を構造的に排除した。

### ③ オフライン・スタンドアロンの意識
Switch リリースを見据え、常時接続やブラウザの標準機能（LocalStorage 等）に依存しない設計を貫いた。
*   **意義:** 認証（JWT）やデータ保存（抽象ストレージ層）が、インターネット環境のないオフラインプレイや、ファイルシステムが異なるコンソール機でも動作するように設計されている。
