## 2. 設計原則

### ① コア・プロトコルの人間統治 (Human-Governed Protocol)
*   **方針:** エンジンの基本命令セット（`Op.ts`）は、人間が Switch でのパフォーマンスと表現力を考慮して厳格に定義・管理する。
*   **理由:** AI によるプロトコルの無秩序な拡張を防ぎ、プラットフォーム間の完全な互換性（Parity）を恒久的に維持するため。

### ② AI 実装を加速する「聖典」と CLI (Sample-Driven AI Implementation)
*   **方針:** AI（Claude Code 等）が実装を担当する際のガイドとなる「高品質なサンプルコード」と「開発者用 CLI」を徹底的に充実させる。
*   **役割:** CLI は品質保証の道具ではなく、AI が「正しいコード」を爆速で量産するためのレールとしての機能に特化する。

### ③ Web エディタへのプレビュー集約 (Web-First Iteration)
*   **方針:** 開発効率向上のため、全てのリアルタイム・プレビューを Web エディタに集約する。
*   **前提:** Web 側で動くものは Switch でも完全に動くという「100% Parity」をエンジンの根幹として保証することで、Switch 側でのリアルタイム・プレビューの必要性を排除し、開発サイクルを高速化する。

### ④ スタンドアロン性能の最適化 (Switch-Native Performance)
*   **方針:** `IRenderer` (C++/OpenGL) を核とし、Live2D、Spine、高度な GLSL 演出をネイティブ環境で最高性能（60fps）で動作させることに特化する。
