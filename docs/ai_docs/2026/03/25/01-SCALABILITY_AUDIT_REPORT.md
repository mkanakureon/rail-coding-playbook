## 2. 三者の AI 監査役による多角的レビュー

### 🛡️ 監査役 A：セキュリティ（Security Auditor）
**「公開プラットフォームとしての防御力の検証」**

*   **レート制限の精密化:** 現在の `apiLimiter` は一律ですが、`auth/register` や `contact` など、悪意のある大量リクエストでサーバーを停止（DoS）させやすいポイントへの個別制限が必要です。
*   **BigInt 境界問題:** Prisma Schema で多用されている `BigInt` は、JavaScript の `Number.MAX_SAFE_INTEGER` を超えた際にフロントエンドで値が壊れるリスクがあります。
*   **サンドボックス:** ユーザーが Live2D やカスタムスクリプトを持ち込む際、他者のブラウザ上で悪意のあるコードが実行されないよう、実行環境を隔離（サンドボックス化）する必要があります。

### 🚀 監査役 B：スケーラビリティ（SRE/Performance Engineer）
**「100万人が同時に叩いた時の物理的限界」**

*   **巨大 JSON カラムの弊害:** `Project.data` に全ページデータを保存する現状では、ページが増えるほど 1 回の保存（UPDATE）が重くなり、DB の I/O を圧迫します。100万 MAU では、DB が真っ先に死にます。
*   **アセット配信の分離:** Node.js サーバー（Hono）で静止画や音声を直接配信（`serveStatic`）するのは、100万 MAU では不可能です。Node.js のスレッドがファイル読み込みで埋まり、API が応答しなくなります。
*   **DB 接続数の枯渇:** サーバーを並列化（スケールアウト）した際、DB への同時接続数が上限に達します。

### 💎 監査役 C：信頼性（Reliability/Frontend Architect）
**「低スペック端末とオフラインの考慮」**

*   **描画負荷の制御:** 53 種のフィルターと Live2D は、古いスマホや低価格なタブレットではメモリ不足（OOM）を引き起こします。
*   **データ整合性:** 電車の中など回線が不安定な環境での「保存失敗」に対する、自動再試行やローカル保存（IndexedDB 等）の仕組みが現状不足しています。
