## 2. アーキテクチャの 3 つの柱（信頼の根拠）

### ① 徹底した「3 層の抽象化」
エンジンの中心部は、以下の 3 層に完全に分離されている。
*   **データ層 (Op/Command):** 命令を純粋な JSON データとして定義。
*   **論理層 (IEngineAPI):** 命令を「どう解釈するか」というプラットフォーム共通のロジック。
*   **描画層 (IRenderer):** WebGL (PixiJS) か OpenGL かを差し替える「実作業」の切り出し。

**【監査結果】** この分離により、特定の描画ライブラリや言語に依存しない「ポータビリティ」が確保されている。これは、大規模なリファクタリングや将来のハードウェア移行において最大の防衛線となる。

### ② モノレポ構造による「知能の集約」
`packages/` 配下の構成（Compiler, Core, Battle, Map 等）が、機能ごとに独立したパッケージとして管理されている。
*   **Compiler パッケージ:** 解析プロセスが 5 つのフェーズ（Tokenizer → Parser → Transformer → Finalizer → Validator）に分かれたパイプライン設計となっており、機能追加時の影響範囲が極めて限定的である。

**【監査結果】** AI がコードを生成する際、関連する小さなパッケージに集中できるため、全体が複雑に絡み合う「スパゲッティ化」を構造的に防いでいる。

### ③ 秩序ある機能統合パターン
Live2D やパーティクルといった複雑な機能の追加においても、常に「Op 定義 → Handler 実装 → テスト追加」というプロジェクト固有のパターンが守られている。

**【監査結果】** AI が過去の成功パターンを学習し、それに則って新機能を追加しているため、ソースコード全体に一貫性（Consistency）があり、人間が読むのと同じレベルの「読みやすさ」と「予測可能性」が維持されている。
