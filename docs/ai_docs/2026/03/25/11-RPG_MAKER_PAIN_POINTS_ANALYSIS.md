# 分析報告：RPG ツクール作者が抱える「3 大ペイン」と次世代エディタへの要求

## 1. はじめに
本レポートは、既存の RPG 制作ツール（主に RPG ツクール系列）を数百時間使用した作者が直面する致命的なストレス（ペインポイント）を抽出し、kaedevn Next が解決すべき課題を定義したものである。

---

## 2. ペイン 1：モーダル地獄による「データの断片化」

### 現状の苦痛
*   アクター、スキル、アイテム、敵キャラの設定が独立したモーダルウィンドウ（ダイアログ）に分断されており、全体的なゲームバランスを俯瞰することが不可能である。
*   「攻撃力に対してこのスキルの威力は適正か？」を確認するために、何度もウィンドウを開閉し、記憶に頼る作業を強いられる。

### 次世代エディタへの要求
*   **スプレッドシート型一括編集:** 全てのデータを Excel のように一覧で比較・編集できるインターフェース。
*   **リレーショナル・リンク:** スキル設定画面から、そのスキルを使用するエネミー一覧へ即座にジャンプできる「繋がり」の可視化。

---

## 3. ペイン 2：不可視のスイッチ・変数による「管理の崩壊」

### 現状の苦痛
*   「スイッチ 121 番」のように ID で管理されるため、外部のメモ帳なしでは変数の用途が分からなくなる。
*   イベントページが複雑な入れ子構造になり、特定のフラグ条件がどこで矛盾しているかを探す「デバッグ作業」に大半の時間が奪われる。

### 次世代エディタへの要求
*   **セマンティック変数名:** ID を廃止し、人間が理解できる名前での完全管理。
*   **イベントのテンプレート化:** 共通の挙動（宝箱、宿屋等）を部品化し、一箇所の修正を全マップに反映させる仕組み。
*   **ビジュアル・デバッガ:** 実行中のイベントフローと変数の中身を、マップ上でリアルタイムにオーバーレイ表示する機能。

---

## 4. ペイン 3：アセット管理の硬直性と UI カスタマイズの壁

### 現状の苦痛
*   素材のインポートに厳格なファイル規格（サイズ、命名規則）を要求され、少しでも外れるとエラーになる。
*   ゲームの「顔」であるメニュー画面やバトル UI のレイアウト変更に、高度なスクリプト（JavaScript/Ruby）の記述が必須となり、非エンジニアが挫折する最大の原因となっている。

### 次世代エディタへの要求
*   **シームレス・キャンバス:** ドラッグ＆ドロップによる自由なアセット登録。
*   **ノーコード UI ビルダー:** パワーポイント感覚で HP バーやボタンの配置をカスタマイズできるビジュアル編集機能。
*   **100% Parity 保証:** エディタで配置した 1 ピクセルが、Switch 実機でも完全に再現される安心感。

## 5. 結論
RPG 制作における真の「作りやすさ」とは、AI が代わりに作ることではなく、**「人間が面白さの設計（試行錯誤）に集中できるよう、事務的な管理コストを極限まで排除すること」** にある。
