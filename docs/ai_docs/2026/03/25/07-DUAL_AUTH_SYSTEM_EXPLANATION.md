# 調査報告：ハイブリッド認証システムの構造と戦略的意義

## 1. はじめに
kaedevn モノレポには、NextAuth.js と Hono JWT という 2 つの独立した認証仕組みが共存している。一見すると冗長に思えるこの設計が、いかにしてプラットフォームの「利便性」と「将来の拡張性」を支えているかを解説する。

---

## 2. 2 つの認証システムの正体

### ① NextAuth.js (apps/next)
*   **役割:** ポータルサイト（ユーザー登録、プロジェクト管理）のフロントエンド認証。
*   **方式:** セッションベース（Cookie）。
*   **意義:** Google ログインや Discord 連携などの SNS 認証を迅速に提供し、ユーザーに「ブラウザを閉じてもログインが維持される」という快適な体験を提供する。

### ② Hono Auth / JWT (apps/hono)
*   **役割:** エディタ (`apps/editor`) およびゲームエンジン実行時の API 認可。
*   **方式:** トークンベース（JSON Web Token / Authorization ヘッダー）。
*   **意義:** API サーバーを Next.js の特定の Cookie ドメインから解放する。これにより、クロスドメインのエディタや、将来の Native アプリ（Switch/iOS/Android）からでも安全に API を利用可能にする。

---

## 3. 認証の「バトンタッチ」：DB による統合
この 2 つのシステムは、背後にある PostgreSQL の `User` テーブルを「共通の真実（Single Source of Truth）」とすることで整合性を保っている。
1.  Next.js で SNS ログインを実行（NextAuth が `User` レコードを作成/更新）。
2.  エディタ起動時、Next.js 側でそのユーザー ID に紐づく短期的な **JWT トークンを発行**。
3.  エディタは以降、そのトークンを Authorization ヘッダーに載せて API（Hono）を叩く。

---

## 4. 戦略的メリット：将来への布石
この設計が「大正解」である理由は、以下の将来的な要件を既にクリアしているためである。

1.  **マルチプラットフォーム対応:** Switch 等の Native 環境ではブラウザの Cookie が使えないが、JWT 方式であればトークンを保存・送信するだけで認証が完結する。
2.  **100万 MAU へのスケーラビリティ:** JWT はステートレス（サーバー側でセッションを保持しない）ため、API サーバーを何台に増やしても認証が壊れず、水平スケーリングが容易である。
3.  **セキュリティの分離:** ポータルの脆弱性が API 全体に波及するリスクを最小限に抑えることができる。

---

## 5. 結論
2 つの認証システムの共存は、技術的な「執念」が生んだ高度な分離設計である。この構造を維持し続けることで、kaedevn は Web の枠を超えた真のクロスプラットフォーム・エンジンとしての地位を確立できる。
