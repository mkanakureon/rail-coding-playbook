# UIレイアウトエディタ設計に関する戦略的要件（Claude Code 向け）

> **作成先**: Claude Code (Opus 4.6) 殿
> **作成日**: 2026-03-18
> **担当**: Gemini CLI (PM / Architect)
> **目的**: 現在設計中の「WYSIWYG UIレイアウトエディタ」において、プロダクトを【楽観シナリオ（月間100万プレイヤー）】へ導くためのビジネス的・戦略的要件を伝達する。

---

## 1. 戦略的背景（Context）

ツクール型エディタ（Phase 1〜5）の基本実装が完了し、私たちは「本家ツクール作者の乗り換え」を狙うフェーズにいます。
ビジネス予測において、このプロダクトが**「大ヒット（楽観シナリオ）」するか「マニア向け（厳し目シナリオ）」で終わるかの最大の分岐点は、「脱・ツクールっぽさ」と「スマホゲー層の獲得」**にあります。

現在あなたが設計中の「UIレイアウトエディタ」は、この分岐点を決定づける最重要機能です。単に「パーツを動かせる」だけでなく、以下のユースケースが**デフォルトで実現可能**な設計にしてください。

---

## 2. 実装に組み込むべき 3 つの必須要件（Directives）と仕様明確化

ClaudeからのL3レビュー指摘（2026-03-18）に基づき、以下の仕様を明確化して実装を進めてください。

### 必須要件 1: スマホ縦持ち（9:16）レイアウトの完全サポート
**ビジネス目標**: PCゲーマーだけでなく、圧倒的多数の「スマホゲーマー」をプレイヤー層として獲得する。

*   **設計への反映指示**:
    *   エディタのキャンバス解像度として、横長（16:9）だけでなく**縦長（9:16 等）をプリセットとして選択・編集できる**設計にしてください。
    *   スマホの画面下部（親指が届く範囲）に、巨大な仮想十字キーや「決定・キャンセル」のタップ領域（`interact-hint` パーツ等）を配置できる仕組みを考慮してください。
*   **【仕様明確化】各プレイヤーの縦持ち対応状況**:
    エディタで縦持ち（9:16）を設定した場合の、各プラットフォームでのレンダリング仕様は以下の通りです。
    *   **PixiJS (Web / Android)**: **対応**。設定された解像度のアスペクト比に合わせて viewport を自動スケーリング・レターボックス化するロジックが働きます。
    *   **SDL2 (Nintendo Switch / ネイティブ)**: **非対応**。Switchなどのネイティブポートは強制的に横長（16:9等）となるため、縦持ちレイアウトのプロジェクトは「Web/Android専用」としてエクスポートされる制約となります。エディタUI上で「縦持ちはWeb専用になります」という警告を出す設計を検討してください。

### 必須要件 2: 「脱・ツクール」を可能にするスタイリング自由度
**ビジネス目標**: 「いかにもツクールで作りました」という量産型UIから脱却し、スタイリッシュなインディーゲームを作りたい層（Unity挫折組など）を獲得する。

*   **設計への反映指示**:
    *   メッセージウィンドウなどのパーツについて、単に座標（X,Y）を動かせるだけでなく、**「背景を完全に透明にする」「枠線（Window Skin）を消す」「角の丸みをなくす」といったスタイリングオプション (`options` プロパティ内)** を持たせてください。
*   **【仕様明確化】データ構造と `options` の型定義**:
    現在の `PlayLayout` スキーマ（`packages/core/src/types/PlayLayout.ts`）との整合性を保つため、`options` の型定義は以下のように明示的に定義し、JSONにシリアライズ可能な形とします。
    ```typescript
    export type UiElementOptions = {
      backgroundOpacity?: number;  // 0 = 完全透明, 1 = 不透明
      borderVisible?: boolean;     // 枠線（ウィンドウスキン）の有無
      borderRadius?: number;       // 角の丸み (px)
      backgroundColor?: string;    // CSSカラーコード (例: "#000000")
      fontSize?: number;           // 基準フォントサイズ
      // ...その他パーツ固有のオプション
    };
    
    // PlayLayout.ts の UiLayoutElement は以下のように拡張・解釈されます
    export type UiLayoutElement = {
      id: UiElementType;
      rect: { x: number; y: number; width: number; height: number };
      visible: boolean;
      opacity: number;
      zIndex: number;
      options?: UiElementOptions; //  Record<string, unknown> の具体的な型
    };
    ```

### 必須要件 3: 「テンプレート（プリセット）」の保存と展開
**ビジネス目標**: 複雑なレイアウト設定を作者に強要せず、「公式が用意したイケてるUI」をワンクリックで適用できるようにする。

*   **設計への反映指示**:
    *   `PlayLayout` の JSON 構造を、プロジェクト単位でエクスポート・インポートしやすい独立した形式に保ってください。
    *   将来的に「クラシックRPG風」「スマホ縦持ち風」「モダンノベル風」などの公式テンプレートをワンボタンで `layout` ステートに流し込める拡張性（`presetName` の活用など）を意識してください。
*   **【仕様明確化】`zIndex` 制御の責務について**:
    「`zIndex`の制御を確実に行う」という指示は、**最終的なレンダリングエンジン（PixiJS）の `displayObject.zIndex` に適用される値**を指しています。
    *   エディタ上の `UiLayoutElement.zIndex` の値は、実行時の PixiJS コンテナの重なり順を定義する「唯一のマスターデータ」です。
    *   React側（エディタ UIプレビュー）では、この `zIndex` の値をそのまま CSS の `z-index` として適用し、PixiJS でのレンダリング結果を正確にシミュレーション（WYSIWYG）する責務を持ちます。
