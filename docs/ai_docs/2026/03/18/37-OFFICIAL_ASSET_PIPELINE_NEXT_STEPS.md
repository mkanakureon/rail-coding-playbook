# 公式アセット自動生成パイプライン 次のステップへの提案書

> **作成日**: 2026-03-18
> **担当**: Gemini CLI
> **宛先**: Claude Code (Opus 4.6)
> **前提ドキュメント**: `36-OFFICIAL_ASSET_GENERATION_PIPELINE.md`

## 1. 概要
Gemini API (Imagen 4.0) と `sharp` を組み合わせた「公式アセット全自動生成パイプライン」の初期基盤（Step 1）が完成し、CLI (`scripts/cli/asset/gen-official-asset.ts`) から高品質なベース画像と 48x48 へのドット絵化・スプライト化が行えるようになりました。

本ドキュメントは、このパイプラインを「真に実用的なゲームアセット（ツクール互換規格）」へと昇華させるため、Claude Code に追加実装を依頼する次のステップ（機能要件）を定義したものです。

---

## 2. 次のステップ（Claude Code への実装依頼事項）

### 📝 1. 4方向（上下左右）の歩行グラフィック生成
現在の実装は「真正面」の画像を1枚生成し、それをコードで上下に揺らして3コマ（足踏み）にしているだけです。
* **要件**:
  - DALL-E/Imagen に対するプロンプトを動的に変更し、「正面 (front)」「背面 (back)」「左向き (left-facing)」「右向き (right-facing)」の4枚の画像を生成（または1枚のシートとして生成）させる。
  - `sharp` でこれらを結合し、ツクール規格である「3列(コマ) × 4行(方向) = 12コマ (144x192 ピクセル)」の完全な歩行スプライトシートを出力する。

### 📝 2. 厳格な減色処理（カラーパレットの強制）
現在の `resize({ kernel: 'nearest' })` だけでは、DALL-E/Imagen が生成した無数の色がそのまま残ってしまい、「レトロなドット絵」としては色が多すぎて浮いてしまいます。
* **要件**:
  - 画像処理パイプラインに「減色（Color Quantization）」のステップを追加する。
  - ツクールやレトロゲームでよく使われる16色〜32色の固定パレット（例: DawnBringer 32 や Arne 16）を定義し、生成された画像の色をピクセル単位で「最も近いパレットの色」に強制置換するスクリプトを実装する。

### 📝 3. 背景透過と輪郭（アウトライン）のクリーンアップ
AI生成画像は純粋な白背景 (`#FFFFFF`) ではないピクセル（影やノイズ）が含まれることがあり、単純な透過ではゴミが残ります。
* **要件**:
  - `sharp` などの画像処理において、単なる Threshold 透過だけでなく、Flood Fill（塗りつぶし）アルゴリズムや、輪郭線の抽出・強調（1pxの黒フチを付ける等）を行い、ゲームエンジン上で背景に綺麗に馴染むようにする。

### 📝 4. マップチップのオートタイル（境界線）対応
現在のタイル生成は「1枚の四角い画像」を生成するだけですが、実際のゲームマップでは「海と陸の境界」などのオートタイル（Auto-tile）が必要です。
* **要件**:
  - 「草地」「水面」などのベース画像を生成したあと、コード側でそれらを切り貼り・合成し、「内角・外角・直線」などを含んだツクール規格のオートタイル形式（例: 2列×3行の結合タイル）に自動マッピングして出力するロジックを実装する。

---

## 3. 実装方針と優先順位
1. **優先度 高**: 「4方向の歩行グラフィック生成」と「厳格な減色処理」。これが完成するだけで、キャラクター素材としての実用性が100倍になります。
2. **優先度 中**: 「背景透過のクリーンアップ」。
3. **優先度 低**: 「オートタイル対応」。まずはキャラクターと単一オブジェクト（宝箱や木など）の完成を優先してください。

Claude Code はこの提案書を読み、`scripts/cli/asset/gen-official-asset.ts` のリファクタリングおよび拡張に着手してください。
