# ツクール型エディタ E2E テスト計画書

> **作成日**: 2026-03-18
> **担当**: Gemini CLI
> **対象**: kaedevn ツクール型エディタ (v1.0)
> **フレームワーク**: Playwright (`tests/shared/editor/tsukuru-editor.spec.ts` 等として実装予定)

本計画書は、実装が完了した Phase 1〜5 の機能がデグレ（機能後退）していないことを継続的に担保するための E2E（End-to-End）テストシナリオを定義します。

---

## 1. テスト環境・前提条件
*   **認証**: 既存テスト同様、API経由でテストユーザーとしてログインし、Tokenを取得してLocalStorageにセットする。
*   **プロジェクト作成**: 各テストスイート開始前に `type: 'tsukuru'` でプロジェクトを新規作成し、クリーンな状態でテストを開始する。
*   **スクリーンショット**: 各重要ステップ（ダイアログを開く、保存する等）でフルページスクリーンショットを取得し、ビジュアルリグレッションテストの基礎とする。

---

## 2. テストスイート（シナリオ群）

### 2.1. シェルと全体レイアウト (Phase 1)
**目的**: エディタの「ガワ」が崩れず、タブ切り替えが正常に行えるかを確認する。

*   **[Test 1] 3大タブの切り替え**
    1. エディタを開く。デフォルトで「🗺️ マップ」タブがアクティブであることを確認。
    2. 「📊 データベース」タブをクリックし、`GameDbPanel` が表示されることを確認。
    3. 「🖼 レイアウト」タブをクリックし、Phase 5のプレースホルダーが表示されることを確認。
*   **[Test 2] マップの新規作成と削除**
    1. 左パネル（マップツリー）の「＋」ボタンをクリックし、新しいマップがリストに追加されることを確認。
    2. 追加されたマップを右クリックし、コンテキストメニューから「削除」を実行。リストから消えることを確認。

### 2.2. データベースエディタ (Phase 2)
**目的**: 複雑なUIを持つ13のタブが、正しくデータを更新・保存できるかを確認する。

*   **[Test 3] タイルセットの通行判定グリッド操作**
    1. データベースタブ →「タイルセット」を開く。
    2. タイルグリッドの最初の要素をクリックし、表示が `○` から `×`（または `☆`）に切り替わることを確認。
    3. プロジェクトを保存し、リロード後にそのタイルの判定が保持されていることをAPIまたは再描画で確認。
*   **[Test 4] トループ（敵グループ）のドラッグ配置**
    1. データベースタブ →「トループ」を開く。
    2. 「メンバー追加」をクリックし、キャンバス上に敵の矩形が出現することを確認。
    3. 矩形をドラッグし、座標（x, y）の表示が追従して変化することを確認。

### 2.3. イベントコマンドシステム (Phase 3)
**目的**: ツクールエディタの心臓部である「ネストされたコマンドリスト」の完全性を確認する。

*   **[Test 5] 条件分岐のネストとコマンド挿入**
    1. マップ上でイベントダイアログを開き、シナリオエディタを表示。
    2. 左のパレットから「条件分岐」を追加する。
    3. リストに表示された「◆ 条件分岐」と「：分岐終了」の間に「文章の表示」コマンドをドラッグまたはダブルクリックで挿入する（ネスト挿入）。
    4. 「文章の表示」をダブルクリックし、16:9 プレビュー領域にテキストが即座に反映されることを確認してOKを押す。
    5. 保存後、プロジェクトデータ（JSON）の階層構造（`thenCommands` の中身）が正しく構築されていることを API でアサートする。
*   **[Test 6] データベース参照ドロップダウン**
    1. パレットから「場所移動」を追加して編集ダイアログを開く。
    2. プレーンテキスト入力ではなく、`<select>` ドロップダウンに [Test 2] で作成したマップのリストが表示されていることを確認する。

### 2.4. ランタイム・テストプレイ (Phase 4)
**目的**: イベントインタプリタと GameState が統合され、ゲームとして機能するかを確認する。

*   **[Test 7] インタプリタの自動実行とデバッガ**
    1. 以下のような単純なコマンド配列を持つイベントを作成し、テストプレイ（PlayOverlay）を起動する。
        - スイッチ[1]をONにする
        - 文章の表示「テスト開始」
        - 変数[1]に 100 を代入
    2. テストプレイ画面に「テスト開始」のメッセージダイアログが（プレビュー通りに）表示されることを確認。
    3. メッセージをクリックしてイベントを最後まで進める。
    4. `F9` キーをエミュレート入力してデバッガを開く。
    5. 「スイッチ」タブで `0001` が `ON` になっていること、「変数」タブで `0001` が `100` になっていることを確認。

---

## 3. 実装優先度

ツクール型エディタはDOM構造が深いため、全てのテストを一度に書くのは非効率です。以下の優先順位で Playwright テストを実装・CI に組み込むことを推奨します。

1.  **高（致命的バグの防止）**: [Test 5] コマンドのネスト挿入と保存。これが壊れるとエディタの存在意義が失われるため。
2.  **中（UXの保証）**: [Test 1] シェルの切り替えと、[Test 7] デバッガでの状態確認。
3.  **低（ビジュアルテスト）**: [Test 3, 4] タイルセットやドラッグUIは、自動テストが脆くなりやすい（Flaky）ため、スナップショット比較を中心に据える。
