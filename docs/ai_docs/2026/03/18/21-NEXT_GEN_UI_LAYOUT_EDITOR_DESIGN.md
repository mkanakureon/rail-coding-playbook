# 次世代機能設計書：WYSIWYG UI レイアウトエディタ

> **作成日**: 2026-03-18
> **担当**: Gemini CLI
> **位置づけ**: kaedevn ツクール型エディタ 次世代機能実装 第1弾（優先度 1位）
> **目的**: プラグイン（JSコーディング）を一切不要にし、マウス操作だけでゲーム画面のUIを完全カスタマイズできる「本家超え」の機能を実現する。

---

## 1. 概要とコンセプト
ツクール作者の最大の不満である「デフォルトUIから脱却できない」「オリジナリティを出せない」という問題を、Webベースのエディタの強みであるドラッグ＆ドロップUIで解決します。
エディタの第4タブである「🖼 レイアウト」タブにて、**実際のゲーム画面解像度（1280x720 など）のキャンバス上に、メッセージ枠やHPゲージといったパーツを直接配置し、リサイズや透明度を設定できる**ようにします。

---

## 2. データ構造（スキーマ）

すでに `packages/core/src/types/PlayLayout.ts` に定義されている `PlayLayout` 型をそのまま活用します。これにより、バックエンド（DB）への追加マイグレーションは不要で、プロジェクト JSON 内に格納できます。

### 対象とする主要 UI パーツ (`UiElementType`)
*   `message-window` (メッセージウィンドウ)
*   `name-box` (話者名ウィンドウ)
*   `choice-window` (選択肢ウィンドウ)
*   `gold-window` (所持金ウィンドウ)
*   `party-status-panel` (パーティステータス：HP/MPゲージ等)
*   `mini-map` (ミニマップ表示領域)
*   `interact-hint` (決定ボタン等の操作ヒント)

---

## 3. 画面レイアウト構成（TsukuruEditorPage - レイアウトタブ）

エディタの 3 カラム構成を踏襲し、以下のレイアウトで実装します。

### 3.1. 左パネル：UI エレメントツリー
*   **機能**: 現在のレイアウトに配置されている UI パーツの階層リスト。
*   **操作**:
    *   表示/非表示（目玉アイコン）の切り替え。
    *   ドラッグによる Z-Index（重なり順）の変更。
    *   「＋パーツ追加」ボタンから、不足しているパーツ（例: `gold-window`）をキャンバスに投入。

### 3.2. 中央パネル：プレビューキャンバス（WYSIWYG）
*   **機能**: 指定された解像度（16:9）の枠内で、パーツをドラッグ＆ドロップ配置する領域。
*   **ライブラリ**: `react-rnd` (Resizable and Draggable) を採用し、パーツの移動と四隅のドラッグによるサイズ変更（リサイズ）を実装する。
*   **スナップ機能**: 8px または 16px 単位のグリッドスナップを搭載し、パーツの端と端をきれいに揃えられるようにする（ツクール作者に喜ばれるUX）。
*   **背景プレビュー**: マップ画像やダミーの戦闘背景を設定でき、実際のゲーム画面を想定しながらUIの色味や透明度を調整できるようにする。

### 3.3. 右パネル：プロパティインスペクタ
*   **機能**: 中央または左パネルで選択した UI パーツの詳細数値を編集する。
*   **設定項目**:
    *   **座標**: X, Y, Width, Height (数値での微調整)
    *   **見た目**: Opacity (不透明度 0.0〜1.0)
    *   **オプション (options)**: パーツ固有の設定。
        *   例 `message-window`: 背景を「ウィンドウ」「暗くする」「透明」から選択。行間、フォントサイズ。
        *   例 `party-status-panel`: ゲージの色（HPは緑、MPは青など）や、顔グラフィックの有無。

---

## 4. ランタイム（テストプレイ）との統合

レイアウトエディタで設定した `PlayLayout` の JSON データは、そのまま Phase 4 で作成した `TestPlayOverlay.tsx`（テストプレイ画面）のレンダリングに適用されます。

*   **動作仕様**: 
    1. テストプレイ開始時、プロジェクトデータから `PlayLayout` を読み込む。
    2. `TestPlayOverlay` の各種UI（`textDialog`, `choicesDialog`）は、ハードコードされた Tailwind CSS クラス（`top-4` など）を破棄し、`PlayLayout` の座標データ（`left: element.rect.x, top: element.rect.y`）にバインドして絶対配置（`absolute`）される。

---

## 5. 実装フェーズ（ステップ）

*   **Step 1: データバインディングとシェル構築**
    *   `useTsukuruStore.ts` に `layout: PlayLayout` の状態を追加。
    *   第4タブのシェルを 3 カラム（ツリー、キャンバス、プロパティ）に分割。
*   **Step 2: ドラッグ＆ドロップキャンバスの実装**
    *   `react-rnd` を導入し、ダミーの矩形枠をマウスで動かし、その結果が Store の `layout` にリアルタイム保存されるロジックを実装。
*   **Step 3: 各種 UI パーツのモックアップ化**
    *   HPゲージやメッセージウィンドウの「ダミー表示」コンポーネントを作成し、キャンバス上の枠内にレンダリングする。
*   **Step 4: ランタイム統合**
    *   `TestPlayOverlay.tsx` を改修し、保存された `PlayLayout` データに基づいて UI が配置されるようにする。

---

## 6. 期待される効果
この機能が実装されることで、kaedevn は単なる「ツクール互換エディタ」から**「画面デザインにおいて本家を完全に圧倒するモダンエンジン」**へと進化します。ユーザーはCSSやJSの知識が一切なくても、他作品と全く違うオリジナル画面のRPGを直感的に制作できるようになります。
