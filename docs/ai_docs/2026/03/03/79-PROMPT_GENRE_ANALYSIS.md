# プロンプトの構造分析と最適化方針：ジャンル共通 vs ジャンル固有

## 1. 現状の構成
現在のプロンプト生成ロジック（`prompts.ts`）は、**「共通の骨格テンプレート」に「ジャンル別の動的なルール」を流し込むハイブリッド構造**になっています。

- **共通テンプレート (`prompts.ts`)**: 
  - システムの役割定義、出力形式（JSON構造）、基本的な執筆・演出ルール。
  - 今回 Gemini 2.0 Flash が自律的に生成した `[表情: ...]` などの指示は、ここにある「演出指示の埋め込み」ルールに基づいています。
- **ジャンル別ルール (`genre-rules/*.yaml`)**:
  - 各ジャンル（コメディ、ファンタジー等）特有のテンポ、比喩の使い方、プロットの進め方。
  - 例（コメディ）: 「ツッコミ役の内心を地の文に入れる」「オチを各話の最後に入れる」。

## 2. 最適化のスコープ
今回の最適化計画（78-GEMINI_PROMPT_OPTIMIZATION_PLAN.md）は、以下の両面からアプローチします。

### A. 全ジャンル共通の「構造化」リファクタリング
Gemini がより正確に制約を守り、高い描写力を発揮できるよう、`prompts.ts` のベース部分を XML タグ形式などで整理します。これは**全ての作品（コメディ以外も）の質を底上げします。**

### B. ジャンル固有ルールの「Gemini最適化」
`comedy.yaml` 等の内容を、「〜しないで」という禁止形から「Gemini が得意なポジティブな行動指針」へと書き換えます。

## 3. 結論：汎用性を保ちつつ質を上げる
プロンプトの修正は「ジャンル共通」の改善が 7 割、「ジャンル固有」の改善が 3 割となります。
これにより、今回コメディで行う実験の成果は、将来的にファンタジーやホラーなどの他ジャンルにもそのまま恩恵として還元されることになります。
