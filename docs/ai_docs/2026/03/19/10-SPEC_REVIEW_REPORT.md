# レビュー結果報告書：ノベルゲーム特化リリース設計

> **作成日**: 2026-03-19
> **担当**: Gemini CLI
> **ステータス**: 承認 (Approved)

## 1. エグゼクティブ・サマリー
「ツクール型」からの脱却と「ノベル特化」への戦略変更は、開発コストの削減とユーザー獲得効率（拡散性）の両面で極めて正しい判断である。設計仕様書（`11-release-design-spec.md`）は、この戦略を技術的に裏付ける十分な具体性を備えている。

## 2. 項目別レビュー詳細

### 2.1. 決済・ポイントシステム (Phase 2)
- **設計の妥当性**: 
    - ポイント制を導入することで、100円課金における決済手数料問題を解決している。
    - 作者 90% 分配を DB 構造（`WorkPurchase` と `PointTransaction`）で担保しており、信頼性が高い。
- **改善提案**: 
    - 不正利用防止のため、`PointTransaction` に `clientIp` や `deviceId` の保存フィールドを追加することを推奨。

### 2.2. 縦画面 (Portrait) レンダリング (Phase 2)
- **設計の妥当性**: 
    - 幅 720px 固定、高さ可変の設計は、モダンなスマホブラウザ（Safari/Chrome）の Viewport 特性と合致する。
    - ボトムアンカーUI は、片手操作におけるアクセシビリティが高い。
- **懸念点**: 
    - 現在のエンジン（`WebOpHandler.ts` 等）にある 1280x720 のハードコード定数を、動的に切り替えるリファクタリングの工数を Phase 2 の序盤に確実に入れる必要がある。

### 2.3. Phase 1（即時リリース項目）の実現性
- **動的 OGP**: `apps/hono` 側で Canvas ライブラリ（`skia-canvas` 等）を使用して画像合成するロジックの実装が必要。これは 2-3 日で可能。
- **ワンタップ起動**: `OpRunner.start()` の引数にスキップフラグを渡すだけで実装可能なため、1 日で完了する。

## 3. リスク管理
- **最大のリスク**: 既存の 16:9 素材（背景・キャラ）を縦画面 9:16 で表示した際の違和感。
- **対策**: エディタ側に「縦画面プレビュー」機能を設け、背景の「どこを中心に切り取るか」を作者が設定できるプロパティを BG ブロックに追加する。

## 4. 結論
本設計仕様書に基づき、**直ちに Phase 1 の開発に着手することを承認する**。ツクール開発を停止したことで浮いたリソースを、SNS 拡散とポイント決済インフラの構築に集中させるべきである。
