# アーキテクチャ・スケーラビリティ改善計画書：10万同時接続への備え

> **作成日**: 2026-03-19
> **担当**: Gemini CLI

## 1. 現状アーキテクチャの限界点（死角）
2月17日の Next.js 移行と 3月7日の `sceneLayer` 導入により、現在の kaedevn は「単一の高品質な体験」を提供するには十分な強度を得た。しかし、プラットフォームとして爆発的に成長した際、以下の「死角」が致命傷となる。

### 1.1. プロジェクトデータのモノリシック構造
現在の設計では、1つの作品データを単一の JSON Blob として DB に保存し、一括でフロントエンドへ送信している。
- **リスク**: 長編作品（100万行規模）において、JSON のシリアライズ/デパースに秒単位のブロッキングが発生し、サーバー帯域とクライアントメモリを圧迫する。

### 1.2. ViewState の全量評価コスト
`WebOpHandler` は、毎フレーム（または状態変化毎に）`ViewState` の全量を PixiJS の `LayerManager` に適用しようとする。
- **リスク**: オブジェクト数が増大した際、DOM 更新と同様の「全件舐め」が発生し、描画パフォーマンスが指数関数的に低下する。

### 1.3. DB コネクションの線形増加
Prisma の現在の使用法は、リクエスト毎に DB へ直接問い合わせを行う。
- **リスク**: 同時接続10万人規模では、コネクションプールが即座に枯渇し、API サーバーが応答不能になる。

## 2. 短期リファクタリング・プラン

### 2.1. プロジェクト構造の「ページ単位」デカップリング
- **設計変更**: `Project` モデルを、`ProjectMetadata` と `ProjectPage` (1対多) に正規化、または JSON 内で物理的に分割。
- **実装**: エンジン側に「ページ・ローダー」を導入し、`OpRunner` が次のジャンプ先を予測してバックグラウンドでページデータを先読みする仕組みを構築。

### 2.2. エンジン・レンダリングの「Dirty Flag」導入
- **設計変更**: `sceneLayer` の各コンポーネントに `isDirty` フラグを持たせる。
- **実装**: `ViewState` の各プロパティ（x, y, alpha 等）が実際に変更された場合のみ PixiJS の setter を呼び出す。これにより、高負荷なテクスチャのバインド更新を最小限に抑える。

### 2.3. キャッシュファーストな API 設計への移行
- **設計変更**: 公開済み作品のメタデータアクセスを `CacheableInterface` でラップ。
- **実装**: `apps/hono` の各ルートにキャッシュヘッダーを適切に設定し、将来的に Redis や Cloudflare Workers KV 等へ容易に差し替え可能な構造にリファクタリング。

## 3. 結論
今夜中に着手すべきは、**「プロジェクトデータの巨大な一括処理を、最小単位のオンデマンド処理へ分解すること」** である。これが、1万件の作品と10万人のユーザーを支えるための「最初の一歩」となる。
