# アップデート通知 ＆ 報告運用ガイドライン

> **作成日**: 2026-03-23
> **ステータス**: 運用指針（通知疲れ防止と透明性の両立）

---

## 1. 課題：毎日 50 回の Push 通知は「ノイズ」になる

開発のスピード感を見せることは重要ですが、通知が多すぎるとユーザーは Discord サーバーをミュートし、最終的に離脱します。

## 2. 戦略：通知の階層化

チャンネルを役割ごとに分離し、ユーザーが「情報の粒度」を選べるようにします。

### A. `#dev-stream`（生ログ・心拍数）
- **役割**: GitHub からの Webhook をそのまま流す。
- **頻度**: 1日 50回以上。
- **設定**: サーバー参加時に「デフォルト通知オフ（ミュート推奨）」を案内。
- **価値**: 「常に動いている」という安心感の提供。

### B. `#changelog`（日次要約・価値の提供）
- **役割**: その日の開発成果を、ユーザー（作者）目線で 3〜5 行に要約して報告。
- **頻度**: 1日 1回（開発終了時）。
- **価値**: 「何が便利になったか」を短時間で把握させる。

### C. `#announcements`（重要告知）
- **役割**: サービス停止、大規模アップデート、イベント告知。
- **頻度**: 週に 1〜2回程度。
- **設定**: `@everyone` を使用。

---

## 3. 「日次要約」の作成ルール

1. **技術用語を翻訳する**: 
   - ×「Prisma の connection_limit を 50 に設定」 
   - ◯「サーバーの安定性を向上させ、混雑時でも止まりにくくしました」
2. **「できるようになったこと」を強調**:
   - 修正や機能追加によって、作者の制作がどう楽になったかを中心に書く。
3. **AI を活用する**:
   - 1日の全コミットメッセージを AI に渡し、ユーザー向けの要約を作成させる。

---

## 4. 結論

毎日 50 回のコミットは、プロジェクトの強み（圧倒的な開発速度）です。それをそのままぶつけるのではなく、**「熱量を維持したまま、ユーザーが消化できる形に料理して届ける」** ことが、作者 100 人を熱狂させ続ける鍵となります。
