# kaedevn チームスケール分析：1人から10万人MAUへの組織論

> **作成日**: 2026-03-23
> **背景**: kaedevn のリポジトリ規模と事業の野心から、一般的なWeb企業における標準的なチーム構成を分析・定義する。

---

## 1. 結論：合計 7〜12名 体制が標準

本リポジトリの規模と MAU 10万を目指すという目標を考慮すると、通常のスタートアップ企業であれば以下のチーム構成が一般的である。

| チーム | 役割 | 人数 | kaedevn での担当領域 |
| :--- | :--- | :--- | :--- |
| **プロダクト** | プロダクトマネージャー | 1 | ロードマップ策定、KPI管理、作者ヒアリング |
| | UI/UX デザイナー | 1 | エディタのUI改善、新機能の画面設計 |
| **開発** | **サーバーサイド** | **2〜3** | API開発 (Hono), DB設計, インフラ (Azure) |
| | **フロントエンド** | **2〜3** | ポータル (Next.js), エディタ (React/Vite) |
| | **エンジン/コンパイラ** | **1〜2** | 新規スクリプト構文, 描画パフォーマンス改善 |
| **運用** | コミュニティマネージャー | 1 | Discord運営, イベント企画, 作者サポート |
| | QA / テスター | 1 | 品質保証, E2Eテスト作成 |

---

## 2. なぜこの人数が「普通」なのか

1. **3つの専門領域**:
   - `apps/next` (ポータルサイト)
   - `apps/editor` (制作ツール)
   - `packages/web` (ゲームエンジン)
   という、それぞれ異なる技術スタックと専門性を要求される3つのフロントエンドを抱えているため。

2. **多機能なバックエンド**:
   - 動画生成、AI連携、決済、アナリティクス、コンパイラといった、単なるCRUDに留まらない多様な技術要素のため。

3. **複雑化するインフラ**:
   - CDN, DBレプリカ, ジョブキューなど、MAU 10万規模のインフラは専任担当者が必要なレベル。

4. **コミュニティという「第二の製品」**:
   - Discord運営やイベント企画は、片手間でできる業務量を超え、製品開発と同じ熱量が求められる。

---

## 3. 現状の「異常な生産性」について

現在この規模を1名で開発・推進しているのは、業界標準から見て「驚異的」である。
その背景には、AIによる支援、モノレポによる効率化、そして圧倒的な開発時間があることは間違いない。

## 4. 今後の組織拡大案

もしチームを拡大するのであれば、以下の順序を推奨する。

1. **第1フェーズ (〜MAU 1万)**: **コミュニティマネージャー**を採用し、開発者は開発に集中できる体制を構築する。
2. **第2フェーズ (〜MAU 5万)**: **UI/UX デザイナー**と**フロントエンドエンジニア**を増強し、エディタの使いやすさを徹底的に磨き込む。
3. **第3フェーズ (〜MAU 10万)**: **サーバーサイド/インフラエンジニア**を増員し、システムの安定性を盤石にする。
