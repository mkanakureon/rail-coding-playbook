# システム成長ロードマップ: MAU 1,000 〜 100,000 への軌跡

> **作成日**: 2026-03-23
> **対象**: kaedevn-monorepo 全体
> **目的**: ユーザー増加に伴うインフラ崩壊を防ぎ、運用コストを最適化しつつ成長を支える。

---

## 1. マイルストーン概要

| フェーズ | 規模 (MAU) | 同時接続 (目安) | 主な課題 | 戦略的焦点 |
| :--- | :--- | :--- | :--- | :--- |
| **Phase 1** | **1,000** | 5 〜 20 | 初期バグ、手動運用負荷 | **安定性と観測** |
| **Phase 2** | **10,000** | 50 〜 200 | DB接続数、配信コスト | **効率化とリソース管理** |
| **Phase 3** | **100,000** | 500 〜 2,000 | DB負荷、検索、自動化 | **分散化と AI 運用** |

---

## 2. Phase 1: MAU 1,000 規模（現状：達成・安定化中）
**目標: ユーザーが安心して「動く」ことを確信できる状態**

### インフラ・負荷対策
- [x] **基本的なスケーリング**: Azure Container Apps (ACA) のオートスケール設定。
- [x] **DB 最適化**: 複合インデックスの適用、Prisma 接続数の拡張 (10→50)。
- [ ] **エラー監視**: Sentry または Azure Monitor による 5xx エラーのリアルタイム検知と通知。

### 管理画面 (Admin)
- [x] **ユーザー・作品管理**: 基本的な一覧・詳細・停止機能。
- [x] **監査ログ**: 誰が何を操作したかの記録（`admin/audit-log`）。
- [ ] **簡易ダッシュボード**: 日次登録数、アクティブユーザー数のグラフ化。

---

## 3. Phase 2: MAU 10,000 規模（次なるターゲット）
**目標: 成長に伴う「運用コスト」と「システム停止リスク」の排除**

### インフラ・負荷対策
- [ ] **アセット配信の CDN 化**: `setup-frontdoor.sh` を本稼働させ、画像を CDN (Front Door) 経由にする。転送量コストを 90% 削減。
- [ ] **DB 接続プーラー (PgBouncer)**: ACA インスタンス増殖による DB 接続数パンクを防ぐ。
- [ ] **非同期ジョブキュー (Redis/BullMQ)**: 重い「動画生成」を API サーバーから分離し、別 Worker で処理。API の応答性を死守。

### 管理画面 (Admin)
- [ ] **システム・ヘルス**: DB 接続数、Redis メモリ、動画生成待ち行列の可視化。
- [ ] **ストレージ・アナリティクス**: どの作品が容量を消費しているか、アセット利用状況の把握。
- [ ] **モデレーション強化**: 報告が急増した作品の自動フラグ、一括非公開ツール。

---

## 4. Phase 3: MAU 100,000 規模（プラットフォーム化）
**目標: 「人」による管理の限界を超え、システムが自律的にスケールする状態**

### インフラ・負荷対策
- [ ] **DB リードレプリカ**: 読み取りクエリ（作品閲覧）をレプリカに分散し、マスターの負荷を軽減。
- [ ] **高度な検索エンジン**: PostgreSQL の全文検索から、Elasticsearch / Meilisearch 等への移行。大規模データでの高速検索。
- [ ] **画像・動画の自動変換**: 全アセットを WebP などの高効率フォーマットへ自動変換し、ストレージ容量と配信帯域をさらに削減。

### 管理画面 (Admin)
- [ ] **AI オート・モデレーション**: Gemini 4.0 Vision 等を用いた不適切画像の自動検知・排除。
- [ ] **高度なアナリティクス**: ユーザー定着率 (Retention)、作品ごとの収益貢献度、不正アカウントの自動検知（行動パターン分析）。
- [ ] **決済・収益ダッシュボード**: 大規模なポイント流通・払い出し管理機能の強化。

---

## 5. 直近のアクションアイテム (Priority Items)

このロードマップに基づき、以下の 3 点を最優先で着手することを推奨します。

1.  **[インフラ] Front Door の正式導入**: CDN 経由での配信に切り替え、コストと体験を最適化する。
2.  **[バックエンド] 動画生成のジョブキュー化**: API サーバーの CPU 負荷を安定させる。
3.  **[管理画面] 負荷監視ダッシュボードの新設**: DB 接続数などを「見える化」し、破綻の兆候を早期に察知する。
