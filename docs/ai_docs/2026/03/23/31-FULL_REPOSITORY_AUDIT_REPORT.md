# kaedevn-monorepo: 全体リポジトリ監査レポート

> **監査日**: 2026-03-23
> **対象**: リポジトリ全域（Architecture, Engine, Editor, Infra）
> **ステータス**: 完了

---

## 1. 監査総評

本リポジトリは、単なる「ツール」の枠を超え、クロスプラットフォーム展開と大規模スケールを見据えた **「堅牢なプラットフォーム・アーキテクチャ」** を備えている。特に、スクリプトから中間形式（Op）への変換パイプラインと、それを各プラットフォームで実行するハンドラの分離が非常にクリーンである。

---

## 2. 領域別評価

### A. アーキテクチャ ＆ モノレポ構成
- **評価**: 非常に高い
- **詳細**: `packages/core` で型を定義し、`compiler` で論理を、`web` で描画を処理する分担が明確。モノレポのワークスペース管理も適切。

### B. コンパイラ ＆ エンジンコア
- **評価**: 高い
- **詳細**: `Compiler.ts` のフェーズ分け（Tokenize -> Parse -> Transform -> Finalize）により、新機能の追加が容易。`WebOpHandler` における命令実行ロジックも網羅性が高い。

### C. エディタ ＆ 状態管理
- **評価**: プロフェッショナル
- **詳細**: Zustand を用いた履歴管理（Undo/Redo）や、プレビュー用のスナップショットスクリプト生成ロジックが洗練されている。

### D. インフラ ＆ デプロイ
- **評価**: 実用的
- **詳細**: Azure Container Apps へのデプロイフローがスクリプト化されており、デプロイ前の自動テストや型チェック、マイグレーション確認が組み込まれている。

---

## 3. 優先改善項目 (Technical Debt)

### 【最優先】WebOpHandler の肥大化（God Object 化）
- **現状**: `WebOpHandler.ts` が 80KB を超え、全命令の実行ロジックを一手に引き受けている。
- **対策**: `CharacterHandler`, `CameraHandler`, `FxHandler` 等に分割し、プラグイン形式で `OpRunner` に登録する構成へのリファクタリングを推奨。

### 【重要】テレメトリの共通化
- **現状**: OpenTelemetry の設定がアプリごとに閉じており、一貫した観測が難しい。
- **対策**: `packages/telemetry` を新設し、モノレポ全体で共通の計装ロジックを利用可能にする。

### 【中長期】エディタストアの型安全性
- **現状**: `useEditorStore.ts` 内に `any` キャストが散見される。
- **対策**: ブロックの型定義を Discriminated Unions で厳密化し、`any` を排除することで、将来のブロック追加時のバグ混入を防ぐ。

### 【中長期】アセット破棄ロジックの検証
- **現状**: 長時間のプレイにおいてテクスチャメモリが蓄積する懸念。
- **対策**: シーン切り替え時（背景変更時等）のテクスチャ解放ロジックをプロファイリングし、モバイル端末でのクラッシュを防止する。

---

## 4. 結論

本プロジェクトは、MAU 1万、10万と成長していくための **「正しい骨組み」** を既に持っている。今後は、機能追加と並行して「肥大化したハンドラの整理」と「型安全性の再強化」を行うことで、さらに長期的なメンテナンス性を確保できる。
