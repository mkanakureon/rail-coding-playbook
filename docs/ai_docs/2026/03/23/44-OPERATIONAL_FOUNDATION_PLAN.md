# 実装計画書：MAU 100 達成のための監視・ログ・復旧対策

> **作成日**: 2026-03-23
> **目標**: ユーザーに「バグった」と言われる前に問題を検知し、数分で復旧できる「プロアクティブな運用体制」を、あなたと AI で構築する。

---

## 1. 監視 (Monitoring)：システムの「心電図」を作る
**目的**: 問題の兆候をリアルタイムに掴む。

### A. 管理者向けヘルス・ダッシュボード (Admin Health Dashboard)
- **場所**: `apps/next/app/(private)/admin/health` (新規作成)
- **表示項目**: DB接続数、API応答時間(p95)、エラーレート(直近1h)
- **実装**: Honoに `/api/admin/health` を新設し、Next.js側で10秒ごとにポーリングしてグラフ表示。

### B. Azure アラート ＆ Discord 通知
- **トリガー**: CPU使用率(5分/80%超)、5xxエラー(1分/5回以上)
- **実装**: Azure Monitorでアラートルールを設定し、通知先をDiscord Webhookに指定。

---

## 2. ログ (Logging)：問題の「フライトレコーダー」を作る
**目的**: 障害発生後、原因をピンポイントで特定する。

### A. フロントエンド・エラーロギング (Sentry.io 導入)
- **役割**: ユーザーのブラウザで発生したJSエラーを収集。
- **実装**: `apps/next` と `apps/editor` に Sentry SDK を導入。

### B. バックエンド・リクエスト追跡 (OTel 強化)
- **役割**: 全てのAPIリクエストに `traceId` を付与し、DBクエリまで一気通貫で追跡。
- **実装**: HonoのOTelミドルウェアで、ログに `userId` を含める。

---

## 3. 復旧 (Recovery)：システムの「自己修復能力」を作る
**目的**: 最悪の事態から迅速に復帰する。

### A. DB 自動バックアップ ＆ リカバリー手順の確立
- **役割**: 人的ミスやディスク障害に備える。
- **実装**: Azure DB for PostgreSQL の「ポイントインタイムリカバリ (PITR)」を有効化。復旧手順書を `docs/07_deployment/RECOVERY_PROCEDURE.md` に作成し、AIと定期的に訓練。

### B. 緊急メンテナンスモード
- **役割**: 重大な脆弱性発見時などに、ユーザーアクセスを一時遮断。
- **実装**: 環境変数 `MAINTENANCE_MODE=true` を用いたミドルウェアをHono/Next.jsに導入。

---
この計画書に基づき、MAU 100 到達前に盤石な運用基盤を構築する。
