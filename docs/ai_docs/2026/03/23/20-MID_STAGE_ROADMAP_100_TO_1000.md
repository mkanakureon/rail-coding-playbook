# 成長期ロードマップ: 100 〜 1,000 MAU（安定と品質の確立）

> **作成日**: 2026-03-23
> **目的**: 多様なユーザーによる「想定外の操作」に耐え、運用の自動化を完了させる。

---

## 1. フェーズ別アクションアイテム

### Phase 1.1: MAU 100 〜 300（エッジケース遭遇期）
- [ ] **エディタのストレージクォータ**: 1ユーザーあたりの合計アセット容量制限の実装。
- [ ] **巨大スクリプト対応**: 数千行の KSC スクリプトをコンパイルした際のメインスレッドブロックの解消（Worker 化の検討）。
- [ ] **メモリリークの徹底排除**: PixiJS のテクスチャ解放（GC）が正しく行われているかのプロファイリング。

### Phase 1.5: MAU 300 〜 600（コミュニティ形成期）
- [ ] **発見性の向上**: カテゴリ別一覧、人気順ソート、タグ検索の UI 実装。
- [ ] **ユーザープロフィール**: 作者ごとの作品一覧ページ、自己紹介、SNS リンク設定。
- [ ] **コメント・フィードバック**: 作品に対する簡易的な感想送信機能。

### Phase 2.0: MAU 600 〜 1,000（運用定型化期）
- [ ] **管理画面のフル機能化**:
    - [ ] アセットブラウザ（全アップロード画像の目視確認ツール）。
    - [ ] ユーザー権限変更（モデレーター権限の付与など）。
    - [ ] お知らせ配信システム（メンテナンス通知、バグ修正報告）。
- [ ] **E2E テストの完全自動化**: 1,000 人のユーザーを抱える中での「デグレード（先祖返り）」をゼロにする。
- [ ] **スケーラビリティ Phase A の完了**: DB 接続プール、インデックス最適化、認証キャッシュの安定稼働確認。

---

## 2. 1,000 MAU 到達時のチェックリスト

- [ ] **バグ発見率の低下**: 新規報告されるバグが、機能そのものではなく「特定の環境」や「極端な操作」に限定されているか。
- [ ] **運用時間の削減**: 管理者が 1日 15分程度の管理画面チェックだけで運営を回せているか。
- [ ] **平均レスポンスタイム**: ユーザー増に関わらず、API レスポンスが 200ms 以下を維持できているか。

---

## 3. 次のステージ（1,000 〜 10,000）への準備

1,000人に達した時点で、**「DB 接続数の推移」**と**「アセット配信の転送量コスト」**を分析し、Phase 2 (10,000 MAU) で予定している CDN 有効化や PgBouncer 導入の具体的な時期を決定する。
