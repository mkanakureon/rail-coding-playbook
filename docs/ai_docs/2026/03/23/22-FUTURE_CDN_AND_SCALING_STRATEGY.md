# 将来のためのインフラ戦略: MAU 10万から 100万への展望

> **作成日**: 2026-03-23
> **ステータス**: 思考メモ（将来のスケール時に参照）
> **目的**: 10万 MAU を超えた際の変化を予測し、現在の Azure 完結戦略の妥当性と、将来の転換点を記録する。

---

## 1. 10万 MAU までは「Azure Front Door 一択」の理由

現時点（MAU 1,000 未満）から 10万 MAU に至るまでは、Azure Front Door を中心とした **「Azure 完結型インフラ」** が最強である。

- **コスト優位性**: Blob Storage や ACA から Front Door への転送量は、Microsoft 内部ネットワークを通るため無料（または極安）。他社 CDN（Cloudflare等）を使うと、Azure から「外に出す」料金だけで多額のコストが発生する。
- **低レイテンシ**: Microsoft の超高速専用ネットワークを通り、公衆インターネットを介さずにエッジからサーバーへ届く。
- **管理の集約**: Managed Identity によるパスワードレス認証や、一元化された WAF ポリシーによる高いセキュリティ。

---

## 2. 10万 MAU を超えた時の「4つのパラダイムシフト」

10万 MAU を超え、大規模プラットフォームとしての安定性が求められる段階では、以下の戦略へ移行を検討する。

### ① 可用性: 「一本足打法」からの卒業
- **戦略**: マルチ CDN（Azure Front Door + Akamai / Cloudflare 等）。
- **理由**: Azure 全域に及ぶような世界規模の障害に備え、DNS レベルで他社インフラへ逃がす経路を確保するため。

### ② コスト: 「EA（エンタープライズ契約）」での直接交渉
- **戦略**: ボリュームディスカウントの交渉。
- **理由**: 標準の従量課金ではなく、年間利用枠を確約することで、1GB あたりの転送単価を大幅に引き下げる。

### ③ 効率: 「エッジ・コンピューティング」の極限活用
- **戦略**: Front Door のエッジ機能を用いた「画像変換」や「認証の一次検証」。
- **理由**: 重い画像処理や不正リクエストの排除を API サーバー（ACA）に届く前にエッジで終わらせ、サーバー増設コストを抑制する。

### ④ 防御: 「オリジン・シールド」の導入
- **戦略**: Front Door Origin Shield の有効化。
- **理由**: 無数のエッジ拠点から直接 Blob Storage にアクセスさせず、特定の「シールド拠点」を経由させることで、ストレージ側のスロットリング（制限）を回避する。

---

## 3. 「Azure 脱出（他社移行）」を検討すべき唯一の指標

もし将来的に Azure Front Door（または Azure 自体）を一部やめるとすれば、その判断基準は以下の通り。

**「Azure の転送量コスト ＞ 他社ストレージ（Cloudflare R2 等）の利用料」になった時**

- **Cloudflare R2 への移行シナリオ**:
  - 画像・動画アセットだけを Azure から Cloudflare R2（下り転送量無料）に移動。
  - CDN も Cloudflare に統一。
  - **メリット**: Azure の高額な転送量をゼロにできる。
  - **デメリット**: サーバー（ACA）とストレージが別クラウドになり、通信のオーバーヘッドが発生する。

---

## 4. 結論：現時点での指針

現在のフェーズ（1,000 MAU 未満）では、他社を検討する手間やネットワークの複雑化を避けるべきである。

1. **まずは Front Door を使い倒す**: キャッシュルール（`/storage/*` は 1年キャッシュ等）を最強の状態にする。
2. **10万 MAU が見えたら再考する**: それまでは Azure のエコシステム内で「スピード」と「一貫性」を最優先とする。
