# 運用・スケーラビリティ・管理機能レビュー (MAU 10,000 規模対応)

## 1. 結論: 現状で MAU 10,000 に耐えられるか？

**「概ね耐えられるが、ピーク時の DB 接続詰まりに懸念あり」**

現在のアーキテクチャは 3-Tier (Next.js / Hono / PostgreSQL) であり、Azure Container Apps + Front Door (CDN) の構成はスケーラビリティに優れています。しかし、1万人規模のユーザーが同時期に活動する場合、以下の 3 点が「運用崩壊」の火種となります。

1. **DB 接続プールの枯渇**: Prisma のデフォルト接続数 (10) は既に 50 に拡張済みですが、サーバーがオートスケールした際に DB 側の最大接続数を超過するリスク。
2. **アセット配信コスト**: Azure Blob Storage への直接アクセスが増えると、下り転送量（Egress）コストが跳ね上がる。
3. **管理業務のパンク**: 手動でのアセット承認や通報対応が 1万人規模では追いつかなくなる。

---

## 2. サーバー負荷とボトルネック分析

### 2-1. DB 接続管理 (The Prisma Bottleneck)
- **現状**: `connection_limit=50` に拡張済み。
- **リスク**: ACA (Azure Container Apps) が 10 インスタンスにスケールすると、50 * 10 = 500 接続。Azure Database for PostgreSQL の最小構成 (2 vCore) では接続数制限に達し、`Can't reach database` でサービスが全停止する。
- **対策**: **PgBouncer** 等のコネクションプーラーの導入、またはサーバー側の `max-replicas` を DB の許容範囲内に制限する。

### 2-2. アセット配信と CDN
- **現状**: `setup-frontdoor.sh` で Front Door の準備がある。
- **リスク**: CDN が未有効化、またはキャッシュ設定が不適切な場合、画像 1 枚ごとに Blob Storage へのリクエストが発生し、レイテンシとコストが増大。
- **対策**: `/storage/*` へのリクエストを Front Door で 1 年間キャッシュ (`immutable`) し、オリジンへのアクセスを 95% 以上削減する。

### 2-3. 重い処理 (CPU/Memory bound)
- **動画生成**: 1万人規模で動画生成が重なると、サーバー CPU が 100% に張り付き、通常の API 応答が遅延。
- **対策**: 動画生成を別インスタンス（Worker）に切り出すか、Azure Functions 等のサーバーレス環境へオフロードする。

---

## 3. 管理画面への追加推奨機能 (Operational Admin)

現在の `apps/next/app/(private)/admin` に以下の機能を追加することで、1万人規模の運用を効率化できます。

### 3-1. システム・モニタリング (Health Dashboard)
- [ ] **DB 接続数監視**: Prisma 経由またはメトリクス API から現在の接続数をグラフ化。
- [ ] **動画生成キューの監視**: 待ち行列の数と平均処理時間を表示。詰まっている場合に「一時停止」するスイッチ。
- [ ] **OTel エラーレート**: Jaeger/Prometheus と連携し、5xx エラーの急増を管理画面上で検知。

### 3-2. アセット・モデレーション (Safety)
- [ ] **AI コンテンツ自動検閲**: アップロードされた画像を Google Cloud Vision 等で自動スキャンし、不適切なものにフラグを立てる。
- [ ] **一括非公開/BAN 機能**: 荒らし行為に対し、ユーザーに関連する全作品を一括で非公開にする機能。

### 3-3. 統計・アナリティクス (Engagement)
- [ ] **リアルタイム接続数**: 今、何人がプレイ中かを表示。
- [ ] **コンバージョン分析**: 作品ページから「プレイ開始」に至った割合の集計。

---

## 4. 運用アクションプラン

### Step 1: 守りを固める (現在〜1ヶ月)
- **Front Door 有効化**: 全アセットを CDN 経由にする。
- **DB 接続監視の強化**: 接続数が 80% に達した際のアラート設定。

### Step 2: 効率化 (1〜3ヶ月)
- **動画生成の非同期化**: Redis + BullMQ (または Azure Service Bus) への移行。
- **管理画面への「統計ダッシュボード」追加**。

### Step 3: 自動化 (3ヶ月〜)
- **画像・動画の自動リサイズ/WebP 変換**: ストレージ容量を 50% 以上削減。
- **AI モデレーション導入**。
