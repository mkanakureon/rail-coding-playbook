# 超初期ロードマップ: 0 〜 100 MAU（生存と検証フェーズ）

> **作成日**: 2026-03-23
> **目的**: サーバー負荷の前に「ユーザーの離脱」を防ぎ、プロダクトマーケットフィット (PMF) の兆候を掴む。

---

## 1. フェーズ別アクションアイテム

### Phase 0.1: MAU 10（アルファ期）
- [ ] **エラー通知の実装**: API エラーやフロントエンドのクラッシュを Discord / Slack に即時通知。
- [ ] **DB マイグレーション・ポリシー**: 開発中の破壊的変更からユーザーデータを保護する手順の確立。
- [ ] **管理者権限の整理**: `admin/*` ページに自分（開発者）だけがアクセスできることを再確認。

### Phase 0.5: MAU 50（ベータ期）
- [ ] **フィードバック・ハブ**: ユーザーが「使いにくい」「バグった」を即座に送れる窓口。
- [ ] **オンボーディング・トラッキング**: 新規登録したユーザーが「作品公開」まで到達したか（離脱ポイント）の可視化。
- [ ] **自動バックアップ**: 1日1回の DB スナップショット保存。

### Phase 1.0: MAU 100（ソフトローンチ期）
- [ ] **モデレーション・フロー**: `admin/reports-abuse`（通報対応）の画面を使いやすく改善。
- [ ] **お知らせ機能 (Announcements)**: 管理画面から全ユーザーへ「新機能追加」「メンテ予告」を表示。
- [ ] **パフォーマンス・ベースライン計測**: 100人時点でのレスポンスタイム (p95) を記録し、今後の劣化を検知できるようにする。

---

## 2. 管理画面 (Admin) への追加優先順位

1. **エラーログ・ビューア**: OTel/Jaeger に行く前の、直近の致命的エラーのサマリー。
2. **ユーザー行動タイムライン**: 特定のユーザーが「どこでエラーに遭遇し、どこで去ったか」を追える簡易ログ。
3. **お知らせエディタ**: マークダウン形式で「運営からのお知らせ」を投稿・管理する機能。

---

## 3. サーバー負荷についての考え方 (MAU 100 以下)

この規模では **サーバー負荷は全く問題になりません**。
Azure の最小インスタンスでも余裕で捌けます。それよりも **「管理者の認知負荷」** を下げること（＝不具合をユーザーに言われる前に気づくこと）にリソースを割くべきです。
