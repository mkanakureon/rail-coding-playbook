# Discord コミュニティ開設・運用ロードマップ

> **作成日**: 2026-03-23
> **ステータス**: 戦略案（MAU 10 時点での開設を推奨）

---

## 1. 目的：なぜ Discord が必要なのか

ビジュアルノベル制作は孤独な作業です。作者が「このエンジンの向こう側には人間（開発者や仲間）がいる」と感じられる場所を作ることで、挫折率を劇的に下げます。

## 2. ステップ別の運用計画

### Step 1: 開発者と「密着」期（MAU 10 〜 50）
- **ターゲット**: 最初のテスター、知人、熱心な早期採用者。
- **目標**: 致命的バグの早期発見と「神対応」によるファン化。
- **主要チャンネル**:
    - `#アップデート`: 毎日・毎回の Git Push レベルの更新を報告。
    - `#質問・サポート`: 開発者が即レスする場所。

### Step 2: 作者同士の「共鳴」期（MAU 50 〜 300）
- **ターゲット**: 一般公開後の新規作者。
- **目標**: 「このエンジンなら反応がもらえる」という空気感の醸成。
- **主要チャンネル**:
    - `#進捗晒し`: 制作中のスクショや動画を投稿。
    - `#プロンプト共有`: AI アセット生成のコツを教え合う。

### Step 3: 公認コミュニティ期（MAU 300 〜 1,000）
- **目標**: 運営コストの低減と自律的な成長。
- **施策**: 作者の中からモデレーターを募り、自治を促す。

---

## 3. 成功のための 3 つのポイント

1. **エディタとの統合**:
   - エディタの UI 内に「困ったら Discord で聞く」ボタンを配置。
   - API エラー時に「Discord で報告する」リンクを表示。
2. **「バグ」を「イベント」にする**:
   - バグ報告を嫌がらず、「素晴らしい発見です！」と称賛し、修正完了を即座にメンションで伝える。
3. **ロードマップの透明化**:
   - 今日作成したようなロードマップを Discord で公開し、開発のワクワク感を共有する。

---

## 4. 結論

Discord は **今すぐ**（あるいは最初の 5 人のユーザーが定着した瞬間に）開設すべきです。大規模になってからでは「開発者と作者の距離」が遠くなります。初期の「泥臭いサポート」こそが、将来の 10 万人を支える 100 人のコア作者を生みます。
