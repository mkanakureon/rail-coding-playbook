# 本格普及期ロードマップ: 1,000 〜 10,000 MAU（スケールと効率の確立）

> **作成日**: 2026-03-23
> **目的**: インフラコストを制御しつつ、1万人規模の同時利用に耐えうる「止まらないシステム」を構築する。

---

## 1. フェーズ別アクションアイテム

### Phase 2.1: MAU 1,000 〜 3,000（スケーラビリティ Tier 1）
- [ ] **Front Door (CDN) 統合**: `/storage/*` へのアクセスを 100% CDN 経由にし、オリジン負荷とコストを削減。
- [ ] **動画生成ジョブキュー (Worker 分離)**: API サーバーから重い処理を完全に切り離し、スケーラビリティを確保。
- [ ] **Redis キャッシュ**: 共通の API レスポンス（人気作品一覧等）を Redis に保持し、DB 負荷を軽減。

### Phase 2.5: MAU 3,000 〜 7,000（コミュニティとエコシステム）
- [ ] **レコメンド・エンジン**: ユーザーのプレイ履歴に基づいた「おすすめ作品」の表示。
- [ ] **アセット自動変換**: アップロード画像をサーバーサイドで WebP/AVIF に変換し、配信帯域を削減。
- [ ] **通報自動処置**: AI (Vision) による不適切画像の一次検閲と、通報数に基づいた自動フラグ立て。

### Phase 3.0: MAU 7,000 〜 10,000（信頼性と経済性）
- [ ] **PgBouncer 導入**: DB コネクションの効率化による同時接続数上限の引き上げ。
- [ ] **カーソルベース・ページネーション**: 全一覧取得 API の高速化と、ディープページング時の負荷解消。
- [ ] **レート制限の高度化**: Redis を用いた、分散環境下での正確な IP/ユーザー単位のレート制限。
- [ ] **マネタイズ・ダッシュボード**: サーバーコストと収益の推移を管理画面で可視化。

---

## 2. 10,000 MAU 到達時のインフラ目標値

- **可用性 (SLA)**: 99.9% 以上（月間停止時間 43分以内）。
- **レスポンスタイム (p95)**: 300ms 以下（全 API 平均）。
- **キャッシュヒット率 (CDN)**: 90% 以上。
- **DB CPU 使用率**: ピーク時平均 60% 以下を維持。

---

## 3. 次のステージ（10,000 〜 100,000）への課題

1万人に達した時点で、**「DB 書き込み負荷の増大（プレイ回数更新等）」**が問題になり始める。Phase 3 (100,000 MAU) に向けて、DB のリードレプリカ導入や、プレイ回数更新のバッチ化（Redis でカウントし、一定時間ごとに DB へ書き戻す）の検討を開始する。
