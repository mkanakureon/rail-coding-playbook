# ソースコード流出防止のための完全分離リポジトリ戦略

## 1. 課題：統合モノレポのリスク
非公開コードとOSSコードを同じリポジトリ（モノレポ）で管理する場合、設定ミスや操作ミスにより、本来秘匿すべきコード（商用ビルドロジックやAPIキー等）が誤って公開リポジトリへ同期されてしまうリスクを完全に排除することは困難です。

## 2. 解決策：物理的分離と一方向依存
リポジトリを物理的に2つに分割し、依存関係を「非公開 → 公開」の一方向に限定することで、流出リスクを構造的にゼロにします。

### A. kaedevn-oss (Public Repository)
- **内容**: エンジンコア、Webレンダラー、エディタ、ローカルツール。
- **公開範囲**: 全世界（GitHub Public）。
- **開発**: コミュニティ主導。機能追加はプルリクエストベースで行われる。

### B. kaedevn-cloud (Private Repository)
- **内容**: 投稿プラットフォーム、商用ビルドサービス、課金システム。
- **公開範囲**: 社内/開発チームのみ（GitHub Private）。
- **依存関係**: `package.json` または `git submodule` で `kaedevn-oss` を参照し、その機能を利用して動作する。

## 3. 安全な開発フロー
1. **OSS機能の開発**: まず `kaedevn-oss` リポジトリでエンジンの修正や機能追加を行い、マージ・公開します。
2. **クラウド機能の開発**: `kaedevn-cloud` リポジトリで、更新されたOSSパッケージを取り込み（`npm update` や `git pull`）、独自のビジネスロジックを実装します。

## 4. AIエージェント（Claude Code）の運用
リポジトリ分割によるコンテキスト分断を防ぐため、ローカル開発環境では両リポジトリを並列に配置した「ワークスペース」を構築します。
- **構成**:
  ```text
  workspace/
  ├── kaedevn-oss/
  └── kaedevn-cloud/
  ```
- **AI指示**: ワークスペースルートでAIを起動し、両方のパスを認識させることで、横断的な理解とコード生成を可能にします。ただし、コミット操作は各リポジトリで独立して行うため、誤送信のリスクはありません。

## 結論
「ソースコード流出は絶対に許されない」という要件においては、利便性（モノレポ）よりも安全性（物理分離）を優先すべきです。この構成であれば、**「非公開コードがOSS側に混入する」という事故は物理的に起こり得ない**ため、安心して開発とOSS運営を両立できます。
