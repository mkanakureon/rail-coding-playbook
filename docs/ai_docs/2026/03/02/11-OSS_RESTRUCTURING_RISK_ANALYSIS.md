# kaedevn-monorepo 再編・OSS化に伴う詳細リスク分析

## 1. 技術的リスク：レンダリング層の密結合
現在の `packages/web` は PixiJS に強く依存しています。これが `core` や `interpreter` に漏れ出している場合、以下のリスクが発生します。
- **Switch移植の困難**: Nintendo Switch 等のネイティブ環境では PixiJS がそのまま動かないため、描画命令の抽象化（IOutputDevice インターフェース等）が不完全だと、ロジックの書き直しが発生します。
- **対策**: 描画命令（Sprite表示、テキスト表示）をコマンド形式で抽象化し、Web版とネイティブ版で完全に実装を分離できる構造を徹底します。

## 2. セキュリティリスク：ローカル環境特有の制約
ブラウザ上で動くエディタからローカルファイル（ユーザーのHDD内の画像等）にアクセスする際、モダンブラウザのセキュリティ制限が壁となります。
- **CORS / Mixed Content**: `localhost` からローカルファイルを直接 `file://` で参照することは制限されています。
- **対策**: Hono-local バックエンドをプロキシとして機能させ、ローカルアセットを一時的な HTTP URL として提供する「仮想アセットサーバー」機能を実装します。

## 3. 法的リスク：OSSライセンスと著作権
リポジトリを公開する際、混入しているライブラリや素材のライセンス管理が重要です。
- **伝播型ライセンス (GPL等)**: もしエンジンの一部に GPL ライセンスのコードが含まれていると、非公開にしたい「商用リーダー」まで OSS 化を迫られるリスクがあります。
- **対策**: 依存関係を MIT/Apache-2.0/BSD などの寛容なライセンスに限定し、ライセンスチェッカーを CI に組み込みます。

## 4. 運用リスク：AIエージェントによるコード品質の低下
Claude Code などの AI が大規模リファクタリング（DI化など）を主導する際、コンテキストの不足により以下の問題が起こり得ます。
- **見えない破壊**: 型定義は通っているが、実行時の挙動（タイミング問題、メモリリーク等）が微妙に変わってしまう。
- **対策**: AI に変更を任せる前に、カバレッジの高い E2E テスト（Playwright）と結合テストを整備し、「AI が壊したことを自動で検知できる環境」を先に構築します。

## 5. コミュニティリスク：フォークとブランドの拡散
エディタを OSS 化することで、酷似した「競合プラットフォーム」が乱立するリスクがあります。
- **対策**: 「エディタ（制作ツール）」は自由に使わせるが、「kaedevn 認定ロゴ」や「公式プラットフォームへの投稿権」といったブランド価値で差別化を図ります。

## 結論
これらのリスクの多くは、**「初期段階での厳格なインターフェース設計」**と**「自動テストの完備」**によって回避可能です。特に AI を開発の主軸に据える場合、人間によるコードレビュー以上に、テストコードの質がプロジェクトの成否を分けることになります。
