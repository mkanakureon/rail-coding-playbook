# 5層パイプライン断絶の現状実態報告書（再確認結果）

## 1. 概要
「新しいコマンドを1つ追加する」という単純な作業において、リポジトリ内の各層がいかに同期しておらず、手作業によるミス（実装漏れ）を誘発しやすい構造になっているかを具体例とともに報告します。

## 2. 実例調査：overlay コマンドの対応状況（2026/03/02現在）

最新のソースコードをスキャンした結果、`overlay` 機能は以下の層で断絶しています。

### A. 疎通している層
- **Compiler層**: トークナイズおよびパースは対応済み。
- **Core層**: 命令（Op）の型定義およびハンドラ定義は対応済み。
- **Web層**: PixiJSを用いた実際の描画ロジックは実装済み。

### B. 断絶している（実装が漏れている）層
- **Interpreter層 (packages/interpreter)**: 
  - `Interpreter.ts` の `executeBuiltin` メソッドに `overlay` の分岐が存在しません。
  - `IEngineAPI.ts` にメソッドが定義されていないため、スクリプトエンジンから呼び出せません。
- **Editor層 (apps/editor)**:
  - `useEditorStore.ts` の `buildPreviewScript` メソッドにおいて、`overlay` 型のブロックを無視しています。
  - これにより、全ページプレビュー実行時にオーバーレイが表示されない不具合が構造的に埋め込まれています。

## 3. 断絶が引き起こす3つの実害

1. **機能の「死に体」化**: エンジン（Web層）に高度な機能が実装されても、スクリプト（Interpreter層）から呼べないため、宝の持ち腐れになっています。
2. **サイレント・デグレード**: エディタの特定のプレビューモードでのみ機能が動かない、といった「部分的な故障」が多発します。
3. **コントリビューターの意欲喪失**: 外部の人がエンジンを改良しても、エディタやパーサーの修正箇所が見つけられず、「このプロジェクトは拡張しにくい」という評価に繋がります。

## 4. 抜本的な解決の方向性

現在の「各層で switch-case を書く」方式を、**「メタデータ駆動」** に変更する必要があります。

- `packages/core` に `COMMAND_DEFINITION` を作成。
- コンパイラ、インタープリタ、エディタの各生成ロジックは、この定義を元に **自動的にルーティングを行う** ように変更。
- 未実装のレイヤーがある場合、ビルド時または実行時に明確な警告を出す仕組みの構築。

## 結論
「5層パイプラインの断絶」は、単なる作業漏れではなく、**現在のアーキテクチャが「漏れを許容してしまっている」**ことに本質的な原因があります。OSS化の前に、この垂直方向の整合性を保証する仕組みを導入することを強く推奨します。
