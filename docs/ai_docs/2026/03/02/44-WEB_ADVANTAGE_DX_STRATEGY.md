# Webモノレポ構成の優位性を活かしたOSS成功戦略：実利とポータビリティの追求

## 1. 概要
高コストな外部APIやGPUリソースに依存せず、現在のWebベースのモノレポ構成が持つ「ポータビリティ」「解析性」「エコシステム」を最大限に活用した、実利的かつ地味に効く改善項目を提案します。

## 2. 戦略①：ゼロ・コンフィグによる「爆速導入」体験
OSSとしての最初のハードルである「環境構築」を徹底的に排除します。

### 具体的な施策
- **DBレス・ローカルモード**: デフォルト設定を PostgreSQL から SQLite (or JSON) に変更し、クローン直後に `npm install && npm start` だけで全機能が動作することを保証。
- **インメモリ・プレビュー**: サーバーサイドへの保存を待たず、ブラウザのメモリ上で即座にゲームを動かす「Instant Preview」の強化。
- **メリット**: 開発者が「ちょっと試してみよう」と思った瞬間に、製品レベルのツールが手元で動き出す感動を提供。

## 3. 戦略②：Web標準技術による「ライブ・デバッグ」の極致
ブラウザが持つ強力な開発者支援機能を、ゲーム制作のワークフローに直結させます。

### 具体的な施策
- **Hot-Module Scenario (HMS)**: シナリオスクリプトを保存した瞬間、ゲームの「続き」から、状態（変数・BGM等）を保持したまま演出だけが更新される仕組み。
- **CSS変数によるテーマ制御**: エンジンのUI（メッセージウィンドウ等）の見た目を、外部ファイルなしでブラウザの開発者ツール（F12）からライブ編集可能に。
- **JS-Bridge**: ブラウザコンソールから直接、実行中のゲーム変数（例: `game.variables.affection = 100`）を操作して、フラグ分岐を確認できる機能。

## 4. 戦略③：アセット・パイプラインの自動化
クリエイターが「面倒だがやらなければならない作業」を、モノレポ内のスクリプトで自動化します。

### 具体的な施策
- **自動最適化**: フォルダに画像を置くだけで、ブラウザに最適な WebP や、ネイティブ向けの軽量フォーマットへ自動変換。
- **アセット逆引き検索**: エディタ上で「この立ち絵を削除したいが、どこかのシーンで使われていないか？」を一瞬で検索できる参照管理の徹底。
- **ハッシュベースの重複排除**: 同じ画像を別名で登録しても、内部で実体を一つに集約し、プロジェクト容量を削減。

## 5. 実装の方向性（Repository Check）
- `packages/compiler` にアセットスキャナーを追加。
- `apps/editor` の保存ロジックにインメモリ・バッファを導入。
- `packages/web` の `WebOpHandler` に、コンソールからのデバッグ・エントリーポイントを公開。

## 結論
「派手なAI」ではなく、**「驚くほど構築が速く、デバッグが心地よく、アセット管理が正確」** という、道具としての基本性能を Web 技術で極めることが、プロ・アマ問わず長く愛される OSS になるための最短ルートです。
