# kaedevn-monorepo 再編・OSS化プロジェクト工数見積もりレポート

## 1. プロジェクト目標
現在のモノレポを「OSS版制作環境（Studio/Engine）」と「非公開版配信基盤（Cloud）」に分離し、個人開発者がローカル環境で自由にノベルゲームを制作・テストできる環境を1ヶ月以内に構築する。

## 2. 工程別詳細見積もり

### Phase 1: 外部依存の抽象化（コア修正）
- **内容**: Azure Blob/PostgreSQLへの直接依存を排除し、Storage/DBインターフェースを導入。
- **工数**: 6日間
- **タスク**: インターフェース定義、既存サービスコードのリファクタリング。

### Phase 2: ローカル実行用バックエンドの開発
- **内容**: ローカルファイルシステム（Node.js fs）およびSQLiteを用いたデータ永続化の実装。
- **工数**: 5日間
- **タスク**: LocalStorageAdapterの実装、Hono用ローカルルーティングの調整。

### Phase 3: 個人開発者用ポータル（マイページ）の構築
- **内容**: Next.jsアプリから認証・SNS機能を省いた軽量ポータルの作成。
- **工数**: 4日間
- **タスク**: UIの整理、ローカルAPIへの接続、不要コンポーネントの削除。

### Phase 4: リポジトリ分割とパッケージング
- **内容**: モノレポの物理的分割と、独立したビルドパイプラインの構築。
- **工数**: 3日間
- **タスク**: Gitリポジトリ分割、package.jsonの再定義、CI設定。

### Phase 5: AIコンテキストおよびドキュメント整備
- **内容**: 外部貢献者（およびAI）のための開発ガイド作成。
- **工数**: 2日間
- **タスク**: GEMINI.md, CLAUDE.mdの配置、READMEの整備。

## 3. リソースと開発手法
- **推奨体制**: シニアエンジニア 1名 ＋ AIエージェント（Claude Code / Gemini CLI）。
- **AI活用戦略**:
  - 定型的なリファクタリング（DI化）はAIが担当。
  - テストコードの自動生成により、手動での動作確認時間を削減。
  - ドキュメントの下書きはAIが既存コードから自動生成。

## 4. リスク要因と対策

- **レンダリング層の抽象化不足**: 
  - **リスク**: PixiJS（Web）に依存したコードがゲームロジック（core/interpreter）に混入していると、Switch等のネイティブ環境への移植コストが急増する。
  - **対策**: `packages/core` におけるインターフェース定義を厳格化し、描画エンジンをプラグイン形式で差し替え可能にする。
- **ローカルブラウザのセキュリティ制限 (CORS/File API)**:
  - **リスク**: ユーザーのPC上のブラウザからローカルファイルにアクセスする際、ブラウザのセキュリティポリシー（CORS等）により画像の読み込みがブロックされる可能性がある。
  - **対策**: ローカル版バックエンド（Hono-local）を「Asset Server」として機能させ、ブラウザに対して適切なHTTPヘッダー付きで素材を配信する仕組みを構築する。
- **ライセンス管理の不徹底**:
  - **リスク**: OSS化したリポジトリに、商用利用不可なサードパーティ製アセットやライブラリが混入した場合、法的リスクが発生する。
  - **対策**: `npm list` や自動スキャンツールを用いて、依存ライブラリのライセンスを精査し、必要に応じて代替ライブラリへ差し替える。
- **AIエージェントの誤学習・誤修正**:
  - **リスク**: AI（Claude Code等）がモノレポの複雑な依存関係を誤解し、不適切なリファクタリングを行うことで、発見しにくいバグが混入する。
  - **対策**: `GEMINI.md` 等のAI用コンテキストファイルに「触れてはいけない領域」と「変更時のルール」を明文化し、AIの行動範囲を制御する。

## 結論
総工数は約20日間（実働4週間）を見込む。AIエージェントを最大限に活用することで、15日間程度まで圧縮し、最短3週間でのOSS公開が可能であると判断する。
