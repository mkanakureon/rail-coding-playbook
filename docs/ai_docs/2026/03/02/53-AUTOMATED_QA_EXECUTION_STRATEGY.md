# 自動QA基盤の実行場所と運用戦略：ローカル・ファーストの原則

## 1. 結論：作者のPC（ローカル環境）での実行を推奨
「演出の崩れ」を検知する自動QA（録画・比較）は、**「作者のPC」での実行**を主軸に据えるべきです。

## 2. ローカル実行を推奨する3つの理由

### A. プライバシーと機密情報の保護
- ノベルゲームの制作において、未発表のシナリオや立ち絵は極めて重要な秘密情報です。
- プラットフォーム（PF）のサーバーにデータを上げることなく、自分のPC内で「崩れ」を確認できることは、クリエイターにとって最大の安心材料になります。

### B. 高コストな計算リソースの回避
- Playwright による「全シーンの自動実行 ＋ 録画」は、サーバーサイドで行うと莫大なCPU/GPUコストがかかります。
- 作者のPC（開発機）であれば、すでにアセットが手元にあり、追加の通信コストなしで高速に処理を回せます。

### C. 修正サイクルの高速化（フィードバックループ）
- サーバーにプッシュしてから「結果」を待つのではなく、ローカルでスクリプトを保存した瞬間に自動で「前回との比較（diff）」が走る環境こそが、最高の開発体験（DX）を生みます。

---

## 3. 運用イメージ：コミット時やプッシュ時の自動実行

作者は以下のようなフローで「自動QA」の恩恵を受けます。

1. **制作**: エディタで演出を調整し、保存。
2. **自動実行 (Git Hook)**: 
   - `git commit`（または特定のコマンド）を実行した際、裏側で `auto-record.sh` が走る。
3. **崩れの検知**: 
   - 前回の「合格済み画像」と、今回の「実行画像」をAIまたは画像比較ツールがチェック。
4. **通知**: 
   - もし演出が意図せず崩れていれば、**OSの通知機能やエディタ上のアラート**で「演出が崩れています！確認してください」とスクリーンショット付きで作者に報告。

## 4. プラットフォーム（PF）サーバーの役割
PF側は、以下の「最終的な品質保証」のみを担います。
- **公開前チェック**: ユーザーが「公開ボタン」を押した際に、サーバー側で最小限のランタイムチェック（スクリプトエラーがないか等）を行う。
- **ビルド依頼時**: Switch版等の製品ビルドを依頼された際、サーバー側で正式なQAキャプチャを生成し、最終確認資料として作者に提供。

## 結論
自動QA基盤の本質は、**「作者の手元で、作者を助けること」**にあります。ローカル実行を基本とすることで、プライバシーを守り、PFのサーバーコストを抑えつつ、クリエイターに「AIが常に演出を監視してくれている」という圧倒的な安心感を提供できます。
