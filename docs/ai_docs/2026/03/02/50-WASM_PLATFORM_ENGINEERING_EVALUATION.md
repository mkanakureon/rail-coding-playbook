# WebAssembly & Platform Engineering 評価報告書：kaedevnへの適用

## 1. 概要
`docs/01_in_specs/0302/web.md` に記載された WebAssembly (Wasm) と Platform Engineering の概念を、現在の kaedevn プロジェクトのアーキテクチャ（Next.js, Vite, Hono, PixiJS）に照らし合わせて評価しました。

## 2. WebAssembly (Wasm) の適用評価

### A. スクリプトVM (KSC) の Wasm 化
- **現状**: 現在の KSC コンパイラおよび VM は TypeScript で実装されています。
- **適用案**: VM の実行コアを Rust 等で実装し Wasm 化することで、ブラウザ上での実行速度を向上させ、かつ OS 非依存の「共通ランタイム」として Switch 版と 100% 同一のロジックを共有できます。
- **メリット**: スクリプトの実行速度向上と、プラットフォーム間での挙動の完全一致。

### B. 安全な「ユーザー拡張・プラグイン」基盤
- **現状**: エフェクトや独自コマンドの追加にはソースコードの修正が必要です。
- **適用案**: Wasm をプラグイン実行基盤として採用します。ユーザーが Rust や C++ で書いた独自の演出ロジックを、エンジンのコアを汚さずに「安全なサンドボックス」内で実行可能にします。
- **メリット**: OSS 利用者が本体をフォークすることなく、高度な拡張機能を配布・利用できるようになります。

## 3. Platform Engineering の適用評価

### A. 作者向け「セルフサービス・ポータル」の構築
- **現状**: `apps/next` がマイページ機能を持っていますが、デプロイやビルドは手動またはスクリプト実行が必要です。
- **適用案**: `apps/next` を「内部開発プラットフォーム (IDP)」として強化します。作者がボタン一つで「Web 公開」「デバッグ用 URL 発行」「Switch 版ビルド依頼」を完結できる仕組みを整えます。
- **メリット**: 制作以外の「インフラ作業」から作者を解放し、創作に集中できる環境を提供します。

### B. 自動化された「品質保証（QA）パイプライン」
- **現状**: Playwright によるテストがありますが、手動実行が主です。
- **適用案**: 以前提案した「自動デモ録画（auto-record.sh）」をさらに進め、コミットごとに「演出の崩れ」を自動検知し、スクリーンショット付きで作者に通知する自動 QA 基盤を構築します。
- **メリット**: AI エージェント（Claude Code 等）が修正した結果を自動で動画化・検証し、人間が承認するだけのフロー（DevEx の最大化）を実現します。

## 4. kaedevn における具体的導入ロードマップ

| フェーズ | 項目 | 内容 |
| :--- | :--- | :--- |
| **短期** | **Platform Eng.** | `scripts/auto-record.sh` の CI 組み込み。作者向けデバッグ用 Snapshot URL 発行機能の整備。 |
| **中期** | **Wasm** | 重い計算（バトルロジックや特殊な画像処理フィルター）の Wasm 化。 |
| **長期** | **Combined** | ユーザーが Wasm でプラグインを書ける「拡張ストア」機能のプラットフォーム統合。 |

## 5. 結論
Wasm は「**実行のポータビリティと安全な拡張**」を、Platform Engineering は「**作者の制作体験（DX）の最大化**」を支える柱となります。特に OSS 化において、これらの技術を「地味だが強力なインフラ」として整備することは、kaedevn が他エンジンに対する技術的優位性を確立する決定的な要因となります。
