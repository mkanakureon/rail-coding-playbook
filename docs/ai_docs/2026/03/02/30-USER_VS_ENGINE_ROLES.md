# 作者が「bg_new」を意識しなくて良い理由：役割の完全分離

## 結論：作者は `@bg` と書くだけでOK
シナリオ作者（スクリプト執筆者）にとって、操作対象は常に **「背景（bg）」** という抽象的な概念だけです。`bg_new` は、エンジンの「中身」が勝手に作り、勝手に消す **「魔法の道具」** です。

## 1. 役割の分担

| 役割 | 使う言葉 | 考えていること |
| :--- | :--- | :--- |
| **シナリオ作者** | `@bg classroom` | 「背景を教室に変えたい。フェードさせたい」 |
| **エンジン内部** | `bg_new` | 「作者が背景を変えたいと言っている。今の画像とぶつからないように、一旦 `bg_new` という名前で準備して、綺麗に混ぜてから最後に入れ替えよう」 |

## 2. なぜ作者に `bg_new` を見せないのか？

もし作者が自分で `bg_new` を管理しなければならないとしたら、スクリプトは以下のように非常に面倒なことになります。

```ks
// もし作者が bg_new を書く必要があったら（非常に大変！）
@load_to_bg_new classroom
@start_crossfade bg bg_new 1000
@wait 1000
@delete_old_bg
@rename_bg_new_to_bg
```

現在の kaedevn（およびティラノスクリプト等）では、これを一行に凝縮しています。

```ks
// 現実（作者はこれだけで良い）
@bg classroom fade 1000
```

## 3. 作者が唯一意識すべきこと
作者が意識すべきなのは、`bg_new` という名前ではなく、**「背景が切り替わっている最中に、次のコマンドを実行しても良いか？」** というテンポの問題だけです。

通常、`@bg` にフェード時間を指定すると、演出が終わるまでスクリプトの進行が一時停止するため、作者は何も心配せずに次のセリフを書くことができます。

## まとめ
- **`bg_new` はエンジンの「私物」** です。
- **作者は `@bg` という「リモコン」** を押すだけで、その裏でエンジンが `bg_new` を使って手品のように背景を入れ替えてくれます。

ですので、作者向けのマニュアルやリファレンスには `bg_new` の記述は不要（むしろ混乱させるので載せない方が良い）です。
