# ヘッドレス・プレビュー実現の難易度と運用デザイン分析

## 1. 技術的難易度の自己評価：【低〜中】
本機能の実現には、ゼロからの開発は不要であり、既存の技術資産の「転用」で対応可能です。

- **Playwrightの活用**: すでに導入済みの Playwright を使い、CLI から `page.screenshot()` を呼び出すラッパーを作成するだけです（工数：1〜2日）。
- **Web技術の恩恵**: ネイティブエンジンと違い、OS 固有のビデオドライバやヘッドレス描画の複雑な設定に悩まされることがありません。

## 2. 「使い勝手」の再定義：AIへの丸投げ
人間がこの機能を使うと「面倒で難しい」と感じますが、AIエージェント（Claude Code等）が使うことを前提とすれば、ユーザー体験（UX）は劇的に向上します。

### 理想的なワークフロー（人間視点）
1. 人間が自然言語で「雰囲気の調整」を指示。
2. AIがコード修正、キャプチャ、画像確認のループを自律実行。
3. 人間は「完成した結果」と「その根拠となる画像」を確認するだけ。

## 3. 実装上の工夫（難しさを解消するポイント）

### A. ヘッドレス実行の高速化
- 重いフルブラウザではなく、最小構成の Chromium をヘッドレスモードで起動。
- アセットのロード完了を待機する `waitForSelector` 等のロジックを共通化し、AIが「真っ白な画面」を撮るミスを防ぐ。

### B. シンプルなCLIインターフェース
AIが迷わないよう、引数を極限まで絞り込んだコマンドを提供します。
`npm run ai:visual-check --project=<id> --page=<num>`

## 4. 結論：難易度よりも「設計思想」の転換
この機能の難しさは技術よりも、**「人間用のGUIエディタ」だけでなく「AI用のCLIエディタ」を同時にメンテナンスしていくという覚悟**にあります。
