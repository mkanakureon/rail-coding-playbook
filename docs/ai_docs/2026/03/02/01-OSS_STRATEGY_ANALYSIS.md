# ノベルゲームエンジンのOSS化戦略分析：エディタ・読者機能の非公開化に伴う影響

## 背景
kaedevnノベルゲームエンジンにおいて、「エンジン（実行コア）はOSS化するが、エディタ（制作ツール）および読者・プラットフォーム機能は非公開とする」という構成をとった場合の影響について分析しました。

## 1. ユーザーへの影響：高い参入障壁
GUIエディタが非公開である場合、開発フローは「VS Code等でのスクリプト記述 ＋ CLIプレビュー」という、プログラミングに近いスタイルに限定されます。
- **クリエイターの脱落**: 非エンジニアのクリエイター（絵作りや演出を重視する層）にとって、座標指定やタイミング調整をすべて数値入力で行うのは極めて困難です。
- **開発効率の低下**: 直感的な配置ができないため、試行錯誤のコストが増大します。

## 2. OSSコミュニティの性質：エンジニア特化
この構成下でのOSS貢献者は、以下の層に限定される可能性が高くなります。
- **移植・最適化志向**: Nintendo Switch等の特定プラットフォームへの移植や、描画パフォーマンスの限界に挑戦したいコアなエンジニア。
- **開発体験の自作**: 既存のエディタがないからこそ、独自のVS Code拡張機能や自動化ツールを自作して公開する層。

## 3. ビジネス戦略としての「SaaSモデル」
一方で、この構成は商業プラットフォームとしては成立しやすいモデルでもあります。
- **戦略**: 「エンジンは自由（無料）だが、生産性を買うなら公式エディタ（有料）を使え」という、UnityやGitHubに近いフリーミアムモデル。
- **優位性**: UI/UXのノウハウを独占し、プラットフォーム内での独自のユーザー体験を維持しつつ、エンジンの透明性を確保して開発者の信頼を得られます。

## 4. 解決策：OSS側で提供すべき「最低限の武器」
エディタなしでもエンジンの利用を促すには、以下のOSS側でのサポートが不可欠です。
1. **高機能なCLIとHot Reload**: 保存と同時に即座に反映されるプレビュー環境。
2. **VS Code 拡張機能 (LSP)**: 補完、エラーチェック、画像プレビューなど。
3. **インゲームデバッガ**: 実行中にマウスでキャラを動かし、その座標をクリップボードにコピーできる等の簡易ツール。

## 5. リスク：プラットフォーム・ベンダーロックイン
読者機能が非公開である場合、「制作したゲームをどこで公開するか」の出口を独占することになります。
- 開発者が「自分のサイトで自由に公開したい」と願う場合、この構成は大きな制約となり、他の自由度の高いエンジン（Ren'Pyやティラノ等）への流出を招くリスクがあります。

## 結論
この戦略をとる場合、kaedevnは「汎用ゲームエンジン」というよりも、**「特定の高品質なプラットフォーム専用のSDK」**としての立ち位置が強くなります。成功の鍵は、**「GUIエディタはないが、VS Codeによる開発体験が最高である」**という、プロ志向・エンジニア志向のクリエイター市場をいかに掴めるかにかかっています。
