# AI時代の「職人技」再構成：検収能力と暗黙知の外部化戦略

## 1. 分析の総括と評価
`docs/01_in_specs/0303/` の分析にある通り、AIによって「コードを書くコスト」が激減した世界では、価値の重心は以下の3点に移動します。
1. **失敗モードの地図（暗黙知）**: 過去に踏んだ罠やドメイン固有の制約を知っていること。
2. **検収能力（収束させる力）**: 出てきた成果物が正しいかを「構造的に」判定できること。
3. **無事これ名馬（運用の継続性）**: 高速な変更サイクルの中で、システムを壊さずに回し続けること。

現在のkaedevnは「作る（Orchestration）」は極めて高いレベルにありますが、「壊さない（Inspection）」の自動化が追いついていないことが、昨日の「5層パイプラインの断裂」などの問題に繋がっています。

## 2. リポジトリへの適用案

### ① 「失敗モードの地図」の言語化（AIへの知識伝承）
AIが「やってはいけないこと」を事前に知るためのドキュメントを整備します。
- **具体策**: `docs/02_references/FAILURE_MODES.md` を作成。
- **内容**: 
  - PixiJSのテクスチャ管理でのGC詰まりの経験。
  - レイヤー順序（bg_new）の失敗例。
  - 5層パイプラインで漏れやすい箇所のリスト。
- **効果**: AIが修正を行う前にこのファイルを読み込むことで、過去と同じ間違いを繰り返さない「シニアの判断」をエージェントに持たせます。

### ② 「検収側」の筋トレ：自動整合性チェックの導入
人間が1層ずつコードを読んで検収するのではなく、パイプラインの断裂を自動検知する仕組みを作ります。
- **具体策**: `packages/core/src/types/Op.ts` を「真実のソース」とし、コンパイラやインタープリタにその命令が実装されているかを静的解析（またはテスト）でチェックする。
- **効果**: 「検収」という作業を、目視から「テストが通るかどうか」の判断にシフトさせます。

### ③ ゴールデンデータ（正解セット）の整備
AIが「何が正しい状態か」を比較参照できるデータセットを構築します。
- **具体策**: 各演出コマンド（@bg, @ch等）に対する「標準的な実行結果画像」を `tests/golden/` に蓄積。
- **効果**: 演出機能の追加・修正時に、Playwrightを用いた自動比較により、AIが自律的に「合格」を判定できるようにします。

## 3. 実装の優先順位
1. **短期**: `FAILURE_MODES.md` の作成（暗黙知の外部化）。
2. **中期**: 5層パイプラインの自動整合性バリデータの作成。
3. **長期**: ビジュアル・リグレッションテストのCI組み込み。

## 結論
「実装量ではなく評価量で育てろ」という分析の提言に従い、kaedevnの開発リソースを **「機能を作るコード」から「機能を評価する構造」** へと大胆にシフトさせます。これにより、人間がコードを1行も書かなくても、1000コミット、10000コミットと「無事に」走り続けられる、真にAIネイティブなエンジンへと進化させます。
