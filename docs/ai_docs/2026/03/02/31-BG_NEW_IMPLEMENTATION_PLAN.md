# 背景遷移システム（bg_new）完全実装計画書

## 1. 現状の課題と「不完全」な点
現在の `bg_new` 実装には、視覚的な整合性を損なう致命的な問題が1点、および堅牢性の不足が数点あります。

- **レイヤー配置の誤り（最優先）**: `WebOpHandler.ts` において、`bg_new` が背景層（backgroundLayer）ではなくキャラクター層（characterLayer）に配置されている。これにより、背景フェード中にキャラクターが背景に隠れてしまう。
- **遷移の中断管理**: 背景遷移のアニメーション中に別の `@bg` コマンドが実行された際、古い `bg_new` が正しく破棄されず、スプライトが蓄積されるリスクがある。
- **初期状態の隠蔽**: `show()` メソッドで追加した瞬間に画面に現れてしまい、フェード開始前の「一瞬のチラつき」が発生する可能性がある。

## 2. bg と bg_new の連携ロジック（修正後）

背景切り替え時の「主役（bg）」と「次鋒（bg_new）」の理想的な連携フローを定義します。

1. **生成**: `bg_new` を **`backgroundLayer` の最前面**（`bg` の上）に作成する。
2. **隠蔽**: 生成直後の `bg_new.alpha` を即座に `0` に設定する（表示させない）。
3. **演出**: 
   - `bg` (旧) を 1.0 → 0.0 へフェードアウト。
   - `bg_new` (新) を 0.0 → 1.0 へフェードイン。
4. **昇格（アトミックな交代）**:
   - `bg` を削除（destroy）。
   - 内部管理マップ (`this.sprites`) から `"bg_new"` のエントリーを消去。
   - 同じスプライトを改めて `"bg"` というキーで登録し直す。

## 3. 実装ステップ

### Step 1: レイヤー判定ロジックの修正 (`WebOpHandler.ts`)
`show()` メソッド内の `isBackground` 判定を拡張し、`bg_new` も背景として扱うようにします。

```typescript
// 修正箇所
const isBackground = id === "bg" || id === "bg_new";
```

### Step 2: bgSet メソッドの堅牢化
遷移開始前に、もし前回の遷移が残っていた場合（`bg_new` が存在する場合）のクリーンアップを追加します。

```typescript
async bgSet(id: string, fadeMs?: number, effect?: string) {
  // 1. もし前回の bg_new が残っていたら強制削除してバグを防ぐ
  this.removeSprite("bg_new");
  
  const oldSprite = this.sprites.get("bg");
  // ... (以下、 Step 2 の連携ロジックに従い実装)
}
```

### Step 3: 背景クリア命令 (`bgClear`) との整合性
背景を消す命令が走った際、もし遷移中であっても `bg` と `bg_new` の両方を安全に消去するようにします。

## 4. 5層パイプラインの確認
`bg_new` は内部的な仕組みであるため、基本的には **Web (Renderer) 層** のみの修正で完結します。

- **Core (Op)**: `BG_SET` 命令に変更なし。
- **Compiler**: 変更なし。
- **Interpreter**: 変更なし。
- **Web (Renderer)**: `WebOpHandler.ts` のロジック修正（本計画のメイン）。
- **Editor**: 変更なし。

## 5. テスト・検証項目
- [ ] **レイヤー順序テスト**: 背景フェード中に、キャラクターが常に背景の前面に表示されているか。
- [ ] **連続切り替えテスト**: 短い間隔で `@bg` を連打しても、メモリリークや画面の乱れが発生しないか。
- [ ] **透明度テスト**: フェード開始の瞬間に、新背景が一瞬だけ 100% で表示されるチラつきがないか。

## 結論
この計画に基づき `WebOpHandler.ts` を修正することで、エンジンの背景演出はプロ仕様のクオリティへと引き上げられます。`bg_new` を「正しく背景層に置く」という単純な修正が、視覚的な没入感を守るための決定打となります。
