# マルチプラットフォームにおける共通ランタイム戦略：WasmとC++/Rustの使い分け

## 1. 核心：同じロジックをどう共有するか？
現在の kaedevn は、Web版（TypeScript）と Switch版（C++）で、スクリプトを実行する「VM（仮想マシン）」のロジックを別々に実装する必要があります。
**Wasm化の狙いは、この「VMの計算ロジック」を1つのソースコードに統合することです。**

## 2. 言語選定の回答

### Q. 全プラットフォーム C++ で実装できるか？
**A. はい、可能です。そして現在の構成（Switch版があること）を考えると、C++ が最も現実的です。**
- **Web版**: C++ で書いた VM を **Emscripten** というツールで `.wasm` にコンパイルします。ブラウザはこれを読み込み、JavaScript から高速に呼び出します。
- **Switch版**: 同じ C++ ソースコードを、任天堂の標準 SDK でコンパイルしてネイティブ動作させます。

### Q. Rust で実装したら Switch は無理ではないか？
**A. 不可能ではありませんが、難易度が高いです。**
- Rust は LLVM をベースとしているため、Switch の CPU (AArch64) 向けにクロスコンパイルすること自体は技術的に可能です。
- しかし、任天堂の公式 SDK は C++ ベースであるため、Rust から SDK の API を叩くための「繋ぎのコード（Binding）」を大量に書く必要があります。
- **結論**: プロジェクトの安定性を重視するなら **C++**、メモリ安全性を追求するなら **Rust** という選択になりますが、現状は C++ が推奨されます。

---

## 3. 「共通ランタイム」のビルドフロー図

どの言語（C++ or Rust）を選んだとしても、以下の流れで「1つのソースから2つの出口」を作ります。

```text
[ 共通VMソース (C++ or Rust) ]
      │
      ├─(A) Web用ビルド (Emscripten / wasm-pack)
      │     └─ [ .wasm ファイル ] ──→ ブラウザで実行 (高速・安全)
      │
      └─(B) Switch用ビルド (Official SDK / LLVM)
            └─ [ ネイティブバイナリ ] ──→ Switch実機で実行 (最高速)
```

## 4. この構成にする具体的メリット

1. **バグの再現性が 100% 一致する**:
   「Web版では動くのに、Switch版では変な挙動をする」という、言語の違いによるバグが原理的に発生しなくなります。
2. **計算負荷の高い処理の高速化**:
   現在の TypeScript 実装の VM よりも、C++/Wasm 実装の方が数倍〜数十倍高速です。複雑なバトルロジックや、大量の演出命令を同時に処理してもカクつきません。
3. **OSS利用者のメリット**:
   「エンジンの心臓部」が C++/Rust のような低レイヤ言語で隔離されているため、Webの知識しかないユーザーが誤ってコアを壊すリスクが減り、逆に高度なエンジニアが最適化に参加しやすくなります。

## 結論
Switch対応を大前提とする kaedevn においては、**「C++ で VM コアを実装し、Web 版には Wasm として提供する」** のが最も賢い選択です。これにより、開発リソースを分散させることなく、両プラットフォームで最高品質のエンジンを維持できます。
