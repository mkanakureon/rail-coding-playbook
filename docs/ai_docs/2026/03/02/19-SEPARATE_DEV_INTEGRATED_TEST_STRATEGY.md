# kaedevn 開発・運用指針：分離開発と統合テスト（Cloud 最終防衛線）戦略

## 1. 開発の原則：リポジトリごとの役割分離
開発スピードよりも「セキュリティ」と「コードの汎用性」を優先し、物理的に分離された2つのリポジトリで開発を遂行します。

### A. kaedevn-oss（公開リポジトリ）: 機能実装の場
- **役割**: ノベルゲームエンジンおよびエディタの「コア機能」の実装。
- **品質管理**: パッケージ単体テスト、Web版エディタのローカル動作確認。
- **目標**: 特定のプラットフォームに依存しない、世界一使いやすい汎用エンジンの構築。

### B. kaedevn-cloud（非公開リポジトリ）: 統合とビジネスの場
- **役割**: OSSコアをインポートし、クラウド保存・認証・商用ビルド等の「製品機能」を実装。
- **品質管理**: **【プロジェクト全体の総合テスト】**の実施場所。
- **目標**: プロの制作に耐えうる、安全で高機能な商用プラットフォームの維持。

## 2. 総合テストの戦略的意義
「総合テストは kaedevn-cloud で行う」ことを徹底します。
- **実環境での検証**: 実際のDBやクラウドストレージ、認証基盤と連携したエンドツーエンド（E2E）テスト。
- **商用機能の保護**: OSS側の更新が、非公開のSwitchビルドロジック等に影響を与えていないかを厳格にチェック。
- **AIによるデバッグ**: 総合テストで不具合が発生した際、AIエージェント（Claude Code等）に両リポジトリを横断解析させ、迅速に修正（パッチ）を適用。

## 3. ワークフローの標準化
1. **コア開発**: `kaedevn-oss` で新機能を実装・テストし、リリース（GitHub/npm等）。
2. **統合**: `kaedevn-cloud` で最新のOSS版をインポート。
3. **製品化**: `kaedevn-cloud` 側で追加の商用機能を実装。
4. **検証**: `kaedevn-cloud` 上で全ての総合テストを実行。パスした場合のみ本番環境へデプロイ。

## 結論
「面倒でも実装は分ける」という規律が、ソースコードの流出を物理的に防ぎ、かつ各コンポーネントの独立性を高めます。そして「総合テストは Cloud で行う」という最終防衛線を張ることで、OSSの勢いと商用製品の信頼性を高次元で両立させます。
