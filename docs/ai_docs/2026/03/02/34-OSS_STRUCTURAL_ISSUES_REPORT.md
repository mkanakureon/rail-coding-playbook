# OSS公開に向けた構造的課題と改善ロードマップ報告書

## 1. 概要
本リポジトリをOSSとして公開し、外部コントリビューターを受け入れるにあたり、現在のモノレポ構造における「拡張の難しさ」「依存の密結合」「デバッグの不透明性」を調査・特定しました。

## 2. 構造的な主要課題

### A. 5層パイプラインの垂直断絶（拡張性の欠如）
新しい演出コマンドを追加する際、CompilerからRendererまで5つのパッケージ（層）を全て手動で更新する必要があります。
- **問題点**: 1箇所でも漏れるとサイレントに失敗するため、新規参加者が「どこを直せば機能が動くのか」を理解するのが極めて困難です。
- **改善策**: コマンド定義を `packages/core` に集約し、各層がそこから情報を自動取得する「レジストリ・ドリブン」な設計への移行。

### B. バックエンド（apps/hono）のクラウド密結合
現在のHono APIは、Azure Blob StorageやPostgreSQL（Prisma）に強く依存しています。
- **問題点**: 外部の開発者が「自分のPCでとりあえず動かす」ために、Azureの契約や複雑なDBセットアップを強いることになり、貢献のハードルを上げています。
- **改善策**: ストレージとDBの抽象化（アダプターパターン）。ローカル実行時は「単なるフォルダ」と「SQLite」で動くようにする。

### C. スクリプト生成ロジックの重複（保守性の低下）
エディタのUIブロックをスクリプト文字列に変換するロジックが、エディタ側とバックエンド側の計4箇所にコピー＆ペーストに近い形で存在しています。
- **問題点**: 修正のたびに4箇所を同期させる必要があり、必ずと言っていいほど修正漏れ（不整合）が発生します。
- **改善策**: `packages/script-builder` を新設し、変換ロジックを一元化する。

### D. サイレント失敗を許容するコンパイラ設計
`KNOWN_COMMANDS` のホワイトリスト方式により、未知のコマンドが「エラーなしで無視される」設計になっています。
- **問題点**: エンジニア以外のクリエイターがタイポ（打ち間違い）をした際に、原因が分からず途方に暮れることになります。
- **改善策**: 未知のコマンドに対する「警告/エラー表示」の義務化。

## 3. OSSとしての「使いにくさ」のポイント

| 項目 | 状況 | OSS化への影響 |
| :--- | :--- | :--- |
| **環境構築** | 多機能な反面、複数のサービス起動が必要 | `npm install` 後の成功体験までの時間が長い |
| **型定義の共有** | `packages/core` にあるが、循環参照が起きやすい | 開発中の型エラー修正に時間がかかる |
| **テストの実行** | E2Eテストが充実しているが、実行が重い | 気軽にプルリクエストを送る心理的障壁 |

## 4. 改善ロードマップ（推奨手順）

1. **第1段階（インフラ）**: DB/ストレージのローカル・プロバイダー実装。
2. **第2段階（コア）**: コマンドレジストリの導入によるパイプラインの自動化。
3. **第3段階（リファクタ）**: スクリプト生成ロジックの共有パッケージ化。
4. **第4段階（ドキュメント）**: `GEMINI.md` / `CLAUDE.md` を活用したAI開発ガイドの完備。

## 結論
現在のリポジトリは「特定の開発者が全体を把握して動かす」には非常に強力ですが、OSSとして「見ず知らずの他人が参加する」には結合が強すぎ、エラーメッセージが不親切です。
**「疎結合化（デカップリング）」**と**「単一ソース（Single Source of Truth）」**を徹底することが、OSSとしての寿命を延ばす鍵となります。
