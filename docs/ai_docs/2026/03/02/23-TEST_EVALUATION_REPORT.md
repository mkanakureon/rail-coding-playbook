# kaedevn テスト評価報告書

## 1. 概要
リポジトリ内の全テストファイルを調査し、テストの網羅性、一貫性、および潜在的な設計上の問題を評価しました。

## 2. カテゴリ別評価

### A. E2E / 結合テスト (`tests/`, `e2e/`)
- **評価**: **極めて高い網羅性**。
- **詳細**: 新機能追加のたびに Playwright テストが追加されており、UIからエンジンまでの結合が実機に近い形で検証されています。特に `ovl-preview.spec.ts` や `battle-play.spec.ts` など、複雑なインタラクションを伴う機能のガードが堅いです。
- **課題**: テストファイル数が非常に多く、CIでの実行時間が課題になる可能性があります。また、重複するログイン処理などの共通化の余地があります。

### B. コンパイラ・インタープリタ (`packages/compiler`, `packages/interpreter`)
- **評価**: **基本的ロジックは堅牢だが、新機能の同期に遅れ**。
- **詳細**: `command-sync.test.ts` は IR への変換を厳格にチェックしており、デグレード防止に役立っています。
- **不足項目**:
  - `packages/interpreter` (KNF): `overlay`, `flash`, `ch_move`, `wait_voice_end` などの新コマンドのテストが欠落しています。
  - `packages/compiler` (KS): `ch_move`, `wait_voice_end` のトークナイズおよびパーステストが未実装です。

### C. レンダリング層 (`packages/web`)
- **評価**: **限定的なテスト**。
- **詳細**: `InventorySystem` や `FlagSystem` などの純粋なロジックはテストされていますが、`WebOpHandler` の描画ロジック（レイヤー順序やスケーリング）は主に E2E テストに依存しています。
- **改善点**: 描画結果のスクリーンショット比較（Visual Regression Testing）の導入により、`bg_new` のレイヤー間違いなどの「見た目の不具合」を早期発見できる可能性があります。

### D. バックエンド API (`apps/hono`)
- **評価**: **AIアシスト関連が非常に充実**。
- **詳細**: RAGやスクリプト生成（assist-*）のテストが手厚く、品質が維持されています。
- **不足項目**: `preview.test.ts` において、`overlay` や `screen_filter` のスクリプト生成ロジックのテストケースが不足しています。

## 3. 特定された「サイレント失敗」のリスクとテストの役割
調査の結果、以下のパターンでテストをすり抜ける不具合が発生しやすいことが判明しました。

1. **パイプラインの断裂**: エンジンには実装されているが、テストが「存在するコマンド」のみを対象としているため、「実装し忘れた層」があることに気づけない。
   - **対策**: `Op.ts` (真実のソース) から、全層の対応状況を自動チェックするメタテストの導入。
2. **アセット未選択のガード漏れ**: `undefined` がスクリプトに混入する問題。
   - **対策**: `useEditorStore` および `preview.ts` の単体テストに、プロパティが欠落したブロックの境界値テストを追加。

## 4. 総評
kaedevn のテストスイートは、個人開発規模としては非常に高い水準にあります。特に E2E テストによる品質担保は強力です。今後の課題は、**「エンジンの全機能をスクリプト層（KS/KNF）へ漏れなく露出させること」**を検証する仕組み作りです。
