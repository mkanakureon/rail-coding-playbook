# OSS同期におけるコミットログ管理戦略：透明性と機密保護の両立

## 1. 概要
統合モノレポ（非公開）からOSSリポジトリ（公開）へコードをエクスポートする際、開発の経緯を示す「コミットログ」をどのように扱うべきかを分析しました。

## 2. コミットログ同期の3つのアプローチ

### A. Squash（履歴集約型）
- **内容**: 複数の修正を1つの「Sync」コミットとして統合。
- **長所**: 機密情報の漏洩リスクがゼロ。履歴がシンプル。
- **短所**: 開発の文脈が失われ、OSSとしての透明性が低い。

### B. Subtree Split（履歴分離型）
- **内容**: 特定ディレクトリ（oss-dist/）に関連するコミット履歴のみを抽出・維持。
- **長所**: 修正の理由や経緯がOSS側にも残り、外部貢献者が理解しやすい。
- **短所**: コミットメッセージ内に非公開コンポーネントへの言及（例: "Fixed for Switch build"）が含まれる可能性がある。

### C. AI-Summarized（AI要約型）
- **内容**: 複数の作業コミットをAIが解析し、意味のある「リリースノート級」のコミットメッセージに再構築。
- **長所**: 非常に読みやすく、質の高い変更履歴を維持できる。
- **短所**: 同期プロセスの自動化にAIのAPI連携が必要。

## 3. 推奨：AI-Filtered Subtree 戦略
以下のステップで、透明性とセキュリティを両立させます。
1. **履歴の抽出**: `git subtree` により、OSS部分の変更履歴を分離。
2. **AIによるログ監査**: 同期前に、AIエージェントがコミットメッセージと差分（diff）をスキャン。
   - 機密情報の混入（APIキー、内部パス、非公開機能名）を検知。
   - 必要に応じて、コミットメッセージを「OSSとして適切な表現」に自動書き換え（`git filter-repo` 等を活用）。
3. **公開リポジトリへのプッシュ**: クリーンな履歴のみを `kaedevn-studio` へ反映。

## 4. 外部からの貢献（PR）の逆同期
OSSリポジトリに届いたプルリクエスト（PR）を、統合モノレポ（非公開）へ取り込むフロー。
- **フロー**: OSSリポジトリのPRを `git remote add` 経由でローカルに取り込み、統合モノレポの `oss-dist/` へマージ。
- **メリット**: AIが統合モノレポ全体のコンテキストでPRをレビューし、非公開機能への影響を即座に確認できる。

## 結論
コミットログはOSSプロジェクトの「信頼の証」です。AIの監査能力を活用することで、**「内部での自由な試行錯誤」と「外部へのプロフェッショナルな履歴公開」**を技術的に分離・統合することが、現代的なOSS運営の最適解となります。
