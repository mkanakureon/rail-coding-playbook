# KSコンパイラおよびJSON生成ロジックの現状調査報告

## 1. 概要
KS（.ks）ファイル内の `@コマンド` を読み取り、ランタイム実行用の JSON (Op命令列) を生成するコンパイラ層（packages/compiler）の整合性と堅牢性を調査しました。

## 2. JSON生成プロセスの評価

### A. トークナイザ (lineClassifier.ts) - 【要警戒】
- **仕組み**: `KNOWN_COMMANDS` という文字列配列に名前が含まれているか否かで、行が「コマンド」か「テキスト」かを判定。
- **問題点**: 判定基準が完全に手動のホワイトリストに依存しており、リストから漏れたコマンドはエラーなく「セリフ」として JSON に書き出されます（サイレント失敗）。
- **未登録の漏れ**: `ch_move`, `wait_voice_end` 等が現在もリストに含まれておらず、これらは JSON 上でコマンドとして認識されません。

### B. パーサー (parseCommand.ts) - 【中難易度】
- **仕組み**: 大規模な `switch-case` により、コマンド名と引数を Op オブジェクトにマッピング。
- **評価**: 引数のパース（正規化）処理が各コマンドの実装内に散在しており、一貫性が低いです。
- **成功例**: `overlay` や `filter` は適切に実装されており、対応する Op が生成される状態にあります。

### C. シリアライザ (JSON書き出し) - 【健全】
- **仕組み**: 生成された Op 配列をそのままシリアライズ。
- **評価**: このレイヤーは純粋なデータ変換のみを担っており、構造的なバグは確認されませんでした。

## 3. コマンド対応状況（JSON書き出しの可否）

| コマンド | トークナイズ判定 | Op変換ロジック | JSON書き出し |
| :--- | :---: | :---: | :---: |
| `@bg`, `@ch`, `@bgm` | OK | OK | ✅ 可能 |
| `@overlay` | OK | OK | ✅ 可能 |
| `@filter`, `@shake` | OK | OK | ✅ 可能 |
| `@ch_move` | **NG** | OK (型のみ) | ❌ 失敗 (TEXT扱い) |
| `@wait_voice_end` | **NG** | **NG** | ❌ 失敗 (TEXT扱い) |

## 4. 構造的改善案
JSON 生成の確実性を高めるため、以下の修正を推奨します。

1. **レジストリからの自動判定**: `KNOWN_COMMANDS` 配列を廃止し、`parseCommand` の定義から判定用リストを自動生成する。
2. **未知のコマンドの警告**: `@` で始まる行がホワイトリストにない場合、無視するのではなく「未知のコマンド」としてコンパイルエラー（または警告）を出す。

## 結論
JSON 生成の仕組み自体は機能していますが、**「トークナイザのホワイトリスト管理」がボトルネック**となり、一部の新機能が JSON まで到達できていません。コンパイラ層の信頼性を OSS レベルに引き上げるには、この手動リストの自動化が急務です。
