# ノベルゲームにおける技術選定の再評価：Wasm不採用の技術的妥当性

## 1. 結論
ノベルゲームを主体とする kaedevn において、実行コアの WebAssembly (Wasm) 化は必須ではなく、現在の TypeScript / JavaScript ベースの開発体制を維持することが、開発効率および OSS としての普及において最適であると判断しました。

## 2. Wasm が不要である技術的根拠

### A. ボトルネックの所在
ノベルゲームのパフォーマンス上のボトルネックは「アセット（画像・音声）のロード」と「描画（GPU処理）」にあります。これらはブラウザ（V8エンジン / WebGL）によって既に高度に最適化されており、ロジック層（VM）をバイナリ化しても、ユーザー体験に与える影響は軽微です。

### B. エコシステムとメンテナンス性
- **学習コスト**: C++/Rust による開発は、Webエンジニア主体の OSS コミュニティにおいて参入障壁となります。
- **デバッグ性能**: ブラウザ標準のデバッガ（Source Map）を活用できる TypeScript の方が、開発中の問題解決が圧倒的に速くなります。

### C. 移植性の代替案
共通ロジックの共有は、Wasm 以外の方法でも実現可能です。
- **QuickJS 等の活用**: Switch 等のネイティブ環境に軽量な JS ランタイムを組み込むことで、Web 版と同じスクリプトロジックを共有できます。

## 3. 今後注力すべき「Web優位性」の方向
Wasm のような「実行速度の追求」ではなく、以下の「制作体験の向上」にリソースを集中すべきです。
- **Platform Engineering**: 録画、プレビュー、自動テスト等の「開発を楽にする基盤」の整備。
- **Web標準技術の深化**: PixiJS のポテンシャルを最大限に引き出す演出機能（bg_new, ch_new 等）の拡充。

## 結論
「ノベルゲーム = 文学 ＋ 演出」である以上、技術的な複雑さを増す Wasm 化よりも、作者が迷わず、AI が助けやすいシンプルな TypeScript アーキテクチャを磨き上げることこそが、OSS としての「成功」への近道です。
