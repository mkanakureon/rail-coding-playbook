# プロンプト再構築計画書：Gemini 2.0 Flash への最適化

## 1. 現状の課題：プロンプトの「秘伝のタレ」化
現在の `prompts.ts` および `settings/*.md` は、歴代モデルへの対応のために場当たり的な指示（否定命令や繰り返しの強調）が蓄積されています。
- **過剰な否定命令**: 「〜しないでください」という指示が多すぎて、AIの創造性を阻害している。
- **コンテキストの散乱**: 似たような世界観設定が複数の場所に重複しており、AIが優先順位を迷う原因となっている。
- **Gemini 2.0 特性の未活用**: Gemini は長いコンテキストと構造化データの理解に長けているが、現在のプロンプトは GPT-3.5/4 時代の「短く・何度も叩き込む」スタイルに留まっている。

## 2. 再構築の基本方針：AI-First & Data-Driven
- **命令の「ポジティブ化」**: 「〜しない」ではなく「〜という制約に従って A という行動をとる」という形式に統一。
- **Gemini 専用フォーマットの採用**: 
  - XMLタグや Markdown の階層構造を多用し、どこが「設定」でどこが「命令」かを視覚的に明確にする。
  - **CoT（Chain of Thought）の明示**: ステップバイステップで考えさせる指示を標準化し、論理的な整合性を高める。
- **プロジェクト設定の整理**: `settings/*.md` を AI が一気に読み込んでインデックス化しやすい「スキーマライクな記述」へリファクタリング。

## 3. 具体的なリファクタリング項目

### A. システムプロンプトの統合 (`prompts.ts`)
現在の継ぎ足しコードを整理し、以下の「3層構造」へ移行します。
1. **Persona**: 熟練の脚本家・演出家としての定義。
2. **Knowledge**: kaedevn のエンジン仕様（Op定義）の注入。
3. **Execution**: 入力データに基づいた具体的な出力フォーマット指定。

### B. 否定命令の撲滅と「ガードレール」の設定
- 例: 「メタ発言をしないで」→「常に劇中の登場人物として、その世界観の言葉のみを用いて出力せよ」
- 例: 「解説を入れないで」→「出力は純粋なスクリプトデータ（JSON/KS）のみとし、前後の挨拶や解説は一切排除せよ」

### C. Gemini 2.0 Flash 向けの特殊指示
- **コンテキスト参照の強化**: 「プロジェクト全体の `_context_initial.json` を常に最優先の事実として参照せよ」という強力なアンカーを打つ。
- **演出タグの標準化**: `[表情: ...]` などのタグ生成を正式な機能として定義し、AIに積極的に推奨する。

## 4. 実行手順（AIエージェントへの指示フロー）
1. **解析**: Claude Code に現在の `prompts.ts` を読み込ませ、「冗長な指示」と「矛盾している箇所」を抽出させる。
2. **プロトタイプ生成**: 抽出結果に基づき、Gemini 2.0 Flash 向けに最適化された新しいテンプレートを 1 ステージずつ生成。
3. **A/Bテスト**: `projects/comedy` を新旧プロンプトで生成し、出力の「ノイズの少なさ」と「描写の質」を比較。

## 5. 実行メタデータの記録 (Evaluation Support)
生成実行ごとに、後の評価や比較の参考となる設定情報を `/output` ディレクトリに JSON 形式で保存します。

- **保存ファイル**: `_generation_metadata.json`
- **記録必須項目**:
  - **Model Info**: 使用した LLM モデル名、プロバイダー（google-ai, openai 等）。
  - **Config**: 生成時のパラメータ（Temperature, Max Tokens, JsonMode 等）。
  - **Environment**: 実行時の環境変数（ASSIST_PROVIDER, ASSIST_MODEL 等）。
  - **Performance**: トークン使用量（In/Out）、総実行時間（Latency）。
  - **Context Summary**: 参照したプロジェクト設定のハッシュ値。

## 結論
プロンプトを「整理された法的文書」のような純度の高い記述にリファクタリングすることで、Gemini 2.0 Flash はその真価を発揮します。また、実行メタデータを厳密に記録することで、感覚的な「良さ」だけでなく、データに基づいたプロンプトの改善ループを回すことが可能になります。
