# AI時代のリポジトリ運用戦略：統合モノレポとOSS自動エクスポート

## 1. 課題：リポジトリ分割によるAIコンテキストの分断
Claude CodeやGemini CLI等のAIエージェントを用いて開発する場合、リポジトリを物理的に分割すると「OSSコアの修正」と「非公開プラットフォームの更新」を同時に、一貫性を持って行うことが困難になります。

## 2. 解決策：統合モノレポ（Private） ＋ OSSミラー（Public）
開発の「真実のソース（Source of Truth）」は単一の非公開モノレポに保持し、特定のディレクトリのみをOSSリポジトリへ自動同期する手法を採用します。

### A. ディレクトリ構造（kaedevn-full / 非公開）
- **oss-dist/ (公開対象)**: 
  - `apps/editor`, `apps/hono-local`, `packages/*`
  - このディレクトリは、外部公開用の `kaedevn-studio` リポジトリと自動同期されます。
- **private/ (非公開対象)**:
  - `apps/portal`, `services/builder-switch`, `internal-packages/`
  - クラウド基盤や商用ビルドロジックなど、機密性の高いコード。

### B. AI開発のワークフロー
1. **一括コンテキスト**: 開発者は `kaedevn-full` のルートでAIエージェントを起動。
2. **横断的修正**: AIはOSSエンジンの修正と、それを利用する非公開サービスの更新を一気通貫で実行。
3. **同期実行**: 開発完了後、同期スクリプト（`npm run oss-sync`等）を実行し、OSSリポジトリへ変更を反映。

## 3. 技術的メリット
- **AIのフルパワー活用**: リポジトリを跨ぐ依存関係をAIが完全に理解した状態で、バグのない修正案を生成可能。
- **開発体験の維持**: 従来のモノレポの利便性（型チェックの一貫性、一括テスト）を保ったまま、OSS戦略を遂行。
- **セキュリティの担保**: 誤って機密コードを公開するリスクを、同期スクリプトと `.gitignore` の二重管理で防止。

## 4. 同期スクリプトの役割
- `git filter-repo` や `git subtree` を活用し、`oss-dist/` ディレクトリのコミット履歴を保持したまま、外部公開リポジトリへプッシュ。
- `OSS_SECRET_KEY` 等の機密情報が誤って混入していないかをスキャンする工程を含める。

## 結論
「同じ場所で開発できない」というAI時代のボトルネックを解消するためには、**「物理的なリポジトリ分割」ではなく「論理的な公開範囲の管理」**が正解です。これにより、開発者はAIを相棒に、爆速でOSSエンジンと商用プラットフォームを同時に進化させることができます。
