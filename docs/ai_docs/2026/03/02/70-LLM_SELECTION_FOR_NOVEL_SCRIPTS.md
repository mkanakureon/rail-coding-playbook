# ノベルゲーム・スクリプト生成におけるLLM選定分析：GPT-4 vs Gemini

## 1. 概要
`projects/fantasy` の多層的なスクリプト構造に基づき、自動執筆および演出補完において GPT-4 と Gemini 1.5 Pro のどちらが最適であるかを評価しました。

## 2. 比較評価

### A. 長期的な整合性（コンテキスト保持能力）
- **判定**: **Gemini 1.5 Pro の圧倒的勝利**
- **理由**: ノベルゲームは Chapter 1 から Chapter 5 以降まで、数万行に及ぶ伏線や口調の維持が必要です。GPT-4 は記憶容量の制限から、中盤以降で設定矛盾を起こしやすいですが、Gemini はプロジェクト全ファイルを一度に読み込めるため、一貫性を保った執筆が可能です。

### B. 独自スクリプト（KS形式）の構文維持
- **判定**: **Gemini の勝利**
- **理由**: 構造解析に強い Gemini は、`@l` や `@p` などの記号配置をデータ構造として正確に把握し、長大な出力でもフォーマットを崩しません。

### C. 文章の表現力・情緒
- **判定**: **GPT-4 の僅差勝利**
- **理由**: 短いシーンにおける感情描写や、比喩表現の豊かさは依然として GPT-4 が優れている場合があります。

## 3. kaedevn における推奨構成

### メインエンジン：Gemini 1.5 Pro (via Google Cloud)
- **用途**: 実際の `.ks` ファイルの生成、全章を通した整合性チェック、エンジンの仕様書を読み込んだ上での正確な演出コマンド付与。
- **メリット**: リポジトリ全体のコンテキストを渡せるため、「シニアな演出家」として機能します。

### 補助エンジン：GPT-4 (via Azure OpenAI)
- **用途**: 各シーンの核心となる「泣けるセリフ」や、独創的なプロットアイデアのブレインストーミング。

## 4. 今後の展望
`apps/hono/src/routes/assist.ts` 等の AI アシスト機能において、リクエストに応じて適切なモデルを切り替える（Orchestration）機能を強化します。特に長編の「続きを書く」機能には、Google Cloud 経由の Gemini 連携が不可欠となります。

## 結論
長編ファンタジー等の複雑な物語を破綻なく完結させるには、**「記憶力の Gemini」**を主軸に据えるのが、技術的にも運用的にも最適解です。
