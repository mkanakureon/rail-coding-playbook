# Session Stats Analysis Report

> **作成日**: 2026-03-15
> **担当**: Gemini CLI

本ドキュメントでは、Gemini CLI の対話セッションで出力される統計情報（Session Stats）の読み方と、 `/stats session` コマンドについての解説をまとめる。

## `/stats session` コマンドとは

Gemini CLI の対話モード（ターミナル）で `/stats session` を入力すると、現在のセッション開始から現在までの利用統計が表示される。

このコマンドは、長時間のセッションにおける **API コストの把握**、**ツール使用の効率性**、および **レートリミット（API 制限）に近づいていないかの確認** に非常に役立つ。

出力される主な項目とその意味は以下の通り：

- **Interaction Summary**:
  - `Tool Calls`: エージェントが実行したツールの総数と成功/失敗の割合。
  - `Success Rate`: ツールの成功率。これが低い場合はプロンプトの改善が必要。
  - `User Agreement`: ツール実行時のユーザー確認（許可）の割合。
  - `Code Changes`: ファイル変更による追加行数と削除行数（Diff）。
- **Performance**:
  - `Wall Time`: セッション開始からの経過時間（実時間）。
  - `Agent Active`: エージェントが思考・実行していた時間の合計。
  - `API Time` / `Tool Time`: ネットワーク通信（API）と、ローカルツール実行（スクリプト実行等）の時間の割合。
- **Usage**:
  - 各モデル（`gemini-2.5-flash-lite`, `gemini-3.1-pro-preview` 等）ごとのリクエスト回数（Reqs）。
  - `Input Tokens` (送信トークン数) と `Output Tokens` (生成トークン数)。
  - `Cache Reads`: コンテキストキャッシングによって再利用されたトークン数。
- **Savings Highlight**:
  - キャッシュ機能によってどれだけのインプットトークンが節約されたか（コスト削減率）。

---

## 2026-03-15 のセッション統計分析

ある長時間の開発セッション（約3時間13分）における統計情報の分析結果を以下に記録する。

### 統計データ概要
- **Wall Time**: 3h 13m 10s
- **Reqs (合計リクエスト数)**: 325回
- **Input Tokens (実質送信)**: 約473万 トークン
- **Cache Reads (キャッシュ利用)**: 約3578万 トークン
- **Savings Highlight**: 88.3% (35,786,910 トークン) がキャッシュから供給された。

### 分析とレートリミット（API 制限）への影響

#### 1. キャッシュの絶大な効果 (88.3%の節約)
最も注目すべき点は、**約3578万トークン（全体の88.3%）がキャッシュから読み込まれた**ことである。
Gemini 3 シリーズ（Pro/Flash）はコンテキストキャッシング（Context Caching）に強力に対応しており、長文のコンテキストを一度送信した後は、それを再利用する。
もしキャッシュが効いていなかった場合、毎回数百万トークンをフルで API に送信することになり、無料・有料枠を問わず「TPM（1分あたりのトークン数制限）」や「TPD（1日あたりのトークン数制限）」に即座に抵触していた可能性が高い。キャッシュにより API の消費枠が劇的に節約されている。

#### 2. 時間当たりのリクエスト数 (RPM)
約193分（3時間13分）で 325回 のリクエストを行っており、平均すると **約 1.6 リクエスト/分** である。
Gemini API の無料枠でも通常 15 RPM (Requests Per Minute) 程度は許容されているため、時間的な分散が非常に上手く効いており、リクエスト頻度の制限（RPMエラー）に引っかかるリスクは極めて低く抑えられている。

#### 3. 今後の注意点（TPD: 1日のトークン上限）
リクエスト数（RPM）や 1分あたりのトークン（TPM）はクリアしているが、**1日あたりの総トークン制限（TPD: Tokens Per Day）** には注意が必要である。
今回消費した実質のインプットトークン（キャッシュ読み込み分を除く）は **約 470万 トークン** である。もし無料枠の API キーを利用している場合、プランや時期によって 1日の上限が数百万〜数千万トークンに設定されていることがある。
現状エラーで停止していないということは、まだ 1日の上限（TPD）には達していない（または有料の Pay-as-you-go プランに設定されている）ことを意味する。

### 結論
長時間のセッションにも関わらず、ツール（`read_file` や `grep_search` 等）で必要な部分だけを読み取ったり、キャッシュが上手く機能しているおかげで、非常に高効率で「エコな」対話が実現できている。現在のペースであれば、レートリミットを過度に気にする必要はなく、快適に開発を進めることができる。
