# 戦略的提言：Web & Native 開発体験の完全統一（Zero-Gap への転換）

## 1. はじめに
本レポートは、kaedevn モノレポにおける Web (PixiJS) と Native (SDL2/Switch) のアーキテクチャを「今この瞬間」に抜本的に統一し、将来の技術的負債を未然に防ぐための戦略的アクションプランを定義したものである。

現在の「Web で先行し、後から Native に移植する」というモデルを、**「Web と Native が同じ心臓（ロジック）を共有し、描画だけを差し替える」**モデルへと進化させる。

---

## 2. 今着手すべき 4 つの「静かなる革命」

### ① スクリプト実行モデルの完全一本化 (Op IR 統一)
*   **現状:** Web はコンパイル済み JSON (Op IR) を実行し、Native は C++ でテキストを逐次解析している。
*   **課題:** 新機能追加時に 2 箇所のパーサー修正が必要であり、挙動の不一致（バグ）の温床となっている。
*   **解決:** Native 側の Parser を廃止し、**Web と共通の Op JSON を読み込んで実行するだけの「C++ 版 OpRunner」**へ移行する。
*   **効果:** 新コマンド追加時の Native 側作業が「switch 文への 1 行追加」のみになり、開発効率が倍増する。

### ② Native エンジンの「ノンブロッキング・ステートマシン化」
*   **現状:** `waitForClick()` 等がスレッドを占有し、アニメーション（Ticker）を停止させている。
*   **課題:** この構造では Live2D の呼吸やパーティクル等の「待ち時間中の演出」が Native 側で実装不可能になる。
*   **解決:** `main.cpp` のループをシングルループ（16ms 徹底）にし、ゲーム状態を `GameScreenState` という型で管理するステートマシンへ移行する。
*   **効果:** Native 側でも Web と同等の「滑らかに動き続ける演出」が解禁される。

### ③ 状態管理と描画の完全分離 (MV(R)P パターンの適用)
*   **現状:** `WebOpHandler`（2,100行）に、演出ロジック（計算）と描画（PixiJS 操作）が混在している。
*   **課題:** Switch 版を作る際に、2,100 行の中から「描画だけ」を抽出するのが極めて困難になっている。
*   **解決:** アニメーションの計算（位置、透明度等の補間）を `packages/core` の共通ロジックに移し、Handler を**「状態を読んで描画するだけ」**の薄いクラスにする。
*   **効果:** Switch 移植が「共通の状態を C++ の描画関数に渡すだけ」の単純作業になる。

### ④ シェーダー資産の「物理的共有」の完遂
*   **現状:** 一部の GLSL コードがまだ TypeScript 内にインラインで記述されている。
*   **課題:** 53 種類のフィルターを Native に移植する際、コピペによる同期ズレが確実に発生する。
*   **解決:** Web 側のコードから GLSL を完全に排除し、`packages/shaders` の外部ファイルを直接参照するパイプラインを完成させる。
*   **効果:** **「1 箇所のシェーダー修正で、全プラットフォームが同時に美しくなる」**理想的なアセット共有を実現する。

## 3. なぜ「今」なのか（判断の根拠）

1.  **Live2D Native 移植の前夜であること:**
    ノンブロッキング化をせずに Live2D を導入すると、呼吸アニメーションの実装で設計上の行き詰まりが発生する。
2.  **負債が「制御可能な規模」であること:**
    Native エンジンが 800 行の今なら 2 日で外科手術が可能だが、3,000 行を超えた後では 2 週間を要する。
3.  **Switch 移植の「負の写経」を防ぐため:**
    今の WebOpHandler を参考に Switch 版を作ると、設計上の欠陥まで Switch にコピーされてしまう。

## 4. 結論：Zero-Gap 開発への到達
この基盤統一が完了した瞬間、オーナー（ユーザー）は以下の価値を手にする。

> **「Web エディタ上で AI と共にリッチな演出を作り込むだけで、それが自動的に（100% の再現性で）Switch 版のバイナリにも反映される」**

これは単なるリファクタリングではなく、kaedevn を「真のクロスプラットフォーム・エンジン」へと昇華させるための、最も重要な戦略的投資である。
