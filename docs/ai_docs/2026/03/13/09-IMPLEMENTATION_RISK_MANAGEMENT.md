# 実装戦略およびリスク管理計画書

**作成日**: 2026-03-13
**対象**: ツクール系サポート（RPG/マップ/アイテム）の実装フェーズ

## 1. コンフリクト（競合）リスク評価

現在の `main` ブランチ（ノベル機能の改善）と、本計画（RPG機能の追加）を並行して進めた際のリスク評価。

### 1.1 リスクが低い理由
1.  **追加型（Additive）の実装**:
    - 新機能の大部分（`MapSystem.ts`, `MapEditorPage.tsx` 等）は新規ファイルとして作成されるため、既存ファイルとの直接的な衝突は発生しにくい。
2.  **定義の拡張が中心**:
    - 既存の重要ファイル（`Op.ts`, `commandDefinitions.ts`）への変更は「リストへの追加」が主であり、Git のマージ機能で自動解消されやすい。
3.  **疎結合なアーキテクチャ**:
    - パッケージ（`core`, `web`, `interpreter`）間の境界が明確であり、影響範囲を特定しやすい。

---

## 2. 大規模差分（Large Diffs）の管理戦略

RPG 機能の追加に伴い、コードベースが数千行規模で増えることが予想されるが、以下の指針で管理する。

### 2.1 「良い大規模差分」の条件
- **既存機能の非破壊**: ノベルゲームエンジンとしての既存ロジックを書き換えるのではなく、新しい命令（Op）への対応を「付け加える」形を徹底する。
- **パッケージ境界の遵守**:
    - `packages/core`: 命令の型定義（ロジックなし）。
    - `packages/web`: PixiJS による描画実装。
    - `apps/editor`: React による UI 実装。
- **TypeScript による安全担保**: `Op` 型や `IOpHandler` インターフェースを先に拡張し、コンパイルエラーを道標として実装を進める。

---

## 3. 実装上の遵守事項

1.  **`IOpHandler` インターフェースの利用**:
    - `packages/core/src/engine/IOpHandler.ts` に新規メソッドを定義し、`WebOpHandler.ts` で実装する。これによりプラットフォーム間の整合性を保つ。
2.  **`SaveData` 構造の継承**:
    - `packages/core/src/types/SaveData.ts` を拡張し、マップIDやプレイヤー座標を保持する。既存のセーブ/ロード画面、クラウド同期機能をそのまま利用可能にする。

---

## 4. 推奨ワークフロー

1.  **ブランチ戦略**: `feature/rpg-support` 等の専用ブランチを使用。
2.  **ボトムアップ実装**:
    - ステップ1: `core` で命令（Op）を定義。
    - ステップ2: `web` で描画・移動ロジックを実装。
    - ステップ3: `editor` で編集 UI を実装。
3.  **継続的検証**: `npm run typecheck` を頻繁に実行し、既存のノベルプロジェクトへの退行（デグレード）がないか確認する。
