# RPGツクールユーザーの不満点と kaedevn による解決アプローチ

**作成日:** 2026-03-16
**対象:** 既存のRPGツクール（MV/MZ）ユーザーおよび新規ゲーム制作者

## 1. はじめに
本ドキュメントは、既存のRPGツクール（以下「本家」）ユーザーが抱える一般的な不満点（ペインポイント）を整理し、kaedevn エンジンがそれらをどのように解決、あるいは差別化しているかを分析したものである。

---

## 2. 主要な不満点と kaedevn の対応

### A. 技術的制約とパフォーマンス
- **本家の不満**: 
  - MVの32bit制限によるメモリ不足（長編でのクラッシュ）。
  - 高解像度（4K等）や大量の画像表示時の著しいラグ。
  - モバイル・Web出力が重く、最適化が困難。
- **kaedevn の対応**:
  - **最新スタック**: 64bitネイティブ環境（Node.js/Switch）および PixiJS 8（WebGL/WebGPU）を採用。
  - **動的リソース管理**: `TextureCache` による効率的なメモリ管理（gc() + destroy() 実装済み）。
  - **高解像度前提**: 1280x720 を論理解像度とし、スケーリングに強い設計。

### B. エディタの UX / 拡張性
- **本家の不満**: 
  - 消しゴムツールがない（透明タイルを選択する手間）。
  - タイルサイズが固定（または選択肢が少ない）。
  - エディタ自体の拡張ができず、メモ欄（Note Tags）による管理が煩冒。
- **kaedevn の対応**:
  - **モダンなツール群**: 消しゴム、バケツ塗り、レイヤー切替、グリッド表示、Undo/Redo を標準搭載。
  - **柔軟なデータ構造**: `GameDb` は JSON ベースで拡張が容易。メモ欄に頼らず、将来的に UI 自体のカスタマイズも可能。
  - **テストプレイの速さ**: エディタと実行エンジンが Web 技術で共通化されているため、起動が極めて高速。

### C. エコシステムと独自性（脱「ツクール臭」）
- **本家の不満**: 
  - 標準素材（RTP）の影響が強く、個性を出すために全ての素材を差し替える手間が膨大。
  - 主要プラグインのブラックボックス化（難読化・有料化）が進み、カスタマイズが困難。
- **kaedevn の対応**:
  - **ノベルエンジン由来の演出**: 立ち絵、バックログ、複雑なテキスト演出が標準機能。これにより「RPGをベースにしたノベル」という独自のジャンルを容易に構築可能。
  - **KSCスクリプト**: プラグインを介さずとも、スクリプト（KSC）とコモンイベントテンプレートにより、ロジックを透明性高く拡張可能。

### D. チーム開発とバージョン管理
- **本家の不満**: 
  - プロジェクトファイル（巨大な JSON）が Git で競合しやすく、複数人での開発が困難。
- **kaedevn の対応**:
  - **モノレポ・モジュール化**: `packages/` ごとにコードが分離されており、データ構造も正規化されているため、本家より Git との親和性が高い設計。

---

## 3. kaedevn が今後優先すべき「不満解消」タスク

1.  **アセット管理の強化**: 名前検索（実装済み）、カテゴリ分類（実装済み）、タグ付け（将来）。
2.  **UIレイアウトエディタ**: プリセット選択（10種、実装済み）+ ビジュアルエディタ（実装済み）で GUI からレイアウト編集可能。
3.  **並列イベント**: 現状は逐次実行。変数+条件分岐で擬似並列が可能。将来的に OpRunner の並列実行を検討。

## 4. 結論
kaedevn は、本家ツクールが抱える「レガシーなアーキテクチャに起因する不満」を、モダンな Web 技術とノベルエンジンの表現力を組み合わせることで解消しようとしている。特にパフォーマンス面と演出の柔軟性において、既存ユーザーへの強力なアピールポイントを持っている。
