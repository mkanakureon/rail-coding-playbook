# 上流レビューおよびプロンプト添削システム 検討書

## 1. 目的
作品設定が固まる前の「企画構想」段階での論理破綻を防止し、かつ AI への指示（インプット・プロンプト）の質を最大化することで、手戻りのない超高精度なシナリオ生成パイプラインを構築する。

## 2. レビュー対象の拡大案

### A. 構想レイヤー（最上流）
- **対象**: `rough_idea.md`, `concept_memo.txt` 等の非構造化データ。
- **添削ポイント**:
    - **独自性（Hook）**: 既存作品の焼き直しになっていないか。
    - **テーマの一貫性**: 結末がテーマを裏切っていないか。
    - **ターゲット適合性**: 想定読者層に刺さる要素が含まれているか。

### B. プロンプトレイヤー（メタ添削）
- **対象**: `prompts.ts` 内のテンプレートや、ユーザーが入力するカスタム指示。
- **添削ポイント**:
    - **具体性**: 「詳細に描写して」ではなく「温度と匂いを含めて描写して」と言い換えられているか。
    - **制約の競合**: 矛盾する命令（例：「簡潔に」と「重厚に」の両立）が含まれていないか。
    - **AI 心理学**: Chain-of-Thought（思考過程の明示）を促す構造になっているか。

---

## 3. 実装メカニズム：自動添削機（Auto-Refiner）

現在の `review-settings` を拡張し、以下の機能を統合する。

### ① インプット・プロンプト・チェッカー
ユーザーが AI に投げようとしている指示文を、まず「プロンプト・エンジニア専門 AI」が読み取り、改善案を提示する。
- **Input**: ユーザーの生プロンプト
- **Output**: 改善されたプロンプト + なぜ改善したかの理由

### ② 段階的階層レビュー
上流工程の出力を、下流工程のインプットとして渡す前に、必ず「ディレクター AI」による承認（パス/リテイク）を挟むゲートキーパー方式。

---

## 4. ワークフローイメージ

1. **[User]** ラフアイデアを入力。
2. **[AI: 分析官]** ラフの論理矛盾を指摘（上流レビュー）。
3. **[User]** アイデアを修正し、生成プロンプトを作成。
4. **[AI: プロンプト職人]** プロンプトの曖昧さを修正（プロンプト添削）。
5. **[AI: 生成]** 高品質な設定/シナリオを出力。

## 5. 公式開発手順（Standard Operating Procedure）

上流工程の品質担保のため、開発者は以下の手順を厳格に順守すること。各ステップで AI のパス（承認）が得られるまで次工程へ進んではならない。

### Step 1: 企画構想(S00)の論理チェック
`s_00_idea.md` を作成後、以下のコマンドを実行し、企画のポテンシャルと論理的欠陥を確認せよ。
```bash
npx tsx scripts/cli/ai/upstream-cli.ts review-idea --input projects/xxx/settings/s_00_idea.md
```

### Step 2: 作品設定(S01-04)の専門家レビュー
設定ファイルを各 Markdown に展開後、以下のコマンドを実行し、地政学・兵站・整合性のレビューを受けよ。
```bash
npx tsx scripts/cli/ai/upstream-cli.ts review-settings --settings projects/xxx/settings
```

### Step 3: 制作プロンプトの最終添削
特定の演出や特殊なシーン生成のため、カスタムプロンプトを自作した場合は、実行前に必ず以下のコマンドで添削を受け、最適化せよ。
```bash
npx tsx scripts/cli/ai/upstream-cli.ts review-prompt --input my_prompt_draft.txt
```

### 補足：内部整合性バリデーション
上記手順の裏側では、以下のシステムが自動的に整合性を担保している。
- **構造チェック**: `md-parser.ts` が設定ファイルの階層構造を解析。
- **スキーマ検証**: `schemas.ts` が AI 出力データの型安全性を Zod で保証。

---

## 6. 結論
企画の初期段階から `upstream-cli.ts` を用いた「構造化レビュー」を繰り返すことで、シナリオ執筆段階での手戻りを最小化できる。今後は、レビュー結果を自動的に設定ファイルへパッチ適用する「オートフィックス機能」の検討も進めるべきである。
