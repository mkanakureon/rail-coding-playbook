# システム的設定更新ワークフロー（Review-Apply Pipeline）検討書

## 1. 背景と課題
人間（AIエージェント）による手動の修正は再現性が低く、設定ファイルが「完成した物語（スクリプト）」に化けてしまうリスクがある。設定ファイルはあくまで「AIが執筆するためのデータ/要件」でなければならない。

## 2. 解決策：構造化パッチによる自動更新
AIレビューの結果を JSON 形式の「パッチ」として出力し、それを専用のプログラムが Markdown に適用する仕組みを構築する。

### ステップ 1: レビューの構造化 (review-settings --json)
AI に以下の JSON 構造で修正案を出力させる。
```json
[
  {
    "file": "/absolute/path/to/s_03_plot.md",
    "target": "第1章 出発",
    "action": "append_requirement",
    "content": "兵站のリアリティ：村人から渡される食料は5日分の干し肉とパンであること"
  }
]
```

### ステップ 2: パッチの自動適用 (apply-review)
新設する `apply-review` コマンドが、上記 JSON を元に設定ファイルを更新する。
- **更新ロジック**: Gemini API に対し「この Markdown の特定セクションに、この要件を**箇条書きのデータとして**追記せよ。ナラティブな文章は書くな」と指示する。
- **整合性の担保**: 更新後、`md-parser.ts` を通してパース可能か自動チェックを行う。

## 3. 再現性の確保
- 全てのレビューと適用ログを `reviews/YYYYMMDD_HHMMSS/` に保存。
- 設定ファイル内の記述は常に「- **要件**: 内容」の形式に統一し、AIのサボりを防止する。

## 4. 実装スケジュール
1. `prompts.ts` にパッチ生成用・パッチ適用用のプロンプトを追加。
2. `upstream-cli.ts` に `apply-review` コマンドを実装。
3. `fantasy` プロジェクトで実際にテスト。
