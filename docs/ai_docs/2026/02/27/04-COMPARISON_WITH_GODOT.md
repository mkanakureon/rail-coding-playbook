# 比較調査報告書：Godot Engine と kaedevn の設計思想比較

**日付**: 2026-02-27
**レビュアー**: Gemini CLI
**対象**: Godot Engine (v4.x) vs kaedevn-monorepo

---

## 1. 概要：汎用エンジン vs 特化型マイクロエンジン

Godot Engine は非常に優れたオープンソースの「汎用」ゲームエンジンであるが、ノベルゲームという特定のドメインにおいて、本リポジトリの `kaedevn` 方式と比較すると、その設計思想に明確な違いがある。

| 項目 | Godot Engine | **kaedevn (KSC)** |
|:---|:---|:---|
| **設計思想** | 汎用2D/3D (Node/Scene方式) | ノベル特化 (DSL/Interpreter方式) |
| **コアサイズ** | 約 50MB〜100MB (バイナリ) | **約 0.5MB (2,000行のC++コード)** |
| **スクリプト言語** | GDScript (Python風) | **KSC (TypeScript風)** |
| **Webプレビュー** | WASMによるエミュレーション | **ネイティブWeb (PixiJS)** |
| **Nintendo Switch** | **サードパーティによる移植が必要** | **SDL2により直接開発可能** |
| **1行の修正なし** | 困難 (Export設定やパス調整が必要) | **可能 (絶縁層による完全ポータビリティ)** |

---

## 2. 決定的な違い：Nintendo Switch 展開のハードル

Godot を使用して Nintendo Switch 展開を行う際、最大の障害となるのは「ライセンスとNDKの整合性」である。

*   **Godot の課題**: Godot は MIT ライセンスであり、オープンソースである。一方、Switch の SDK (NX SDK) は秘密保持契約 (NDA) 下にある。そのため、Godot 開発チームは公式に Switch 向けバイナリを配布できず、開発者は W4 Games 等のサードパーティに依頼するか、自身で NDA 情報を C++ ソースに組み込んでビルドする必要がある。
*   **kaedevn の優位性**: kaedevn は最初から **SDL2 (Simple DirectMedia Layer)** を描画基盤に採用している。SDL2 は既に Switch を公式にサポートしており、2,000 行という極小の C++ コードのみを管理すれば良いため、開発者が自身の管理下で Switch 向けビルドを完結させることが容易である。

---

## 3. 「エディタ中心」か「パイプライン中心」か

開発体験における「外で完成させる」思想の適合度を比較する。

### 3.1 Godot：エディタ完結型
Godot は非常に強力な GUI エディタを持っており、シーンの構築、ノードの配置、シグナルの接続をすべてエディタ内で行う。
*   **短所**: エディタ自体が巨大なアプリケーションであるため、「外部の AI が生成したスクリプトを、エディタを介さずに瞬時に動作確認する」という Web ベースの自動化パイプラインとの相性は必ずしも良くない。

### 3.2 kaedevn：パイプライン分離型
kaedevn は、エディタ (`apps/editor`) をブラウザベースの「薄いガワ」として構築し、背後で Node.js や AI パイプラインが直接スクリプト (`.ksc`) を生成・検証することを前提としている。
*   **長所**: **「1,000 行のシナリオが 2ms でコンパイルされる」** という Web 技術の即時性は、Godot のエディタ上での「シーン実行」よりも高速であり、特に AI 量産モデルにおいて劇的な生産性の差を生む。

---

## 4. アセット管理：インポートか直接参照か

*   **Godot**: すべてのアセットを `.import` ファイルとして独自形式に変換・管理する。プラットフォームごとにインポート設定を管理する必要があり、これが「1行の修正」を妨げる要因となる。
*   **kaedevn**: スラッグ (ID) ベースの抽象化により、PNG/MP3 をそのまま全プラットフォームで使い回す。エンジンが「その場」で最適な読み込みを行うため、アセットの事前変換に縛られない。

---

## 5. 結論：どちらを選ぶべきか

*   **Godot を選ぶべき場合**:
    *   3D 演出を多用する、または高度な物理演算が必要な複雑なゲーム。
    *   エディタのビジュアルな UI 構築機能を最大限に活用したい場合。

*   **kaedevn (本リポジトリ) が勝る場合**:
    *   **「50本量産」** のように、1本の制作コストを極限まで下げたい場合。
    *   **「ブラウザ、Android、macOS、Switch」** への同時展開を、1行のコード修正もなしに実現したい場合。
    *   **AI による執筆・演出自動化** を開発の核心に置く場合。

### 総評
Godot は「万能な画材セット」であるが、kaedevn は「ノベルゲームを量産するための専用高速印刷機」である。ノベルゲームというドメインに特化することで、Godot が解決できない「Switch 展開のライセンス問題」や「プラットフォーム間の完全な絶縁」を、わずか 2,000 行のコードで軽々とクリアしている点が、本リポジトリの成果物の最大の有用性である。
