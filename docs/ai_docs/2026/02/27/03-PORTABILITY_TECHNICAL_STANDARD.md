# 技術標準：完全ポータビリティ維持のための設計指針

**日付**: 2026-02-27
**レビュアー**: Gemini CLI
**目的**: 「1行の修正もなく全プラットフォームで動く」状態を恒久的に維持するための技術規約を定義する。

---

## 1. 思想：シングルソース・マルチランタイム

本プロジェクトの核心は、**「コンテンツ（KSC）は1つであり、プラットフォームごとに実装（Runtime）が異なるだけである」**という点にある。これを維持するため、以下の4つの同期を絶対原則とする。

---

## 2. 4つの同期原則

### 2.1 ロジックの同期 (Logic Sync)
TypeScript版（Web）とC++版（Native）のインタプリタは、同一の命令列に対し**1ピクセルの差異もなく**同じ状態遷移を行わなければならない。

*   **規約**: 新しいコマンド（例：`filter()`）を追加する場合、必ず両方の言語で実装し、同一のパラメータで同一の結果が得られることをユニットテストで保証すること。
*   **検証**: `native-engine` の `unit_tests` と、`packages/interpreter` の `vitest` で共通のテストケース（JSON形式）を読み込み、期待される `GameState` の差分が一致することを確認する。

### 2.2 データ（セーブ）の同期 (State Sync)
「ブラウザでプレイを開始し、外出先でAndroid版で続きから遊ぶ」ことを可能にする。

*   **規約**: セーブデータは共通の `SaveData` スキーマ（JSON）に従う。
*   **禁止事項**: プラットフォーム固有のオブジェクト（PointerEventやWindowハンドル等）をGameStateに含めてはならない。
*   **型定義**: `packages/core/src/types/` に定義された型のみを永続化対象とする。

### 2.3 アセットの同期 (Asset Sync)
解像度と座標系を「論理座標」で固定し、物理的な画面サイズから絶縁する。

*   **論理基準**: 1280x720 (16:9) を 1.0 とした相対座標。
*   **規約**: エンジン側は必ず `Letterbox` または `Pillarbox` を実装し、アスペクト比の異なる端末でも「ゲーム画面内の座標」を不変に保つ。
*   **実証**: `SDL_RenderSetLogicalSize` (Native) と PixiJS の `scale` (Web) で同一の倍率計算ロジックを使用すること。

### 2.4 入力の同期 (Input Sync)
入力デバイスの物理的な差異を「仮想アクション」で吸収する。

*   **抽象層**: `IInput` インターフェースを介した `Action.OK`, `Action.CANCEL`, `Action.SKIP` への変換。
*   **規約**: ゲームロジック側で `mousedown` や `keydown` を直接リッスンすることを厳禁とする。必ず抽象化されたアクションイベントを購読すること。

---

## 3. 「一行の修正もなし」を破壊するリスクとその回避策

調査により、将来的にポータビリティを損なう可能性のある「地雷」と、その回避策を特定した。

| リスク要因 | 影響 | 回避策 |
|:---|:---|:---|
| **浮動小数点の精度誤差** | 計算結果の微細なズレ | 座標計算や変数操作において、必要以上に高い精度に依存しない。重要な判定は整数で行う。 |
| **ファイルシステムの差異** | パス区切り文字 (`/` vs ``) | 必ずスラッグ（ID）を使用し、物理パスの解決は `AssetProvider` に完全に委任する。 |
| **フォントレンダリング** | テキストの溢れ、改行位置のズレ | Noto Sans JP 等のフォントを全プラットフォームで同梱し、フォントメトリクスを強制的に一致させる。 |
| **外部ライブラリの露出** | 特定環境でのみ動くコードの混入 | `IEngineAPI` を介さない外部ライブラリ（ブラウザの `window` や NDK の直接呼び出し）を `Interpreter` 層から排除する。 |

---

## 4. 評価：本リポジトリの有用性

本リポジトリの成果物は、以下の点において「完成されたポータビリティ基盤」としての価値がある。

1.  **検証済みスタック**: Web, macOS, Android で既に同一の演出が再現されており、理論ではなく「実績」に基づいている。
2.  **極小のフットプリント**: C++版エンジンが 2,000 行程度と極めて小さいため、新しいプラットフォーム（Switch, iOS等）への移植が「数日」の作業で完了する。
3.  **AIフレンドリー**: 人間がプラットフォーム間の差異を気にせず、AIに「純粋な物語と演出」だけを書かせることができる。

---

## 5. 結論

「1行の修正もなく動く」ことは、徹底した**「副作用の排除」**と**「徹底した抽象化」**の産物である。この技術標準を遵守することで、kaedevn は今後プラットフォームが増えても、制作コストを一定に保ち続けることができる。
