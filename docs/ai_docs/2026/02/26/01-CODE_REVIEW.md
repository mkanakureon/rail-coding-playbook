# kaedevn-monorepo ソースコードレビュー報告書

## 1. 概要
本プロジェクトは、Nintendo SwitchおよびWebをターゲットとしたクロスプラットフォーム・ビジュアルノベルエンジンです。独自のスクリプト言語（Kaede Script / .ksc）のインタプリタを中心に、モジュラリティの高いモノレポ構成で実装されています。

今回のレビューでは、プロジェクトの核心部である `packages/core` および `packages/interpreter` の実装を中心に分析しました。

---

## 2. 評価項目別分析

### 2.1 抽象化とポータビリティ（アーキテクチャ）
- **評価**: 非常に優れている
- **詳細**: `IInput`, `IAudio`, `IStorage` といったインターフェース層が定義されており、プラットフォーム固有の処理（WebのPixiJSやSwitchのネイティブ実装）がコアロジックに混入しないよう厳格に分離されています。
- **特筆点**: ゲームロジックを `dispatch(Action)` を通じたアクションベースの駆動に制限している点は、Switch移植時の入力遅延対策やマッピング変更に非常に強い設計です。

### 2.2 インタプリタの実装 (`Interpreter.ts`)
- **評価**: 高い完成度（柔軟性とデバッグ機能の両立）
- **詳細**: `Interpreter` クラスが `Parser`, `Evaluator`, `GameState` を協調させる伝統的なインタプリタパターンを採用しています。
- **メリット**:
    - **非同期処理の管理**: `handleDialogue` や `executeBuiltin` など、演出（ウェイトやクリック待ち）を伴う処理を適切に `async/await` で処理しており、演出の同期性が担保されています。
    - **デバッグ支援**: フェーズ7で追加された `Debugger` や `ErrorHandler` により、未定義変数への提案機能など、スクリプト制作者に親切な設計になっています。
- **懸念点**:
    - **正規表現によるパース**: `handleExpression` 内で正規表現によるマッチングを多用しています。スクリプトが複雑化した場合、パースの優先順位やエッジケース（文字列内の `=` など）の処理で限界が来る可能性があります。

### 2.3 状態管理 (`GameState.ts`)
- **評価**: 堅実なスコープ管理
- **詳細**: ローカルスコープ（関数・サブルーチン）とグローバル変数の分離が明確です。`getVar`/`setVar` で内側のスコープから再帰的に検索する実装は標準的かつ直感的です。
- **改善提案**: 現状 `Map<string, unknown>` ですが、エンジン仕様（v2.1）にある「number/boolean/string/nullのみ」という制約を型システム上、またはバリデーション層でより厳密に強制しても良いでしょう。

---

## 3. 具体的な改善提案・アドバイス

### 3.1 パーサーの進化
現在の実装は行単位の走査が中心ですが、将来的に複雑な制御構造（ループや高度な式評価）を導入する場合、現在の `Parser.ts` を抽象構文木（AST）を生成する本格的なパーサーに移行することを検討してください。

### 3.2 保存データの整合性 (`SaveData.ts`)
`SaveData` インターフェースには `save_schema_version` が含まれており、後方互換性への配慮が見られます。さらに、`interpreter` の状態（コールスタックやPC）をシリアライズする際、関数の再帰呼び出し中にセーブした場合の復元処理について、より詳細な単体テストがあると安心です。

### 3.3 型安全性の強化
`unknown` 型が各所で使用されています。`packages/core` 内にスクリプト内で扱える「リテラル型」のUnion（`string | number | boolean | null`）を定義し、それを `GameState` や `Evaluator` で徹底して使用することで、ランタイムエラーを未然に防ぐことができます。

---

## 4. 結論
本プロジェクトのソースコードは、**「ポータビリティ（移植性）」**と**「開発体験（デバッグしやすさ）」**を両立させた、非常にプロフェッショナルな設計となっています。

特にインタプリタ層において、エラーメッセージを親切にする工夫や、プラットフォームを意識した抽象化は、中〜大規模な開発にも耐えうる堅牢な基盤と言えます。今後の実装では、パース処理の堅牢化と型制約の厳密化を進めることで、さらに高い信頼性を得られるでしょう。
