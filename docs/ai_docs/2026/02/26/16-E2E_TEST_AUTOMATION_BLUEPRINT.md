# シナリオ駆動型 E2E 自動テスト戦略案

**作成日**: 2026-02-26
**対象**: AI 生成シナリオの動作保証と Playwright による自動検証

## 1. 課題
AI が生成したスクリプト（.ks）には、時としてコンパイラが通らない構文ミスや、存在しないアセットの参照が含まれる可能性があります。これらを手動でチェックするのは非効率です。

## 2. 自動検証のフロー

### 2.1 静的解析 (Linter)
生成直後に `packages/ksc-compiler` を実行し、構文エラーがないかを確認します。また、`assets.md` に存在しないアセット slug が使われていないかを静的にクロスチェックします。

### 2.2 動作検証 (headless Playwright)
生成されたストーリーごとに、以下の Playwright スクリプトを自動生成または実行します。

1.  **全編早送りテスト**: `Skip` アクションを連続実行し、エラー（画像の読み込み失敗、スクリプトの停止など）なしで最後まで到達できるかを確認。
2.  **選択肢網羅テスト**: 有向グラフとしてシナリオを走査し、全ての選択肢の分岐先が正しくロードされるかを確認。
3.  **ビジュアル回帰テスト**: 各シーンの代表的なフレームでスクリーンショットを撮り、背景とキャラクターの重なりが異常でないか（キャラが画面外にいる等）を簡易チェック。

## 3. Playwright 拡張の設計
`tests/` ディレクトリに、`.ks` ファイルを引数に取る「汎用シナリオテストランナー」を作成することを推奨します。

```typescript
// イメージ
test('AI Generated Story Verification', async ({ page }) => {
  await page.goto(`http://localhost:5175/test-runner?script=${targetScript}`);
  await page.click('[data-action="auto-play"]');
  // エラーログの監視
  // 終点フラグの確認
});
```

## 4. 結論
この戦略により、大量のストーリーを自動生成しても、品質の低い（壊れた）スクリプトがユーザーに届くのを防ぐことができます。
