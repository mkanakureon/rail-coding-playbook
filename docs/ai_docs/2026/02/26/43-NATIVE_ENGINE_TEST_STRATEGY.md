# ネイティブエンジン：テスト戦略と自動検証計画

**作成日**: 2026-02-26
**対象**: C++ 実装における品質保証
**核心**: 「画面を見なくても動作が保証されている」状態の構築

## 1. テストスタック
- **フレームワーク**: Google Test (GTest)
- **モックライブラリ**: Google Mock (GMock)
- **計測**: Valgrind / LLVM LeakSanitizer (メモリリーク検出)

## 2. テストの階層

### 2.1 ユニットテスト (Logic)
- **対象**: `GameState`, `Parser`, `VariableStore`
- **内容**: 
    - KSC スクリプトのパース結果が正しいか。
    - 変数の加算、条件分岐（if）が正しく評価されるか。
    - コールスタック（call/ret）の深さと復元。

### 2.2 モックテスト (SDL2 Side-Effects)
SDL2 の関数を直接呼ばず、インターフェース越しに呼ぶことで「副作用」を検証します。
- **検証方法**: `EXPECT_CALL(mockRenderer, drawTexture("forest", ...))` 
- **利点**: GPU やウィンドウシステムがない CI 環境でも、描画ロジックの正しさを 100% 検証可能。

### 2.3 統合テスト (Scenario)
- **対象**: `NativeEngine` 全体
- **内容**: 短いテスト用 KSC を読み込み、特定の行まで実行した後の `GameState` が期待通りであることを検証。

## 3. 重点テスト項目
- **メモリリーク**: テクスチャのロードと破棄を 10,000 回繰り返し、メモリ使用量が増加しないことを検証。
- **異常系**: 壊れた JSON、存在しないアセット slug、不正な KSC 構文に対する堅牢なエラーハンドリング。
