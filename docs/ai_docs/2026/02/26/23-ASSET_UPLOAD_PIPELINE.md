# アセット・パイプライン：アップロードから実行まで

**作成日**: 2026-02-26
**対象**: アセット管理システムの全体像とデータフロー

## 1. アセット・ライフサイクルの概要
kaedevn における画像・音声アセットは、以下の 5 ステップを経てゲーム内で使用されます。

1.  **UI (Editor)**: ユーザーがファイルをドラッグ＆ドロップ。
2.  **API (Hono)**: ファイルをストリームとして受信し、バリデーション（サイズ、形式）を実施。
3.  **Storage (Azure)**: Blob Storage へ永続化。
4.  **Database (Prisma)**: メタデータ（URL、サイズ、種類、slug）を `Asset` テーブルに記録。
5.  **Engine (Web)**: `.ks` スクリプトの slug に基づき DB から URL を取得し、PixiJS でロード。

## 2. 特殊なアップロード処理

### 2.1 FrameSet (アニメーション)
アニメーションスプライトは ZIP 形式で一括アップロードされます。
- Hono 側で解凍し、各フレームを個別の Blob として保存。
- `FrameSet` モデルと複数の `Asset` レコードを自動生成・紐付け。

### 2.2 プレビュー用キャッシュ
デプロイ環境では Azure を使用しますが、ローカル開発時は `apps/hono/public/uploads` へのファイルシステム保存も並行してサポートされており、環境変数で切り替え可能です。

## 3. スクリプトからの参照
シナリオ内では物理パスではなく、DB 上の **slug** を使用します。
```ksc
bg("forest_evening") // slug で参照
```
エンジン側で Hono API を叩き、`forest_evening` に紐付く最新の Azure Blob URL を解決します。

## 4. 今後の課題と最適化
- **画像圧縮**: アップロード時に `sharp` 等を使用して、WebP や KTX2 へ自動変換する処理の導入。
- **CDN 連携**: Azure Front Door または CDN を介した配信により、グローバルなロード時間の短縮。
- **重複検知**: ハッシュ計算により、同一ファイルの二重アップロードを防止するロジックの実装。
