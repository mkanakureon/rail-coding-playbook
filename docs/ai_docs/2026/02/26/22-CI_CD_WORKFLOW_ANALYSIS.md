# CI/CD ワークフロー分析と改善提案報告書

**作成日**: 2026-02-26
**対象**: GitHub Actions (ci.yml, check-hardcoded-urls.yml)

## 1. 現状のワークフロー分析

### 1.1 CI (ci.yml)
- **特徴**: モノレポ構造を考慮したジョブ分割がなされています。
- **Typecheck**: 依存関係（`@kaedevn/core` 等）のビルドを先行させず、いきなり全体をチェックしているため、エラーが出るリスクがあります。
- **Unit Tests**: `ubuntu-latest` 上で Docker サービス（PostgreSQL）を立ち上げ、Prisma マイグレーションまで実行する本格的な統合テスト環境が構築されています。

### 1.2 URL 管理チェック (check-hardcoded-urls.yml)
- **特徴**: ローカル開発用のポート（8080）が本番用ビルドに含まれていないかを `grep` で厳格にチェックしています。
- **意義**: Azure へのデプロイ時に「接続できない」という致命的なミスを未然に防ぐ、非常に優れたガードレールです。

## 2. 課題と懸念事項
- **E2E テストの不在**: Playwright 等を用いたブラウザ動作検証が CI に含まれていません。
- **キャッシュの最適化**: `npm ci` を全ジョブで繰り返しており、ジョブ間の成果物（`dist/`）の受け渡しが `needs` だけで解決していない可能性があります。
- **Switch 環境のシミュレーション**: Switch 移植を目指すなら、モバイル端末相当の制限下でのテスト（低スペックシミュレーション）が将来的に必要です。

## 3. 改善提案
1.  **ビルド済みパッケージの再利用**: `typecheck` や `unit-tests` で共通して使用する `core` や `interpreter` を一度だけビルドし、ジョブ間で `actions/upload-artifact` を使って共有することで時間を短縮する。
2.  **Playwright CI の追加**: `tests/` にあるテストコードを PR ごとに実行するジョブを追加。
3.  **Linter の強化**: `eslint` だけでなく、`knip` 等を導入して「未使用の依存関係やエクスポート」を自動検出する。

## 4. 結論
現在の CI 環境は、開発初期段階としては非常に堅牢（特に DB 統合テストと URL チェック）です。今後は E2E テストの統合とビルド効率化を進めることで、より高速で安全な開発サイクルが実現できます。
