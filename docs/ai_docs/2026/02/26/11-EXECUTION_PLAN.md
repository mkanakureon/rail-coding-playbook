# Gemini CLI 活用実行計画書 (2026 Q1-Q2)

**作成日**: 2026-02-26
**目的**: 大規模コンテキスト処理能力を活かしたプロジェクト管理と AI 執筆支援の高度化

## 1. フェーズ1：プロジェクト・ガバナンスの確立（即時〜1ヶ月）

本リポジトリの整合性を維持し、開発効率を最大化するための基盤を構築します。

- **1.1 アーキテクチャ監視（ガードドッグ）**
    - `packages/core` のインターフェース遵守状況を定期的に分析し、Switch 移植を妨げる「プラットフォーム依存コード」の混入を早期発見するレポートを作成。
- **1.2 既存ドキュメントの統合と整理**
    - `docs/` 直下に散在する 40 以上のマークダウンファイルを精査・要約し、最新の設計思想と矛盾する古いドキュメントを特定・アーカイブ推奨リストとして報告。
- **1.3 意思決定ログの自動生成**
    - セッションごとの重要な設計変更や議論を `docs/10_ai_docs` に連番付きで記録し、後から「なぜこの設計になったか」を辿れる環境を維持。
- **1.4 外部知識の取り込みとリサーチ**
    - Google Search を活用し、PixiJS の最新最適化手法、Azure の新機能、Nintendo Switch 開発の公開技術情報などを調査し、技術選定の根拠となるリサーチレポートを随時提供。
    - ノベルゲーム市場のトレンドや、他社エンジンの機能を分析し、執筆支援システムの競争力を高めるための機能提案を実施。

## 2. フェーズ2：AI 執筆支援システムの高度化（1ヶ月〜3ヶ月）

現状の「4段階生成」を、より複雑な長編作品に対応できるよう進化させます。

- **2.1 「長編記憶システム」の論理設計**
    - ベクトル DB (RAG) に頼り切らない、プロンプトチェーン内での「伏線管理・回収」のアルゴリズム策定。
    - 章を跨いでもキャラクターの「秘密」や「感情の変化」がリセットされないための累積コンテキスト管理の最適化。
- **2.2 ジャンル別ルールの継続的ブラッシュアップ**
    - 4つのジャンルレビューで判明した「フレーズの繰り返し」等の課題を解決するための YAML ルールの追加定義。
- **2.3 ストーリー整合性チェッカーの構築**
    - 生成された全話テキストを解析し、設定（overview.md, characters.md）との矛盾（性別の食い違い、場所の矛盾、既知情報の忘却など）を自動検出し警告する機能の設計。

## 3. フェーズ3：検証とQAの自動化（3ヶ月以降）

コードとシナリオの両面から、プロジェクトの品質を保証します。

- **3.1 シナリオ駆動型 E2E テストの設計図作成**
    - 生成された `.ks` ファイルから「期待される画面遷移・選択肢」を抽出し、Playwright で自動テストするためのシナリオ・スペックを生成。
- **3.2 パフォーマンス・シミュレーション報告**
    - PixiJS の描画負荷やアセット管理が Switch のリソース制限（2048px制約等）に適合しているかをソースコードレベルで監査。
