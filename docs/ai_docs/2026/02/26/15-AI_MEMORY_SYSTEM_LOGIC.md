# AI 執筆支援：長編作品向け「伏線・記憶管理」ロジック設計

**作成日**: 2026-02-26
**目的**: 10万文字を超える長編作品において、設定の矛盾や伏線の未回収を防ぐ高度なコンテキスト管理の実現

## 1. 現状の限界
現在の「章要約」の累積のみでは、物語が後半に進むにつれて最新の章の要約がトークンウィンドウを圧迫し、初期の重要な設定（第1章で提示された謎など）が LLM の記憶から脱落するリスクがあります。

## 2. 提案する「ハイブリッド型記憶管理」

### 2.1 伏線レジストリ (Plot Thread Registry)
要約とは別に、**「未解決の謎 (Open Threads)」** と **「解決済みの事実 (Resolved Threads)」** を構造化データ（JSON）として管理します。

- **属性**: `id`, `description`, `introducedInChapter`, `status`, `resolvedInChapter`
- **更新ロジック**: 各章の終了後に「この章で新しく提示された謎」と「解決された謎」を LLM に抽出させ、レジストリを更新します。

### 2.2 階層型要約の導入
全ての要約を同じ重みで渡すのではなく、以下の優先順位でコンテキストを構築します。

1.  **直近 2 シーンの全文**: 動作や会話の連続性を担保。
2.  **直前の 1 話分の要約**: 舞台設定や現在の目的を維持。
3.  **作品全体の「骨格」要約**: 全5章〜36章の大まかな流れ。
4.  **未解決の伏線レジストリ（全件）**: LLM に「今何を解決すべきか」を常に意識させる。

## 3. キャラクター・ナレッジ・グラフ
キャラクターごとに「誰が何を知っているか」を追跡します。

- **用途**: 主人公がまだ知らないはずの情報を、LLM が誤って主人公のセリフに出してしまう「知識の汚染」を防ぎます。
- **実装**: 各シーン生成時に、そのシーンに登場するキャラの「既知情報リスト」のみをプロンプトに注入します。

## 4. 期待される効果
このロジックにより、RAG (ベクトル検索) を導入せずとも、数万文字単位の整合性を維持したシナリオ生成が可能になります。特に「第1章の伏線が第10章で完璧に回収される」といったカタルシスを、AI だけで安定して生成できるようになります。
