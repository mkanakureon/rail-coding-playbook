# データベーススキーマ・モデリング報告書

**作成日**: 2026-02-26
**対象**: Prisma スキーマに基づくデータ構造とリレーションの分析

## 1. 核心的データ構造
本プロジェクトのデータベースは、ユーザー（User）、プロジェクト（Project）、公開作品（Work）の 3 つを軸に構成されています。

### 1.1 プロジェクトと公開作品の分離
- **Project**: 制作中の「生データ」。エディタの全情報（ページ、未公開設定等）を保持。
- **Work**: 公開された「パッケージ」。一般ユーザーが閲覧・評価（Like）する単位。
- **設計の意図**: 編集中のデータが即座に公開作品に反映されないよう、明示的な「公開」プロセスを介する構造になっています。

## 2. アセット・キャラクター管理のリレーション

### 2.1 階層的アセット構造
アセットは `Asset` モデルで一括管理されますが、キャラクター表現（Expression）と密接に紐付いています。
- `Character` (1) ↔ `Expression` (N)
- `Expression` (1) ↔ `Asset` (1) または `FrameSet` (1)
- **特徴**: 静止画だけでなく、アニメーションスプライト（FrameSet）を同一の Expression インターフェースで扱える柔軟な設計です。

### 2.2 ユーザーライブラリ
`UserAsset`, `UserCharacter` モデルの存在により、プロジェクトを跨いで共通利用できる「ユーザー専用素材庫」の概念が実装されています。

## 3. AI 執筆支援システムの永続化層
AI 関連のデータは以下の 4 つのテーブルで管理されています。
- `WorkSetting`: AI へのインプットとなる作品設定（JSON）。
- `GenerationResult`: 各ステージ（章・話・テキスト）の生成結果。ハッシュ値による再生成検知をサポート。
- `RagChunk`: RAG (検索拡張生成) 用の知識断片。ベクトル検索への拡張性を考慮（Json型）。
- `AiRequest`: API 使用量とレイテンシの監査ログ。

## 4. 特筆すべき設計
- **ID 体系**: すべての ID が `VarChar(31)` で定義されており、ULID 等のソート可能な文字列 ID の使用が想定されています。
- **タイムスタンプ**: `BigInt` (Unix time) を使用。JavaScript の `Date` オブジェクトとの親和性と、DB 間での移植性を考慮した選択です。
- **整合性**: `onDelete: Cascade` が適切に設定されており、プロジェクト削除時に紐付くアセットや生成結果が自動掃除されるクリーンな設計です。
