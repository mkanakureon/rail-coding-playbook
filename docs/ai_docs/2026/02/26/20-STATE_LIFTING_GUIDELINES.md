# React 状態管理ガイドライン：State Lifting と共有設計

**作成日**: 2026-02-26
**対象**: `apps/editor` 開発者

## 1. 原則：Lifting State Up
kaedevn のエディタは、多数の「ブロック」が互いに依存し合う複雑な UI を持ちます。状態管理の混乱を避けるため、以下の原則を遵守してください。

### 1.1 共通の最小祖先での管理
2 つ以上のコンポーネントが同じデータを必要とする場合、それらの**共通の親コンポーネント**に状態を保持させ、Props 経由でデータと更新関数（Setter）を渡します。

- **例**: `BlockList` (親) がブロック配列を持ち、`ChBlockCard` や `BgBlockCard` (子) にその一部を渡す。

## 2. 実装パターン

### 2.1 ブロックデータの更新
各ブロックカード（子）は、自身が持つ ID と新しい値を親の `onUpdate` 関数に返し、親側で配列全体を更新するパターンを推奨します。

```tsx
// 子コンポーネントの例
const handleValueChange = (newVal: string) => {
  onUpdate(id, { ...data, value: newVal });
};
```

### 2.2 Context API の使い分け
広範囲（アプリ全体）で共有が必要な情報は Context を使用し、局所的な共有は State Lifting を優先します。

| 共有範囲 | 推奨手法 | 内容 |
| :--- | :--- | :--- |
| **グローバル** | `AuthContext` | ユーザー認証情報、テーマ設定 |
| **画面単位** | `ProjectContext` | 編集中のプロジェクト ID、権限情報 |
| **コンポーネント群** | **State Lifting** | ブロックの順序、ドラッグ＆ドロップ状態 |

## 3. パフォーマンスへの配慮
State Lifting は親の再レンダリングを引き起こします。不要な再描画を防ぐため、以下の対策を検討してください。
- `React.memo` による子コンポーネントのメモ化。
- 大きな状態配列を操作する場合は、`useCallback` で更新関数をラップする。

## 4. 禁止事項
- `window` オブジェクトへの直接的な状態の保存。
- 子コンポーネント間での `refs` を介した直接通信。
- 同一データを複数の `useState` で冗長に保持すること（情報の不整合の原因となります）。
