## はじめに

ビジュアルノベルエンジンを作るなら、スクリプト言語の設計は避けて通れない。

既存の選択肢——吉里吉里の TJS、Ren'Py の Python ベーススクリプト、ティラノスクリプトのタグ形式——を検討したうえで、kaedevn では独自の言語 **KSC（Kaede Script）** を設計した。

この記事では、KSC の文法設計と、その背後にあるトレードオフを解説する。

## 設計原則

KSC の設計原則は 3 つだ。

1. **1 行 1 命令**: コンパイルなしにインタプリタが直接実行できる
2. **テキストファースト**: セリフ表示がエンジンの主要機能なので、セリフの記述が最も簡潔であるべき
3. **JavaScript ライクな式評価**: 学習コスト削減のため、条件式や変数代入は JavaScript に近い構文

## セリフブロック: `#speaker...#`

KSC 最大の特徴は **セリフブロック構文** だ。

```ksc
#hero
おはよう。
今日もいい天気だな。
#
```

`#` の後に話者名を書き、`#` 単体で終了する。このシンプルな構文には設計上の意図がある。

### なぜタグ形式にしなかったか

ティラノスクリプトのようなタグ形式だと、セリフは以下のようになる。

```
[name]hero[/name]
[text]おはよう。[br]今日もいい天気だな。[/text]
```

一方 KSC では：

```ksc
#hero
おはよう。
今日もいい天気だな。
#
```

違いは **ノイズの量** だ。ビジュアルノベルのスクリプトではセリフが全体の 70% 以上を占める。セリフの記述にタグのノイズが入ると、シナリオライターの可読性が著しく低下する。

### 地の文の扱い

話者名が空（`#` の後にスペースのみ）の場合は地の文として扱われる。

```ksc
#
ここは学校の屋上だ。
風が心地よい。
#
```

これにより、ナレーションとキャラクターのセリフを同じ構文で統一的に記述できる。

### 文字列補間

セリフブロック内では `{式}` による文字列補間が有効だ。

```ksc
name = "太郎"
score = 100

#narrator
{name}のスコアは{score}点です。
合計は{score + 50}点です。
#
```

補間は**セリフブロック内のみ**で有効とした。通常の式文で文字列補間を許可すると、パーサーの複雑度が跳ね上がるためだ。式文では通常の文字列結合 `"Hello " + name` を使う。

補間内で関数呼び出しも可能にした。これは実際のシナリオで頻出するパターン——好感度に応じた表情名を動的に挿入するなど——に対応するためだ。

```ksc
def mood(aff) {
  if (aff >= 8) {
    return "happy"
  }
  return "normal"
}

ch("heroine", mood(affection), "center")
```

## ラベルとジャンプ

ラベルは `*` で始まる行で定義する。

```ksc
*chapter1
bg("school_gate")

#hero
今日も学校に来たな
#

jump("chapter2")

*chapter2
bg("classroom")

#hero
教室に着いた
#
```

### なぜ `*` を選んだか

`*` は TyranoScript/吉里吉里のラベル記法を踏襲している。ビジュアルノベル制作者にとって馴染み深い記号だ。

代替として `:chapter1` や `@chapter1` も検討したが、`:` はオブジェクトリテラルと紛らわしく、`@` はデコレータ記法との混同が懸念された。VN 業界の慣例に倣い `*` を採用した。

### ラベル名の重複

同名ラベルの重複定義は**後勝ち**とした。

```ksc
*greeting
#hero
最初の挨拶
#

*greeting
#hero
上書きされた挨拶
#
```

この場合、`jump("greeting")` は後者にジャンプする。厳密にはエラーにすべきだが、開発中のイテレーション速度を優先し、寛容な設計とした。将来的に警告を出す仕組みを追加する予定だ。

## 変数と代入

変数宣言は不要で、代入時に自動生成される。

```ksc
affection = 0
name = "Player"
flag = true
```

複合代入も対応している。

```ksc
affection += 2
score -= 10
multiplier *= 1.5
```

### 型システム

KSC の型は 4 つだけだ。

| 型 | リテラル例 | 説明 |
|---|---|---|
| `number` | `42`, `3.14` | 浮動小数点数（JavaScript の `number`） |
| `string` | `"hello"`, `'world'` | 文字列 |
| `boolean` | `true`, `false` | 真偽値 |
| `null` | -- | 未定義変数参照時にエラー |

配列やオブジェクトは**意図的に排除**した。VN スクリプトで配列操作が必要になるケースは稀であり、型の複雑さがシナリオライターのハードルになる。もし将来的に配列が必要になったら、組み込み関数で対応する方針だ。

### Truthy/Falsy 判定

JavaScript に準拠した truthy/falsy 判定を採用している。

```
true   → truthy
false  → falsy
0      → falsy
非ゼロ → truthy
""     → falsy
非空   → truthy
null   → falsy
```

これにより `if (affection)` のような省略記法が自然に動く。

## 条件分岐

if 文は JavaScript ライクな構文だが、**ブレース位置に制約がある**。

```ksc
if (affection >= 5) {
  // 高好感度ルート
} else if (affection >= 3) {
  // 中好感度ルート
} else {
  // 低好感度ルート
}
```

`} else if` と `} else {` は**閉じブレースと同じ行に記述**しなければならない。

```ksc
// NG: これは動かない
}
else {
  ...
}
```

この制約は意図的だ。KSC のパーサーは行単位で分類を行うため、`} else {` が 1 行にまとまっていないとブロックの終了位置を正しく検出できない。パーサーの複雑度を抑えるために、構文に制約を課した。

## 選択肢: choice 構文

VN エンジンの核心機能である選択肢は、専用の `choice` 構文で記述する。

```ksc
choice {
  "一緒に帰る" {
    affection += 2
    jump("go_home")
  }
  "図書室に行く" if (affection >= 3) {
    flag_library = true
    jump("library")
  }
  "用事がある" {
    affection -= 1
    jump("decline")
  }
}
```

### 条件付き選択肢

各選択肢に `if (条件)` を付与でき、条件が false の選択肢は**非表示**になる。

```ksc
"屋上で話そう" if (affection >= 5) {
  // affection が 5 未満ならこの選択肢は表示されない
}
```

表示可能な選択肢が 0 件の場合はランタイムエラーとした。無限ループに陥るよりも、エラーで止めた方がデバッグしやすいという判断だ。

### なぜ JSON 的な構文にしなかったか

代替として以下の構文も検討した。

```
choice:
  - text: "一緒に帰る"
    action: jump("go_home")
  - text: "図書室に行く"
    condition: affection >= 3
    action: jump("library")
```

YAML ライクな構文は可読性が高いが、**ブロック内でのセリフ表示や変数操作ができない**。KSC の choice 構文はブロック内に任意の文を書けるため、選択肢を選んだ直後のリアクションを自然に記述できる。

```ksc
"一緒に帰る" {
  affection += 2
  #friend
  やった！一緒に帰ろう
  #
  jump("go_home")
}
```

## def と sub の分離

KSC では**値を返す関数（def）** と **副作用専用サブルーチン（sub）** を明確に分離した。

```ksc
// def: 値を返す
def mood(aff) {
  if (aff >= 8) {
    return "happy"
  }
  return "normal"
}

// sub: 副作用のみ（return expr は禁止）
sub greet() {
  #hero
  こんにちは！
  #
}
```

### なぜ分離したか

理由は 2 つ。

**1. 意図の明示**: `sub` を見た瞬間に「これは演出用で戻り値がない」と分かる。VN スクリプトでは演出関連のサブルーチン（共通イベント、トランジション）が多用されるため、意図が明示されていると保守性が上がる。

**2. エラーの早期検出**: `sub` 内で `return expr` を書くとコンパイル時ではなく実行時にエラーになる。しかし「値を返したつもりだったのに返っていなかった」というバグよりは、「return expr は sub では使えません」というエラーの方がデバッグしやすい。

### ラベルベースのサブルーチン（call/ret）

def/sub に加えて、ラベルベースの `call`/`ret` も用意している。

```ksc
call("common_event")
// call 先から ret() で戻ってくる

*common_event
bg("school")
#narrator
共通イベント
#
ret()
```

これは TyranoScript/吉里吉里の `call`/`return` と同じ概念だ。既存の VN エンジンからの移行を容易にするために残している。

注意点として、`ret()` は `call()` の復帰専用であり、`def`/`sub` 内では使えない。`def`/`sub` では `return` を使う。この制約は紛らわしいが、スタックフレームの種類（`label` vs `function`/`subroutine`）を明確に区別するための設計判断だ。

## 演算子優先順位

JavaScript と同じ優先順位テーブルを採用した。

| 優先度 | 演算子 | 説明 |
|--------|--------|------|
| 1（低） | `\|\|` | 論理 OR |
| 2 | `&&` | 論理 AND |
| 3 | `==` `!=` | 等価比較 |
| 4 | `>` `>=` `<` `<=` | 大小比較 |
| 5 | `+` `-` | 加算・減算 |
| 6 | `*` `/` `%` | 乗算・除算 |
| 7（高） | `!` `-`（単項） | 否定・符号反転 |

`===`（厳密等価）は**採用しなかった**。KSC の型システムは単純なので、暗黙の型変換が問題になるケースが少ない。`==` で十分だという判断だ。

## 予約語

予約語は意図的に少なく保っている。

```
true  false  if  else  while  def  sub  return  choice
```

`while` は現在未実装だが、予約語として確保している。VN スクリプトでループが必要になるケースは限られる（RPG 要素を持つシナリオなど）が、将来の拡張に備えて予約した。

## 組み込み関数名の予約

以下の名前はユーザー定義関数名に使えない。

```
bg  ch  ch_anim  ch_hide  ch_clear  bgm  bgm_stop
se  voice  wait  waitclick  timeline  timeline_play
battle  jump  call  ret
```

ユーザーが `def bg()` のような定義をすると、実行時エラーになる。組み込み関数の上書きを許可すると、デバッグが極めて困難になるためだ。

## 完全なシナリオ例

以下は KSC の全機能を使った学園恋愛ノベルのサンプルだ。

```ksc
// 変数初期化
affection = 0
name = "太郎"

// 好感度判定関数
def mood(aff) {
  if (aff >= 8) {
    return "happy"
  }
  return "normal"
}

// 挨拶サブルーチン
sub morning_greeting() {
  #heroine
  おはよう、{name}くん！
  #
  affection += 1
}

// メインストーリー
*start
bg("school_gate")
bgm("daily_life")

call("morning_greeting")

choice {
  "一緒に帰ろう" {
    affection += 2
    jump("go_home")
  }
  "図書室に寄らない？" if (affection >= 3) {
    jump("library")
  }
}

*go_home
bg("street_evening")
ch("heroine", mood(affection), "center")

#heroine
今日は楽しかったね
#

if (affection >= 5) {
  jump("true_ending")
}

jump("normal_ending")
```

セリフ、変数、関数、選択肢、条件分岐、ラベルジャンプ——VN に必要な要素が、最小限のノイズで記述できていることが分かる。

## まとめ

KSC の設計は「何を入れるか」よりも「何を入れないか」が重要だった。

- 配列・オブジェクトは排除
- `===` は不採用
- 行中コメントは未サポート
- セリフブロック外の文字列補間は無効

これらの制約は、言語を**シナリオライターにとって安全な道具**に保つためのものだ。VN 専用言語に汎用言語の全機能を持ち込むと、複雑さが本来の目的——ストーリーを書くこと——を妨げてしまう。

---

KSC の文法設計を一通り解説した。セリフブロック構文と def/sub 分離が最も議論の多かった設計判断で、前者は可読性、後者は意図の明示性を優先した結果だ。言語設計は「正解がない」分野だが、ユースケースを絞り込むことで、少なくとも「ひどくない」設計には到達できたと思う。
