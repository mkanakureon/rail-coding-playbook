## はじめに

OSS プロジェクトの README は、利用者が最初に目にするページです。GitHub リポジトリを開いたとき、README の第一印象でプロジェクトの信頼性を判断されます。

ビジュアルノベルエンジン「kaedevn」の OSS リポジトリでは、README を「プロジェクトの顔」として設計しました。この記事では、実際の README の構成と、各セクションの配置意図を解説します。

## セクションごとの設計意図

### 1. タイトル + 一行説明（日英併記）

```markdown
# kaedevn

.ksc (Kaede Script) で動くビジュアルノベルエンジン。
プラットフォーム抽象化（IEngineAPI）により Web / Console / Switch に対応。

Visual novel engine powered by .ksc (Kaede Script) with platform abstraction (IEngineAPI).
```

日本語メインで書きつつ、英語の一行説明を添えています。理由は以下です。

- **主要ターゲットは日本語圏**: ビジュアルノベルは日本発の文化。日本語話者が多い
- **GitHub のグローバル性**: 英語の一行があれば、非日本語圏の人も概要を把握できる
- **IEngineAPI への言及**: 技術的な差別化ポイント（プラットフォーム抽象化）を冒頭で伝える

### 2. パッケージテーブル

Monorepo 構成のため、パッケージ一覧をテーブルで示しています。将来パッケージが増えた場合も、この表に追加するだけです。

### 3. 実行結果のスクリーンショット

README に画像を 1 枚入れるだけで、「このプロジェクトは何をするのか」が視覚的に伝わります。テキストだけの README と比べて、理解速度が格段に上がります。

スクリーンショットの選び方のポイントは以下です。

- **動作結果を見せる**: コードではなく、実行した結果を見せる
- **軽量にする**: PNG で数百 KB 以内に抑える
- **リポジトリ内に配置**: 外部 URL は切れるリスクがある

### 4. クイックスタート

3 行で「インストール → ビルド → 実行」を完結させています。初めて訪れた人が 1 分以内に動かせることが目標です。

### 5. スクリプト例

.ksc の特徴が分かる短いスクリプト例を掲載しています。含まれる要素は以下です。

- `bg()`, `ch()` — 背景・キャラクターの表示（ビジュアルノベルらしさ）
- `#hero ... #` — セリフブロック（独自構文の特徴）
- `choice { ... }` — 選択肢（ゲームらしさ）
- `if (affection >= 3)` — 条件付き選択肢（プログラミング言語としての表現力）

1 つのスクリプト例で「このエンジンで何ができるか」を凝縮して伝えています。

### 6. ドキュメントリンクテーブル

30 ファイル以上あるドキュメントの中から、利用者が最初に読むべき 8 ファイルを厳選してリンクしています。全ドキュメントへのリンクは `docs/README.md` に委ねることで、README の見通しを保っています。

### 7. サンプルスクリプト一覧

「example があるのは知っているが、何が入っているか分からない」を防ぐため、テーブルで各サンプルの内容を示しています。

### 8. 利用方法

「Fork 推奨」というスタンスを明記し、CONTRIBUTING.md へのリンクを張っています。PR は受け付けない方針のため、「利用方法」というタイトルにしています。

## README の設計原則

### 1. 上から順に読んで理解が深まる構造

```
何をするプロジェクトか（タイトル + 一行説明）
  ↓
実行するとどうなるか（スクリーンショット）
  ↓
どう使うか（クイックスタート）
  ↓
何が書けるか（スクリプト例）
  ↓
もっと知りたい（ドキュメントリンク）
  ↓
具体例を見たい（サンプル一覧）
```

### 2. 画像は 1 枚だけ

スクリーンショットやバッジを大量に並べると、逆に情報過多で読まれなくなります。1 枚の画像で「動く」ことを伝えるのが効果的です。

### 3. リンクはテーブルで整理

箇条書きでリンクを並べると読みにくくなります。テーブルで「カテゴリ | リンク」を並べることで、一覧性が高まります。

### 4. 日本語と英語のバランス

日本語圏向けのプロジェクトでも、英語の一行説明を入れることで、GitHub の検索結果やSNS でシェアされた際に非日本語圏の人にも伝わります。

## まとめ

README は「プロジェクトの顔」です。以下の要素を意識して設計しました。

| 要素 | 効果 |
|---|---|
| スクリーンショット | 「何ができるか」を視覚的に伝える |
| クイックスタート | 1 分で動かせる安心感 |
| スクリプト例 | 言語の特徴を凝縮して伝える |
| ドキュメントリンク | 「ちゃんと設計されている」信頼感 |
| サンプル一覧 | 学習リソースの存在を示す |
| 利用方法 | Fork 推奨のスタンスを明確にする |

個人 OSS でここまで README を作り込む必要があるかは議論の余地がありますが、「最初に訪れた人が 30 秒で概要を理解し、3 分で動かせる」ことを目標にすると、自然とこの構成になりました。

---

README はプロジェクトの「第一印象」を決めるドキュメントです。スクリーンショット 1 枚、クイックスタート 3 行、スクリプト例 1 つ。これだけで「使ってみよう」と思ってもらえるかどうかが決まります。情報量よりも「伝わるかどうか」を優先した構成を心がけました。
