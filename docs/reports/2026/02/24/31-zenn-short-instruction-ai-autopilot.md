## はじめに

Claude Code を使っていて、最も驚いた瞬間は「進んで」の 3 文字で AI が正しく動いたときでした。

インタプリタの Phase 7 実装中、テストを 1 つ通した後に「進んで」と打つだけで、次のテストケースを実装し、テストを実行し、失敗箇所を修正し、全テストが通るまで自走しました。一方で、別のプロジェクトで同じ「進んで」と打っても、AI は何をすべきか分からず止まりました。

この違いは何か。**コンテキストが整っているかどうか**です。本記事では、「短い指示で AI が動く条件」と、その条件を意図的に作る方法を解説します。

## 「進んで」が機能する条件

### 条件 1: CLAUDE.md にプロジェクトルールが書かれている

このプロジェクトのルートには `CLAUDE.md` というファイルがあります。

```markdown
# CLAUDE.md

## Project Overview
kaedevn-monorepo は Nintendo Switch / Web 向けビジュアルノベルエンジン。

## Architecture
### Core Abstractions
| Interface | Purpose | Web Implementation |
|-----------|---------|-------------------|
| IInput    | Unified action dispatch | PixiJS pointer/keyboard events |
| IAudio    | BGM/SE/VOICE playback   | Web Audio API |
| IStorage  | Save/Load abstraction   | IndexedDB |

## Development Commands
npm install
npm run typecheck
npm run build

## Rules
### Testing
- expect で期待する状態を明確に検証する
- スキップ・フォールバック・エラー握りつぶしは禁止
- waitForTimeout でごまかさず、正しいセレクタや条件を待つ
- 失敗したら原因を調査してコードを修正する
```

この `CLAUDE.md` は Claude Code が**自動的に読み込む**ファイルです。プロジェクトのアーキテクチャ、開発コマンド、テストルールが書かれていれば、AI はこれらを暗黙的に理解した上で作業を進めます。

### 条件 2: 直前の会話にコンテキストがある

「進んで」が機能するのは、直前の会話で「Phase 7-1 のエラーハンドリングテストを実装中」という文脈がある場合です。AI は以下を暗黙的に理解しています。

- 現在実装中のフェーズ（Phase 7-1）
- 対象モジュール（ErrorHandler）
- 何をすべきか（次のテストケースの実装）
- テストの実行方法（CLAUDE.md に書かれている）
- テストのルール（期待値を明確に検証する）

### 条件 3: ソースコードが「次に何をすべきか」を示している

テストファイルに TODO コメントがあったり、テスト計画書に未実装のケースが列挙されていたりすると、AI は「次にやるべきこと」を判断できます。

```typescript
// test/phase7-1.test.ts
describe('ErrorHandler', () => {
  it('should suggest similar variable names', () => { /* 実装済み */ });
  it('should include stack trace', () => { /* 実装済み */ });
  // 次はここ:
  it('should show source context around error', () => { /* TODO */ });
});
```

## 「進んで」が機能しない条件

逆に、以下の場合は「進んで」が機能しません。

| 条件 | 原因 |
|------|------|
| CLAUDE.md がない | プロジェクトの文脈が分からない |
| 会話が新規 | 直前のコンテキストがない |
| 何をすべきか曖昧 | 計画書や TODO がない |
| 複数の選択肢がある | どの方向に「進む」か分からない |

## 短い指示のバリエーション

実際に使っている短い指示のパターンを紹介します。

### 「テストして」

```
→ npm run test を実行
→ 失敗したテストを確認
→ 原因を分析
→ コードを修正
→ 再テスト
→ 全テスト通過まで繰り返す
```

これが機能するのは、CLAUDE.md の Testing ルールに「失敗したら原因を調査してコードを修正する」と書いてあるからです。

### 「型チェックして」

```
→ npm run typecheck を実行
→ エラーを確認
→ 型エラーを修正
→ 再チェック
```

Development Commands に `npm run typecheck` が書いてあるので、AI はこのコマンドを使います。

### 「続けて」

「進んで」と似ていますが、こちらは「前回の作業の続き」を意味します。Phase 実装の途中で会話が切れた場合に使います。

### 「確認して」

```
→ git status で変更ファイルを確認
→ 変更内容を要約
→ 問題がないか分析
```

## CLAUDE.md の設計: コンテキストを意図的に作る

短い指示で AI が動くようにするには、CLAUDE.md を戦略的に設計する必要があります。

### 1. プロジェクトの全体像を書く

```markdown
## Project Overview
kaedevn-monorepo は Nintendo Switch / Web 向けビジュアルノベルエンジン。
```

これにより、AI は全ての作業を「ビジュアルノベルエンジン開発」の文脈で解釈します。

### 2. アーキテクチャの制約を書く

```markdown
### Core Abstractions (mandatory)
Rendering (PixiJS) can be used directly for now, but input, audio, and storage
must go through abstractions to avoid full rewrites at Switch porting time.
```

「Switch 移植を考慮せよ」という制約が書かれていることで、AI が直接 Web API を呼ぶコードを生成するのを防ぎます。

### 3. 開発コマンドを書く

```markdown
## Development Commands
npm install
npm run typecheck
npm run build
```

「テストして」と言われたとき、AI はここを見てコマンドを選びます。

### 4. テストルールを書く

```markdown
### Testing
- expect で期待する状態を明確に検証する
- スキップ・フォールバック・エラー握りつぶしは禁止
- 失敗したら原因を調査してコードを修正する
```

これがないと、AI はテストが失敗したときにスキップしたり、エラーを無視したりする可能性があります。

### 5. State の配置ルールを書く

```markdown
### State の配置（React）
1. その state を参照または更新するコンポーネントを全てリストアップする
2. 共通の最小祖先を特定し、そこに state を置く
3. EditorPage.tsx の render 構造を確認する
```

React の state 管理は AI が間違えやすいポイントです。ルールを明文化しておくことで、「このコンポーネントに state を置くべきか」を AI が自分で判断できます。

## 具体例: Phase 7 の実装過程

Phase 7（統合テスト・デバッグ）の実装過程を振り返ります。

### 指示 1: 開始

```
Phase 7-1 のエラーハンドリングを実装してください。
packages/interpreter/docs/KNF_Interpreter_Phase7_Plan.md を参照してください。
```

これが最初の詳細な指示です。計画書の場所を示すことで、AI は何を実装すべきかを理解します。

### 指示 2-10: 「進んで」

```
進んで
```

Phase 7-1 の各テストケースを順番に実装していきます。AI は以下のサイクルを自走します。

1. 次のテストケースを特定
2. テストを書く
3. 実装コードを書く
4. テストを実行
5. 失敗したら修正
6. 全テスト通過を確認

### 指示 11: 「Phase 7-2 に進んで」

```
Phase 7-2 に進んで
```

フェーズの切り替え時は、フェーズ番号を明示します。AI は計画書から Phase 7-2 の内容（デバッグモード実装）を読み取り、作業を開始します。

### 指示 12-20: 「進んで」

```
進んで
```

Phase 7-2 も同様に「進んで」で自走します。

### 最終結果

```
Phase 7-1: 15 テスト（エラーハンドリング）
Phase 7-2: 18 テスト（デバッグモード）
Phase 7-3: 74 テスト（統合テスト・デモ）
合計: 107 テスト
```

約 20 回の指示で 107 テストを実装しました。そのうち 15 回は「進んで」の 3 文字です。

## 短い指示の限界

短い指示が機能しない場面もあります。

### 設計判断が必要な場面

「キャラクターをアセットに統合すべきか、別テーブルのまま維持すべきか」のような設計判断は、「進んで」では解決できません。これは人間が判断し、明確に指示する必要があります。

### コンテキストが切れた場面

長い会話の途中でコンテキストウィンドウの上限に達すると、前半の文脈が失われます。この場合、「Phase 7-2 のブレークポイント実装の続きをやってください。test/phase7-2.test.ts を見てください」のように、コンテキストを再設定する必要があります。

### 新しい機能の開始

全く新しい機能の実装を始める場合、「進んで」では方向が定まりません。「タイムラインエディタを実装してください。仕様は docs/timeline-spec.md にあります」のように、具体的な指示が必要です。

## 指示の粒度と出力品質の関係

```
粒度         指示例                     出力品質
─────────────────────────────────────────────────
極細         "setBg の第2引数を optional に"   正確（ほぼ100%）
細           "エラーハンドリングを追加して"      高い（90%+）
中           "Phase 7-1 を実装して"            高い（85%+）
粗           "進んで"                         コンテキスト依存（70-95%）
極粗         "作って"                         低い（50%以下）
```

「進んで」の品質が 70-95% と幅があるのは、コンテキストの充実度に完全に依存するからです。CLAUDE.md が充実し、計画書があり、テストファイルが構造化されていれば 95%。何もなければ 70%。

## まとめ: コンテキスト設計 > 指示の工夫

「短い指示で AI を動かす」テクニックは、指示の工夫ではなく、**コンテキストの設計**です。

| コンテキスト要素 | 効果 |
|----------------|------|
| CLAUDE.md | プロジェクトルールの暗黙的な適用 |
| 計画書・仕様書 | 「次に何をすべきか」の判断材料 |
| テストファイル | 実装の方向性とゴールの明確化 |
| 直前の会話 | 現在の作業フェーズの文脈 |

これらが揃っていれば、2-3 文字の指示で AI は正しく自走します。整っていなければ、どんなに詳細な指示を書いても、AI は迷います。

「進んで」が機能するプロジェクトは、ドキュメントと構造が優れたプロジェクトです。短い指示は、プロジェクトの質を映す鏡なのです。

「進んで」の 3 文字で AI が自走する体験は衝撃的でした。しかし振り返ると、その背景には CLAUDE.md のルール設計、計画書の整備、テストファイルの構造化という地道な準備がありました。短い指示は「魔法」ではなく、コンテキスト設計の結果です。AI 協働の本質は、「どう指示するか」ではなく「どうコンテキストを作るか」にあると確信しています。
