## この記事で得られるもの

Claude Code の skills 機能を使って、コミット・デプロイ・レポート保存・OSS 同期などの定型作業を自動化する方法。実際に運用している 8 つのスキルの設計と、トリガー設計で失敗した話。

## skills とは

Claude Code の skills は `.claude/skills/{name}/skill.md` に置く Markdown ファイルだ。特別なフォーマットはない。ただの Markdown。

Claude Code はスキルファイルを読み、そこに書かれた手順に従って動く。プログラミング不要で、自然言語の手順書がそのまま自動化ツールになる。

## 8 つのスキルの全体像

```
.claude/skills/
├── commit/              # git コミット
├── push/                # git プッシュ
├── save-report/         # 作業レポート保存
├── sync-oss/            # OSS リポジトリ同期
├── zenn/                # Zenn 記事作成
├── deploy-azure/        # Azure デプロイ
├── dev-server/          # 開発サーバー起動
└── playwright-e2e-test/ # E2E テスト
```

この 8 つで日常の開発ワークフローをほぼカバーしている。

## スキルの書き方 — commit を例に

最も使用頻度が高い commit スキルの構造を見てみよう。

### 手順を番号付きで書く

```markdown
## 手順

1. `git status` で変更ファイルを確認
2. `git diff` で差分を確認
3. 最近のコミットメッセージのスタイルを確認
4. ステージング（.env, credentials は除外）
5. コミットメッセージを作成
6. コミット実行
7. `git status` で成功を確認
```

Claude Code は番号付きリストを順番に実行する。「差分を確認 → メッセージ作成 → コミット」の順序が保証される。

### フォーマットを具体例で示す

```markdown
## コミットメッセージのフォーマット

{プレフィックス}: {日本語の説明}

- 変更点の箇条書き

---
Claude の感想（1〜3行、ゆるいトーンでOK）

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

抽象的な説明よりも具体的なフォーマットの方が再現性が高い。「日本語でコミットメッセージを書いて」より、実際のテンプレートを見せる方が正確。

### 除外ルールを明記する

`.env` やクレデンシャルファイルのステージングを禁止するルールを書いておく。これがないと `git add -A` で全部追加されるリスクがある。

## 「Claude の感想」— ゆるさの設計

コミットメッセージに Claude Code 自身の感想を入れるルールを設けた。

```
---
このスキル、書いてるときは「こんなの要るかな」と思ったけど、
使い始めたら毎回の手作業が消えて驚いた。

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

`---` で区切って、ゆるいトーンで 1〜3 行書く。技術的な内容ではなく、作業の正直な感想。

なぜ入れたか:

- **コミットログが読んで面白くなる。** 技術的な変更点だけだと味気ない
- **AI が書いたコードの透明性。** Co-Author に加えて、AI の「考え」が記録に残る
- **開発のモチベーション。** 一人開発だと会話相手がいないが、感想があると会話っぽくなる

実際のコミットログから抜粋:

```
docs: Zenn 投稿ネタに予想文字数を追加

- 38 ネタ全てに予想文字数（5,000〜15,000字）を追記
- 合計予想: 335,000字

---
38本で33万字、1日3本ペースでも2週間かかる計算。なかなかの執筆量。

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

## sync-oss — 保護ファイルという概念

OSS リポジトリへの同期スキルで、設計上もっとも気を使った部分がある。

```markdown
## kaedevn ルートの保護ファイル（絶対に上書き・削除しない）

| ファイル | 用途 |
|---|---|
| README.md | OSS リポジトリのトップ README |
| CONTRIBUTING.md | コントリビューションガイド |
| LICENSE | MIT ライセンス |
| package.json | ルート package.json |

コミット時は git add packages/interpreter/ のみ使う。
git add -A は禁止。
```

monorepo → OSS リポジトリの一方通行同期で、rsync `--delete` を使う。つまり **monorepo に存在しないファイルは OSS 側で削除される。**

しかし OSS リポジトリのルートにある README や LICENSE は monorepo には存在しない。保護しないと同期のたびに消える。

解決策は単純: **rsync のスコープを `packages/interpreter/` に限定** し、ルートファイルは触らない。さらに安全策として `git add -A` を禁止し、必ず `git add packages/interpreter/` のみを使うルールにした。

スキルに「やってはいけないこと」を書くのは、「やってほしいこと」を書くより重要だ。

## save-report — トリガー設計の失敗と改善

スキルの発動条件（トリガー）は自然言語で書く。

```markdown
## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "commit this" | コミット実行 |
| "push it" | プッシュ実行 |
| "OSSに同期して" | OSS 同期実行 |
```

最初は少数のトリガーだけ書いていたが、実際に使うと「保存して」「ここに保存して」「ドキュメント保存して」「まとめて」などバリエーションが多い。

save-report スキルでは最終的に 12 個のトリガーバリエーションを書いた:

```markdown
| "振り返り文書を作成して" | 保存 |
| "レポート書いて"         | 保存 |
| "まとめを書いて"         | 保存 |
| "ここに保存して"         | 保存 |
| "保存して"               | 保存 |
| "文書作成して"           | 保存 |
```

さらに「暗黙のトリガー」ルールを追加した:

```markdown
### 重要: 暗黙のトリガー

ユーザーが「保存」「文書」「ドキュメント」「md」「まとめ」などの
キーワードを含む指示をした場合、保存先が明示されていなくても
docs/09_reports/ を保存先とみなして発動する。
```

これで「ここにまとめて」のようなあいまいな指示でも正しく動くようになった。

教訓: **トリガーは最初に多めに書くか、暗黙のルールで補完する。** 少なすぎるとスキルが発動しない。

## zenn — 2 箇所保存の設計

Zenn 記事作成スキルには「2 箇所に必ず保存する」ルールがある。

```markdown
## 保存先（2箇所・両方必須）

### 1. 元ネタ
docs/09_reports/{年}/{月}/{日}/{連番}-zenn-{タイトル}.md

### 2. Zenn ドラフト
docs/zenn/drafts/{ULID先頭10文字}-{タイトル}.md
```

元ネタ（プロジェクト内の作業記録）と Zenn ドラフト（プッシュ元）を分離した。

最初は 1 箇所だけだったが、「レポートとして検索したい」「Zenn にプッシュしたい」という 2 つの用途が衝突した。ファイル名の命名規則が違う（レポートは日付連番、Zenn は ULID スラッグ）ので、2 箇所に同じ内容を置くことにした。

## スキルの保守コスト

8 つのスキルを運用して分かったこと:

1. **スキルは育つ。** 最初は 10 行程度だったものが、例外処理やエッジケースの追加で 100 行近くになる。sync-oss は最終的に 108 行になった
2. **スキル間の一貫性が重要。** commit スキルで決めた「感想コメント」のルールを sync-oss にも適用する、というように横断的なルールが必要。一つのスキルだけ違うフォーマットだと混乱する
3. **Markdown だから修正が簡単。** コードを書き換えるのと違い、手順の追加・変更は 1 行の Markdown 編集で済む。デプロイも不要

## スキルを書き始めるなら

おすすめの順序:

1. **commit スキルから始める** — 毎日使う。効果を実感しやすい
2. **プレフィックスとフォーマットだけ書く** — 最初は 20 行でいい
3. **問題が起きたらルールを追加** — 「.env がステージングされた」→ 除外ルール追加
4. **横展開する** — commit で学んだパターンを他のスキルに適用

## まとめ

- Skills は `.claude/skills/{name}/skill.md` に置く Markdown ファイル
- 手順を番号付きで、フォーマットを具体例で、除外ルールを明記する
- トリガーは多めに書く。暗黙のトリガールールも有効
- 保護ファイルのような「やってはいけないこと」を書くのが最も重要
- Markdown なので保守コストが低い
