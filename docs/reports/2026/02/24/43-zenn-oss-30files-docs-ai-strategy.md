## はじめに

個人で開発しているビジュアルノベルエンジン「kaedevn」のインタプリタパッケージ（`@kaedevn/interpreter`）を OSS として公開するにあたり、ドキュメントの整備が必要になりました。結果として、`packages/interpreter/docs/` 直下に 30 ファイル以上のドキュメントを Claude Code と協働して作成しました。

この記事では、「どのような順序で」「どのような依存関係を意識して」ドキュメントを生成したかの戦略を解説します。

## ドキュメントの全体構成

最終的に作成したドキュメントは以下の 6 カテゴリ、24 ファイル（OSS 向け）+ 内部ドキュメント群です。

| カテゴリ | ファイル数 | 内容 |
|---|---|---|
| 要求仕様書 | 3 | プロジェクト目的、スクリプト言語要件、プラットフォーム抽象化要件 |
| 仕様書 | 7 | KSC 言語仕様、コマンド仕様、IEngineAPI 仕様、式評価、状態管理、エラーハンドリング、出力モデル |
| 設計書 | 6 | アーキテクチャ、パーサー、式評価器、デバッグシステム、拡張性、クラス対応 |
| 実装ガイド | 4 | クイックスタート、IEngineAPI 実装ガイド、スクリプト記述ガイド、ConsoleEngine ガイド |
| テスト関連 | 3 | テスト計画、テスト手順、テストカバレッジ |
| リファレンス・運用 | 5 | API リファレンス、FAQ、変更履歴、用語集、ドキュメント索引 |

合計で 5,080 行以上のドキュメントが生成されました。

## 依存順生成の戦略

### なぜ順序が重要か

ドキュメントを AI に生成させるとき、「全部一度に作って」と言ってもうまくいきません。理由は以下です。

1. **参照の一貫性**: 用語集（glossary.md）で定義した用語を、他のドキュメントが正しく参照する必要がある
2. **詳細度の段階**: 要求仕様で「何を作るか」を定義してから、仕様書で「どう動くか」を書く
3. **コンテキストの蓄積**: 先に書いたドキュメントが、後のドキュメントのコンテキストになる

### Phase 構成

```
Phase A: 基盤ドキュメント（他のドキュメントが参照する土台）
  1. glossary.md          ← 全ドキュメントが参照する用語定義
  2. requirements-overview.md  ← プロジェクトの全体像
  3. spec-ksc-language.md      ← 言語仕様（コア）
  4. spec-engine-api.md        ← エンジン API 仕様（コア）

Phase B: コア仕様・設計（Phase A を参照）
  5-11. requirements-*, spec-*, design-architecture.md

Phase C: 詳細設計（Phase B を参照）
  12-15. design-parser.md, design-evaluator.md, etc.

Phase D: 実装ガイド・テスト（Phase A-C を参照）
  16-21. guide-*, testing-*

Phase E: リファレンス・運用（最後に作成）
  22-24. api-reference.md, faq-troubleshooting.md, changelog.md
```

### Phase A が最重要

Phase A の 4 ファイルは「他の全ドキュメントの土台」です。特に `glossary.md`（用語集）を最初に作ることで、以下のような一貫性が保証されます。

- 「KSC」と「.ksc」の使い分けが統一される
- 「IEngineAPI」のような固有名詞が揺れない
- 「GameState」「CallFrame」などの内部用語が正確に参照される

### ドキュメント間の参照関係

```
glossary.md
  ↓ 用語定義を参照
requirements-overview.md ← spec-ksc-language.md
  ↓ 機能要件を参照        ↓ 文法定義を参照
spec-builtin-commands.md  design-parser.md
  ↓ コマンド仕様を参照     ↓ パーサー設計を参照
guide-scripting.md        design-evaluator.md
  ↓ スクリプト例を参照
guide-getting-started.md
  ↓ クイックスタートを参照
api-reference.md
```

この依存関係を意識することで、「guide-scripting.md が参照する spec-builtin-commands.md がまだ存在しない」といった問題を回避できます。

## docs/README.md -- ドキュメント索引

全ドキュメントへのリンクを整理した索引ファイルを作成しました。この索引があることで、利用者は「何がどこにあるか」を一目で把握できます。

## 実際の作業フロー

### 1. 計画書の作成

まず `oss-docs-plan.md` を作成し、24 ファイルの一覧・作成順序・検証方法を定義しました。

### 2. Phase ごとの生成

Claude Code に「Phase A を作成して」と指示し、4 ファイルを生成。次に「Phase B を作成して」と進めました。各 Phase では、前の Phase で作成したファイルを参照コンテキストとして渡しています。

### 3. 検証

- 全 24 ファイルが `packages/interpreter/docs/` 直下に存在すること
- `docs/README.md` に全ファイルの索引が含まれること
- ドキュメント間のリンクが切れていないこと

## ドキュメント整備の効果

### 1. AI との協働品質の向上

ドキュメントが充実すると、Claude Code が「spec-ksc-language.md を読んで」と指示するだけで正確なスクリプト例を生成できるようになります。ドキュメントが AI のコンテキストとなり、出力精度が上がるフライホイール効果です。

### 2. OSS としての信頼性

README だけでは「使い方が分からない」と離脱されがちです。体系的なドキュメントがあることで、「このプロジェクトはちゃんと設計されている」という信頼感を与えられます。

### 3. 自分自身の理解の深化

ドキュメントを書く過程で、「この仕様は曖昧だな」「この設計判断を明文化しておくべきだった」といった発見がありました。ドキュメント整備は、実装の品質向上にも寄与します。

## 学んだこと

### 「まず計画書を書く」

24 ファイルを一気に作るのではなく、まず計画書（oss-docs-plan.md）を書き、ファイル一覧・作成順序・検証方法を定義しました。これにより、作業の全体像が明確になり、AI への指示も「Phase A を作成して」と簡潔にできました。

### 「依存順に作る」

用語集 → 要求仕様 → 仕様書 → 設計書 → ガイド → リファレンスの順で作ることで、参照の一貫性が保証されます。逆順（ガイドを先に作ってから仕様書を書く）だと、用語の揺れやリンク切れが発生しやすくなります。

### 「索引を作る」

30 ファイル以上あると、「どこに何があるか」が分からなくなります。README.md に全ファイルのリンクテーブルを用意することで、利用者も開発者もナビゲーションできます。

## まとめ

個人 OSS で 30 ファイルのドキュメントを整備することは、一見すると過剰に見えるかもしれません。しかし、ドキュメントは「未来の自分」と「利用者」へのメッセージです。AI と協働すれば、体系的なドキュメントを短時間で整備できます。重要なのは「何を・どの順で書くか」の戦略であり、実際の執筆は AI に任せられます。
