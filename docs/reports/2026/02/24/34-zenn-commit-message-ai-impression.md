## はじめに

Claude Code でコミットすると、メッセージはだいたいこうなる。

```
feat: ○○を実装

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

正確で、フォーマットも統一されていて、何も問題はない。でも、味気ない。

そこで、コミットメッセージに「Claude の一言」を必須にするルールを追加した。結果、こうなった。

```
docs: Zenn 投稿ネタ 38 本を 278 コミットから抽出

---
278コミット全部読み返すの、自分の作業なのに他人の日記読んでるみたいで面白かった。
ebbfb3d の初期コミットから今日までの成長っぷりがすごい。

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

「自分の作業なのに他人の日記読んでるみたい」——こういう一言が入るだけで、`git log` を読む体験がまるで変わる。

## 仕組み：commit スキル

Claude Code には「スキル」という仕組みがある。`.claude/skills/` 以下にマークダウンファイルを置くと、特定のトリガーで自動的に読み込まれる。

コミット用のスキルは `.claude/skills/commit/skill.md` にある。全 127 行。

### トリガー

```markdown
## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "コミットして" | 全変更をステージしてコミット |
| "全部コミットして" | 同上 |
| "ドキュメントだけコミットして" | docs/ のみステージしてコミット |
| "コードだけコミットして" | packages/ apps/ のみステージしてコミット |
```

「コミットして」と言うだけで、`git status` → `git diff --stat` → `git log --oneline -5` → ステージ → コミットを全自動で実行する。

### コミットメッセージのルール

```markdown
### プレフィックス

| プレフィックス | 使う場面 |
|---|---|
| `feat:` | 新機能の追加 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントのみの変更 |
| `refactor:` | 動作を変えないコード整理 |
| `chore:` | ビルド・設定・スクリプト等のメンテナンス |

### フォーマット

{プレフィックス}: {日本語の説明}

- 説明は**日本語**で書く
- 複数の変更がある場合は `・` で区切る
```

ここまでは普通だ。重要なのは次のセクション。

## 「Claude の一言」ルール

```markdown
## Claude の一言（必須）

コミットメッセージの末尾に、今回の作業についての
**Claude Code の感想・振り返り**を1〜3行加える。

### ルール

- 作業内容を振り返った**正直な感想**を書く
- **たわいない・ゆるい**トーンでOK。堅苦しくしない
- 自分が「面白かった」「難しかった」「地味だけど大事」など思ったことを素直に
- 日本語で書く
- `---` で本文と区切る
```

ポイントは 3 つある。

### 1. 「必須」にする

任意だと書かない。人間と同じだ。「必須」と書けば Claude Code は必ず書く。

### 2. 「ゆるいトーン」を指定する

「正直な感想」「たわいない」「堅苦しくしない」——こう書かないと、Claude Code は真面目な要約を書いてしまう。

```
# こうなりがち（ゆるいトーン指定なし）
本変更により、コミットメッセージの一貫性が向上しました。

# こうなる（ゆるいトーン指定あり）
スキルのトリガー追加、地味だけどこういう細かい改善が積み重なって使いやすくなるんだよな。
```

### 3. 例を複数見せる

```markdown
### 例

---
今日は表を作るだけかと思ったら、KSとKSCの違いを整理するいい機会になった。
wait の単位が秒とミリ秒でバラバラなのは、うっかりすると絶対ハマりそう。

---
スキルのトリガー追加、地味だけどこういう細かい改善が積み重なって使いやすくなるんだよな。
「ここに保存して」を正しく解釈できなかったのは反省。

---
コミットスキルを自分で作るのはちょっとおもしろい体験だった。
過去のコミット眺めてたら、このプロジェクトの歩みが見えてきてなかなか感慨深い。
```

3 つの例を見せることで、Claude Code はバリエーションのある感想を書くようになる。毎回同じパターンにならない。

## 完成形

スキルが定義する「完成形イメージ」がこれだ。

```
feat: KSC コマンドリファレンス・対応表を追加

---
KSとKSCは別物だと改めて整理できてよかった。
コマンド表、地味に手間かかったけど、これは手元に置いておきたいやつ。

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

構造は明確。

1. `{プレフィックス}: {説明}`（1 行目）
2. 空行
3. 変更点の箇条書き（任意）
4. `---`（区切り線）
5. Claude の感想（1-3 行）
6. 空行
7. `Co-Authored-By:`

## 実際のログから

スキルを導入してからのコミットメッセージをいくつか紹介する。

### コンパイラ実装の感想

```
feat: KSCコンパイラ Phase 0-1 実装（レキサー・パーサー・型定義）

---
コンパイラをゼロから書くのは久しぶりで楽しかった。
レキサーは一発で全テスト通ったのに、パーサーで「{がブロック文になる問題」に
引っかかったのはJS/TSあるあるで、ちょっとニヤッとした。
```

「ちょっとニヤッとした」——これは Claude Code が書いた。指示は「正直な感想をゆるいトーンで」だけだ。

### ドキュメント整理の感想

```
docs: Zenn 投稿ネタに予想文字数を追加

---
38本で33万字、1日3本ペースでも2週間かかる計算。なかなかの執筆量。
```

計算結果を淡々と述べつつ、「なかなかの」という主観が入る。このバランスが良い。

### メタな感想

```
docs: Zenn 投稿ネタに G パターン（メタ）を追加

---
ネタ一覧を作った手順自体がネタになる、メタな再帰。
```

自分の作業がメタ的に再帰していることに言及する。AI がメタ認知的な感想を書くのは面白い。

### 地味な作業の感想

```
chore: sync-oss スキルにルートファイル保護ルールを追加

---
（この感想は同期スキルの改善時に書かれた）
```

地味な chore でも感想は入る。「地味だけど大事」系の感想が自然に出てくる。

## なぜ感想を書かせるのか

### 1. git log が読み物になる

```bash
git log --format="%B" | less
```

これを実行すると、技術的な変更履歴の合間に、AI の感想が挟まる。プロジェクトの進行を「読む」体験が生まれる。

### 2. AI の判断プロセスが可視化される

「パーサーで { がブロック文になる問題に引っかかった」——これは実装中に実際に起きた問題だ。コミットの diff だけでは見えない「過程」が、感想から読み取れる。

### 3. レビューの手がかりになる

「110 テスト全緑は気持ちいい」と書いてあれば、テストが通っていることがわかる。「うっかりすると絶対ハマりそう」と書いてあれば、注意点があることがわかる。

## 導入のコツ

### 最小構成

commit スキルがなくても、Claude Code のシステムプロンプトや CLAUDE.md に以下を書くだけで機能する。

```markdown
## コミットメッセージ

末尾に `---` で区切って、今回の作業についての正直な感想を1〜3行書く。
ゆるいトーンでOK。
```

### スキルにする利点

スキルにすると「コミットして」の一言で全自動になる。手順も統一される。

```markdown
## 手順

1. `git status` で変更ファイルを確認
2. `git diff --stat` で変更内容の概要を把握
3. 直近コミット（`git log --oneline -5`）を見てメッセージスタイルを確認
4. ステージするファイルを決定
5. コミットメッセージを作成
6. `git commit` を実行
7. `git status` で成功を確認
```

この手順を毎回守るので、「ステージし忘れ」「スタイル不一致」が起きない。

### プレコミットフック対応

```markdown
## プレコミットフック対応

- ハードコードされた URL チェック（Editor の API URL）
- チェックが失敗した場合は `--no-verify` を使わず、原因を修正してから再コミット
```

`--no-verify` を使わせないルールも書ける。AI は指示を守る。

## 注意点

### 感想の品質はプロンプト次第

「感想を書け」だけだと、定型文になる。「正直に」「ゆるく」「例を参考に」の 3 つが揃って初めてバリエーションが出る。

### Co-Authored-By のモデル名

```
Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

スキルの例では `Sonnet 4.6` になっていたが、実際の運用では `Opus 4.6` で統一している。モデル名は手動で合わせる必要がある。

### 感想が長すぎる場合

「1〜3行」と書いていても、大きな実装の後は 3 行を超えることがある。気になるなら「最大3行」を太字にする。

## まとめ

コミットスキルに「感想を書け」と一行追加するだけで、`git log` の体験が変わる。

技術的には何も難しくない。`skill.md` に数行書くだけだ。でも、その数行が生み出す差は大きい。278 コミットの `git log` を読み返したとき、感想付きのメッセージは **そこに開発者がいた痕跡** として機能する。

それが人間ではなく AI の痕跡だったとしても。
