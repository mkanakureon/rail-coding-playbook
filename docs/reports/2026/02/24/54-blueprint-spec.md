# Blueprint 方式 — 生成AI時代の画面定義フォーマット

> Generated by Claude Opus 4.6

## Blueprint とは

**Blueprint（ブループリント）** は、生成AIに画面改修を指示するとき「どこに何があるか」の解釈ブレを防ぐための軽量フォーマット。

tsx ファイルと 1:1 対応する `.bp` ファイルに画面の構造を定義し、改修指示は自然言語で書く。

### 従来の画面仕様書との違い

| | 従来の画面仕様書 | Blueprint |
|---|---|---|
| 読み手 | 人間 | AI |
| 書き手 | 設計者 | 開発者（=指示者） |
| 詳細度 | 網羅的（数十ページ） | 構造のみ（5〜15行） |
| 目的 | 「何を作るか」を伝える | 「どこに何があるか」だけ伝える |
| 指示方法 | 仕様書に従って実装 | Blueprint + 自然言語 |
| 更新タイミング | 設計フェーズのみ | tsx 変更のたびに |

### なぜこれが必要か

生成AIが画面を実装するとき、最大のボトルネックは「どこに何があるか」の推測ミス。

- 「Header」と言ったとき、AIは共通ヘッダーなのかページ固有のヘッダーなのか迷う
- 「フォームの下に」と言ったとき、AIはフォーム全体の下なのか各入力欄の下なのか迷う
- 「アセットパネル」と言ったとき、AIは EditorPage 全体を対象にするかもしれない

Blueprint は**この1点だけを潰す**。指示内容（何をするか）は自然言語のままでいい。

## ファイル構成

### 拡張子: `.bp`

- 中身はプレーンテキスト（Markdown 互換）
- コードでも文書でもない「AIへの構造ヒント」という新カテゴリ
- `.md` と区別することで、ドキュメントと混ざらない

### ディレクトリ: リポジトリ直下の `blueprint/`

tsx のディレクトリ構造をミラーする。

```
blueprint/
├── next/
│   ├── public/          # apps/next/app/(public)/
│   │   ├── index.bp     # → apps/next/app/page.tsx
│   │   ├── login.bp     # → apps/next/app/(public)/login/page.tsx
│   │   ├── works.bp     # → apps/next/app/(public)/works/page.tsx
│   │   ├── play-id.bp   # → apps/next/app/(public)/play/[id]/page.tsx
│   │   └── ...
│   ├── private/          # apps/next/app/(private)/
│   │   ├── mypage.bp    # → apps/next/app/(private)/mypage/page.tsx
│   │   ├── projects-id.bp
│   │   └── ...
│   └── admin/            # apps/next/app/(private)/admin/
│       ├── index.bp      # → ダッシュボード
│       ├── users.bp
│       └── ...
└── editor/               # apps/editor/src/
    ├── editor-page.bp    # → pages/EditorPage.tsx
    ├── asset-panel.bp    # → components/panels/AssetPanel.tsx
    ├── character-panel.bp
    ├── settings-panel.bp
    └── timeline-panel.bp
```

### ファイル命名規則

| tsx のパス | .bp のファイル名 |
|---|---|
| `app/page.tsx` | `index.bp` |
| `app/(public)/works/page.tsx` | `works.bp` |
| `app/(public)/play/[id]/page.tsx` | `play-id.bp` |
| `app/(private)/mypage/messages/page.tsx` | `mypage-messages.bp` |
| `app/(private)/admin/users/[id]/page.tsx` | `users-id.bp` |
| `pages/EditorPage.tsx` | `editor-page.bp` |
| `components/panels/AssetPanel.tsx` | `asset-panel.bp` |

ルール:
- ディレクトリ区切りはハイフンに変換: `mypage/messages` → `mypage-messages`
- 動的パラメータは `-id` に変換: `[id]` → `-id`
- ルートの `page.tsx` は `index.bp`

## .bp ファイルのフォーマット

```
TARGET: <URL> — <tsx ファイルパス>
LAYOUT:
- <ブロック名>: <中身の要素>
- <ブロック名>: <中身の要素>
STATES: <該当する状態>
```

| フィールド | 必須 | 説明 |
|-----------|------|------|
| TARGET | Yes | URL + tsx ファイルパスの両方を書く |
| LAYOUT | Yes | UI ブロックの構成。`-` で列挙 |
| STATES | No | loading / empty / error / default のうち該当するもの |

### TARGET の書き方

```
# 基本
TARGET: /works — apps/next/app/(public)/works/page.tsx

# パネル・タブ単位で絞る場合
TARGET: /projects/editor/:workId (AssetPanel — アセットタブ) — apps/editor/src/components/panels/AssetPanel.tsx
```

URL だけでなく tsx パスを明記することで、AI が正確にファイルを特定できる。

### LAYOUT の書き方

```
LAYOUT:
- Header: H1 "作品一覧" + マイページリンク + ソートタブ（新着順 / 人気順）
- Main: WorkCard グリッド（4カラム、サムネイル / タイトル / 説明）
- Pagination: 前へ / ページ N / 次へ
```

記法:
- `/` — 並列する要素: `ソートタブ（新着順 / 人気順）`
- `+` — 連結する要素: `H1 "作品一覧" + マイページリンク`
- `()` — 補足説明: `WorkCard グリッド（4カラム）`
- `,` — 列挙: `ボタン（Admin, Messages, Logout）`

---

## 記述ルール

1. LAYOUT のブロック名は**画面に見える単位**で区切る（Header / Filter / List / Footer など）
2. ブロック内の要素は `/` `+` `()` `,` で区切る
3. **変えない部分も LAYOUT に書く**（AI に全体像を伝える）
4. **1 指示 = 1 TARGET**（複数画面をまたぐ場合は分割する）
5. パネル・タブ単位で TARGET を絞ると、複雑な画面でもブレない

## 使い方: Blueprint + 自然言語で改修指示

### ステップ 1: .bp ファイルを参照する

改修したい画面の `.bp` ファイルを開く。LAYOUT にブロック名が並んでいる。

### ステップ 2: 自然言語で指示を書く

LAYOUT のブロック名を使って位置を指定し、「〜して」「〜しないで」で指示を書く。

### ステップ 3: AI に渡す

```
以下の Blueprint と指示を反映して。
推測で仕様追加やリファクタはしない。
完了後、変更ファイル一覧を報告して。

<.bp の内容をコピペ>

<自然言語の指示>
```

### 例1: 作品一覧の検索UI簡素化

**Blueprint ファイル**: `blueprint/next/public/works.bp`

```
TARGET: /works — apps/next/app/(public)/works/page.tsx
LAYOUT:
- Header: H1 "作品一覧" + マイページリンク + ソートタブ（新着順 / 人気順）+ カテゴリフィルター pill（すべて + カテゴリ一覧）
- Main: WorkCard グリッド（4カラム、サムネイル / タイトル / 説明 / タグ / 作者 / 再生数 / いいね数）+ InListAd
- Pagination: 前へ / ページ N / 次へ
- StickyBottomAd: 広告枠
STATES: error バナー / loading スピナー / empty（作品がありません）/ default
```

**自然言語の指示**:
```
Header のソートタブに「いいね順」ボタンを追加して。
Header のカテゴリフィルター pill は横スクロールにして、折り返しはしない。
Main のカードサイズは変えないで。
```

**ポイント**: `.bp` のブロック名（Header, Main）を指示で参照するので、AI が「どのソートタブ？」で迷わない。

```
- EmptyState: フィルター別メッセージ
- DeleteConfirmSheet: 削除確認シート
```

**自然言語の指示**:
```

FilterButtons に「未使用のみ」チェックボックスを追加して。
SlugEditor の挙動は変えないで。
```

**ポイント**: TARGET にパネル名まで書くことで、EditorPage 全体ではなく1タブだけを対象と特定。

### 例3: ログイン画面のバリデーション改善

**Blueprint ファイル**: `blueprint/next/public/login.bp`

```
TARGET: /login — apps/next/app/(public)/login/page.tsx
LAYOUT:
- H1: ログイン
- Form: メールアドレス + パスワード + ログインボタン
- RegisterLink: 新規登録リンク
- InContentAd: 広告枠
STATES: success バナー（registered=true）/ error バナー / loading スピナー
```

**自然言語の指示**:
```
バリデーションエラーを error バナーではなく Form の各入力欄の直下に赤字で表示して。
Form のログインボタンは送信中に disabled + スピナーにして。
RegisterLink の文言は変えないで。
```

**ポイント**: LAYOUT のブロック名を使って「error バナーではなく Form の入力欄の直下に」と位置を正確に指定。

### 例4: マイページのプロジェクト一覧改修

**Blueprint ファイル**: `blueprint/next/private/mypage.bp`

```

- HeaderBanner: グラデーション + アバター頭文字 + ユーザー名 + メール + ボタン（管理画面 if admin / メッセージ（未読バッジ）/ ログアウト）
```

## Blueprint を書くときのコツ

| コツ | 理由 |
|-----|------|
| 変えないブロックも書く | AI が全体像を把握し、触らない範囲が明確になる |
| ブロック名を自然言語の指示で参照する | 「error バナーではなく Form の…」のように位置を正確に伝えられる |
| パネル・タブ単位で TARGET を絞る | EditorPage 全体より「AssetPanel」と書くほうがブレない |
| STATES は状態管理があるページだけ | 静的ページや設定画面では不要 |
| tsx が変わったら .bp も更新する | 対応が崩れると AI の推測ミスが復活する |

---

## 更新ルール

| タイミング | やること |
|-----------|---------|
| tsx の UI ブロック構成が変わった | 対応する `.bp` の LAYOUT を更新 |
| 新しいページ/パネルを追加した | 新しい `.bp` ファイルを作成 |
| ページ/パネルを削除した | 対応する `.bp` ファイルを削除 |
| URL やファイルパスが変わった | `.bp` の TARGET 行を更新 |

---

## まとめ

Blueprint 方式の本質:

1. **AI が迷うのは「どこに何があるか」の1点** — それだけを `.bp` で固定する
2. **指示は自然言語のまま** — 人間の記述コストはほぼゼロ
3. **tsx と 1:1 対応** — 構造が崩れない
4. **Figma でも Storybook でもない** — AI の入力として最適化された画面定義という新カテゴリ
