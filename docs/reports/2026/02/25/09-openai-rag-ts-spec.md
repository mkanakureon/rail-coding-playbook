# TS/Node だけで OpenAI API + RAG を実装する設計

> Generated by Claude Opus 4.6

## 背景

kaedevn（ノベルゲーム PF）は TypeScript で開発中。AI 機能（モデレーション・執筆支援・NPC 会話）を追加するにあたり、LangChain 等の AI ライブラリを使わず TS/Node + OpenAI API + PostgreSQL(pgvector) だけで RAG を実装する方針を整理した。

元資料: `docs/01_in_specs/0225/TS_Node だけで OpenAI API 呼び出し.md`

---

## 構成（TS のみで完結）

| レイヤー | 方式 |
|----------|------|
| **生成** | OpenAI Responses API（または Chat Completions）を HTTP で呼ぶ |
| **埋め込み** | OpenAI Embeddings API |
| **ベクトル検索** | PostgreSQL + pgvector（推奨: 運用が単純） |
| **RAG パイプライン** | チャンク化 → 埋め込み → upsert → 検索 → コンテキスト化 → 回答生成 |

```

  src/answer.ts            # context → 生成
```

## PostgreSQL スキーマ

pgvector 拡張を使用する。

```sql
-- 拡張
CREATE EXTENSION IF NOT EXISTS vector;

-- チャンク格納
CREATE TABLE rag_chunks (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id    text NOT NULL,
  work_id      text NOT NULL,
  scene_id     text NULL,
  source_id    text NULL,           -- JSON の docId など
  chunk_index  int  NOT NULL,
  content      text NOT NULL,
  content_hash text NOT NULL,        -- 差分更新用（sha256）

  -- scope 分離（越境防止の要）
  scope_type   text NOT NULL,        -- 'work' | 'tenant_policy' | 'global' | 'npc'
  scope_id     text NULL,            -- work_id / npc_id など

  -- メタ（検索で使うキーは列にも持つ）
  meta         jsonb NOT NULL DEFAULT '{}'::jsonb,

  -- 埋め込み（dimension は使うモデルに合わせる）
  embedding    vector(1536) NOT NULL,

  created_at   timestamptz NOT NULL DEFAULT now(),
  updated_at   timestamptz NOT NULL DEFAULT now()
);

-- ユニーク制約：同一ソースの同一チャンクを重複させない
CREATE UNIQUE INDEX rag_chunks_uq
  ON rag_chunks(tenant_id, work_id, source_id, chunk_index);

-- ベクトル検索インデックス（cosine 距離）
CREATE INDEX rag_chunks_embedding_ivfflat
  ON rag_chunks USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);

-- フィルタ用インデックス
CREATE INDEX rag_chunks_filter_idx
  ON rag_chunks(tenant_id, work_id, scene_id);

-- 監査・コスト管理
CREATE TABLE ai_requests (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id    text NOT NULL,
  user_id      text NOT NULL,
  endpoint     text NOT NULL,        -- 'moderate' | 'assist' | 'npc_chat'
  model        text NOT NULL,
  tokens_in    int  NOT NULL,
  tokens_out   int  NOT NULL,
  cost_usd     numeric(10,6),
  created_at   timestamptz NOT NULL DEFAULT now()
);
```

> `embedding` の次元（1536）は一例。実際に使うモデルに合わせて `vector(N)` を変更する。

## 類似検索 SQL（フィルタ込み）

```sql
SELECT
  id, content, meta,
  1 - (embedding <=> $1) AS score
FROM rag_chunks
WHERE tenant_id = $2
  AND work_id   = $3
  AND (scene_id = $4 OR $4 IS NULL)
ORDER BY embedding <=> $1
LIMIT $5;
```

## チャンク設計（ノベル PF 向け）

| 項目 | 方針 |
|------|------|
| サイズ | 300〜800 tokens / chunk |
| 境界 | 同一シーン内で作る（別作品・別シーンを混ぜない） |
| 種別 | `dialogue`（発話）/ `narration`（地の文）/ `command`（演出）を分けて抽出 |
| メタデータ | `type`, `speaker`, `lineNo`, `tags`, `r18`, `visibility` |

### 検索戦略

1. **フィルタ検索**: tenant → work → scene で絞る
2. **ベクトル類似**: topK = 8〜20
3. **再ランキング**（任意）: BM25 + ベクトルの加重（AI 不要、TS で実装可）

## 越境防止: scope 設計

RAG 検索は必ず以下のフィルタを通す。

| scope_type | 用途 | scope_id の例 |
|------------|------|---------------|
| `work` | 作品テキスト | `work_id` |
| `tenant_policy` | 規約・運営ルール | `tenant_id` |
| `global` | 共通ナレッジ（FAQ 等） | — |
| `npc` | NPC 固有知識 | `npc_id` |

## API 設計（3 機能、共通基盤）

すべての機能が内部で同じ `ai-gateway.generate()` + `rag-core.search(scope...)` を呼ぶ。

### Phase 1: 管理者向けモデレーション（最優先）

```
POST /admin/moderate
```

- 入力: 作品テキスト / 画像メタ
- 出力: フラグ（R18 / 暴力 / 個人情報）、理由、対応（非公開 / 要修正）
- RAG 不要でも動く。`ai_requests`（監査ログ）を最初に整備する
- 導入が最も軽い

### Phase 2: 作者向け執筆支援

```
POST /author/assist
```

- シーンの整合性チェック
- 登場人物の口調統一
- 伏線・設定の参照（RAG）
- scope: `work` のみ（作品内限定）

### Phase 3: 読者向け NPC 会話（最も難易度が高い）

```
POST /play/npc/chat
```

- セッション制御（1 会話 = 1 セッション、上限回数）
- プロンプト注入対策、出力制限
- NPC 記憶: 会話ログの要約 + 埋め込み（長期は要約で圧縮）
- scope: `npc` + `work`（作品設定）を合成検索
- 別途 `npc_memory` テーブルが必要（最初は JSONB で十分）

## 安全・運用ルール

### 必須

- **API で統制**: ブラウザ（Next.js / Editor）から直接 OpenAI / DB を叩かない
- **コスト上限**: tenant / user / endpoint ごと（読者: 1 日 n 回、作者: 月 m 回）
- **監査ログ**: `ai_requests` に全リクエストを記録
- **R18 / 公開範囲フィルタ**: DB 検索段階で強制
- **tenant_id フィルタ**: WHERE に必ず入れる（RAG の典型事故防止）

### 注意点

| 項目 | 対策 |
|------|------|
| コスト管理 | 埋め込みは更新差分のみ。`content_hash` で同一チャンクの再生成を防ぐ |
| プロンプト注入 | 検索結果は「引用扱い」。命令として実行しない方針を固定 |
| ストリーミング | Responses API の stream を使う（UX 向上） |
| ゲーム再生への直結 | しない。レイテンシ・障害時の UX 悪化を避けるため「作者向け補助」or「任意機能」から開始 |

## 導入ステップ

1. pgvector + `rag_chunks` / `ai_requests` テーブル追加
2. `packages/rag-core` 追加（検索だけ先に）
3. Hono に `POST /admin/moderate` 追加（Phase 1）
4. UI は「検索結果を作者が確認できる」程度から開始
5. Phase 2（執筆支援）→ Phase 3（NPC 会話）と段階ロールアウト
