### 1. `projects/slice-of-life` — 日常・青春の会話劇

**nano 適性: 最高（成功率 1 位）**

| 項目 | 内容 |
|------|------|
| ジャンル | `日常・青春` |
| YAML | `slice-of-life.yaml`（既存） |
| 章数 | 5 章 × 3 話 |
| 選択肢 | なし（`interactive: false`） |
| nano の強み | 掛け合い、空気感、テンポ |
| 事故リスク | 低（整合性要求が低い） |

#### コンセプト

**「最後の文化祭」** — 高校3年生の秋、文化祭の準備を通じて4人の関係が少しずつ変わる。
大事件は起きない。ただ、季節が変わり、卒業が近づく中で、
言えなかった言葉が少しずつ言葉になっていく。

#### 設計方針

- 1 シーン 1 テーマ（例：相談→決断、誤解→仲直り）
- 毎章に「季節の変化」を入れる（9月残暑→10月金木犀→11月初冬）
- キャラ 4 人。全員が主人公の友人だが、互いの関係にも動きがある
- 緊張要素はゼロ。代わりに「小さな気づき」を章ごとの開示情報にする
- 食べ物・飲み物の描写を多用（nano が得意な生活感）

#### 章構成案

| 章 | テーマ | 核になる出来事 |
|----|--------|-------------|
| 1 | 始動 | 文化祭の出し物が決まらない。4人の意見が割れる |
| 2 | 衝突 | 準備中に本音がぶつかる。一人が「もうどうでもいい」と言う |
| 3 | 距離 | 気まずい空気。別行動になるが、一人が差し入れを持ってくる |
| 4 | 収束 | 前日。全員が集まって夜通し作業。疲れた中で笑いが戻る |
| 5 | 当日 | 文化祭本番。完璧ではないが、4人にしかわからない達成感 |

### 2. `projects/fantasy` — ファンタジー冒険（クエスト型）

**nano 適性: 高（成功率 4 位）**

| 項目 | 内容 |
|------|------|
| ジャンル | `ファンタジー・冒険` |
| YAML | `fantasy.yaml`（既存） |
| 章数 | 5 章 × 3 話 |
| 選択肢 | なし（`interactive: false`） |
| nano の強み | 目的→移動→障害→報酬の定型進行 |
| 事故リスク | 中（固有名詞の管理が必要） |

#### コンセプト

**「枯れた泉の守り人」** — 辺境の村の泉が枯れた。
薬草師の見習い・ルカは、泉の源流を辿って「大樹の根」を目指す旅に出る。
道連れは口の悪い元兵士と、言葉を話す狐。
世界を救う壮大な話ではなく、小さな村のための小さな冒険。

#### 設計方針

- 固有名詞は最小限：地名 4 つ、種族名 2 つ、アイテム名 3 つ以内
- 魔法体系は「1 キャラ 1 能力」に限定（破綻防止）
- 章ごとに「移動→遭遇→解決」の定型
- 戦闘より「交渉」「知恵」で解決する展開（nano は戦闘描写で破綻しやすい）
- 用語辞書を world.md で固定

### 3. `projects/comedy` — コメディ会話劇

**nano 適性: 高（成功率 5 位）**

| 項目 | 内容 |
|------|------|
| ジャンル | `コメディ・日常` |
| YAML | `slice-of-life.yaml` + `_default.yaml`（コメディ専用は不要） |
| 章数 | 5 章 × 3 話 |
| 選択肢 | なし（`interactive: false`） |
| nano の強み | ノリ、勢い、勘違い・すれ違い・ツッコミの型 |
| 事故リスク | 中（同じギャグの繰り返しに注意） |

#### コンセプト

**「幽霊部員が多すぎる」** — 廃部寸前の「何でも研究部」に、
なぜか毎週新しい幽霊部員が現れる。全員「入部届は出した」と言い張るが、
部長の手元には1枚もない。5人の自称部員による勘違いコメディ。

#### 設計方針

- 1 章 1 騒動（毎回小さい問題が起きて、全員の勘違いで拡大し、偶然解決する）
- 「繰り返しのギャグ」は 1 章 1 回まで（nano のギャグ擦り防止）
- 各章で新しい「勘違いパターン」を導入（同じ型を使い回さない）
- ツッコミ役を固定（部長）、ボケは分散（全員がボケ要素を持つ）
- シリアス要素ゼロ。最後まで軽く終わる

| 2 | 部室争奪戦 | 隣の部と部室がダブルブッキング。交渉が全て裏目に出る |
| 3 | 文化祭の出し物 | 5人が別々の出し物を準備している。誰も確認しない |
| 4 | 合宿 | 行き先が5人とも違う。全員が正しい集合場所にいると信じている |
| 5 | 存続審査 | 顧問の先生に活動報告するが、活動内容の認識が全員バラバラ |

## 実装チェックリスト

### 共通（各プロジェクト）

- [ ] `settings/overview.md` — frontmatter + 概要（`interactive: false`, `chapters: 5`）
- [ ] `settings/characters.md` — キャラ設定（固有情報・機能を含む）
- [ ] `settings/world.md` — 場所の五感描写
- [ ] `settings/plot.md` — 5章の構成と章ごとの必須開示情報
- [ ] Mock テスト実行確認（`npx tsx scripts/assist-cli.ts all --settings <path> --mock`）

### ジャンルルール YAML

| ジャンル | YAML | 対応 |
|---------|------|------|
| 日常・青春 | `slice-of-life.yaml` | 既存で十分 |
| ファンタジー | `fantasy.yaml` | 既存で十分 |
| コメディ | なし | `comedy.yaml` を新規作成 |

### comedy.yaml の設計案

```yaml
keywords:
  - コメディ
  - ギャグ
  - お笑い

stage2Rules:
  - 各話に勘違い・すれ違いの起点を1つ設定する
  - 問題の拡大→収束のリズムを守る（起点→混乱→偶然の解決）
  - 同じギャグパターンを2話連続で使わない

descriptionRules:
  - テンポ重視。地の文は短く、会話の比率を高くする
  - ツッコミの地の文は1文で完結させる
  - 大げさな比喩をコメディ効果として積極的に使う

characterRules:
  - ツッコミ役の内心ツッコミを地の文に入れる
  - 各キャラのボケパターンを固定し、予測可能性で笑いを取る
  - キャラ同士の勘違いが連鎖する構造にする

directionRules:
  - オチは各話の最後に必ず入れる
  - シリアスに見せかけてコメディで落とすフェイントを1章1回入れる
  - 最終章は全キャラの勘違いが同時に発覚する構成にする
```

## 作成順序

| 順 | プロジェクト | 理由 |
|----|------------|------|
| 1 | `slice-of-life` | nano 最適性 1 位。YAML 既存。最も安定した出力が期待できる |
| 2 | `fantasy` | 定型進行で nano と相性良。YAML 既存 |
| 3 | `comedy` | YAML 新規作成が必要。ギャグの品質は実 API テストで要確認 |

## 検証計画

各プロジェクト作成後、以下の順で検証する。

1. **Mock テスト** — パイプラインが通ることを確認
2. **実 API テスト（nano）** — `npx tsx scripts/assist-cli.ts all --settings <path>` で全章生成
3. **品質レビュー** — 生成された `全話テキスト.md` を読み、以下を確認:
   - ジャンルの雰囲気が出ているか
   - 章ごとの必須開示情報が反映されているか
   - 同じ描写・台詞の繰り返しがないか
   - キャラの差別化ができているか
4. **ジャンル横断比較** — 全ジャンルの出力を並べて nano の得意・不得意を分析
