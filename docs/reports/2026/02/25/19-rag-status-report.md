# AI 執筆パイプラインにおける RAG の現状

> Generated by Claude Opus 4.6

## 結論

RAG（Retrieval-Augmented Generation）は**使っていない**。設定ファイルを**全文挿入**でプロンプトに注入する方式で動いている。現状の規模（5章完結・設定ファイル4本）ではこの方式で十分であり、変更の必要はない。

## 現在のアーキテクチャ

### 設定ファイルの注入方式

| ファイル | パース先 | プロンプトへの注入方法 |
|---------|---------|-------------------|
| `overview.md` | `settings.overview` | 各ステージの system prompt に概要テキストとして丸ごと挿入 |
| `characters.md` | `settings.characters[]` | キャラごとに名前・slug・口調・性格・経歴を展開して挿入 |
| `world.md` | `settings.worldSetting` | 全文を世界観として挿入 + `extractLocationDetail()` でシーンの場所に該当する五感描写を抽出 |
| `plot.md` | `settings.plotOutline` | Stage 0/1 で章構成の指針として全文挿入 |
| `history.md` | `settings.historicalFacts` | 歴史的事実として挿入（存在する場合のみ） |
| ジャンル YAML | `resolveGenreRules()` | Stage 2/3 でジャンル固有ルール（description/character/direction）を挿入 |

### 各ステージでの使われ方

```
Stage 0 (skeleton)  : overview + plot + characters（名前リスト）
Stage 1 (chapters)  : overview + plot + characters + world + genre
Stage 2 (episodes)  : overview + characters + world + genre YAML rules + chapter context
Stage 3 (scene text): overview + characters（該当キャラのみ） + world（該当場所の五感描写） + genre rules
Stage 4 (.ks変換)   : characters（名前→slug マッピング）のみ。LLM不使用
```

### extractLocationDetail() — 唯一の「検索的」処理

`world.md` の全文をプロンプトに入れるのではなく、シーンの `location` に該当する箇所だけを抽出する関数がある。

```typescript
// prompts.ts
if (settings.worldSetting) {
  const locationDetail = extractLocationDetail(settings.worldSetting, scene.location);
  if (locationDetail) {
    // 該当場所の五感描写だけを system prompt に追加
  }
}
```

これはベクトル検索ではなく、Markdown の見出しとキーワードの文字列マッチ。RAG ではないが、「必要な部分だけ取得する」という思想は同じ。

## RAG を導入しない理由

### 1. コンテキスト長に収まる

| 設定ファイル | 平均サイズ | 5章プロジェクト合計 |
|------------|----------|-----------------|
| overview.md | 500〜800字 | — |
| characters.md | 1,500〜3,000字 | — |
| world.md | 1,500〜2,500字 | — |
| plot.md | 2,000〜4,000字 | — |
| **合計** | — | **約6,000〜10,000字** |

gpt-4.1-nano のコンテキスト長は 128K トークン。設定ファイル全文（約1万字 ≒ 5,000〜7,000トークン）は余裕で収まる。

### 2. 分割するとかえって品質が下がる

RAG でチャンクに分割すると、以下のリスクがある：

- キャラの性格と口調が別チャンクに分かれ、一貫性が崩れる
- 世界観の全体像がなくなり、場所間の関係性が見えなくなる
- プロットの伏線と回収が別チャンクになり、整合性チェックができない

5章完結の短編では、全文を入れた方が LLM が文脈を理解しやすい。

### 3. インフラが不要

RAG を導入すると以下が必要になる：

- ベクトル DB（Pinecone, pgvector, Qdrant 等）
- Embedding API の呼び出し（追加コスト）
- チャンク分割・インデックス更新のパイプライン

現状のファイル読み込み → 全文挿入は、追加インフラなしで動く。

## RAG が必要になるケース（将来）

以下の条件に該当した場合、RAG 導入を検討する：

| 条件 | 閾値 | 理由 |
|------|------|------|
| 章数の増加 | 20章以上 | plot.md が肥大化し、コンテキストを圧迫する |
| キャラ数の増加 | 15人以上 | characters.md の全文挿入でトークンが膨れる |
| 世界観の複雑化 | 地名20箇所以上 | world.md から該当箇所の抽出精度が必要 |
| 複数プロジェクトの横断 | 設定の共有 | プロジェクト間で共通設定を検索する必要 |
| ユーザー入力からの検索 | 自由文→設定マッチ | ユーザーが「あのキャラ」と言ったら該当設定を返す |

## まとめ

- 現在の方式: **設定ファイル全文をプロンプトに挿入**（RAG なし）
- 唯一の検索的処理: `extractLocationDetail()` による場所の文字列マッチ抽出
- 変更不要の理由: 5章完結・設定4ファイル・合計1万字以下はコンテキストに余裕で収まる
- 導入判断基準: 章数20以上 or キャラ15人以上 or 地名20箇所以上になったら再検討
