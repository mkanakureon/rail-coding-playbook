# AI 執筆支援プロンプトパイプライン設計書

> Generated by Claude Opus 4.6

## 概要

4 Stage + 5 補助プロンプトの計 9 種類のプロンプトで、設定ファイルからノベルゲームのシナリオ（.ks）を自動生成する。

```
設定ファイル群 (.md)
    │
    ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 1   章プロット生成                     ×1 回      │
│  ────────────────────────────────────────────────────── │
│  入力: 作品設定(全文) + キャラ設定(全文)                  │
│  出力: chapters[] (JSON)                                │
│  API: 1 call                                            │
└──────────────┬───────────────────────────────────────────┘
               │ chapters.json
               ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 2   話プロット生成                     ×章数      │
│  ────────────────────────────────────────────────────── │
│  入力: 設定(概要) + キャラ + 全章プロット                 │
│       + 前章あらすじ + 未解決伏線 + 必須新情報            │
│  出力: episodes[].scenes[] (JSON)                       │
│  API: 1 call / 章                                       │
└──────────────┬───────────────────────────────────────────┘
               │ ch{N}_episodes.json
               ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 3   シーンテキスト生成          ×(章数×話数×シーン数) │
│  ────────────────────────────────────────────────────── │
│  入力: シーンプロット + 口調ガイド + 場所詳細             │
│       + 前シーン要約 + 前話あらすじ + キャラ状態          │
│       + 前章あらすじ + 未解決伏線 + ジャンルルール         │
│  出力: lines[] (JSON)                                   │
│  API: 1 call / シーン + 1 要約 + 1 キャラ状態抽出        │
│       = 3 calls / シーン                                │
├──────────────────────────────────────────────────────────┤
│  補助: シーン要約      (各シーン後、次シーンのコンテキスト) │
│  補助: キャラ状態抽出  (各シーン後、累積更新)             │
│  補助: 話要約          (各話後、次話のコンテキスト)       │
└──────────────┬───────────────────────────────────────────┘
               │ ch{N}_ep{M}_text.json
               ▼
┌──────────────────────────────────────────────────────────┐
│  Stage 4   .ks スクリプト変換               ×話数        │
│  ────────────────────────────────────────────────────── │
│  入力: Stage 3 の JSON                                  │
│  出力: .ks ファイル                                      │
│  API: 0 calls（テンプレート変換、決定的処理）             │
└──────────────┬───────────────────────────────────────────┘
               │
               ▼
┌──────────────────────────────────────────────────────────┐
│  章末処理                                     ×章数      │
│  ────────────────────────────────────────────────────── │
│  補助: 章要約          (章全体のまとめ → 次章コンテキスト) │
│  補助: 伏線抽出・更新  (open/resolved の追跡)            │
│  API: 2 calls / 章                                      │
├──────────────────────────────────────────────────────────┤
│  篇末処理（chaptersPerArc 章ごと）            ×篇数      │
│  ────────────────────────────────────────────────────── │
│  補助: 篇要約          (複数章を圧縮 → 長期コンテキスト)  │
│  補助: キャラ状態圧縮  (knowledge 最大5件に絞る)          │
│  API: 2 calls / 篇                                      │
└──────────────────────────────────────────────────────────┘
```

## API コール数の計算式

```
1 章あたりの API コール数:
  Stage 1: 0 (全章で 1 回だけ)
  Stage 2: 1
  Stage 3: シーン数 × 3 (テキスト + シーン要約 + キャラ状態)
           + 話数 × 1 (話要約)
  Stage 4: 0 (テンプレート)
  章末:    2 (章要約 + 伏線抽出)
  篇末:    2 / chaptersPerArc (篇要約 + キャラ圧縮)

例: 3話/章 × 3シーン/話 = 9シーン
  → 1 + 27 + 3 + 2 + 0.67 ≈ 34 calls/章
  → 3章で Stage1 含め 1 + 102 = 103 calls
```

## 各プロンプトの詳細

### Stage 1: buildStage1Prompt

**目的**: 作品全体の章プロットを一括生成

| 項目 | 内容 |
|------|------|
| 関数 | `buildStage1Prompt(settings)` |
| API | 1 call（全体で 1 回） |
| JSON モード | あり |
| ソース | `prompts.ts:102-140` |

**system プロンプト構成**:

```
ロール: ノベルゲームのシナリオライター
タスク: {N}章の章プロットを作成

出力形式: JSON { chapters: [{ chapterNumber, title, summary, keyEvents, emotionalArc, activeCharacters }] }

ルール:
- 各章に明確な目的と転換点
- 起承転結を意識
- ラフプロットがある場合はそれを尊重
- activeCharacters には slug を使う
- 各章の keyEvents は前章と重複しない新規の具体的事実を最低3つ    ← NEW
- 情報は章ごとにエスカレーション                                   ← NEW
- 同じ発見や同じリアクションを複数章で繰り返さない                   ← NEW
```

**user プロンプト構成**:

| セクション | ソース | 備考 |
|-----------|--------|------|
| 作品設定 | `formatSettings(settings)` | 概要・ジャンル・時代・トーン・世界観・歴史・ラフプロット |
| キャラクター | `formatCharacters(settings)` | 名前・slug・役割・性格・口調・台詞パターン・微反応・内面・固有情報・経歴・関係 |

### Stage 2: buildStage2Prompt

**目的**: 1 章を複数話×シーンに分割

| 項目 | 内容 |
|------|------|
| 関数 | `buildStage2Prompt(settings, chapters, targetChapter, prevSummaries?, plotThreads?)` |
| API | 1 call / 章 |
| JSON モード | あり |
| ソース | `prompts.ts:144-218` |

**system プロンプト構成**:

```
ロール: ノベルゲームのシナリオライター
タスク: 第{N}章を{M}話の話プロットに分割

出力形式: JSON { episodes: [{ episodeNumber, title, summary, scenes: [{ sceneNumber, location, time, characters, event, hasChoice }], endHook }] }

ルール:
- 各話に 2〜4 シーン
- 1 話あたり 1〜2 箇所の選択肢シーン
- characters には slug を使う
- location は作品設定の場所名をそのまま使う
- [ジャンルルール: stage2Rules]                                     ← YAML 注入
```

**user プロンプト構成**:

| セクション | ソース | 条件 |
|-----------|--------|------|
| 作品設定（概要） | `formatSettings(settings, true)` | 常に |
| キャラクター | `formatCharacters(settings)` | 常に |
| 前章までのあらすじ | `previousChapterSummaries` | 第 2 章以降 |
| 未解決の伏線 | `plotThreads (open)` | あれば |
| 全章プロット | `chapters` 全件の要約 | 常に |
| 対象章 | `chapter.summary + keyEvents` | 常に |
| この章で必ず導入する新情報 | `extractChapterNewInfo(roughPlot, N)` | plot.md に記載あれば ← NEW |

### Stage 3: buildStage3Prompt

**目的**: 1 シーンの地の文・セリフ・選択肢を生成

| 項目 | 内容 |
|------|------|
| 関数 | `buildStage3Prompt(settings, scene, episode, prevContext, prevEpSummary?, retrieved?, charStates?, prevChSummaries?, plotThreads?)` |
| API | 1 call / シーン |
| JSON モード | あり |
| ソース | `prompts.ts:222-338` |

**system プロンプト構成**:

```
ロール: ノベルゲームのシナリオライター
タスク: シーンプロットに基づいてテキストを書く

出力形式: JSON { lines: [ { type: "narration"/"dialogue"/"choice", ... } ] }

ルール:
- セリフは各キャラの口調を忠実に再現
- 同じ台詞パターンを連続使用しない
- 地の文は 10〜30 行
- dialogue.speaker は slug、ただし地の文にはキャラ名を日本語で書く   ← NEW
- 選択肢は必ず 2 つ以上、異なる行動と異なる結果                     ← NEW
- 選択肢の result は具体的な展開を書く                              ← NEW
- 前シーンの描写表現を繰り返さない                                  ← NEW
- キャラの固有情報をそのキャラの視点として反映                       ← NEW

描写ルール: [ジャンルルール: descriptionRules]                       ← YAML 注入
キャラクター描写ルール: [ジャンルルール: characterRules]              ← YAML 注入
演出ルール: [ジャンルルール: directionRules]                         ← YAML 注入
```

**user プロンプト構成**:

| セクション | ソース | 条件 |
|-----------|--------|------|

| キャラクター口調ガイド | `sceneChars` の性格・口調・固有情報 | 常に |
| この話のあらすじ | `episode.summary` | 常に |
| このシーンのプロット | 場所・時間帯・登場人物（名前形式）・展開・選択肢有無 | 常に |

## コンテキスト管理の仕組み

```
┌─── 篇 (Arc) ───────────────────────────────────────┐
│                                                     │
│  篇要約 (500〜800字)      ← 篇末に生成              │
│  圧縮キャラ状態           ← 篇末に圧縮              │
│                                                     │
│  ┌─── 章 (Chapter) ──────────────────────────────┐  │
│  │  章要約 (300〜500字)  ← 章末に生成             │  │
│  │  伏線リスト           ← 章末に更新             │  │
│  │                                               │  │
│  │  ┌─── 話 (Episode) ─────────────────────────┐ │  │
│  │  │  話要約 (100〜200字) ← 話末に生成         │ │  │
│  │  │                                          │ │  │
│  │  │  ┌─── シーン (Scene) ──────────────────┐ │ │  │
│  │  │  │  シーン要約 (200字)  ← シーン後      │ │ │  │
│  │  │  │  キャラ状態          ← シーン後(累積) │ │ │  │
│  │  │  └────────────────────────────────────┘ │ │  │
│  │  └──────────────────────────────────────────┘ │  │
│  └────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**Stage 2/3 に渡される有効コンテキスト**:

```
effectiveSummaries = [過去の篇要約...] + [現在の篇内の章要約...]
```

篇をまたぐと章要約は篇要約に圧縮され、`currentArcChapterSummaries` がリセットされる。
これによりコンテキストサイズの線形膨張を防ぐ。

## ジャンルルール注入の仕組み

```
apps/hono/src/lib/assist/genre-rules/
  _default.yaml       ← 全ジャンル共通（常に結合）
  mystery.yaml         ← keywords: [ミステリー, サスペンス, 推理, 探偵]
  romance.yaml         ← keywords: [恋愛, ラブコメ, 純愛]
  fantasy.yaml         ← keywords: [ファンタジー, 異世界, 魔法]
  horror.yaml          ← keywords: [ホラー, 怪談, 恐怖]
  slice-of-life.yaml   ← keywords: [日常, ほのぼの, 青春]
```

**解決ロジック** (`genre-rules.ts`):

1. 全 YAML ファイルを読み込み（初回のみ、以降キャッシュ）
2. `settings.genre` 文字列に含まれる keyword でマッチ
3. 複合ジャンル対応（例: `群像劇・ミステリー` → mystery がマッチ）
4. マッチしたルール + `_default` を結合（重複排除）
5. 結合結果を Stage 2 の `stage2Rules` と Stage 3 の `descriptionRules` / `characterRules` / `directionRules` に注入

## 設定ファイル → プロンプトのマッピング

| 設定ファイル | パースされるフィールド | 使用される Stage |
|-------------|----------------------|-----------------|
| `overview.md` | YAML frontmatter (genre, era, tone, themes, chapters, episodes_per_chapter, chapters_per_arc) + 本文 (overview) | 1, 2, 3 |
| `characters.md` | name, slug, role, personality, speechStyle, 台詞パターン, 微反応, 内面, 固有情報, 経歴, 関係 | 1, 2, 3 |
| `world.md` | 場所ごとの五感情報 (視覚/聴覚/嗅覚/触覚/温度) | 3 (場所詳細) |
| `plot.md` | ラフプロット + 篇ごとの緊張核・開示する矛盾・章ごとの必須新情報 | 1 (全文), 2 (必須新情報) |
| `history.md` | 歴史的事実 + 矛盾・疑問点 + 段階開示テーブル | 1 (全文) |
| `assets.md` | 場所名 → アセット slug マッピング | 4 (.ks 生成) |
