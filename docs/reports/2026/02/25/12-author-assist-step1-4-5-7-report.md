## 実装の詳細

### Step 1: `packages/ai-gateway`

LLM プロバイダーを統一インターフェースで扱うパッケージ。

```typescript
// LLMClient インターフェース
interface LLMClient {
  readonly provider: Provider;
  generate(params: GenerateParams): Promise<GenerateResult>;
}

// ファクトリ
createClient('openai', { apiKey, defaultModel })   → OpenAIClient
createClient('anthropic', { apiKey, defaultModel }) → AnthropicClient
createClient('mock')                                → MockClient
```

**プロバイダー別の JSON mode 対応**:

| プロバイダー | JSON mode の実装 |
|---|---|
| OpenAI | `response_format: { type: "json_object" }` |
| Anthropic | system prompt に `<json>` タグ指示 → レスポンスから抽出 |
| Mock | 直接 JSON 文字列を返す |

**リトライ対応**: OpenAI（429/5xx）、Anthropic（429/529/5xx）に対して `maxRetries` 回リトライ。`retry-after` ヘッダーも尊重。

**MockClient のステージ検出ロジック**:

```
system prompt を順に検査:
  1. "要約" を含む      → { summary: "..." }
  2. "話プロット" を含む → { episodes: [...] }  ← 先に検査（Stage 2 の user には "章プロット" が含まれるため）
  3. "章プロット" を含む → { chapters: [...] }
  4. "テキスト" を含む   → { lines: [...] }
  5. どれにも一致しない  → { message: "..." }
```

### Step 4: 型定義 + Zod + .md パーサー + プロンプト

**5 ファイル**を `apps/hono/src/lib/assist/` に作成。

#### types.ts — 型定義

```
WorkSetting          作品設定（overview, genre, characters, assetMapping 等）
CharacterSetting     キャラ設定（name, slug, role, personality, speechStyle 等）
AssetMapping         場所名 → bg slug のマッピング
Stage1Result         章プロット配列
Stage2Result         話プロット配列（scenes 付き）
Stage3Result         話テキスト（scenes > lines 配列）
Line                 narration | dialogue | choice の判別共用体
ChoiceOption         選択肢（flagSet, jumpTo は nullable で将来拡張対応）
```

#### schemas.ts — Zod バリデーション

各ステージ出力の構造・制約をスキーマで定義:

| スキーマ | 主な制約 |
|---|---|
| `Stage1ResultSchema` | chapters: 1〜10件、summary: 30〜800字、keyEvents: 1〜5件 |
| `Stage2ResultSchema` | episodes: 1〜10件、scenes: 1〜5件、summary: 20〜600字 |
| `Stage3ResultSchema` | discriminatedUnion で narration/dialogue/choice を判定、choice: 2〜4選択肢 |
| `SummaryResultSchema` | summary: 10〜500字 |

#### parser.ts — JSON 抽出 + バリデーション

```
extractJson(raw)  →  3段階で JSON を抽出:
  1. 直接 JSON.parse（OpenAI response_format の場合）
  2. <json>...</json> タグ抽出（Anthropic の場合）
  3. ```json...``` コードブロック抽出（フォールバック）

parseStageResult(raw, schema)  →  extractJson + Zod safeParse
  失敗時は詳細なパス付きエラーメッセージを返す
```

#### md-parser.ts — .md ファイルパーサー

```
parseFrontmatter(content)     YAML frontmatter を分離（インライン配列、ブロック配列、数値対応）
parseCharacters(content)      ## Name（slug） ヘッダー + 箇条書きをパース
parseAssetMapping(content)    マークダウンテーブルから場所マッピング抽出
loadWorkSettings(dir)         ディレクトリから全 .md を読み込み → WorkSetting に統合
```

#### prompts.ts — プロンプトテンプレート

| 関数 | ステージ | system の特徴 |
|---|---|---|
| `buildStage1Prompt` | 章プロット | ジャンル・テーマ・全設定を含む。JSON 出力形式を明示 |
| `buildStage2Prompt` | 話プロット | 対象章 + 全章コンテキスト。シーン構成の制約ルール |
| `buildStage3Prompt` | テキスト | 該当キャラのみ口調ガイド。前シーン/前話コンテキスト対応 |
| `buildSummaryPrompt` | 要約 | 100〜200字で次話向けあらすじ生成 |

### Step 5: .ks ジェネレーター

LLM を使わず TypeScript テンプレートで変換:

```
narration  →  テキスト\n@l\n
dialogue   →  {名前}：{セリフ}\n@l\n   （slug → 表示名に変換、全角コロン）
choice     →  choice {\n  "選択肢" {\n    @set flag += value\n    結果\n    @l\n  }\n}
シーン先頭 →  @bg {bg_slug} fade 500    （assetMapping がある場合のみ）
シーン末尾 →  @p
```

### Step 7: CLI テストスクリプト

```bash

npx tsx scripts/assist-cli.ts chapters  --settings projects/demo/settings/
npx tsx scripts/assist-cli.ts episodes  --settings projects/demo/settings/ --chapters output/chapters.json --target 1

npx tsx scripts/assist-cli.ts ks        --settings projects/demo/settings/ --text output/ch1_ep1_text.json

# 全ステージ一気通貫
npx tsx scripts/assist-cli.ts all       --settings projects/demo/settings/
```

```
3章 × 3話/章 = 9話分を生成

  ├── ch{N}_ep{M}_text.json   (Stage 3: テキスト + 自動要約)

```

## デバッグで判明した問題と修正

### 1. MockClient のステージ検出順序

**問題**: Stage 2 のプロンプトの user に「全章プロット」というテキストが含まれるため、Mock が「章プロット」（Stage 1）に先にマッチして `{ chapters: [...] }` を返していた。

**修正**:
- 検出対象を `combined`（system + user）から **`system` のみ** に変更
- Stage 2 の検出を Stage 1 より先に配置

### 2. 要約プロンプトとテキストプロンプトの競合

**問題**: 要約プロンプトの system に「テキストを…要約」と書いてあるため、「テキスト」のキーワードで Stage 3 テキスト生成にマッチしていた。

**修正**: 要約チェックを Stage 3 テキストチェックの**前**に配置。

### 3. Zod v4 の `.optional().default()` 挙動

**問題**: テストデータの `summary` が Zod の `min()` 制約を満たしていなかった。`.optional().default(false)` は Zod v4 でも問題なく動作するが、テストデータの方が不備だった。

**修正**: テストデータの summary を制約を満たす長さに変更。スキーマは `.default(false)` に簡略化。

## 教訓

1. **Mock のキーワード検出は system prompt のみで行う** — user prompt には前ステージの出力がコンテキストとして含まれるため、意図しないステージにマッチする。検出順序も、より具体的なキーワードを先にチェックする。

2. **Zod のテストデータは制約を満たす長さにする** — `min(30)` のような文字数制約があるスキーマに対して、テストデータが短いと「テストの問題」なのか「スキーマの問題」なのか切り分けが面倒になる。正常系テストデータは制約を余裕を持って満たすようにする。

3. **CLI テスト先行の効果** — DB・API 不要で全ステージの動作確認ができたため、プロンプトテンプレートやスキーマの問題を素早く検出・修正できた。API 統合時にはコアロジックの信頼性が確保された状態で進められる。
