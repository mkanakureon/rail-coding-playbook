## 発見したアンチパターン

### 1. `expect([200, 500]).toContain()` — 絶対に失敗しないテスト

6 ファイルで発見。HTTP ステータスが 200 でも 500 でもテストが通る。

**発生原因:** テスト環境に DB がなく、Prisma がエラーを返して 500 になる。「ルーティングが通ることだけ確認」という判断で `[200, 500]` を許容していた。

**該当ファイル:**

| ファイル | パターン | 修正 |
|---------|---------|------|
| `official-assets.test.ts` | `expect([200, 500]).toContain()` x6 | Prisma モック → exact 200 |
| `users.test.ts` | `expect([200, 500]).toContain()` | テスト削除（DB依存） |
| `works.test.ts` | `expect([200, 500]).toContain()`, `expect([404, 500]).toContain()` | テスト削除（DB依存） |
| `api-structure.test.ts` | `expect([200, 500])`, `expect([400, 401])`, `expect([401, 404])` 等 | exact ステータス + メッセージ検証 |
| `assets.test.ts` | `expect([401, 404, 405]).toContain()` | exact 401 |
| `preview.test.ts` | `expect([404, 500]).toContain()` | Prisma モック → exact 404 |

### 2. `toHaveProperty('error')` のみ — メッセージ内容未検証

約 20 箇所で `body.error` の存在だけを確認し、メッセージ内容を検証していなかった。

**修正:** `expect(body.error).toContain('認証が必要')` 等、具体的なメッセージ検証を追加。

### 3. `expect([400, 429]).toContain()` — レートリミッターとの競合

`auth.test.ts` と `contact.test.ts` で、テスト実行順によってレートリミッターが発動し、バリデーションテストが 429 を返す問題。

**修正:** テスト数をレートリミットの上限内に収めた（register: 3件/hour → 3テストに削減、contact: 5件/hour → 5テストに削減）。

### 4. `if (status === 400) ... else expect(429)` — 条件分岐アサーション

`contact.test.ts` で、レスポンスのステータスによって異なるアサーションを実行。片方のパスしか通らない。

**修正:** テスト数を制限内に収め、exact 400 で検証。

## 教訓

1. **`expect([200, 500])` は何もテストしていない** — 正常でも異常でも通るテストは価値がゼロ
2. **DB 依存はモックで排除する** — `vi.mock` で Prisma を差し替えれば DB なしでも正確なテストが書ける
3. **レートリミッターとテストの競合に注意** — テスト数がリミット上限を超えると結果が非決定的になる
4. **テストの定期監査が必要** — 弱いアサーションは書いた時点では合理的に見えるが、時間とともにリスクが蓄積する
5. **Live テストは TypeScript で書く** — curl より Vitest のアサーション機能が豊富で CI に組み込みやすい
