## 方針

1. **output 形式は変更しない** — `Stage3Result`, `Line`, `.ks` 生成は既存のまま
2. **パイプラインのフロー・ステージ数・API 呼び出し回数は変更しない**
3. 改善手段は 2 つ:
   - **インプット文書**（`projects/longstory/settings/`）の情報密度向上
   - **ジャンル別 YAML 外部ルール** + TS プロンプトへの注入

### 設計上の重要な意思決定

| 論点 | 検討 | 決定 |
|------|------|------|
| ジャンル固有ルールの配置 | TS ハードコード vs 外部ファイル | **外部 YAML** — ビルド不要で差替え可能 |
| 外部ファイル形式 | JSON vs YAML | **YAML** — 日本語の可読性、コメント可、編集ミスしにくい |
| TS の役割 | ルール定義 + 注入 | **ローダー + 注入のみ**（ルールの中身は一切持たない） |
| ルールの作者への公開 | 見せる vs 見せない | **見せない** — システム側だけで管理 |
| `_default.yaml` の扱い | フォールバックのみ | **常にマッチ結果に結合** — 共通ルールが全ジャンルに乗る |
| frontmatter パーサー | 自前維持 vs js-yaml | **js-yaml に置換** — ネスト YAML 対応の副次効果 |

## 教訓

- **プロンプトの文言差し替えだけなら TS にハードコードしない** — 外部ファイルにすればビルド不要で調整可能
- **YAML > JSON**（この用途では）— 日本語の長文ルールを人が読み書きする場合、引用符不要・コメント可の YAML が優位
- **`_default.yaml` を常に結合する設計**にすれば、各ジャンル YAML には固有ルールだけ書けばよく、共通ルールの重複が消える
- **レビュー指摘の「プロンプトが良いから成立している」は重要な示唆** — モデル性能ではなく設計品質がボトルネック。プロンプト改善は直接的に出力品質に繋がる
