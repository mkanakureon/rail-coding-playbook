# AI 執筆支援システム — 破綻なく生成できる文字数の分析

> Generated by Claude Opus 4.6

## 背景

AI 執筆支援 v1（4 Stage パイプライン + RAG）が完成した。プロット設計からシーンテキスト生成、.ks スクリプト変換までの全工程が MockClient で動作する。

次のステップとして、**このシステムで物語の一貫性を保ったまま生成できる文字数の上限**を見積もる必要がある。トークン上限ではなく、LLM 呼び出し間のコンテキスト伝達設計がボトルネックになるため、その分析を行った。

## パイプライン構造

```
Stage 1 (gpt-4o-mini)   : 作品設定 → 章プロット         [1回の呼び出し]
Stage 2 (gpt-4o-mini)   : 章プロット → 話プロット         [章ごとに1回]
Stage 3 (claude-sonnet)  : シーンプロット → テキスト       [シーンごとに1回]
Summary (gpt-4o-mini)    : 話テキスト → 前話のあらすじ      [話ごとに1回]
Stage 4 (TSテンプレート)  : テキスト → .ks               [LLM不使用]
```

各ステージの出力トークン上限は **4,096 トークン**。入力コンテキスト窓は gpt-4o-mini が 128K、claude-sonnet が 200K。いずれも十分大きく、トークン上限がボトルネックになることはない。

## ボトルネック: LLM 呼び出し間の情報伝達

破綻の原因はトークン制限ではなく、**シーン間で引き継がれるコンテキストの情報量**にある。

### 現在の引き継ぎ設計

| 引き継ぎ区間 | 渡されるもの | 情報量 | 保持率 |
|------------|------------|-------|-------|
| シーン → シーン（話内） | `previousContext`: 直前シーンの最後の台詞 3 行 | ~100 文字 | ~12% |
| 話 → 話（章内） | `previousEpisodeSummary`: 前話の要約 | 100〜200 文字 | ~8% |
| 章 → 章 | **なし**（元の章プロット計画のみ、実際の生成内容は渡されない） | 0 文字 | 0% |

各ホップで 88〜92% の物語情報が失われる。3 ホップ（3 話）経過すると元の情報の 0.1% 以下になる。章をまたぐと完全にゼロリセットされる。

### コンテキスト構造の詳細

**Stage 3 プロンプトに渡される情報（1 シーンあたり）:**

| 項目 | トークン数（推定） | 性質 |
|------|----------------|------|
| システムプロンプト（出力形式・ルール） | ~500 | 固定 |
| 作品設定（ジャンル・トーン） | ~100 | 固定 |
| キャラクター口調ガイド（該当キャラのみ） | ~50-100/人 | 固定 |
| この話のあらすじ | ~100-200 | 話内で固定 |
| シーンプロット（場所・時間・展開） | ~50 | シーン固有 |
| 前のシーンまでの流れ（台詞 3 行） | ~50-100 | **可変・薄い** |
| 前話のあらすじ（100-200 字） | ~100 | **可変・薄い** |
| RAG 検索結果（関連設定チャンク） | ~200-500 | 可変 |
| **合計** | **~1,000-1,500** | |

固定情報（設定・口調・世界観）は常に正確に渡される → **キャラクターの声・世界観の一貫性は保たれる**。
可変情報（前シーン・前話の流れ）が極端に薄い → **プロットの連続性が損なわれやすい**。

## 破綻リスクの予測

### 判定基準

- **破綻なし**: キャラクターの知識・感情・関係性が矛盾なく連続し、同じイベントの繰り返しがない
- **一部不整合**: 細部の矛盾はあるが、読者が気づきにくいレベル
- **破綻**: キャラクターが知っているはずの情報を知らない、同じ展開が繰り返される、選択の結果が反映されない

### 規模別の判定

| 規模 | 総文字数 | 判定 | 根拠 |
|------|---------|------|------|
| **3 章 × 3 話** | ~21,000 文字 | **破綻なし** | 話間要約で主要プロットは伝達可能。章数が少なく情報喪失の累積が小さい |
| **5 章 × 3 話** | ~36,000 文字 | **一部不整合** | 1-2 章の伏線が 4-5 章で忘れられる。キャラクターの成長が平坦化する |
| **7 章以上** | ~50,000 文字〜 | **破綻** | 章間に情報伝達がなく、初期の出来事が完全に失われる |

## 具体的な破綻パターン

### パターン 1: 台詞の繰り返し

デモ出力で既に発生している:

```
ch3_ep2.ks:12  主人公：ここに来るとは思わなかったな。
ch3_ep3.ks:46  主人公：ここに来るとは思わなかったな。
```

```
ch3_ep2.ks:21  涼：んー、いろいろあってね。でも大丈夫。
ch3_ep3.ks:24  涼：んー、いろいろあってね。でも大丈夫。
```

**原因:** シーン間コンテキストが台詞 3 行のみのため、LLM が同じパターンに収束する。

### パターン 2: キャラクターの記憶喪失

1 章で涼の過去（美咲と小学校時代の同級生）が明かされたとする。3 章のプロンプトには 1 章の生成内容が含まれないため、主人公が再び同じ秘密に驚く展開が生成されうる。

**原因:** `previousEpisodeSummary` のチェーン（200 字 → 200 字 → 200 字...）で細部が圧縮消失し、章をまたぐとゼロリセットされる。

### パターン 3: 選択肢の無効化

```json
{ "type": "choice", "options": [
  { "text": "素直に話す", "flagSet": { "name": "trust", "operator": "+=", "value": 1 } },
  { "text": "はぐらかす", "flagSet": null }
]}
```

`trust` フラグは設定されるが、次シーン/次話のプロンプトにフラグ状態が渡されない。物語上の分岐が実質機能しない。

### パターン 4: キャラクターアークの平坦化

各シーンで設定ファイルの `性格: 穏やかだが芯が強い` から再スタートする。1 章で「芯の強さ」を見せた主人公が、2 章で同じレベルから成長し直す。

**原因:** キャラクターの状態（現在の感情・知っている情報・他キャラとの関係変化）を追跡する仕組みがない。

## 結論

| | 文字数 | 条件 |
|---|---|---|
| **現状の安全ライン** | **~20,000 文字**（3 章 9 話） | 改修なし |
| **改善後の想定** | **~50,000 文字〜**（7 章以上） | 下記 3 施策の実装後 |

## 改善施策（優先度順）

### 施策 1: 章要約の追加

各章の全話完了後に「章要約」（300〜500 字）を生成し、次章の Stage 2・Stage 3 プロンプトに注入する。これにより章間の情報伝達がゼロから 300-500 字に改善される。

**変更箇所:**
- `prompts.ts` に `buildChapterSummaryPrompt` 追加
- `assist-cli.ts` の `runAll()` で章完了ごとに章要約を生成・蓄積
- `buildStage2Prompt` / `buildStage3Prompt` に `previousChapterSummaries?: string[]` パラメータ追加

### 施策 2: previousContext の強化

現在の「台詞 3 行」を「シーン要約 200 字」に変更する。地の文の情景・感情描写も引き継がれるようになる。

**変更箇所:**
- `assist-cli.ts` の `runStage3()` で、シーン完了後に LLM で 200 字要約を生成
- `buildStage3Prompt` の `previousContext` に要約テキストを渡す

### 施策 3: キャラクター状態の追跡

各シーン完了後に、登場キャラクターの「知っていること」「関係性の変化」「感情状態」を LLM に抽出させ、次シーンのプロンプトに注入する。

**変更箇所:**
- `types.ts` に `CharacterState` 型追加
- `prompts.ts` に `buildCharacterStatePrompt` 追加
- `assist-cli.ts` でシーンごとにキャラクター状態を更新・蓄積
- `buildStage3Prompt` に `characterStates?: CharacterState[]` パラメータ追加

### 施策の効果予測

| 施策 | 情報保持率の改善 | 安全ライン向上 |
|------|--------------|-------------|
| 章要約 | 章間: 0% → ~30% | +2 章（5 章まで） |
| previousContext 強化 | シーン間: 12% → ~25% | シーン内の一貫性向上 |
| キャラクター状態追跡 | キャラ情報: 0% → ~50% | キャラ破綻の大幅削減 |
| **3 施策合計** | — | **7 章以上（~50,000 文字〜）** |

### 施策 4（将来）: フラグ状態の注入

選択結果のフラグを次シーンのプロンプトに含める。施策 1-3 の後に実装する。
