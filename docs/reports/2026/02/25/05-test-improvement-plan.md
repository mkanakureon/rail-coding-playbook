## テスト追加計画

### Phase 0: 既存テスト資産の整理（即時）

**目的:** 未コミットのテストをコミットし、既知問題を対処する。

| # | タスク | 優先度 |
|---|---|---|
| 0-1 | `apps/hono/test/` の未コミット 10 ファイルをコミット | 高 |
| 0-2 | `interpreter/Integration.test.ts` の Phase5 再帰テストを `it.skip` に変更 | 中 |
| 0-3 | テストファイル一覧（`51-test-file-inventory.md`）を最新状態に更新 | 低 |

### Phase 1: API ルートのテスト補完（1-2 日）

**目的:** 未テストの 4 ルートにテストを追加し、API カバレッジを 100% にする。

| # | 対象 | テストファイル | テスト内容 |
|---|---|---|---|
| 1-1 | `my-characters.ts` | `test/my-characters.test.ts` | CRUD（一覧・作成・更新・削除）、認証必須、ユーザースコープ |
| 1-2 | `official-assets.ts` | `test/official-assets.test.ts` | 一覧取得、カテゴリフィルター、サブカテゴリフィルター、管理者 CRUD |
| 1-3 | `preview.ts` | `test/preview.test.ts` | プレビューデータ取得、存在しない ID、アクセス権限 |
| 1-4 | `user-assets.ts` | `test/user-assets.test.ts` | アップロード・一覧・削除、認証必須、ユーザースコープ |

**テスト方針:**
- 既存の `test/helpers.ts` と `test/setup.ts` を活用
- 認証付きリクエストのテストは `auth.test.ts` のパターンを踏襲
- DB 依存テストは `schema-sync.test.ts` と同様に実 DB 接続

### Phase 2: デプロイ前バリデーション（1 日）

**目的:** 今回のデプロイ障害で判明した 4 つのチェックをスクリプト化する。

| # | チェック | コマンド | 検知できるエラー |
|---|---|---|---|
| 2-1 | TypeScript 型チェック | `npx tsc --noEmit` (editor, next) | unused import、型不一致 |
| 2-2 | ESLint | `npx next lint` (next) | rules-of-hooks 等 |
| 2-3 | Prisma マイグレーション状態 | `npx prisma migrate status` | スキーマ不整合 |
| 2-4 | schema-sync テスト | `npx vitest run test/schema-sync.test.ts` | DB カラム不足 |

**成果物:** `scripts/pre-deploy-check.sh` — デプロイスクリプトの前に実行するバリデーションスクリプト。

```bash
#!/bin/bash
set -e
echo "=== Pre-deploy checks ==="

echo "[1/4] TypeScript type check..."
npx tsc -b apps/editor --noEmit
npx tsc -b apps/next --noEmit

echo "[2/4] ESLint..."
cd apps/next && npx next lint && cd ../..

echo "[3/4] Prisma migrate status..."
cd apps/hono && npx prisma migrate status && cd ../..

echo "[4/4] Schema sync test..."
cd apps/hono && npx vitest run test/schema-sync.test.ts && cd ../..

echo "=== All checks passed ==="
```

### Phase 3: Editor ユニットテスト追加（2-3 日）

**目的:** `apps/editor` にユニットテストを追加する。現在ゼロ。

**方針:** すべてのコンポーネントをテストするのではなく、以下の優先度でビジネスロジックに集中する。

| # | 対象 | テストファイル | テスト内容 | 根拠 |
|---|---|---|---|---|
| 3-1 | `useEditorStore` (zustand) | `test/store.test.ts` | ブロック追加・削除・並び替え、プロジェクト読み込み、undo/redo | ストアがエディタ全体の状態管理。バグの影響範囲が最大 |
| 3-2 | `config/api.ts` | `test/api.test.ts` | URL 構築、認証ヘッダー付与、エラーハンドリング | デプロイ障害の直接原因になりうる |
| 3-3 | ブロック変換ロジック | `test/block-transform.test.ts` | KSC ブロック → スクリプト変換、インポート・エクスポート | データ破損防止 |
| 3-4 | タイムライン計算 | `test/timeline-calc.test.ts` | シーク位置計算、キーフレーム補間、ループ処理 | Zenn 記事 A-05 で実装した Tween タイムラインの検証 |

**テスト環境:** `vitest.config.ts` は既に `jsdom` 環境で設定済み。React コンポーネントのレンダリングテストには `@testing-library/react` を追加。

| # | テストファイル | テスト内容 |

### Phase 5: CI 構築（将来）

**目的:** GitHub Actions で push 時に自動テストを実行する。

| ステージ | 内容 | 実行時間目安 |
|---|---|---|
| lint | `tsc --noEmit` + `next lint` | 30 秒 |
| unit | Vitest 全パッケージ | 1-2 分 |
| schema | `schema-sync.test.ts`（DB 接続必要） | 10 秒 |
| build | `npm run build` | 3-5 分 |
| e2e | Playwright（サーバー起動必要） | 5-10 分 |

**優先順:** lint → unit → build の順で導入。schema と e2e は DB / サーバー依存があるため後回し。
