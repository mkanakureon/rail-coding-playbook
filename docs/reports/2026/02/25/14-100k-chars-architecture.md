# 10 万字生成アーキテクチャ設計

## 現状の問題

現在の安全ライン: ~50,000 字（3 章 9 話）。10 万字（~18 章 or 6 章 x 6 話）を一発生成するには、以下の 3 つの破綻要因を解決する必要がある。

### 破綻要因 1: コンテキストの線形膨張

| データ | 5 章時点 | 10 章時点 | 15 章時点 | 問題 |
|--------|---------|----------|----------|------|
| chapterSummaries | ~1,400 字 | ~3,500 字 | ~5,250 字 | Stage 2 で全量渡し（上限なし） |
| characterStates.knowledge | ~300 字/キャラ | ~750 字/キャラ | ~1,250 字/キャラ | 追記のみ、圧縮なし |
| characterStates.relationships | ~150 字/キャラ | ~375 字/キャラ | ~625 字/キャラ | 同上 |

10 章時点で Stage 2 プロンプトのコンテキスト部分だけで ~5,000 トークン。Stage 3 は ~6,000 トークン。プロンプト全体で 10,000+ トークンになり、生成品質が低下する。

### 破綻要因 2: 遠い過去の忘却

Stage 3 は最新 2 章分の章要約しか渡さない。第 15 章を書くとき、第 1 章の伏線や第 5 章の重要イベントは完全に消失する。

### 破綻要因 3: プロットスレッドの断絶

「美咲と涼の過去」「父の会社倒産」など、複数章にまたがる伏線を明示的に追跡していない。LLM は渡されたコンテキストにない伏線を回収できない。

---

## 解決策: 3 層コンテキスト階層

```
Arc（篇）         ← 新規追加
  └─ Chapter（章）
       └─ Episode（話）
            └─ Scene（シーン）
```

### 層 1: Arc（篇）要約 — 遠い過去を保持

3〜5 章ごとに「篇」としてグループ化し、篇要約（500〜800 字）を生成する。

```
第一篇「出会い編」（第 1〜3 章）→ 篇要約 500 字
第二篇「葛藤編」（第 4〜6 章）→ 篇要約 500 字
第三篇「決着編」（第 7〜9 章）→ 篇要約 500 字
```

**効果**: 9 章分の章要約（~3,150 字）が 1 篇要約（500 字）に圧縮される。15 章時点でも篇要約 3 つ + 直近章要約 2 つ = ~2,500 字。

### 層 2: キャラ状態の圧縮 — 膨張を制御

篇の区切りで「キャラ状態圧縮プロンプト」を実行:

- knowledge: 最重要 5 件に絞り、残りは 1 行サマリーに
- relationshipChanges: 現在の関係性だけ保持（変化履歴は篇要約に吸収済み）
- emotionalState: 最新を保持（変わらず）

**効果**: 1 キャラあたり ~200 字で固定。5 キャラでも ~1,000 字。

### 層 3: プロットスレッド追跡 — 伏線の断絶を防止

明示的に「未解決のプロットスレッド」を管理:

```typescript
interface PlotThread {
  id: string;
  description: string;        // 「美咲と涼の小学校時代の関係」
  introducedIn: number;        // 章番号
  status: 'open' | 'resolved';
  resolvedIn?: number;
}
```

- 各章完了後に「プロットスレッド抽出プロンプト」で新規スレッドを検出
- 解決済みスレッドは `resolved` にして次章以降は渡さない
- open スレッドのみ Stage 2/3 に渡す（常に小さい）

**効果**: 伏線を忘れない。open スレッドは通常 5〜10 件（~500 字）で収まる。

---

## コンテキスト予算（15 章時点）

| コンテキスト | 現在 | 改修後 | 削減率 |
|-------------|------|--------|--------|
| 章要約 (Stage 2) | ~5,250 字 | ~2,500 字（篇要約 + 直近 2 章） | -52% |
| キャラ状態 | ~6,250 字 | ~1,000 字（圧縮済み） | -84% |
| プロットスレッド | 0 字 | ~500 字 | 新規 |
| **合計** | **~11,500 字** | **~4,000 字** | **-65%** |

Stage 3 プロンプト全体: 固定部分 ~2,000 字 + コンテキスト ~4,000 字 = ~6,000 字（~1,500 トークン）。gpt-4.1-nano の入力上限に対して十分な余裕がある。

## 実装計画

### Step 1: 型定義

```typescript
// types.ts に追加
interface PlotThread {
  id: string;
  description: string;
  introducedIn: number;
  status: 'open' | 'resolved';
  resolvedIn?: number;
}

interface ArcSummary {
  arcNumber: number;
  title: string;
  chaptersFrom: number;
  chaptersTo: number;
  summary: string;
}
```

### Step 2: 新規プロンプト 3 種

| 関数 | 入力 | 出力 | タイミング |
|------|------|------|-----------|
| `buildArcSummaryPrompt` | 篇内の章要約 + キャラ状態 | `{ summary }` | 篇区切り（3〜5 章ごと） |
| `buildCompressCharStatesPrompt` | 現在のキャラ状態 | 圧縮された `CharacterState[]` | 篇区切り |
| `buildExtractPlotThreadsPrompt` | 章要約 + 既存スレッド | 更新された `PlotThread[]` | 各章完了後 |

### Step 3: Zod スキーマ

```typescript
ArcSummaryResultSchema
CompressedCharStatesResultSchema
PlotThreadsResultSchema
```

### Step 4: CLI フロー改修 (`runAll`)

```
for 各章:
  Stage 2（篇要約 + 直近章要約 + open スレッド を渡す）
  for 各話:
    Stage 3（圧縮キャラ状態 + 直近章要約 + open スレッド を渡す）
    シーン要約 + キャラ状態抽出
  章要約生成
  プロットスレッド抽出

  if 篇の区切り（3〜5 章ごと）:
    篇要約生成
    キャラ状態圧縮
    古い章要約を破棄（篇要約に吸収済み）
```

### Step 5: Stage 1 の 2 段階化（10 章超の場合）

10 章を超える場合、Stage 1 を 2 段階にする:

1. **Stage 1a**: 篇プロット生成（3〜5 篇のあらすじ）
2. **Stage 1b**: 篇ごとに章プロット生成（篇プロットをコンテキストに）

これにより Stage 1 のプロンプトも膨張しない。

### Step 6: `overview.md` 拡張

```yaml
---
genre: 恋愛
chapters: 18
episodes_per_chapter: 3
chapters_per_arc: 3        # ← 追加: 篇あたりの章数
---
```

## LLM 呼び出し回数

| 単位 | 現在 | 改修後 | 増分 |
|------|------|--------|------|
| 1 シーン | 3 回 | 3 回 | ±0 |
| 1 話（3 シーン） | 10 回 | 10 回 | ±0 |
| 1 章（3 話） | 31 回 | 32 回（+スレッド抽出） | +1 |
| 1 篇（3 章） | 93 回 | 97 回（+篇要約 +キャラ圧縮 +スレッド×3） | +4 |
| 18 章（6 篇） | 558 回 | 582 回 | +24 |

増分は gpt-4.1-nano での要約・抽出なので、コスト増は微小。

## 検証計画

1. **Mock テスト**: 18 章構成で `all` を実行し、出力構造を検証
2. **コンテキストサイズ計測**: 各 Stage のプロンプトトークン数をログ出力
3. **一貫性チェック**: 最終章の生成テキストに第 1 章の伏線が反映されているか
4. **実 API テスト**: gpt-4.1-nano で 18 章を一括生成し、人手で品質確認

## 実装優先順

```
Phase A（必須・効果大）:
  1. キャラ状態圧縮（膨張の主因を解決）
  2. プロットスレッド追跡（伏線断絶を解決）

Phase B（10 章超で必要）:
  3. Arc 篇要約（章要約の膨張を解決）
  4. Stage 1 の 2 段階化
```

Phase A だけで ~80,000 字は安全に生成可能。Phase B まで実装すれば 10 万字 + α。
