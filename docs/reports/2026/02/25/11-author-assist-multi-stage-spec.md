# 作者向け執筆支援 — 多段階プロンプト生成 仕様書

> Generated by Claude Opus 4.6

## 概要

作者が「概要・プロット・世界観・キャラ設定・歴史的事実」などをインプットとして与えると、多段階のプロンプトチェーンで最終的に `.ks` ファイル（ノベルゲームスクリプト）まで自動生成する機能。

```
インプット（作品設定）
  ↓ Stage 1  ← gpt-4o-mini（構造化出力、コスト優先）
章プロット（全体の章構成）
  ↓ Stage 2  ← gpt-4o-mini（構造化出力、コスト優先）
話プロット（各話のあらすじ）
  ↓ Stage 3  ← claude-sonnet（テキスト品質・口調再現重視）
話テキスト（地の文 + セリフ）※シーン単位で分割生成
  ↓ Stage 4  ← TS テンプレート（LLM 不使用）
.ks ファイル（演出コマンド付きスクリプト）
```

各ステージの出力を作者が確認・編集してから次のステージに進む。完全自動ではなく **人間 in the loop** の設計。

---

## インプット仕様

### 方針: ローカル .md ファイルから読み込む

作品設定（世界観・プロット・キャラ等）は **ローカルの Markdown ファイル**として管理する。最初は CLI でファイルパスを指定して動作確認する。TS アプリへの組み込みは後回し。

```
projects/{project-name}/
  settings/
    overview.md         # 必須: 作品概要 + ジャンル
    characters.md       # 必須: キャラクター設定
    world.md            # 任意: 世界観
    plot.md             # 任意: ラフプロット
    history.md          # 任意: 歴史的事実・史実
    assets.md           # 任意: アセットマッピング
```

### .md ファイルフォーマット

各 .md ファイルは **YAML frontmatter + 本文** の形式。CLI パーサーが frontmatter をメタデータとして読み取り、本文を自由記述のコンテンツとして扱う。

#### `overview.md`（必須）

```markdown
---
genre: 恋愛
era: 現代
tone: シリアス
themes:
  - 友情
  - 裏切り
  - 成長
chapters: 3
episodes_per_chapter: 3
---

# 作品概要

高校二年生の春、転校生がやってきたことで主人公の日常が変わり始める。
幼馴染との関係、新たな出会い、そして隠された過去。
三角関係の中で、主人公は自分の本当の気持ちに向き合うことになる。

舞台は地方の港町。海と坂道が印象的な風景の中で、
青春の甘さと苦さが交差する物語。
```

#### `characters.md`（必須）

```markdown
# キャラクター設定

## 主人公（hero）

- 役割: 主人公
- 性格: 穏やかだが芯が強い。人の話をよく聞く
- 口調: タメ口。「〜だな」「〜かな」が多い
- 経歴: 港町で生まれ育った。父は漁師、母は喫茶店を経営
- 関係: 美咲とは幼馴染。小さい頃から一緒にいた

## 美咲（misaki）

- 役割: ヒロイン
- 性格: 明るく活発。でも一人になると考え込むことがある
- 口調: タメ口。「〜だよ！」「〜でしょ？」が多い。感情が声に出やすい
- 経歴: 主人公の隣の家に住んでいる。水泳部のエース
- 関係: 主人公に密かに好意を持っている

## 涼（ryo）

- 役割: ライバル / 転校生
- 性格: クールで知的。人を寄せ付けない雰囲気だが根は優しい
- 口調: 敬語混じり。「〜ですね」「〜かもしれません」。打ち解けると砕ける
- 経歴: 東京から転校してきた。家庭の事情で引っ越してきた
- 関係: 美咲と小学校時代の同級生だった（主人公は知らない）
```

> **パース規則**: `## 名前（slug）` をヘッダーとして認識。slug は半角英数。箇条書きの `- キー: 値` をフィールドとして抽出。

### CLI での実行（テスト用）

```bash
# Stage 1: 章プロット生成
npx tsx scripts/assist-cli.ts chapters \
  --settings projects/demo/settings/

# Stage 2: 話プロット生成（Stage 1 の出力を指定）
npx tsx scripts/assist-cli.ts episodes \
  --settings projects/demo/settings/ \
  --chapters output/chapters.json \
  --target 1

# Stage 3: 話テキスト生成
npx tsx scripts/assist-cli.ts text \
  --settings projects/demo/settings/ \
  --chapters output/chapters.json \
  --episode output/ch1_episodes.json \
  --target 1

# Stage 4: .ks 変換
npx tsx scripts/assist-cli.ts ks \
  --settings projects/demo/settings/ \
  --text output/ch1_ep1_text.json

# 出力先: output/ ディレクトリに JSON / .ks を保存
```

CLI スクリプト (`scripts/assist-cli.ts`) が .md ファイルを読み込み → `WorkSetting` 型に変換 → ai-gateway を呼び出す。

### .md パーサー


### 内部データ型（WorkSetting）

.md ファイルから読み込んだ内容は最終的にこの型に変換される。API で使う際も同じ型。

```typescript
interface WorkSetting {
  projectId: string;
  version: number;         // 設定変更履歴（1 から開始、更新ごとにインクリメント）

  // 必須（overview.md から）
  overview: string;        // 作品概要（本文部分）
  genre: string;           // frontmatter: genre

  // 任意（各 .md ファイルから。ファイルが存在すれば読み込む）
  roughPlot?: string;      // plot.md の本文
  worldSetting?: string;   // world.md の本文
  era?: string;            // overview.md frontmatter: era
  historicalFacts?: string; // history.md の本文
  themes?: string[];       // overview.md frontmatter: themes
  tone?: string;           // overview.md frontmatter: tone
  targetLength?: {
    chapters: number;      // overview.md frontmatter: chapters（デフォルト: 3）
    episodesPerChapter: number; // overview.md frontmatter: episodes_per_chapter（デフォルト: 3）
  };

  // キャラ設定（characters.md から）
  characters: CharacterSetting[];

  // アセットマッピング（assets.md から）
  assetMapping?: AssetMapping;
}

interface CharacterSetting {
  name: string;            // 表示名（例: "主人公"）— ## ヘッダーの名前部分
  slug: string;            // スクリプト ID（例: "hero"）— ## ヘッダーの括弧内
  role: string;            // 箇条書きの「役割」
  personality: string;     // 箇条書きの「性格」
  speechStyle?: string;    // 箇条書きの「口調」
  background?: string;     // 箇条書きの「経歴」
  relationships?: string;  // 箇条書きの「関係」
}

interface AssetMapping {
  // 場所名（日本語）→ 背景アセット slug
  locations: Record<string, string>;
  // assets.md の「背景」テーブルから抽出
}
```

### .md ↔ API の関係

| フェーズ | インプット方式 |
|---------|-------------|
| **v1（CLI テスト）** | ローカル .md ファイルをパスで指定 |
| **v2（API 統合）** | .md の内容を API リクエストで送信、または `work_settings` テーブルに保存 |
| **v3（エディタ UI）** | エディタ上で設定フォームを用意し、保存時に `work_settings` へ |

v1 → v2 への移行時、`WorkSetting` 型は変えない。入力ソースが .md → API リクエスト に変わるだけ。

## LLM プロバイダ抽象化

### LLMClient インターフェース

OpenAI / Anthropic を統一的に扱うための抽象層を `packages/ai-gateway` に設ける。

```typescript
interface LLMClient {
  generate(params: GenerateParams): Promise<GenerateResult>;
}

interface GenerateParams {
  model: string;
  systemPrompt: string;
  userPrompt: string;
  jsonMode: boolean;        // OpenAI: response_format, Claude: XML ラッパー指示
  maxTokens?: number;
  temperature?: number;
}

interface GenerateResult {
  content: string;
  usage: {
    tokensIn: number;
    tokensOut: number;
  };
  model: string;
  provider: 'openai' | 'anthropic';
}
```

### プロバイダ実装

```typescript
// packages/ai-gateway/src/providers/openai.ts
class OpenAIClient implements LLMClient {
  // fetch ベース、response_format: { type: "json_object" } 対応
  // リトライ（429 / 5xx）、タイムアウト 30 秒
}

// packages/ai-gateway/src/providers/anthropic.ts
class AnthropicClient implements LLMClient {
  // fetch ベース、JSON 出力は XML ラッパー指示 + パース
  // リトライ（429 / 529）、タイムアウト 30 秒
}
```

### ステージ別モデル使い分け

| Stage | 推奨モデル | 理由 |
|-------|-----------|------|
| 1: 章プロット | `gpt-4o-mini` | 構造化出力が安定、コスト低 |
| 2: 話プロット | `gpt-4o-mini` | 同上 |
| 3: 話テキスト | `claude-sonnet-4-6` | テキスト品質・キャラ口調再現に優れる |
| 4: .ks 変換 | — (LLM 不使用) | — |

モデルは `work_settings` または API リクエストで上書き可能にする。

**環境変数**:
```
OPENAI_API_KEY=<YOUR_OPENAI_API_KEY>...
ANTHROPIC_API_KEY=<YOUR_ANTHROPIC_API_KEY>...
ASSIST_STAGE1_MODEL=gpt-4o-mini
ASSIST_STAGE2_MODEL=gpt-4o-mini
ASSIST_STAGE3_MODEL=claude-sonnet-4-6
```

## JSON 出力の安定性

LLM に JSON を出力させる際の方針:

### OpenAI

- `response_format: { type: "json_object" }` を必ず指定
- プロンプトに「JSON で出力してください」を含める（OpenAI の要件）

### Anthropic (Claude)

- システムプロンプトに XML ラッパー指示を含める:
  ```
  出力は以下の形式で、<json> タグ内に JSON のみを記述してください。
  <json>
  { ... }
  </json>
  ```
- レスポンスから `<json>...</json>` を抽出してパース

### パース・バリデーション共通

```typescript
// apps/hono/src/lib/assist/parser.ts
function parseStageResult<T>(
  raw: string,
  schema: z.ZodSchema<T>,
  provider: 'openai' | 'anthropic'
): T {
  // 1. プロバイダに応じて JSON 文字列を抽出
  //    - OpenAI: そのままパース
  //    - Anthropic: <json>...</json> タグから抽出
  //    - フォールバック: マークダウンコードブロック ```json ... ``` から抽出
  // 2. JSON.parse
  // 3. zod の safeParse でバリデーション
  // 4. 失敗時: 1 回リトライ（LLM を再呼び出し）
}
```

### エンドポイント

```
POST /api/assist/generate/chapters
```

### 入力

```json
{
  "projectId": "01JXXX...",

}
```

```
あなたはノベルゲームのシナリオライターです。
以下の作品設定に基づいて、{N}章の章プロットを作成してください。

## 作品設定
- 概要: {overview}
- ジャンル: {genre}
- 世界観: {worldSetting}
- 時代: {era}
- 歴史的事実: {historicalFacts}
- テーマ: {themes}
- トーン: {tone}
- ラフプロット: {roughPlot}

## キャラクター
{characters の一覧}

## 出力形式
JSON で出力してください。
{
  "chapters": [
    {
      "chapterNumber": 1,
      "title": "章タイトル",
      "summary": "この章で起きること（200〜400字）",
      "keyEvents": ["主要イベント1", "主要イベント2"],
      "emotionalArc": "この章の感情の流れ（上昇/下降/転換）",
      "activeCharacters": ["hero", "heroine"]
    }
  ]
}

## ルール
- 各章に明確な目的と転換点を持たせる
- 起承転結を意識する
- キャラクターの成長や関係性の変化を盛り込む
- ラフプロットがある場合はそれを尊重する
```

### 出力型（Zod スキーマ）

```typescript
const ChapterPlotSchema = z.object({
  chapterNumber: z.number(),
  title: z.string(),
  summary: z.string().min(100).max(800),
  keyEvents: z.array(z.string()).min(1).max(5),
  emotionalArc: z.string(),
  activeCharacters: z.array(z.string()),
});

const Stage1ResultSchema = z.object({
  chapters: z.array(ChapterPlotSchema).min(1).max(10),
});
```

## Stage 2: 話プロット生成

### エンドポイント

```
POST /api/assist/generate/episodes
```

### 入力

```json
{
  "projectId": "01JXXX...",
  "settings": { /* WorkSetting */ },
  "chapters": [ /* Stage 1 の出力（作者が編集済み） */ ],
  "targetChapter": 1
}
```

### プロンプト設計

```
あなたはノベルゲームのシナリオライターです。
以下の章プロットに基づいて、第{N}章の各話のプロットを作成してください。

## 作品設定
（Stage 1 と同様）

## 全体の章構成
{chapters の要約}

## 対象の章
{targetChapter の詳細}

## 出力形式
JSON で出力してください。
{
  "episodes": [
    {
      "episodeNumber": 1,
      "title": "話タイトル",
      "summary": "この話で起きること（100〜300字）",
      "scenes": [
        {
          "sceneNumber": 1,
          "location": "場所",
          "time": "時間帯（朝/昼/夕/夜）",
          "characters": ["hero", "heroine"],
          "event": "このシーンで起きること（50〜100字）",
          "hasChoice": false
        }
      ],
      "endHook": "次の話への引き（読者を引きつける要素）"
    }
  ]
}

## ルール
- 1話 = エディタの1ページに対応
- 各話に1〜3シーンを持たせる
- シーンは場所や時間が変わるタイミングで分ける
- 会話が中心になるようにする（ノベルゲームなので）
- 選択肢を入れるべき箇所があれば hasChoice: true にする
```

### 出力型（Zod スキーマ）

```typescript
const ScenePlotSchema = z.object({
  sceneNumber: z.number(),
  location: z.string(),
  time: z.string(),
  characters: z.array(z.string()),
  event: z.string(),
  hasChoice: z.boolean().optional().default(false),
});

const EpisodePlotSchema = z.object({
  episodeNumber: z.number(),
  title: z.string(),
  summary: z.string().min(50).max(600),
  scenes: z.array(ScenePlotSchema).min(1).max(5),
  endHook: z.string(),
});

const Stage2ResultSchema = z.object({
  episodes: z.array(EpisodePlotSchema).min(1).max(10),
});
```

## Stage 3: 話テキスト生成（シーン単位分割）

### エンドポイント

```
POST /api/assist/generate/text
```

### シーン単位の分割生成

トークン上限の問題を回避するため、**1 シーンずつ生成する**。API 内部でシーン単位のループを回す。

```
episode.scenes をループ:
  scene[0] → LLM 生成 → result[0]
  scene[1] → LLM 生成（scene[0] の結果をコンテキストとして渡す） → result[1]
  scene[2] → LLM 生成（scene[0-1] の要約をコンテキストとして渡す） → result[2]
→ 全シーンの結果を結合して返却
```

### 入力

```json
{
  "projectId": "01JXXX...",
  "settings": { /* WorkSetting */ },
  "chapter": { /* 対象の章プロット */ },
  "episode": { /* 対象の話プロット（作者が編集済み） */ },
  "previousEpisodeSummary": "前話のあらすじ（あれば。自動生成も可）"
}
```

### プロンプト設計（1 シーンごとに呼び出し）

```
あなたはノベルゲームのシナリオライターです。
以下のシーンプロットに基づいて、実際のテキスト（地の文とセリフ）を書いてください。

## 作品設定
（省略形: overview + tone + 関連キャラのみ）

## キャラクター口調ガイド
{characters の name, slug, speechStyle, personality を一覧}

## このシーンのプロット
- 場所: {scene.location}
- 時間帯: {scene.time}
- 登場人物: {scene.characters}
- 展開: {scene.event}
- 選択肢あり: {scene.hasChoice}

## 前のシーンまでの流れ
{previousContext}

## 前話のあらすじ
{previousEpisodeSummary}

## 出力形式
JSON で出力してください。
{
  "lines": [
    { "type": "narration", "text": "地の文" },
    { "type": "dialogue", "speaker": "hero", "text": "セリフ" },
    { "type": "choice", "options": [
      { "text": "選択肢テキスト", "result": "選択後の短い展開", "flagSet": null, "jumpTo": null }
    ]}
  ]
}

## ルール
- セリフは各キャラの口調を忠実に再現する
- 地の文は簡潔に（ノベルゲームなので長い描写は避ける）
- 10〜30行程度
- 感情の動きがわかるように書く
- speaker には slug を使う（name ではない）
- 選択肢がある場合、options にフラグや分岐先を含めてよい
```

### 出力型（Zod スキーマ）

```typescript
const ChoiceOptionSchema = z.object({
  text: z.string(),
  result: z.string(),
  flagSet: z.object({
    name: z.string(),
    operator: z.enum(['=', '+=', '-=']),
    value: z.number(),
  }).nullable().optional(),
  jumpTo: z.string().nullable().optional(),
});

const LineSchema = z.discriminatedUnion('type', [
  z.object({ type: z.literal('narration'), text: z.string() }),
  z.object({ type: z.literal('dialogue'), speaker: z.string(), text: z.string() }),
  z.object({ type: z.literal('choice'), options: z.array(ChoiceOptionSchema).min(2).max(4) }),
]);

const SceneTextSchema = z.object({
  lines: z.array(LineSchema).min(1),
});

const Stage3ResultSchema = z.object({
  title: z.string(),
  scenes: z.array(z.object({
    sceneNumber: z.number(),
    location: z.string(),
    time: z.string(),
    lines: z.array(LineSchema).min(1),
  })),
  summary: z.string().optional(),  // 自動生成した話の要約（次話への引き継ぎ用）
});
```

### 自動要約（previousEpisodeSummary の自動生成）

Stage 3 の生成完了後、生成結果全体を要約して `generation_results` に `summary` フィールドとして保存する。次の話を生成する際に自動的に `previousEpisodeSummary` として渡す。

```
生成後の後処理:
  1. 全シーンの lines を結合
  2. LLM で 100〜200 字に要約（低コストモデル: gpt-4o-mini）
  3. generation_results.result.summary に保存
```

作者が手動で `previousEpisodeSummary` を指定した場合はそちらを優先する。

## Stage 4: .ks ファイル生成

### エンドポイント

```
POST /api/assist/generate/ks
```

### 入力

```json
{
  "projectId": "01JXXX...",
  "settings": { /* WorkSetting（キャラ情報 + assetMapping を使用） */ },
  "episodeText": { /* Stage 3 の出力（作者が編集済み） */ }
}
```

`assetMapping` は `settings.assetMapping` から取得する（リクエストごとに別途渡す必要はない）。

### 変換ロジック（TS で実装、LLM 不使用）

Stage 4 は **LLM を使わず TS のテンプレートエンジンで変換する**。理由:
- .ks のフォーマットは固定仕様であり、LLM の「揺れ」を許容できない
- コンパイラが通らない .ks を生成するリスクを排除する

```typescript
function generateKs(
  episodeText: EpisodeText,
  characters: CharacterSetting[],
  assetMapping?: AssetMapping
): string {
  let ks = '';
  const charMap = new Map(characters.map(c => [c.slug, c.name]));

  for (const scene of episodeText.scenes) {
    // 背景
    const bgId = assetMapping?.locations[scene.location];
    if (bgId) ks += `@bg ${bgId} fade 500\n\n`;

    for (const line of scene.lines) {
      switch (line.type) {
        case 'narration':
          ks += `${line.text}\n@l\n\n`;
          break;
        case 'dialogue':
          const charName = charMap.get(line.speaker) ?? line.speaker;
          ks += `${charName}：${line.text}\n@l\n\n`;
          break;
        case 'choice':
          ks += `choice {\n`;
          for (const opt of line.options) {
            ks += `  "${opt.text}" {\n`;
            // フラグ設定
            if (opt.flagSet) {
              ks += `    ${opt.flagSet.name} ${opt.flagSet.operator} ${opt.flagSet.value}\n`;
            }
            ks += `    ${opt.result}\n`;
            ks += `    @l\n`;
            // 分岐ジャンプ
            if (opt.jumpTo) {
              ks += `    @jump ${opt.jumpTo}\n`;
            }
            ks += `  }\n`;
          }
          ks += `}\n\n`;
          break;
      }
    }
    ks += `@p\n\n`;
  }
  return ks;
}
```

```ks
@l
```


`version` は設定を更新するたびにインクリメントする。`generation_results` の `input_hash` と組み合わせて、設定変更後の再生成判定に使う。

## DB テーブル: `generation_results`

各ステージの生成結果を保存し、次のステージへの入力にする。

```sql
CREATE TABLE generation_results (
  id           VARCHAR(31) PRIMARY KEY,
  project_id   VARCHAR(31) NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  user_id      VARCHAR(31) NOT NULL,
  stage        VARCHAR(20) NOT NULL,  -- 'chapters' | 'episodes' | 'text' | 'ks'
  stage_key    VARCHAR(50) NOT NULL,  -- 'all' | 'ch1' | 'ch1_ep1' 等
  input_hash   VARCHAR(64) NOT NULL,  -- 入力のハッシュ（再生成判定用）
  result       JSONB NOT NULL,        -- 生成結果（stage ごとに型が異なる → アプリ層で Zod バリデーション）
  is_edited    BOOLEAN NOT NULL DEFAULT false,  -- 作者が編集したか
  created_at   BIGINT NOT NULL,
  updated_at   BIGINT NOT NULL,
  UNIQUE(project_id, stage, stage_key)
);

CREATE INDEX gen_results_project_idx ON generation_results(project_id, stage);
```

**`result` の型安全性**: `stage` カラムの値に応じて、アプリ層で対応する Zod スキーマ（`Stage1ResultSchema`, `Stage2ResultSchema`, `Stage3ResultSchema`）でバリデーションする。DB 取得時に `parseStageResult()` を通すことで型安全を保証する。

## 制約・注意点

| 項目 | 内容 |
|------|------|
| .ks 生成は LLM 不使用 | Stage 4 は TS テンプレートで変換。構文の正確性を保証する |
| スピーカー名 | 全角コロン（：）で統一。LLM 出力で半角になった場合は後処理で変換 |
| Stage 3 はシーン単位生成 | 1 シーンずつ LLM を呼び出す。前シーンのコンテキストを次に渡す |
| JSON パース | OpenAI: `response_format`、Claude: XML タグ抽出 → Zod `safeParse` → 失敗時リトライ 1 回 |
| 再生成 | 各ステージは何度でも再生成可能。`input_hash` + `work_settings.version` で変更検知 |
| モデル使い分け | Stage 1-2: gpt-4o-mini（コスト）、Stage 3: claude-sonnet（品質） |
| 作品設定の保存 | `work_settings` テーブルに永続化。`version` で変更履歴を追跡 |
| choice の拡張性 | v1 では `flagSet` / `jumpTo` フィールドを nullable で用意。将来の分岐マージに備える |
