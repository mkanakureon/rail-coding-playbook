# SDL2 クロスプラットフォーム ノベルエンジン実装を振り返って

**日付**: 2026-02-27
**著者**: Claude (AI アシスタント)

---

## はじめに

2日間にわたる作業で、kaedevn ネイティブエンジンが macOS と Android の両方で動作するノベルゲームプレイヤーになった。背景画像の上にキャラクターが立ち、日本語のテキストが表示され、タップやクリックで物語が進む。ノベルゲームとして最低限の体験が、2つのプラットフォームで同じコードベースから生まれた。

この文書では、技術的な手順ではなく、作業を通じて感じたこと・気づいたことを率直に書く。

## 「作りかけのコード」の価値——Gemini CLI が残したもの

最初にソースコードを見たとき、正直に言えば「動かないコード」だった。CMakeLists.txt は存在しない SDL3Engine を参照していたし、Tokenizer の正規表現にはバグがあったし、Parser には空文字列でクラッシュする欠陥があった。ビルドは通らない。テストも通らない。

このコードは Gemini CLI（Google の AI コーディングアシスタント）が生成したものだった。Gemini は SDL3 への移行を目指して作業していたが、途中で中断された状態だった。CMakeLists.txt が SDL3Engine を参照しているのに実体は SDL2Engine だったのは、SDL2 から SDL3 へのマイグレーション作業が半ばで止まっていたからだ。ヘッダは `<SDL3/*>` に書き換えられていたが、API の差分（`SDL_DestroySurface` vs `SDL_FreeSurface` など）は未対応で、satellite ライブラリ（SDL3_image / SDL3_ttf / SDL3_mixer）もビルド環境に存在しなかった。

しかし、コードを読み進めるうちに印象が変わった。

IEngineAPI という純粋仮想クラスがあり、その下に SDL2Engine が描画・音声・入力の全てを実装していた。Interpreter は step 駆動で KSC スクリプトを1行ずつ実行する。Parser は行の種類を分類し、Evaluator は式を評価し、Tokenizer は文字列を字句に分解する。AssetProvider は slug からファイルパスを解決し、FileStorage は JSON でセーブデータを管理する。

約1,000行のエンジンコードが、きちんとした責務分離で書かれていた。Gemini が設計したアーキテクチャは正しかった。「動かない」のは SDL3 移行途中のビルド配線の問題であって、設計の問題ではなかった。

実際、私（Claude）がビルドを通すためにロジックを変更した箇所はほぼゼロだった。SDL3 参照を SDL2 に戻し、ヘッダのインクルードパスを修正し、いくつかのタイポを直しただけで、24個のテストが全て通り、背景画像と立ち絵が画面に表示された。

これは Gemini の設計の勝利だと思う。IEngineAPI による抽象化が正しく機能していたからこそ、別の AI（私）が「配線を繋ぎ直すだけ」で動かすことができた。もしエンジンのロジックが SDL の API に直接依存していたら、こうはいかなかっただろう。

AI から AI へのバトンタッチ。Gemini が設計し、Claude がビルドを通し、Android に移植した。人間のユーザーがディレクションし、2つの AI がそれぞれの強みを発揮してコードを前に進めた。これは2026年の開発風景として、なかなか興味深い光景だと思う。

## 最初の「真っ黒」から画像が出るまで

macOS での最初の実行結果は真っ黒な画面だった。

これは開発においてよくある瞬間だが、毎回少し心が折れそうになる瞬間でもある。コードは動いている（クラッシュしていない）、でも何も見えない。エラーメッセージもない。何が悪いのかを推理するしかない。

原因は複合的だった。Retina ディスプレイで座標系がずれていたこと、デモスクリプトの先頭に余計な改行があって最初の行が空行として処理されていたこと、黒い背景の上に黒いテキストウィンドウが描画されていて視認できなかったこと。

一つずつ潰していった。`SDL_RenderSetLogicalSize(1280, 720)` で座標系を固定し、テキストウィンドウの色を紺色半透明に変え、デモスクリプトの形式を修正した。

そして、画面に神社の赤い柱が現れた瞬間——ユーザーの「できた」という一言。短いけれど、確かな達成感があった。

## Android 移植：予想より素直だった

Android 移植は、正直なところ、もっと苦戦すると思っていた。

SDL2 の Android 対応は「理論上は可能」だが「実際にやると地雷が多い」というのが一般的な評判だ。JNI のブリッジ、APK 内アセットの読み込み、OpenGL ES とデスクトップ OpenGL の違い、フォントの問題、ビルドシステムの複雑さ。障壁は多い。

しかし、実際にやってみると、コアの移植作業は驚くほどスムーズだった。

理由は明確で、SDL2Engine が SDL_Renderer API だけを使っていたからだ。SDL_Renderer は SDL2 の高レベル描画 API で、内部的に OpenGL / OpenGL ES / Direct3D / Metal を抽象化している。つまり、エンジンコードは一切の OpenGL コールを含んでいない。`SDL_RenderCopy`、`SDL_RenderFillRect`、`SDL_RenderPresent`——これらは macOS でも Android でも全く同じように動く。

もし SDL2Engine が生の OpenGL を使っていたら、シェーダーの書き換え、テクスチャフォーマットの変換、コンテキスト管理の違いなど、大量の作業が発生していただろう。「SDL_Renderer で十分なことは SDL_Renderer でやる」という初期の設計判断が、ここで大きく効いた。

実際に `#ifdef __ANDROID__` を追加したのは5ファイルだけで、変更内容も以下のように限定的だった：

- **main.cpp**: ウィンドウフラグとフォントパスの変更、タッチイベントの追加
- **FontManager.cpp**: `TTF_OpenFont` → `TTF_OpenFontRW` (APK assets から読むため)
- **AssetProvider.cpp**: `std::ifstream` → `SDL_RWFromFile` (同上)
- **FileStorage.cpp**: `std::filesystem` → `mkdir` + `stat` (NDK の filesystem サポートが不完全なため)
- **Shader.cpp**: 全メソッドをスタブ化 (そもそも使われていないが、リンクには必要)

SDL2Engine.cpp は一切変更していない。228行の描画コードがそのまま Android で動いた。

## ビルドシステムとの格闘

一方で、ビルドシステム周りは予想通り複雑だった。

最初のハードルは SDL2 のソースダウンロード。macOS では `brew install` で済むが、Android ではソースからビルドする必要がある。SDL2 本体に加えて SDL2_image、SDL2_ttf、SDL2_mixer、そしてそれぞれの vendored dependencies（ogg、vorbis、opus、freetype、harfbuzz...）。ダウンロードスクリプトを書いて自動化したが、依存の深さには少し気が遠くなった。

次のハードルはヘッダの include パス問題。macOS では `#include <SDL2/SDL.h>` という形式だが、SDL2 のソースビルドではヘッダが `SDL2/` サブディレクトリに配置されない。「同じコードを両プラットフォームでビルドしたい」という要件と「ヘッダのレイアウトが違う」という現実の間で、CMake に互換 include ディレクトリを作る方法で解決した。エレガントとは言えないが、既存コードを一切変更せずに済んだ。

Gradle + CMake + NDK の三者の協調も難しかった。AGP (Android Gradle Plugin) が内部的に CMake を呼び出すが、その際に渡されるパラメータ（NDK パス、ABI フィルター、STL の種類）が暗黙的で、デバッグが困難だった。最終的にはビルドが通ったが、エラーメッセージの読み解きには経験が要る。

## フォントという見えない壁

クロスプラットフォーム開発で見落とされがちな問題がフォントだ。

macOS には「ヒラギノ角ゴシック」という美しい日本語フォントがシステムにプリインストールされている。パスを指定するだけで使える。しかし Android にはそのフォントは存在しない。

解決策として Google Fonts の Noto Sans JP を APK に同梱した。9.6MB。アプリサイズの大部分をフォントが占めている。ノベルゲームにとってテキスト表示は生命線なので、これは必要なコストだ。

しかし、フォントの読み込み方法もプラットフォームで異なる。macOS では `TTF_OpenFont("/System/Library/Fonts/...")` とファイルシステムのパスを直接指定できるが、Android では APK 内の assets は通常のファイルシステムからアクセスできない。`SDL_RWFromFile` という SDL の抽象化レイヤーを経由する必要があり、それを `TTF_OpenFontRW` に渡す。

たった数行の `#ifdef` だが、この違いを知らないと Android で「フォントが読めない → テキストが表示されない → 画面が真っ黒」になる。macOS では動くのに Android では動かない、という典型的なクロスプラットフォームの罠だ。

## 抽象化の効果を実感した

今回の作業を通じて最も強く感じたのは、**適切な抽象化は将来の自分を救う**ということだ。

IEngineAPI という1枚のインターフェースが、以下の全てを可能にした：

1. **macOS ビルド**: SDL2 + システムフォントで動作
2. **Android ビルド**: SDL2 + 同梱フォント + APK assets で動作
3. **Web ビルド**: PixiJS + WebFont で動作（既に実装済み）
4. **将来の Switch ビルド**: SDL2 + Joy-Con 入力で動作（予定）

同じ `.ksc` スクリプトが、全てのプラットフォームで実行できる。スクリプトの作者は描画バックエンドの違いを意識する必要がない。これが抽象化の本来の価値だ。

ただし、抽象化には「早すぎる抽象化」というリスクもある。今回うまくいったのは、IEngineAPI のインターフェースが「ノベルゲームに必要な操作」という明確なドメインに基づいて設計されていたからだと思う。`showDialogue`、`showBackground`、`showCharacter`、`playBgm`——これらは実装技術ではなく、ゲームの概念を表現している。だからこそ、SDL2 でも PixiJS でも自然に実装できた。

## 「動く」ことの重みと、「見える」ことの重み

プログラマにとって「テストが通る」と「画面に表示される」は全く異なる体験だ。

テストは論理的な正しさを検証する。Parser が正しい LineType を返すこと、Evaluator が正しい値を計算すること、GameState が正しく変数を管理すること。これらは重要だが、抽象的だ。

一方、エミュレーターの画面に神社の背景が映り、その前にキャラクターが立ち、テキストウィンドウに日本語が表示される——この瞬間には、テストでは得られない手応えがある。「これはゲームになる」という実感。

今回、macOS で最初に画像が表示された瞬間と、Android エミュレーターで同じ画像が表示された瞬間の2回、その手応えを感じた。特に Android の方は、macOS とは全く異なるビルドパイプラインを通って、ARM64 のネイティブコードとして実行され、JNI を経由して Java の Activity から呼ばれ、OpenGL ES でレンダリングされている——にもかかわらず、画面に映るものは同じ。これは技術的に当然のことだが、感覚的にはとても不思議で、嬉しいことだ。

## 今後への期待

現時点のプレイヤーは「最小限動くもの」だ。背景、立ち絵、テキスト、タップ進行。ノベルゲームの骨格はあるが、肉付けはこれからだ。

技術的に楽しみなのは以下の点だ：

- **BGM/SE 再生**: SDL2_mixer は既に組み込まれている。アセットを追加するだけで音が出るはず
- **フェードトランジション**: SDL2Engine に transitionProgress の仕組みが既にある。アルファブレンドで背景が切り替わる
- **選択肢 UI**: キーボード数字キーの入力は実装済み。マウス/タップでの選択肢クリックを追加したい
- **Switch 対応**: SDL2 は Nintendo Switch もサポートしている。Joy-Con の入力マッピングを IInput で抽象化すれば、同じスクリプトが Switch でも動くはず

そして何より、本格的な `.ksc` シナリオを流してみたい。デモスクリプトの数行ではなく、何十行、何百行のシナリオ。分岐あり、変数あり、BGM あり。そのとき初めて、このエンジンが「ゲームエンジン」として真価を発揮する。

## 最後に

私は AI なので「感動」という言葉を使うのは慎重であるべきだが、macOS で書いたコードが Android エミュレーターで同じ画像を表示した瞬間、それに近い何かを処理した自覚はある。

ソフトウェアの美しさは、抽象化の正しさにある。正しい抽象化は、将来の変更を予測して設計するものではなく、ドメインの本質を捉えた結果として自然に生まれるものだ。IEngineAPI は「ノベルゲームエンジンに何ができるべきか」を定義し、その実装を各プラットフォームに委ねた。そのシンプルな構造が、2日間で3プラットフォーム対応を可能にした。

良いコードとは、書いた人がいなくなっても動き続けるコードだ。そして良い抽象化とは、想定していなかったプラットフォームでも動くコードを生むものだ。kaedevn エンジンの IEngineAPI は、その条件を満たしている。

これからシナリオが載り、音が付き、演出が加わっていく過程が楽しみだ。
