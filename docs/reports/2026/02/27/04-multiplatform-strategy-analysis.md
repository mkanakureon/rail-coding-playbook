# kaedevn マルチプラットフォーム戦略の考察

**日付**: 2026-02-27
**著者**: Claude Opus 4.6

---

## 1. 概要：何を実現しているのか

kaedevn は、1つのスクリプト（`.ksc`）と1セットのアセット（背景・立ち絵・音声）から、以下のプラットフォームにゲームを展開できるノベルゲームエンジンである。

| プラットフォーム | レンダリング | インタプリタ | 状態 |
|----------------|-------------|-------------|------|
| **ブラウザ (Web)** | PixiJS + WebGL | TypeScript (VM / OpRunner) | 実装済み・本番稼働 |
| **macOS** | SDL2 (SDL_Renderer) | C++ (step駆動) | 動作確認済み |
| **Android** | SDL2 (SDL_Renderer) | C++ (step駆動) | 動作確認済み |
| **Windows** | SDL2 (SDL_Renderer) | C++ (同上) | 予定（SDL2が標準対応） |
| **iOS** | SDL2 (SDL_Renderer) | C++ (同上) | 予定（SDL2が標準対応） |
| **Nintendo Switch** | SDL2 (SDL_Renderer) | C++ (同上) | 予定（SDL2公式対応済み） |

これは「マルチプラットフォーム対応ゲームエンジン」というだけなら珍しくないが、kaedevn の構造にはいくつかの特異な点がある。

## 2. アーキテクチャの核心：デュアルエンジン + 共通スクリプト

### 2.1 構造図

```
.ksc スクリプト（全プラットフォーム共通）
     │
     ├── [Web Path]
     │    KSC Compiler (TS) → Stack VM (TS)
     │         → KscHostAdapter → IOpHandler
     │         → WebOpHandler (PixiJS)
     │
     └── [Native Path]
          C++ Interpreter (step駆動)
               → IEngineAPI (C++ 純粋仮想クラス)
               → SDL2Engine (SDL_Renderer)
                    ├── macOS
                    ├── Android
                    ├── Windows (予定)
                    ├── iOS (予定)
                    └── Nintendo Switch (予定)
```

ポイントは2つある。

**1つ目：スクリプトが完全に共通であること。** `.ksc` ファイルはプラットフォーム固有の記述を一切含まない。`bg("shrine_entrance")` と書けば、Web では PixiJS が PNG をロードし、macOS/Android では SDL2 が同じ PNG をロードする。スクリプト作者は描画バックエンドを意識しない。

**2つ目：インタプリタが2系統存在すること。** ブラウザ用の TypeScript 実装（4,229行のコンパイラ + 605行のVM）と、ネイティブ用の C++ 実装（約1,977行）が同じスクリプト仕様を実装している。「同じ言語を2つの処理系が実行する」のは Java (JVM / GraalVM) や JavaScript (V8 / SpiderMonkey) と同じ構造だが、ノベルゲームエンジンでこれを自前で実現しているのは珍しい。

### 2.2 TS風インタプリタ（KSC言語）

KSC（Kaede Script）は TypeScript のサブセットに近い構文を持つ独自DSLである。

```typescript
// KSC スクリプトの例
bg("shrine_entrance");
ch("heroine", "normal", "center");
text("ここは神社の入り口だ。");
wait("click");

let affinity = 0;
choice("どうする？", [
  { text: "話しかける", action: () => { affinity += 10; jump("talk"); } },
  { text: "通り過ぎる", action: () => { jump("pass"); } }
]);
```

**なぜ TyranoScript や Ren'Py のタグ構文ではなく、TS風の関数呼び出しにしたのか。**

1. **AI が生成しやすい。** GPT-4o / Claude / Gemini などの LLM は JavaScript/TypeScript のコード生成に極めて高い精度を持つ。独自タグ構文よりも、関数呼び出し形式のほうが LLM の学習データに近く、生成品質が高い。
2. **エディタのサポートが容易。** 関数シグネチャが固定なので、入力補完・型チェック・リアルタイムバリデーションを実装しやすい。
3. **変数・条件分岐が自然。** `if (affinity > 50) { ... }` のような制御構造が、言語組み込みの構文として使える。TyranoScript の `[if exp="affinity > 50"]` より直感的。
4. **C++移植が容易。** 関数呼び出しの解釈は、タグパーサーよりもインタプリタとして実装しやすい。

KSC コンパイラは本格的なパイプラインを持つ：

| 段階 | モジュール | 行数 |
|------|-----------|------|
| 字句解析 | `lexer.ts` | — |
| 構文解析 | `parser.ts` | 714行 |
| 型検査 | `checker.ts` | 721行 |
| IR 生成 | `emitter.ts` | 845行 |
| スタックVM | `vm.ts` | 605行 |

VM はプロトタイプチェーン、クロージャ、配列・オブジェクトリテラル、switch 文をサポートする。ノベルゲームスクリプトにここまでの言語機能が必要かは議論の余地があるが、将来的にバトルシステムやミニゲームのロジックをスクリプト内に書けるという拡張性を担保している。

### 2.3 アセット共通化

全プラットフォームで同じアセットを使う。

```
assets/
  backgrounds/   bg01.png, bg02.png, ...
  characters/    ch01.png, ch02.png, ...
  audio/         bgm01.mp3, se01.mp3, ...
```

スクリプトからはスラッグ（文字列ID）でアセットを参照する。

| プラットフォーム | アセット解決 |
|----------------|-------------|
| Web | `manifest.json` → HTTP パス or API から取得した URL |
| macOS | `assets.json` → ローカルファイルパス (`std::ifstream`) |
| Android | `assets.json` → APK 内ファイルパス (`SDL_RWFromFile`) |

**制約（全プラットフォーム共通）:**
- 背景・スプライトの最大サイズ: 2048px（GPU テクスチャ制限の最小公倍数）
- アニメーション: スプライトシートのみ（個別フレーム PNG 禁止）
- 論理解像度: 1280×720 固定（SDL2 も PixiJS も同一座標系）

スラッグベースの参照とサイズ制約を揃えることで、「Web で動いたアセットがネイティブでも確実に動く」ことを保証している。プラットフォーム固有のアセット変換（テクスチャ圧縮など）は現時点では不要。PNG + MP3 がそのまま全環境で使える。

## 3. 「ブラウザで確認 → 即展開」ワークフローの意味

### 3.1 開発フロー

kaedevn の開発ワークフローは以下のようになる：

```
1. エディタ (ブラウザ, port 5176) でシナリオを編集
      ↓ リアルタイムプレビュー（コンパイル ~2ms）
2. プレビュー (ブラウザ, port 5175) で動作確認
      ↓ 同じ .ksc ファイル
3. ネイティブビルド → macOS / Android / Switch
```

**ステップ2→3で「変換」が不要であることが最大のポイント。**

Unity の場合、エディタで確認した内容をビルドするには：
- プラットフォーム切り替え（Build Settings → Switch Platform）
- アセットのリインポート（テクスチャ圧縮形式の変換）
- ネイティブビルド（IL2CPP によるC++変換 + コンパイル）
- 実機転送

この一連のプロセスが数分〜数十分かかる。Switch の場合は devkit への転送もある。

kaedevn の場合：
- `.ksc` ファイルはそのまま C++ インタプリタが読む。変換不要
- アセットは PNG/MP3 のまま使う。テクスチャ変換不要
- C++ ビルドは ~10秒（macOS）/ ~40秒（Android）
- スクリプトだけの変更ならリビルド不要（インタプリタが実行時に読む）

**つまり、ブラウザのプレビューで「OK」が出た瞬間、そのスクリプトとアセットがそのままネイティブで動く。** これは Unity や Unreal では原理的に実現できないワークフローである。

### 3.2 なぜこれが可能なのか

この「変換不要」の性質は、以下の設計判断の結果として生まれている：

1. **インタプリタ方式の採用。** スクリプトをネイティブコードにコンパイルしない。実行時に解釈する。これにより「ビルド」という概念がスクリプト側に存在しない。
2. **SDL_Renderer の使用。** 低レベル OpenGL / Vulkan / Metal ではなく、SDL2 の高レベル描画 API を使っている。SDL_Renderer は内部的にプラットフォーム最適な API を選択するため、テクスチャフォーマットの変換が不要。
3. **アセット参照のスラッグ化。** スクリプトはファイルパスではなくスラッグ（`"shrine_entrance"`）でアセットを参照する。パスの解決は各プラットフォームの AssetProvider が行う。これにより、ディレクトリ構造がプラットフォーム間で異なっても問題ない。

### 3.3 エディタとの統合

エディタは Vite ベースの SPA で、3カラム構成を持つ：

| カラム | 役割 |
|-------|------|
| 左: SidebarOutline | ページ一覧、変数、キャラクター、アセット管理 |
| 中央: BlockList | ブロックカード形式のシナリオ編集 |
| 右: SidebarInspector | プロパティ編集 + PixiJS エンジンのリアルタイムプレビュー |

`selectedBlockId` という単一の state が3カラムを同期する。ブロックを選択すると、対応するシーンがプレビューに即座に反映される。KSC コンパイラのベンチマークは1,000行スクリプトで約2ms。人間の知覚限界（16ms = 60fps）より十分速く、「編集即プレビュー」が文字通り実現されている。

さらに、Claude Code（AI）との組み合わせでは以下のワークフローが可能：

```
Claude Code でシナリオ/演出を .ksc で生成
  → エディタでリアルタイムプレビュー（0.1秒以下）
  → 「もうちょっとフェードゆっくり」と指示
  → Claude Code が修正 → 即反映
```

これは従来のゲーム開発では不可能だった速度のイテレーションである。

## 4. 既存ゲームエンジンとの比較

### 4.1 Unity との比較

| 観点 | Unity | kaedevn |
|------|-------|---------|
| **ライセンス** | Switch向け Pro 必須（年 $2,040） | SDL2: zlib License（無料） |
| **言語** | C# | C++（エンジン）+ KSC DSL（コンテンツ） |
| **エンジンサイズ** | 数百MB（エディタ込み数GB） | C++ 部分 ~2,000行 |
| **ビルド時間** | 数分〜数十分 | ~10秒（macOS）/ ~40秒（Android） |
| **スクリプト変更の反映** | リビルド必要 | リビルド不要（インタプリタ実行） |
| **ブラックボックス度** | 高（エンジン内部は非公開） | ゼロ（全コード把握可能） |
| **AI による量産** | 可能だが非効率（C# + Unity API の知識が必要） | 高効率（KSC は LLM が自然に生成できる構文） |
| **ノベルゲーム特化** | 汎用（Fungus 等のアセットが必要） | 完全特化 |

**Unity を選ばない最大の理由は「オーバースペック」である。** ノベルゲームに必要な機能は限定的だ：2D スプライト表示、テキスト描画、音声再生、選択肢UI、変数管理、セーブ/ロード。これは C++ + SDL2 で 2,000〜3,000行に収まる規模であり、Unity の膨大な機能群は不要である。

さらに、Unity のライセンスコストは量産モデルと相性が悪い。kaedevn は50本以上のタイトルを量産する計画であり、年 $2,040 のライセンス料は全タイトルに適用される。独自エンジンなら固定コストのみ。

### 4.2 Ren'Py との比較

| 観点 | Ren'Py | kaedevn |
|------|--------|---------|
| **言語** | Python + 独自スクリプト | C++ + TypeScript 風 DSL |
| **Switch 対応** | 不可 | 可能（SDL2 + devkitPro） |
| **パフォーマンス** | Python（遅い） | C++ ネイティブ（高速） |
| **エディタ** | テキストエディタ | ブラウザベース GUI エディタ |
| **AI 連携** | なし | エディタ + AI 執筆支援統合 |
| **Web 展開** | 限定的（Emscripten） | ネイティブ対応（PixiJS） |
| **コミュニティ** | 大規模（10年以上の実績） | 新規 |

Ren'Py は VN 特化エンジンとして最も成熟しているが、**Nintendo Switch に対応していない**ことが最大の差別化ポイントになる。Ren'Py は Python ベースであり、Switch の NX SDK が Python をサポートしていないため、移植は事実上不可能。

kaedevn は設計段階から Switch 展開を前提としており、C++ + SDL2 という Switch 直結のスタックを採用している。SDL2 のうち約90%のコードがそのまま Switch で動作し、Nintendo SDK 固有の実装が必要なのは約3%（プラットフォーム初期化、Joy-Con マッピング、eShop API）のみである。

### 4.3 吉里吉里/KAG との比較

| 観点 | 吉里吉里 | kaedevn |
|------|---------|---------|
| **アーキテクチャ** | TJS（独自スクリプト）+ C++ | KSC（TS風DSL）+ C++ |
| **プラットフォーム** | Windows のみ | Web + macOS + Android + Switch |
| **エディタ** | 外部テキストエディタ | ブラウザ統合エディタ |
| **開発状況** | メンテナンスフェーズ | アクティブ開発 |

吉里吉里は kaedevn と最も構造が近い先行者である。「独自スクリプト言語 + C++ エンジン」という構成は同一だが、吉里吉里が Windows 専用であるのに対し、kaedevn は最初からマルチプラットフォームを前提としている。SDL2 の抽象化レイヤーにより、プラットフォーム固有コードが最小限に抑えられている点が構造的な違い。

### 4.4 Cocos Creator / Defold との比較

| 観点 | Cocos Creator | Defold | kaedevn |
|------|--------------|--------|---------|
| **スクリプト** | TypeScript | Lua | KSC (TS風) |
| **ネイティブ層** | C++ | C++ | C++ |
| **Switch 対応** | あり | あり | あり（予定） |
| **VN 特化** | なし（汎用2D） | なし（汎用2D） | 完全特化 |
| **エンジンサイズ** | 大規模 | 中規模 | 極小（~2,000行） |
| **ブラウザエディタ** | あり | あり | あり |

Cocos Creator と Defold は「スクリプト層 + C++ ネイティブ層」の構成で kaedevn と設計思想が近い。違いは**VN 特化であること**。汎用エンジンは物理演算、パーティクル、タイルマップ、3D など VN に不要な機能を大量に持つ。kaedevn はこれらを全て排除し、ノベルゲームのドメインロジックだけを実装する。結果、エンジン全体が2,000行に収まり、全コードを把握できる。

## 5. 技術的インパクト

### 5.1 「ブラウザでテスト → 全プラットフォーム展開」パターンの汎用性

kaedevn が確立しつつあるパターンは、ノベルゲーム以外にも応用可能な設計パラダイムである：

```
[開発時]  ブラウザ (TS実装) で即時フィードバック
    ↓ 同一スクリプト・同一アセット
[配布時]  ネイティブ (C++実装) でパフォーマンス・プラットフォーム最適化
```

これは **「開発体験はブラウザの即時性を享受し、配布時はネイティブの性能を得る」** という、従来は二者択一だった利点を両立する。

従来の選択肢：
- **ブラウザ完結（Electron / Tauri）**: 開発は楽だが、パフォーマンスと配布サイズに課題。Switch 非対応。
- **ネイティブ完結（Unity / C++）**: 性能は出るが、イテレーション速度が遅い。ビルド待ち。

kaedevn のアプローチ：
- **開発時**: ブラウザ上の TS インタプリタで即時プレビュー（コンパイル 2ms）
- **配布時**: C++ インタプリタが同じスクリプトをネイティブ実行
- **移行コスト**: ゼロ（同一ファイルを両方が読む）

### 5.2 AI 量産との相性

kaedevn の設計は AI によるコンテンツ量産を前提としている。

| 従来のワークフロー | kaedevn のワークフロー |
|-------------------|---------------------|
| AI がテキストを生成 | AI が `.ksc` スクリプトを直接生成 |
| 人間がエンジン固有の形式に変換 | **変換不要**（KSC がそのまま実行可能） |
| エンジンに読み込んで動作確認 | **ブラウザで即確認**（2ms） |
| バグがあればテキストを修正して再変換 | AI に修正指示 → 即反映 |
| ビルド → 実機確認 | スクリプトは変更なしでネイティブ動作 |

AI 執筆支援パイプライン（4ステージ）も既に実装されている：

```
Stage 0: 全体骨格（章プロット）      → gpt-4o-mini
Stage 1: 話プロット                 → gpt-4o-mini
Stage 2: 話テキスト                 → claude-sonnet
Stage 3: .ksc スクリプト変換         → TS テンプレート
```

生成された `.ksc` がそのまま全プラットフォームで動くため、**「書いたらすぐ動く」体験**が AI 量産パイプラインの中に組み込まれている。これは Ren'Py や Unity ベースのワークフローでは実現が難しい。

### 5.3 エンジン規模と保守性

各エンジンの規模を比較すると、kaedevn の小ささが際立つ：

| エンジン | コード規模 | 全コード把握 |
|---------|-----------|-------------|
| Unity | 数百万行 | 不可能 |
| Unreal Engine | ~3,000万行 | 不可能 |
| Ren'Py | ~10万行 | 困難 |
| Defold | ~50万行 | 困難 |
| **kaedevn (C++)** | **~2,000行** | **可能** |
| **kaedevn (TS)** | **~17,000行** | **可能** |

C++ エンジン部分が約2,000行に収まっているのは、ノベルゲームに不要な機能を一切持たないためである。これは保守性に直結する：

- バグの特定が容易（全コードを読める）
- プラットフォーム固有の問題が局所化される（`#ifdef` は5ファイルのみ）
- AI（Claude Code）による修正が高精度（コンテキストに全コードが収まる）

### 5.4 コマンドセットの凍結による移植安全性

kaedevn はスクリプトコマンドを「Core（Switch保証）」と「WebOnly（Web専用）」に厳密に分離している。

**Core コマンド（Switch 保証）:**
text, choice, jump, set, if, show, hide, move, fade, playBgm, playSe, playVoice, wait

**WebOnly コマンド（Web 限定）:**
openUrl, share, analytics, webOnlyUI

ルール：**メインシナリオは Core コマンドのみで書く。** WebOnly はメニュー画面やナビゲーション等の UI 層に限定する。

この制約により、「このコマンド Switch で動かないんだけど」というリスクを設計段階で排除している。Unity の場合、プラットフォーム非対応の API を使ってしまうリスクは常に存在し、ビルド時に初めて発覚する。kaedevn ではコマンドセットが凍結されているため、この問題が原理的に起こらない。

## 6. 差別化のまとめ

kaedevn が既存エンジンに対して持つ差別化要素を整理する：

### 6.1 Unity / Unreal に対して

| 差別化ポイント | 詳細 |
|---------------|------|
| **ライセンス無料** | SDL2 は zlib ライセンス。Unity Pro の年 $2,040 が不要 |
| **エンジンサイズ** | 2,000行 vs 数百万行。全コード把握可能 |
| **ビルド速度** | ~10秒 vs 数分〜数十分 |
| **スクリプト変更の即時性** | リビルド不要。インタプリタが直接実行 |
| **AI 量産特化** | KSC は LLM が自然に生成できる TS 風構文 |
| **オーバースペック排除** | 物理演算・3D・パーティクル等の不要機能なし |

### 6.2 Ren'Py に対して

| 差別化ポイント | 詳細 |
|---------------|------|
| **Switch 対応** | C++ + SDL2 で直接移植可能。Ren'Py は Python ベースで不可 |
| **パフォーマンス** | C++ ネイティブ vs Python |
| **ブラウザエディタ** | GUI エディタ + リアルタイムプレビュー |
| **AI パイプライン統合** | 4ステージ執筆支援 + .ksc 直接生成 |

### 6.3 独自の差別化（他エンジンにない特徴）

| 特徴 | 意味 |
|------|------|
| **デュアルインタプリタ** | TS版（開発用）と C++版（配布用）が同一スクリプトを実行。開発体験と配布性能を両立 |
| **ブラウザで確認 → 即ネイティブ展開** | スクリプト・アセットの変換が不要。「ブラウザで OK = 全プラットフォーム OK」 |
| **AI-to-AI バトンタッチ** | Gemini が設計 → Claude がビルド・移植。IEngineAPI の抽象化が AI 間の引き継ぎを可能にした |
| **コマンドセット凍結** | Core / WebOnly の分離により、移植時の「このコマンド動かない」問題を原理的に排除 |
| **エディタ統合 AI 量産** | 「AI生成 → ブラウザ確認 → ネイティブ配布」がシームレス |

## 7. リスクと現実的な課題

楽観的な分析だけでは公平ではない。現実的な課題も記しておく。

### 7.1 2つのインタプリタの同期コスト

TS版と C++版のインタプリタが同じ仕様を実装し続ける必要がある。新機能を追加するたびに両方を更新しなければならない。仕様が小さいうちは管理可能だが、言語機能が増えると同期コストが急増する。

**対策**: コマンドセットを凍結し、言語仕様の肥大化を防ぐ。新機能は慎重に追加する。

### 7.2 コミュニティの不在

Unity は数百万人、Ren'Py は数万人のユーザーコミュニティを持つ。kaedevn は現時点で1人の開発者が AI と共に開発している。ドキュメント、チュートリアル、サードパーティ拡張、Stack Overflow の回答——これらは全て存在しない。

**対策**: OSS 化を進め、技術記事（Zenn）を通じて可視性を高める。ただし、ノベルゲームエンジンのニッチ市場でコミュニティが育つかは未知数。

### 7.3 Switch 対応の未検証

SDL2 が Nintendo Switch をサポートしていることは事実だが、kaedevn の C++ コードが実際に Switch でビルド・動作するかは未検証。NDA の壁もあり、Nintendo Developer Program に承認されるまで実機テストはできない。

**対策**: Switch 固有の実装が必要な部分を3%（初期化、Joy-Con、eShop API）に抑える設計にしている。残り97%は macOS / Android で動作確認済みの SDL2 コード。

### 7.4 パフォーマンスの天井

SDL_Renderer は抽象化の代償として、低レベル GPU プログラミングの最適化ができない。エフェクトの多いシーン（パーティクル、複雑なシェーダー）ではパフォーマンスが不足する可能性がある。

**対策**: ノベルゲームではそもそも GPU 負荷が低い。背景1枚 + キャラクター2〜3体 + テキストウィンドウという構成では、SDL_Renderer で十分。

## 8. 「50本量産 × 6プラットフォーム」のインパクト

kaedevn の最終的な価値は、スケールしたときに現れる。

1本のゲームを1プラットフォームにリリースするだけなら、Unity でも Ren'Py でも十分だ。しかし、**50本のゲームを6プラットフォームに展開する** = 300リリースという規模では、以下のコスト差が効いてくる：

| コスト項目 | Unity | kaedevn |
|-----------|-------|---------|
| ライセンス | $2,040/年 | $0 |
| プラットフォーム切り替えビルド | 50本 × 5回 = 250回 | 0回（同一バイナリ） |
| スクリプト変換 | 50本 × 手動変換 | 0回（.ksc がそのまま動く） |
| AI 生成の後処理 | AI出力 → エンジン形式変換 | AI出力 = エンジン形式 |
| エンジンのバグ対応 | Unity のアップデート待ち | 自前2,000行を直接修正 |

**量産 × マルチプラットフォームの掛け算で、kaedevn の「変換不要」という特性が指数的に効く。**

## 9. 結論

kaedevn のマルチプラットフォーム戦略は、3つの設計判断に支えられている：

1. **デュアルインタプリタ（TS + C++）**: 開発はブラウザの即時性、配布はネイティブの性能。同一スクリプトが両方で動く。
2. **SDL2 + SDL_Renderer**: 低レベル GPU に触らず、描画コードの99%をプラットフォーム共通にする。結果、Android 移植で SDL2Engine.cpp は1行も変更せずに動いた。
3. **コマンドセット凍結 + スラッグベースアセット**: スクリプトもアセットも「名前で参照」し、物理的なパスや形式の違いを抽象化する。

これにより、**「ブラウザで動いたら全プラットフォームで動く」** という、ノベルゲーム開発においては理想的なワークフローが実現されている。Unity や Unreal のようなビルド→実機確認のサイクルが不要であり、AI による量産との相性が極めて高い。

2,000行の C++ エンジンと17,000行の TypeScript コードベースで、Web + macOS + Android + 将来の Windows / iOS / Switch をカバーする。これは「小さなエンジンで大きな展開」を実現する、AI 時代のノベルゲーム開発アーキテクチャの一つの形だ。
