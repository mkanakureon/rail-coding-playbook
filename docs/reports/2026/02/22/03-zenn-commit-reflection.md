Claude Code と一緒に開発していると、ふと思った。

「このコミット、何を考えながら作ったか、後で絶対忘れるな」

## コミットメッセージに感想を入れるようにした

Claude Code に、コミットするたびに**今回の作業についての一言感想**を末尾に入れてもらうようにした。

ルールはシンプル。

```
{プレフィックス}: {日本語の説明}

---
（Claude Code の感想・振り返り、2〜3行）

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

実際のコミットメッセージはこんな感じになる。

```
docs: KS/KSC コマンド対応表・リファレンス追加、インタープリタ更新

---
KSとKSCは別物だと改めて整理できてよかった。
コマンド表、地味に手間かかったけど、これは手元に置いておきたいやつ。
wait の単位が秒とミリ秒でバラバラなのは、うっかりすると絶対ハマりそう。

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

```
feat: ksc スクリプトブロックをエディタに追加

---
スクリプトブロックの追加、思ったよりすんなりいった。
エディタがここまで育ってきたのを実感する作業だった。
次はユーザーが実際に書いてみたときの動作確認が楽しみ。

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

## これ、雑音じゃないの？

最初に思ったのはこれ。コミットメッセージは簡潔であるべきというのが一般的なベストプラクティスだし、感想を入れるなんて邪道では？と。

でも使ってみると、**意外と邪魔にならない**。

`git log --oneline` では表示されないし、`git log` で詳細を見たいときだけ目に入る。そしてそのとき、「あ、このコミット、こういう気持ちでやってたのか」とわかる。

ドキュメントを整備したコミットに「地味だけど大事」と書いてある。バグ修正のコミットに「原因に気づいたときは思わず声が出た」と書いてある。そういうのが残っていると、後で読み返したとき**コードの変遷だけじゃなく、開発の温度感が伝わってくる**。

## Claude Code のスキルとして定義した

毎回指示するのは面倒なので、`.claude/skills/commit/skill.md` にルールをまとめた。

Claude Code には「コミットするとき、末尾に今回の作業の感想をゆるいトーンで書いて」と指示している。堅苦しくしないこと、たわいない感想でいいことも明記。

そうすることで Claude Code は毎回、少し違うトーンで、でも作業内容に対して正直な一言を書いてくれる。

## コミットログが開発日記になった

結果として、`git log` を開くのが少し楽しくなった。

過去の自分（とClaude Code）が何を考えていたか、何が大変で、何がうまくいったか。それがコードの変更と一緒に残っている。

雑音か信号かで言えば、**読む人の文脈による**とは思う。チームで使っているリポジトリには向かないかもしれない。でも個人や少人数プロジェクトでは、コミットログが「記録」以上の何かになる。

試してみる価値はあると思う。
