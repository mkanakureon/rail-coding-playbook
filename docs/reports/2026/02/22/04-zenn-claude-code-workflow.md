Claude Code を使って数ヶ月経ったころ、「毎回同じことを指示している」と気づいた。

レポートを保存するとき、コミットするとき、Zenn 記事を書くとき。いつも同じような指示を打っている。それなら最初から「こういうときはこうして」とルールを書いておけばいい。それが Claude Code の**スキルシステム**だった。

## Claude Code のスキルとは

`.claude/skills/{スキル名}/skill.md` にルールを書いておくと、Claude Code がそのルールに従って動いてくれる。特定のキーワードや `/スキル名` で呼び出せる。

```
.claude/skills/
  save-report/skill.md   # レポート保存ルール
  commit/skill.md        # コミットルール
  zenn/skill.md          # Zenn記事作成ルール
```

カスタムコマンドをプロジェクトに埋め込んでおく、くらいのイメージ。

## save-report: 作業記録を日付フォルダに自動保存

最初に作ったのがこれ。作業のたびに `docs/09_reports/2026/02/22/01-xxx.md` のような形式でレポートを保存するスキル。

```
docs/09_reports/{年}/{月}/{日}/{連番}-{タイトル}.md
```

ところが最初はうまく動かなかった。「ここに文書を保存して」と言ったら `docs/` のルートに保存してしまったのだ。

原因はトリガーの定義が甘かったこと。スキルには「こういう言葉で発動する」というトリガー表現を書くのだが、「ここに保存して」という言葉が入っていなかった。

```markdown
| ユーザーの表現 | 動作 |
|---|---|
| "振り返り文書を作成して" | レポートを保存 |
| "docs/09_reports に保存して" | 明示的な保存先指定 |
```

これでは「ここに」「保存して」のような短い表現は拾えない。

修正後は「保存」「文書」「md」「まとめ」などのキーワードを含む指示なら保存先が明示されていなくても `docs/09_reports/` を使うよう追記した。さらに**デフォルト保存先の明確化**として「`ここ` は常に `docs/09_reports/` を指す」と明記した。

スキルは使いながら育てるものだと実感した。

## commit: コミットルールと、ちょっと面白い仕掛け

次に作ったのがコミットスキル。過去のコミット履歴を眺めて、このプロジェクトのスタイルを定義した。

```
feat: ksc スクリプトブロックをエディタに追加
fix: タイムライン編集UI改善・チャンネル生成共通化
docs: KS/KSC コマンド対応表・リファレンス追加
```

プレフィックスは `feat` / `fix` / `docs` / `refactor` / `chore`、説明は日本語、複数変更は `・` で区切る。これをスキルに定義した。

ここに**ちょっとした遊び心**を加えた。コミットメッセージの末尾に、Claude Code の作業感想を入れてもらうようにしたのだ。

```
docs: KS/KSC コマンド対応表・リファレンス追加

---
KSとKSCは別物だと改めて整理できてよかった。
コマンド表、地味に手間かかったけど、これは手元に置いておきたいやつ。
waitの単位が秒とミリ秒でバラバラなのは、うっかりすると絶対ハマりそう。

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

「これ、雑音じゃない？」と自分でも思った。でも `git log --oneline` では表示されないし、詳細を見たいときだけ目に入る。そしてそのとき、「このコミット、こういう温度感でやってたのか」とわかる。

コードの変遷だけじゃなく、**開発のその時の気持ち**が残る。コミットログが開発日記になった感覚がある。

チームのリポジトリには向かないかもしれない。でも個人プロジェクトなら試す価値はあると思っている。

## zenn: この記事自体もスキルで書いている

そしてこの記事を書くために Zenn スキルを作った。

作るきっかけは「Zenn に投稿したい」という一言から始まった相談だった。コミットメッセージの感想を Zenn に投稿したら面白いか、雑音か、という話。

正直に言えば**単独では薄い**。「AIにコミット感想を書かせた」だけでは記事として密度が足りない。でも「スキルシステムを使って開発ワークフローを整えていく過程」の一エピソードとして入れるなら面白い。

そういう判断もスキルに書いておくことにした。

```markdown
### 記事が「雑音」になる条件（避ける）
- アイデア1つだけで記事にする（エピソードが少ない）
- 「こんなことができます」だけで終わる（実用性が薄い）
```

## やってみてわかったこと

スキルシステムを使い始めてから、Claude Code への指示が短くなった。「コミットして」「保存して」で意図通りに動く。

ただ、最初からうまく定義できるわけではない。「ここに保存して」で失敗したように、使いながら足りない部分に気づいて育てていく。スキルファイルはコードと同じで、最初のバージョンが完成形ではない。

もう一つ気づいたのは、スキルに書くことで**自分の作業ルールが言語化される**ということ。「コミットメッセージはこういう形式で」「レポートはここに保存」というルールは頭の中にあったが、スキルとして書くことで初めて整理された。

Claude Code のためのドキュメントを書いているようで、実は自分のための作業メモになっている。

---

スキルファイルは `.claude/skills/` に置くだけで使えるので、試してみる場合はそこから始めると手軽です。
