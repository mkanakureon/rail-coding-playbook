# 転送量最適化 実装計画書

## Context

現在、画像アセットは API サーバー経由（ローカル）または Azure Blob 直 URL で配信されている。
CDN（Front Door）導入と合わせて、Cache-Control ヘッダ・ハッシュファイル名・ASSET_BASE_URL 一元管理を実装し、転送費を最小化する。

Front Door 自体はAzure上でのみ設定可能だが、**アプリケーション側の変更はすべてローカルでテスト可能**な設計にする。

## 実装フェーズ

### Phase 1: ASSET_BASE_URL 導入（URL生成の一元化）

**目的:** アセットURLの生成を1箇所に集約し、CDN/Blob/ローカルの切替を環境変数だけで行えるようにする

**変更ファイル:**
- `apps/hono/src/lib/config.ts` — `resolveAssetUrl()` を `ASSET_BASE_URL` 環境変数で制御
- `apps/hono/.env` — `ASSET_BASE_URL` 追加（ローカル: 未設定 → `/uploads/` のまま）
- `apps/editor/src/config/api.ts` — `getAssetUrl()` は変更不要（すでに http 判定済み）
- `apps/next/app/(private)/admin/assets/page.tsx` — `resolveUrl()` も同様

**ロジック:**
```
ASSET_BASE_URL 設定あり → `${ASSET_BASE_URL}/${path}`
ASSET_BASE_URL 未設定 + Azure → `https://{account}.blob.core.windows.net/assets/${path}`
ASSET_BASE_URL 未設定 + Local → `/uploads/${path}`
```

**ローカルテスト:**
- `ASSET_BASE_URL` 未設定 → `/uploads/bg/xxx.png` が返ること
- `ASSET_BASE_URL=http://localhost:8080/uploads` → 完全URL が返ること
- ユニットテスト: `resolveAssetUrl()` の各パターン

### Phase 2: Cache-Control ヘッダ付与

**目的:** Blob アップロード時とローカル静的配信時に適切な Cache-Control を設定

**変更ファイル:**
- `apps/hono/src/lib/azure.ts` — `uploadBlobFromBuffer()` / `uploadBlobFromStream()` に `blobCacheControl` 追加
- `apps/hono/src/index.ts` — `/uploads/*` 静的配信に Cache-Control ミドルウェア追加
- `apps/hono/src/routes/assets.ts` — API レスポンスに `Cache-Control: no-store` 追加（不要な中間キャッシュ防止）

**ヘッダ値:**

| 対象 | Cache-Control |
|------|--------------|
| 画像・音声・フォント | `public, max-age=31536000, immutable` |
| JSON（manifest 等） | `public, max-age=60` |
| API レスポンス | `no-store` |

**ローカルテスト:**
```bash
# 静的ファイルのヘッダ確認
curl -I http://localhost:8080/uploads/bg/xxx.png
# → Cache-Control: public, max-age=31536000, immutable

# API レスポンスのヘッダ確認
curl -I http://localhost:8080/api/assets/xxx
# → Cache-Control: no-store
```

### Phase 3: コンテンツハッシュファイル名

**目的:** ファイル内容のハッシュをファイル名に含め、キャッシュ破棄を URL の変更で行う（CDN immutable 戦略の前提）

**変更ファイル:**
- `apps/hono/src/routes/assets.ts` — アップロード時にSHA256ハッシュ生成、ファイル名を `{hash}.{ext}` に変更
- `apps/hono/src/lib/hash.ts` — 新規: `generateContentHash(buffer: Buffer): string`（SHA256の先頭16文字）

**ロジック:**
```
現在: bg/{ULID}.png
変更後: bg/{sha256-16chars}.png
```

既存アセットは `normalizeAssetUrl()` で引き続き解決可能（後方互換）。

**ローカルテスト:**
- 同じファイルを2回アップロード → 同じハッシュファイル名（冪等性）
- 異なるファイル → 異なるハッシュファイル名
- ユニットテスト: `generateContentHash()` のハッシュ値検証

### Phase 4: 画像最適化（WebP 変換）

**目的:** アップロード時に画像を WebP に変換し、転送サイズを削減

**変更ファイル:**
- `apps/hono/package.json` — `sharp` 依存追加
- `apps/hono/src/lib/image.ts` — 新規: `optimizeImage(buffer, options)` → WebP 変換 + リサイズ
- `apps/hono/src/routes/assets.ts` — アップロードフローに最適化ステップ追加

**ルール:**
- 画像のみ変換（bg, ch）。音声（bgm）はスキップ
- 最大幅 2048px（CLAUDE.md の Asset Constraints 準拠）
- 元画像が WebP より小さい場合は元画像を採用
- 品質: 80（調整可能）

**ローカルテスト:**
- PNG をアップロード → WebP で保存されること
- ファイルサイズが縮小されていること
- 2048px 超の画像 → リサイズされること
- ユニットテスト: `optimizeImage()` の出力形式・サイズ検証

### Phase 5: ローカル静的配信の圧縮

**目的:** ローカル開発時も gzip/brotli 圧縮を有効にし、本番 CDN の挙動を模擬

**変更ファイル:**
- `apps/hono/src/index.ts` — Hono `compress` ミドルウェア追加

**ローカルテスト:**
```bash
curl -H "Accept-Encoding: gzip" -I http://localhost:8080/uploads/bg/xxx.webp
# → Content-Encoding: gzip
```

### Phase 6: Front Door（CDN）設定スクリプト

**目的:** Azure Front Door のデプロイと設定を自動化（Azure 上でのみ実行）

**変更ファイル:**
- `scripts/setup-frontdoor.sh` — 新規: Front Door プロファイル・エンドポイント・ルール作成

**設定内容:**
- `/assets/*`, `/works/*` → Blob Origin、長期キャッシュ、圧縮ON
- `/api/*` → Container Apps Origin、キャッシュOFF
- Query string: Ignore

**テスト:** Azure 上で `curl -I https://cdn.example.com/assets/bg/xxx.webp` でヘッダ確認

## テスト戦略

### ユニットテスト（Phase 1-4、全てローカル）

| テスト | ファイル |
|--------|---------|
| `resolveAssetUrl()` 全パターン | `apps/hono/src/lib/config.test.ts` |
| `generateContentHash()` | `apps/hono/src/lib/hash.test.ts` |
| `optimizeImage()` | `apps/hono/src/lib/image.test.ts` |

### 結合テスト（ローカル）

| テスト | 方法 |
|--------|------|
| アップロード → WebP 変換 → ハッシュ名保存 | `curl` でファイルアップロード、レスポンスのURL確認 |
| Cache-Control ヘッダ | `curl -I` で静的ファイル・API レスポンスのヘッダ確認 |
| 圧縮 | `curl --compressed` でレスポンスサイズ比較 |
| URL 生成 | `ASSET_BASE_URL` の有無でレスポンスのURL形式確認 |

### Azure テスト（Phase 6 のみ）

| テスト | 方法 |
|--------|------|
| CDN キャッシュ | `curl -I` で `X-Cache: HIT` 確認 |
| Origin → CDN 経路 | 2回目リクエストで Blob egress が増えないこと |

## 実装順序と依存関係

```
Phase 1 (ASSET_BASE_URL) ← 他の全 Phase の前提
  ↓
Phase 2 (Cache-Control) ← Phase 6 の前提
  ↓
Phase 3 (ハッシュファイル名) ← Phase 2 の immutable 戦略に必要
  ↓
Phase 4 (WebP 変換) ← 独立、いつでも可
  ↓
Phase 5 (圧縮) ← 独立、いつでも可
  ↓
Phase 6 (Front Door) ← Phase 1-2 完了後に実施
```
