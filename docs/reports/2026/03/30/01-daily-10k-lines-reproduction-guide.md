## 2. なぜ可能になったか — 5つの構造的要因

### 要因① CLAUDE.md / GEMINI.md — 「AIの憲法」

AIに毎回プロジェクトの文脈を説明する必要がない。CLAUDE.md（202行）がセッション開始時に自動ロードされ、以下を固定する:

- アーキテクチャ（3層分離、IInput/IAudio/IStorage 抽象化）
- コマンドセット（frozen: 変更不可）
- Save Schema（frozen）
- 解像度・レイアウトルール
- デプロイ手順（GitHub Actions 限定）
- バグ修正の手順（推測変更禁止）
- DB接続情報・サーバー構成

**効果**: AIが「どう書くべきか」を迷う時間がゼロになる。10回のセッションで10回とも同じ品質基準で出力される。

### 要因② カスタムスキル30種 — 定型作業の完全自動化

`.claude/skills/` に30種のスキルが定義されている:

| カテゴリ | スキル例 | 効果 |
|---------|---------|------|
| Git操作 | `/commit`, `/push`, `/pr` | コミットメッセージの品質・Co-Author 付与を自動化 |
| テスト | `/test` | 変更ファイルから影響範囲を自動判定→関連テストのみ実行→報告書保存 |
| 文書 | `/save-report`, `/devlog`, `/qiita`, `/zenn` | 報告書・日誌・記事を定型フォーマットで自動生成 |
| デプロイ | `/deploy-azure`, `/test-azure` | Azure デプロイ・ヘルスチェックをスキル1発で実行 |
| RAG検索 | `/rag-search` | docs/ 10万行以上をベクトル+キーワードのハイブリッド検索 |
| エディタ | `/edit-blocks`, `/character`, `/map`, `/asset` | ゲームコンテンツの CRUD 操作をCLIで完結 |
| 品質 | `/browser-verify`, `/broken-memo` | Playwright で UI 検証、壊れたテストの追跡 |

**効果**: 人間が「コミットして」と言うだけで、staging → メッセージ生成 → Co-Author 付与 → フック通過 → 完了確認まで一貫実行される。手順ミスがゼロになる。

### 要因③ モノレポ + 疎結合パッケージ構造

```
packages/
  core/          ← 型定義・インターフェース（変更頻度：低）
  compiler/      ← KSCスクリプトコンパイラ
  interpreter/   ← ランタイム実行エンジン
  web/           ← Web実装（PixiJS）
  map/           ← マップエンジン
  battle/        ← バトルシステム
  ksc-compiler/  ← KSCコンパイラ
  ai-gateway/    ← AI API 抽象化
apps/
  editor/        ← ビジュアルエディタ（React）
  hono/          ← バックエンドAPI
  next/          ← 認証・プロジェクト管理
```

**効果**:
- AIが1つのパッケージを修正する際、他パッケージのコンテキストを読む必要がない
- `packages/core` のインターフェースが決まれば、実装は全パッケージに**並列展開**可能
- 50種のフィルターを追加するとき、パターン1つを理解すれば残り49は「コピー＆アダプト」

### 要因④ Hooks による自動品質ゲート

`.claude/settings.json` で2つのフックが常時稼働:

```json
{
  "hooks": {
    "PreToolUse": [
      // .env, .eslintrc, settings.json への書き込みをブロック
    ],
    "PostToolUse": [
      // .ts/.tsx/.js/.jsx の編集後に oxlint を自動実行
    ]
  }
}
```

加えて:
- **Pre-push hook** (`.husky/pre-push`): typecheck + lint が通らないと push できない
- **Pre-commit hook**: ハードコードURL チェック

**効果**: AIが「壊す」ことへの恐れなく全速力で書ける。壊れたら即座に検知される。

### 要因⑤ 「不読不踏」監査モデル — docs/10_ai_docs + docs/09_reports

AIが実装するたびに:
1. **設計書** → 実装前に書く（`docs/10_ai_docs/`）
2. **テスト報告書** → テスト後に自動保存（`docs/09_reports/`）
3. **開発日誌** → 1日の終わりに自動生成（`docs/landing/devlog/`）

**効果**: 人間は1万行のコードを読む代わりに、AIが書いた1ページの報告書を読むだけで Go/No-Go を判断できる。レビューがボトルネックにならない。

## 3. 再現手順 — ゼロから構築するための具体的ステップ

### Step 1: AI用憲法を書く（所要: 2〜4時間）

プロジェクトルートに `CLAUDE.md`（Claude Code 用）を作成する。

```markdown
# CLAUDE.md

## Project Overview
（プロジェクトの目的と技術スタック）

## Architecture
（パッケージ構成、依存関係、抽象化ルール）

## Development Commands
（ビルド、テスト、起動のコマンド）

## Rules
（コーディング規約、バグ修正手順、禁止事項）
```

**書くべき内容:**

| セクション | 内容 | なぜ必要か |
|-----------|------|-----------|
| Architecture | パッケージ構成、インターフェース一覧 | AIが正しい場所にコードを書く |
| Frozen Schema | 変更不可のデータ構造 | AIが勝手に互換性を壊さない |
| Commands | ビルド・テスト・起動の正確なコマンド | AIが正しいコマンドを実行する |
| Rules | バグ修正手順、禁止事項 | AIが推測で変更しない |
| DB/Credentials | 接続情報 | AIが環境構築を自分で行える |

**コツ:**
- 「AIが迷うポイント」を先回りして書く
- 200行以内に収める（長すぎるとAIが重要な部分を見落とす）
- 「禁止事項」は具体的に書く（「推測でコード変更禁止」など）

### Step 2: カスタムスキルを作る（所要: 1日）

`.claude/skills/{スキル名}/skill.md` にスキル定義を置く。

**最初に作るべき5つ:**

#### 1. `/commit` — コミットスキル

```markdown
---
description: Triggers on "コミットして", "commit"
---

# コミットスキル

1. git status → git diff → git log（スタイル確認）
2. ステージ → メッセージ生成 → コミット
3. メッセージ末尾に感想 + Co-Authored-By を付与
```

#### 2. `/test` — スマートテスト

```markdown
# テストスキル

1. git diff で変更ファイルを取得
2. 変更パスからテスト対象ワークスペースを判定
3. 対象テストのみ実行
4. 報告書を docs/09_reports/ に保存
```

#### 3. `/save-report` — レポート保存

```markdown
# レポート保存スキル

保存先: docs/09_reports/{YYYY}/{MM}/{DD}/{連番}-{タイトル}.md
フォーマット: 背景 → 方針 → 変更一覧 → 詳細 → 教訓
```

#### 4. `/push` — プッシュ

#### 5. `/devlog` — 開発日誌

**コツ:**
- スキルは「手順書」ではなく「AIの行動プロトコル」として書く
- トリガーワード（日本語・英語）を `description` に明記する
- 出力フォーマットを具体例付きで定義する

### Step 3: モノレポ + 疎結合構造にする（初期設計時）

```
your-project/
  packages/
    core/       ← 型・インターフェースのみ（実装なし）
    impl-web/   ← Web実装
    impl-xxx/   ← 別プラットフォーム実装
  apps/
    frontend/   ← UI
    backend/    ← API
```

**ルール:**
- `packages/core` は型定義とインターフェースのみ。実装を入れない
- 各パッケージは `core` にのみ依存する（パッケージ間の横の依存を避ける）
- AIが1パッケージの中だけで完結する変更ができる粒度にする

### Step 4: 品質ゲートを設置する（所要: 1時間）

#### a. Claude Code Hooks

```json
        "matcher": "Write|Edit",
```

#### b. Git Hooks


#### c. CI/CD

```yaml

  push:
    branches: [main]
```

**「AIが壊しても即座に分かる」状態を作ることが最重要。** これがないとAIは慎重になりすぎて速度が出ない。

### Step 5: 監査ドキュメント体系を構築する（所要: 30分）

```
docs/
  09_reports/       ← AI が生成する報告書（日付+連番管理）
  10_ai_docs/       ← AI が生成する設計書・分析書
  landing/devlog/   ← 開発日誌（公開用）
```

**ルール:**
- AIが実装したら、必ず報告書もセットで生成させる
- 人間はコードではなく報告書をレビューする
- 報告書に「残存リスク」セクションを必須にする

### Step 6: RAG検索を導入する（所要: 半日）

ドキュメントが1万行を超えたら、Grep では追いつかなくなる。

- PostgreSQL + pgvector でベクトル検索基盤を作る
- docs/ の Markdown を chunk化 → embedding → DB に格納
- MCP エンドポイントまたは CLI で検索可能にする

**効果**: AIが「過去にこの問題をどう解決したか」を自分で調べられるようになる。同じバグを2度踏まない。

## 4. 日常の運用フロー

```
人間: 「フィルター50種追加して」
  ↓
AI: /rag-search で過去のフィルター実装を検索
  ↓
AI: packages/core にインターフェース定義を確認
  ↓
AI: 50種のフィルター実装を packages/web に一括生成
    （パターン1つ理解 → 残り49は並列展開）
  ↓
AI: /test で影響範囲のテストを自動実行
  ↓
AI: /save-report でテスト報告書を保存
  ↓
AI: /commit でコミット（感想 + Co-Author 付き）
  ↓
人間: 報告書を1枚読んで「OK」
  ↓
人間: /push → GitHub Actions でデプロイ
```

**人間の作業: 指示1行 + 報告書レビュー1枚。**
**AI の作業: 実装 + テスト + 文書化 + コミット。**

## 5. よくある失敗パターンと対策

| 失敗 | 原因 | 対策 |
|------|------|------|
| AIが毎回違うスタイルで書く | CLAUDE.md がない or 曖昧 | 規約を具体例付きで明文化する |
| AIが既存コードを壊す | テストがない | pre-push で typecheck + lint を必須にする |
| レビューが追いつかない | diff を全部読んでいる | 報告書ベースのレビューに移行する |
| AIが同じバグを何度も踏む | 過去のインシデント記録がない | docs/ に記録 + RAG検索で参照可能にする |
| AIが間違った場所にファイルを作る | ディレクトリ構造の説明がない | CLAUDE.md に構成図を書く |
| 秘密情報をコミットする | 保護がない | Hooks で .env 等をブロックする |
