# Playwright MCP 導入による E2E テスト精度向上計画

> Generated by Claude Opus 4.6

## 背景

### 現状の問題

E2E テストでスクリーンショットを撮影し、Claude が画像判定で検証するフローを運用しているが、**画像判定の精度にブレがある**。

- テストが OK 判定だが、実際には UI が壊れている（偽陽性）
- 微妙なレイアウト崩れ、テキスト欠け、要素の非表示を見逃す
- 画像のピクセル比較ではなく AI の主観的判定に依存している

**「テスト OK だけど実際は NG」が最も危険なパターン。** テストへの信頼が失われると、テスト自体が形骸化する。

### 根本原因

| 判定方式 | 問題 |
|---------|------|
| スクリーンショット画像判定 | AI が「見た目 OK」と判断しても DOM 上は壊れている場合がある |
| ピクセル比較 | レンダリング差異（フォント、アンチエイリアス）で不安定 |
| 手動目視確認 | スケールしない、属人的 |

## 方針

**画像判定（曖昧） → DOM / アクセシビリティツリー判定（正確）に切り替える。**

Playwright 公式 MCP サーバー（`@playwright/mcp`）を Claude Code に接続し、ブラウザの DOM 構造を直接取得・検証する。

## 導入内容

### 1. Playwright MCP サーバー（実施済み）

`.mcp.json` に `@playwright/mcp` を headless モードで設定済み。

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest", "--headless"]
    }
  }
}
```

### 2. browser-verify スキル（実施済み）

`.claude/skills/browser-verify/skill.md` を新規作成済み。

- `playwright-e2e-test` スキルとは完全独立（依存なし）
- DOM / アクセシビリティツリーによる構造的検証
- 検証結果をテーブル形式でレポート

## 検証精度の比較

| 検証項目 | 画像判定（旧） | DOM 判定（新） |
|---------|--------------|--------------|
| テキスト内容 | 画像から OCR 的に推測 | `textContent` で exact match |
| 要素の存在 | 目視で「ありそう」 | セレクタで有無を確定 |
| 入力値 | スクショから読み取り | `value` プロパティで取得 |
| 非表示要素 | 判定不可 | `display:none` / `visibility` 検出可 |
| レイアウト崩れ | 微妙な差は見逃す | `getBoundingClientRect` で座標検証可 |
| 動的コンテンツ | タイミング依存 | `browser_wait_for` で確実に待機 |

## 利用可能な MCP ツール

| ツール | 用途 | 精度向上への貢献 |
|-------|------|----------------|
| `browser_navigate` | ページ遷移 | — |
| `browser_snapshot` | アクセシビリティツリー取得 | **核心**: DOM 構造で判定 |
| `browser_click` | クリック操作 | インタラクション後の状態検証 |
| `browser_type` | テキスト入力 | フォーム入力の検証 |
| `browser_press_key` | キーボード操作 | ショートカットの動作確認 |
| `browser_select_option` | セレクト操作 | ドロップダウンの値検証 |
| `browser_hover` | ホバー | ツールチップ・ホバー UI の確認 |
| `browser_evaluate` | JS 実行 | カスタム検証ロジック・赤枠アノテーション |
| `browser_wait_for` | 待機 | 非同期ロード完了の確実な検出 |
| `browser_file_upload` | ファイルアップロード | アセットアップロードの検証 |

## 2 つのスキルの棲み分け

| | playwright-e2e-test | browser-verify |
|---|---|---|
| **目的** | 再実行可能なテストファイル作成 | その場の対話的 UI 検証 |
| **判定方式** | スクショ画像 + `expect` | DOM / アクセシビリティツリー |
| **出力** | `.spec.ts` ファイル | 検証レポート（テーブル） |
| **依存** | `npx playwright test` | Playwright MCP ツール |
| **再現性** | 何度でも実行可能 | 1 回限りの確認 |
| **適した場面** | CI/CD・リグレッション防止 | 開発中の UI 確認・デバッグ |

### 使い分けフロー

```
開発中の UI 確認
  → browser-verify（対話的に DOM 検証）
  → 問題なければ OK

リグレッション防止
  → playwright-e2e-test（.spec.ts 作成）
  → CI で自動実行
```

## 想定される活用シーン

### シーン 1: エディタ UI の変更確認

```
ユーザー: 「エディタのブロック追加メニューが正しく表示されてるか確認して」

browser-verify:
1. browser_navigate → http://localhost:5176/projects/editor/{id}
2. browser_click → ブロック追加ボタン
3. browser_snapshot → メニュー内の選択肢を DOM で確認
4. 全ブロックタイプ（text, bg, ch, ...）の存在を構造的に検証
```

### シーン 2: プレビュー画面の表示検証

```
ユーザー: 「プレビューでテキストブロックが正しく描画されてるか見て」

browser-verify:
1. browser_navigate → http://localhost:5175/ksc-demo.html
2. browser_wait_for → Canvas レンダリング完了待ち
3. browser_evaluate → Canvas 上のテキスト要素を JS で検証
4. browser_snapshot → 周辺 UI（スキップボタン等）の存在確認
```

### シーン 3: 認証フローの検証

```
ユーザー: 「ログイン → マイページ遷移が動いてるか確認して」

browser-verify:
1. browser_navigate → http://localhost:3000/login
2. browser_type → email, password 入力
3. browser_click → ログインボタン
4. browser_wait_for → リダイレクト完了
5. browser_snapshot → マイページの要素確認
```

## リスクと対策

| リスク | 対策 |
|-------|------|
| MCP サーバーの起動失敗 | `--headless` で安定性確保、エラー時は `npx playwright test` にフォールバック |
| Canvas ベースの描画（PixiJS）は DOM に現れない | `browser_evaluate` で Canvas API 経由の検証を併用 |
| MCP ツールの応答が遅い | タイムアウト設定、必要な検証のみに絞る |
| headless と headed で挙動が異なる | 基本 headless、問題時のみ headed に切り替え |

## 拡張機能（実装済み）

| 拡張 | 概要 | スクリプト |
|------|------|-----------|
| 検証結果の永続化 | レポートを JSON + MD で `docs/09_reports/` に自動保存 | `scripts/browser-verify/save-report.ts` |
| スナップショット比較 | 前回の DOM スナップショットとの差分検出 | `scripts/browser-verify/compare-snapshots.ts` |
| CI 連携 | browser-verify の検証項目を `.spec.ts` に変換 | `scripts/browser-verify/generate-spec.ts` |
| Azure 環境対応 | browser-verify スキルに本番 URL を追加、環境切り替え対応 | `browser-verify/skill.md` |

### 検証結果の永続化

```bash
# 検証結果 JSON を stdin から渡してレポート保存
cat report.json | npx tsx scripts/browser-verify/save-report.ts --stdin
```

保存先: `docs/09_reports/{YYYY}/{MM}/{DD}/verify-{slug}.json` + `.md`
スナップショット付きの場合: `tests/snapshots/{YYYY}/{MM}/{DD}/verify-{slug}.snapshot.txt`

### スナップショット比較

```bash
# 画面 slug を指定して最新 2 つのスナップショットを比較
npx tsx scripts/browser-verify/compare-snapshots.ts <screen-slug>
```

### CI 連携

```bash
# 検証レポート JSON → Playwright .spec.ts に変換
npx tsx scripts/browser-verify/generate-spec.ts <report.json>
```

生成先: `tests/shared/verify/verify-{slug}.spec.ts`（セレクタの手動調整が必要な場合あり）

### Azure 環境対応

browser-verify スキルに Azure 本番 URL を追加済み。「本番確認」「Azure 検証」等のキーワードで自動切り替え。
