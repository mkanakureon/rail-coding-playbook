# メメントモリ ベンチマーク分析 & IRenderer 設計

> Generated by Claude Opus 4.6

## 背景

kaedevn の運営型ゲームアプリ事業において、メメントモリ級の描画リッチさをどこまで再現できるかを評価し、Web（PixiJS）と Native（SDL2）の描画資産を統一するためのアーキテクチャを設計する。

---

## メメントモリの描画技術

| 技術 | メメントモリでの使い方 |
|------|---------------------|
| **Live2D / Spine** | キャラの呼吸・瞬き・髪揺れ・待機モーション |
| **パーティクルエンジン** | 花びら・光の粒子・魔法エフェクト・ガチャ演出 |
| **ポストプロセス** | Bloom/Glow・被写界深度・光条(ゴッドレイ) |
| **パララックス** | 背景の多層スクロール（奥行き感） |
| **バトルVFX** | スキルエフェクト・加算合成・画面シェイク |
| **UIアニメーション** | スムーズなトランジション・背景パーティクル |
| **音楽連動演出** | BGM に合わせた視覚エフェクト切り替え |

## kaedevn Web エンジン（PixiJS）との比較

### 今すぐ再現できるもの

| 機能 | 状態 | 詳細 |
|------|------|------|
| **ポストプロセス** | 再現可能 | Bloom・CRT・Glitch・ChromaticAberration 等 10種のカスタム GLSL シェーダー |
| **カラーグレーディング** | 再現可能 | 39種フィルター（時間帯・天候・雰囲気・レトロ）、3層同時合成 |
| **画面シェイク・フラッシュ** | 再現可能 | screenFx で実装済み |
| **加算合成** | 再現可能 | blend mode `add` 対応（バトルエフェクト向き） |
| **スプライトシートアニメ** | 再現可能 | BattleAnimPlayer でフレームごとの変形＋SE同期 |
| **カメラ制御** | 再現可能 | ズーム・パン・ピボット＋イージング |
| **品質自動調整** | 再現可能 | FPS 監視で high/medium/low 自動切替 |

### 追加実装が必要なもの

| 機能 | 工数 | 方法 |
|------|------|------|
| **Spine 骨格アニメ** | 1-2週間 | `@pixi/spine` プラグイン追加（Web）、spine-sdl（Native） |
| **パーティクルエンジン** | 1週間 | `@pixi/particle-emitter` or カスタム GLSL（雨シェーダーの延長） |
| **パララックス背景** | 2-3日 | LayerManager に複数背景レイヤー＋スクロール速度差を追加 |
| **ゴッドレイ** | 2-3日 | カスタム GLSL（既存のシェーダー基盤で書ける） |
| **被写界深度(DoF)** | 3-5日 | focusBlur の拡張（レイヤー別にぼかし強度を変える） |

### 再現が難しいもの

| 機能 | 理由 |
|------|------|
| **Live2D** | ライセンス料＋専用ランタイムが必要。代替は Spine で十分 |
| **3D 表現** | 2D エンジンの範囲外。ただしメメントモリも本質は 2D |

### 評価

| 観点 | 評価 |
|------|------|
| メメントモリの描画の **70-80%** | 今の資産で再現可能（フィルター・シェーダー・バトルアニメ・画面演出） |
| 残り **20-30%**（Spine + パーティクル） | 2-3週間の追加実装で対応可能 |
| Live2D の呼吸・瞬き | Spine で代替すれば同等の表現力 |

## パーティクル（粒子演出）の具体例

### ホーム画面・待機演出

| 演出 | 内容 |
|------|------|
| **光の粒** | キャラの周囲にゆっくり浮遊する小さな光点（ホタルのような） |
| **花びら** | 画面上から下へ舞い落ちる。回転＋横揺れ付き |
| **塵・埃** | 光に照らされて漂う微粒子（空気感を出す） |
| **キラキラ** | キャラの髪や装飾品から放射状にきらめく星型パーティクル |

### バトル・スキル演出

| 演出 | 内容 |
|------|------|
| **火の粉** | 炎スキルで大量の火花が飛散、加算合成で光る |
| **氷の結晶** | 凍結エフェクトで六角形の粒子が広がる |
| **斬撃の軌跡** | 剣の軌道に沿って光の粒が残る（トレイル） |
| **爆発の破片** | 中心から放射状に飛散→減速→消滅 |
| **回復の光** | 下から上へ螺旋状に昇る緑の光球 |

### ガチャ演出

| 演出 | 内容 |
|------|------|
| **集束する光** | 大量の粒子が中心に集まる→爆発的に拡散 |
| **虹色の粒子** | レア演出で色が時間変化する粒子の渦 |
| **星の雨** | 最高レア確定で画面全体に降り注ぐ |

### 実装方法（2択）

| 方法 | メリット | デメリット |
|------|---------|----------|
| **GLSL シェーダー**（既存の雨フィルターの延長） | GPU で高速、数千粒子OK | 表現の自由度がやや低い |
| **pixi-particle-emitter** プラグイン | JSON で演出定義、デザイナーが調整しやすい | CPU 負荷がやや高い（数百粒子まで） |

## SDL2 側の現状（Native エンジン）

### Web 側とのギャップ

| 機能 | Web (PixiJS) | SDL2 (native-engine) |
|------|:-----------:|:-------------------:|
| 背景表示 | Cover + トランジション | Cover + αフェード |
| キャラ表示 | 表情切替・フォーカス・Tint | 表示のみ（Tint なし） |
| フィルター（39種） | 全実装 | **なし** |
| カスタムシェーダー（10種） | GLSL3.0 | **macOS のみ**（iOS/Android は no-op） |
| パーティクル | 雨（GLSL procedural） | **なし** |
| アニメーション・イージング | fadeTo/lerpPosition/shake | **なし**（即切替のみ） |
| バトルアニメ | BattleAnimPlayer | **なし** |
| 加算合成 | blend mode `add` | SDL_BLENDMODE_BLEND のみ |
| カメラ制御 | ズーム・パン・ピボット | **なし** |

**SDL2 側は「背景 + キャラ + テキスト + 音声」のノベルゲーム最低限。**

### SDL_Renderer のままでできること（短期）

| 演出 | 方法 |
|------|------|
| スプライトアニメ | SDL_RenderCopy でフレーム切替 |
| αフェード | SDL_SetTextureAlphaMod（既にある） |
| 加算合成 | `SDL_BLENDMODE_ADD` に変えるだけ |
| Tint（色乗算） | `SDL_SetTextureColorMod(tex, r, g, b)` |
| パーティクル（CPU） | 小さなテクスチャを数百個 RenderCopy で描画 |
| 画面シェイク | render 時に全体の描画位置をオフセット |
| パララックス | 背景レイヤーごとにスクロール速度差 |

### SDL_Renderer の限界

`SDL_Renderer` は内部的に OpenGL を使っているが、ユーザーにシェーダーを触らせない設計。フレームバッファ → シェーダーで後処理、というポストプロセスのパイプライン自体が不可能。

## IRenderer 設計 — Web と Native の描画統一

### 問題

SDL_Renderer を捨てて OpenGL ES 直接描画に移行すれば、Web の GLSL シェーダー資産がそのまま Native で使える。

### 描画パイプラインの統一

```
Web:    PixiJS → WebGL2 (= OpenGL ES 3.0 のブラウザ版)
Native: SDL2  → OpenGL ES 3.0 (直接)
Switch: SDL2  → NVN (シェーダーは GLSL から変換)
```

WebGL2 と OpenGL ES 3.0 はほぼ同じ仕様。

| 資産 | 今（SDL_Renderer） | OpenGL ES 移行後 |
|------|:------------------:|:----------------:|
| Web の GLSL シェーダー 10 本 | 使えない | **ほぼコピペで動く** |
| フィルター合成ロジック | 再実装が必要 | **同じアルゴリズム** |
| 新しいエフェクト追加時 | Web と Native で2回書く | **1回書けば両方動く** |

### アーキテクチャ

```
IRenderer (インターフェース)
  │
  ├─ GLRenderer    ← OpenGL ES 3.0 直接（macOS / iOS / Android / Switch）
  ├─ WebRenderer   ← PixiJS（Web）← 既にある WebOpHandler
  └─ SoftRenderer  ← SDL_Renderer（フォールバック / デバッグ用、今のやつ）
```

### IRenderer インターフェース（最小限）

```cpp
class IRenderer {
public:
    virtual ~IRenderer() = default;

    // テクスチャ
    virtual TextureHandle loadTexture(const std::string& path) = 0;
    virtual void destroyTexture(TextureHandle tex) = 0;

    // 描画
    virtual void drawSprite(TextureHandle tex, const SpriteParams& params) = 0;
    //   params: x, y, scaleX, scaleY, rotation, alpha, tint, blendMode, anchor

    // ポストプロセス（フィルター）
    virtual void beginScene() = 0;          // フレームバッファに描画開始
    virtual void applyFilter(FilterType type, const FilterParams& params) = 0;
    virtual void endScene() = 0;            // 画面に出力

    // パーティクル
    virtual void emitParticles(const ParticleEmitterDef& def) = 0;
    virtual void updateParticles(float dt) = 0;
};
```

10 メソッド前後で十分。

### 責務の分離

| やること | IRenderer の責務 | IRenderer の外（エンジン層） |
|----------|:---------------:|:------------------------:|
| 三角形を描く | **中** | |
| シェーダーを切り替える | **中** | |
| フレームバッファを管理する | **中** | |
| 「Bloom フィルターをかける」 | **中** | |
| 「キャラを表示する」 | | **外** |
| 「ガチャ演出を再生する」 | | **外** |
| イージング計算 | | **外**（共通 C++ コード） |
| パーティクルの物理 | | **外**（共通 C++ コード） |

レンダラーは「描け」と言われたものを描くだけ。演出のロジック（何をいつどう動かすか）は共通コードに置く。

### Web 側との対応関係

```
TypeScript (Web)                 C++ (Native)
─────────────────               ─────────────────
WebOpHandler                     SDL2Engine (演出ロジック)
  └─ PixiJS (描画)                └─ IRenderer (描画)
                                      └─ GLRenderer (OpenGL ES)
```

### GLSL シェーダーの共有

```
packages/
  shaders/                        ← 新設。GLSL ソースを1箇所に
    bloom.frag
    crt.frag
    rain.frag
    ...
  web/src/renderer/filters/       ← ここから import
  native-engine/src/engine/       ← ここからファイル読み込み
```

Web は PixiJS の Filter にソースを渡す、Native は `glShaderSource` にソースを渡す。同じ .frag ファイルから両方ビルドできる（ES 3.0 と WebGL2 はほぼ同じ）。

### SDL2 の影響範囲

変わるのは `SDL2Engine.cpp` の描画部分だけ。SDL2 の他の機能はそのまま:

| SDL2 の機能 | 影響 |
|------------|------|
| ウィンドウ管理 | 変わらない |
| 入力 (SDL_Event) | 変わらない |
| 音声 (SDL_mixer) | 変わらない |
| ファイル (SDL_RWops) | 変わらない |
| PAK アーカイブ | 変わらない |
| **描画だけ** | SDL_RenderCopy → glDrawElements + シェーダー |

SDL2 のウィンドウで OpenGL コンテキストを作るのは SDL2 の標準機能（`SDL_GL_CreateContext`）。SDL2 を捨てるわけではなく、**SDL_Renderer だけを捨てて OpenGL ES を直接叩く**。

### SoftRenderer（今の SDL_Renderer）は残す

開発・デバッグ時に OpenGL なしで動く経路があると便利。今の `SDL2Engine::render()` をそのまま `SoftRenderer` として残しておけば、シェーダーが壊れたときの切り分けにも使える。

## パーティクル実装方式と IRenderer の関係

### 2方式の位置づけ

パーティクル（粒子演出）には GLSL 方式と CPU 方式の2つがあるが、**どちらも IRenderer の上（消費者側）にいる。** IRenderer の中には入らない。

```
                  共通
─────────────────────────────────
  演出ロジック
  「花びらを100個、sin波で落とす」
  「爆発の破片を放射状に飛ばす」
─────────────────────────────────
          │
    ┌─────┴─────┐
    │           │
 GLSL粒子    CPUパーティクル
 (環境演出)   (バトル・ガチャ)
    │           │
    │       位置・速度・寿命を
    │       毎フレーム計算
    │           │
    │       drawSprite() × N個
    │           │
─────────────────────────────────
          │
      IRenderer  ← 境界
          │
─────────────────────────────────
  GLRenderer  /  PixiJS  /  Soft
```

### IRenderer との関わり方

| 方式 | IRenderer の使い方 | 粒子の位置計算 |
|------|-------------------|--------------|
| **GLSL 粒子** | `applyFilter()` で呼ぶ。シェーダー内で位置計算も描画も完結 | GPU（sin/cos/hash） |
| **CPU パーティクル** | エンジン層で位置計算 → `drawSprite()` を粒子の数だけ呼ぶ | CPU（C++ / TypeScript） |

### 演出ごとの方式選択

| 演出 | GLSL（GPU） | CPU | 理由 |
|------|:-----------:|:---:|------|
| 光の粒（浮遊） | **最適** | ○ | 位置が sin/cos で決まる。個別制御不要 |
| 花びら（落下） | **最適** | ○ | 回転＋横揺れも sin で表現できる |
| 塵・埃 | **最適** | ○ | ノイズ関数で浮遊。既存の `GLSL_NOISE2D` がそのまま使える |
| キラキラ | **最適** | ○ | ハッシュ関数でランダム配置＋α点滅 |
| 雨 | **実装済み** | — | 既に 3 層の procedural rain がある |
| 虹色の渦 | **最適** | △ | 色の時間変化＋渦はシェーダーの得意分野 |
| 星の雨 | **最適** | ○ | 大量＋単純な動き。GPU 向き |
| 火の粉 | ○ | ○ | 加算合成＋放射。どちらでも可 |
| 回復の光（螺旋） | ○ | ○ | sin/cos で螺旋は書ける。どちらでも可 |
| 氷の結晶 | △ | **向き** | 六角形の形状が必要。テクスチャスプライトの方が自然 |
| 斬撃の軌跡 | △ | **向き** | 剣の位置に追従。ゲームロジックとの連携が必要 |
| 爆発の破片 | △ | **向き** | 物理（減速・重力）が必要 |
| ガチャ集束→拡散 | △ | **向き** | フェーズ切替（集束→爆発）があり、タイミング制御が必要 |

### まとめると

```
GLSL 共有で実装（Web と Native で同じ .frag）:
  → 環境演出: 光の粒、花びら、塵、キラキラ、雨、虹色の渦、星の雨
  → 既存の雨シェーダーの延長線上

CPU パーティクル + IRenderer.drawSprite で実装:
  → バトル演出: 斬撃、爆発、氷の結晶
  → ガチャ演出: 集束→拡散
  → ゲームロジックと連動する必要があるもの
```

### プラットフォーム間の互換性

| 方式 | Web | Native | 互換性 |
|------|-----|--------|--------|
| **GLSL 粒子** | PixiJS Filter に `.frag` を渡す | `glShaderSource` に同じ `.frag` を渡す | **同じファイル** |
| **CPU パーティクル** | TypeScript で計算 → PixiJS の `addChild` | C++ で計算 → `drawSprite()` | **ロジックは別だが IRenderer の API が同じ** |

GLSL 粒子は文字通り同じコードが動く。CPU パーティクルは言語が違う（TS vs C++）が、やることは「座標と速度を足して `drawSprite` を呼ぶ」だけなので、ロジックが単純すぎて互換性の問題が起きようがない。

**IRenderer が境界を守っている限り、その上で何をやっても下に影響しない。** それがインターフェースを噛ませる意味。

## 推奨ロードマップ

```
今（SDL_Renderer）
  │
  ├─ 短期: SDL_Renderer のままで加算合成・Tint・CPU パーティクル・シェイクを追加
  │         → ノベルゲーム+αの演出。1-2週間
  │
  └─ 中期: IRenderer + GLRenderer 実装
            → Web の GLSL シェーダーをそのまま移植
            → メメントモリ級のポストプロセスが Native でも動く
            → 2-3週間
```

## まとめ

| 項目 | 結論 |
|------|------|
| メメントモリの再現度 | Web 側で 70-80% は今すぐ可能、Spine + パーティクル追加で 95%+ |
| SDL2 側の課題 | SDL_Renderer ではシェーダーが使えない |
| 解決策 | IRenderer インターフェース（10メソッド）+ GLRenderer |
| 効果 | Web の GLSL 10 本がそのまま Native で動く。エフェクト追加が1回で済む |
| リスク | SoftRenderer として SDL_Renderer を残すので既存動作は壊れない |
| インフラ | 運営型ゲームアプリが増えてもインフラは1つ（共有プラットフォーム） |
