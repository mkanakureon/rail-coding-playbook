### 回路設計・組み込み26年が効いている場所

Web エンジニアからスタートした人にはない思考が kaedevn の設計に現れている。

| 組み込みの経験 | kaedevn での発現 |
|-------------|----------------|
| DDA直線描画回路の設計（全TTL、基板3枚のGPU） | **レンダリングパイプラインの最下層を知ってる。** `sprite.position.set(x,y)` の裏で何が起きてるかをTTLゲートのレベルから理解 |
| D-RAMへのピクセル書き込み | スプライト・テクスチャ管理の勘所。「2048px 上限」「スプライトシートのみ」のルール設計 |
| CPUデバッグ用にダンプツール・メモリエディタを自作、16進でテストプログラムを手打ち | **ツールを自分で作る発想。** kaedevn のコンパイラ・エディタ・CLI を全部自作したのと同じ思考 |
| メモリ制約の中で動かす | Switch 対応設計（IInput/IAudio/IStorage の抽象化） |
| レイヤー分離の設計 | コンパイラ → エンジン → ランタイムの3層構造 |
| 少ないリソースで最大の効果 | 1人+AI で全部作る発想 |
| 通信プロトコルの設計経験 | API 設計、セーブスキーマの凍結判断 |

**「制約の中で動くものを作る」のは回路設計と同じ思考。** 26年間それをやってきたから、kaedevn の設計が最初から Switch を見据えた抽象化になっている。

## 「1人」は戦略じゃなく制約だった

```
会社員（副業で開発）
  → チームを組む時間も金もない
    → 1人でやるしかない
      → 生成AIが出てきた
        → 1人でも全部作れるようになった
          → 結果的に利益率77%の構造ができた
```

| 制約 | 普通はデメリット | kaedevn では強みに |
|------|-------------|-----------------|
| 1人しかいない | 開発が遅い | 生成AIで解決。利益率77% |
| 副業だから時間がない | 機能が少ない | 本当に必要なものだけ作った |
| お金がない | 人を雇えない | 固定費ゼロ。利益を再投資に回せる |
| チームがいない | レビューがない | 生成AIがレビュー相手 |

**「1人でやろう」と決めたのではなく「1人しかいなかった」。** その制約が結果的に最大の強みになった。

## 生成AIのタイミング

### kaedevn 以前の失敗

kaedevn の前に、Claude Code を使って2つのプロジェクトに挑戦し、どちらも完成しなかった。

| プロジェクト | 技術 | 結果 | 学び |
|------------|------|------|------|
| NPC会話システム | Python + Claude Code | **10万行を超えて破綻** | 巨大な単一コードベースは崩壊する |
| コマンドバトルゲーム | Unity + Claude Code | **完成せず** | Unity は重い/複雑すぎて1人では回しきれない |

この2つの失敗が kaedevn の設計を決めた:
- 10万行で破綻 → **最初から monorepo + packages 分割**
- Unity で回しきれない → **PixiJS/WebGL。軽量で自分でコントロールできるもの**
- 完成しなかった → **「動くもの」を最短で作ることを最優先**
- Claude Code は使い慣れていた → **ツールの学習コストゼロ**

### きっかけ

ある日、「TypeScript が生成AIとの開発に最適」という記事を読んだ。Python で10万行書いて破綻した直後だから、「TSに乗り換える」判断が即座にできた。TS でノベルゲームを作ってみたら、1日でプロトタイプができた。そこから kaedevn が始まった。

### 1日でプロトタイプができた理由

記事は誰でも読めた。TS も誰でも書ける。生成AIも誰でも使える。

でも「読んだ → ノベルゲーム → 1日で動くもの」まで行けたのは:

| 要素 | なぜ速かった |
|------|------------|
| 「TSがいい」と判断できた | エンジニアの経験 |
| 「ノベルゲームを作ろう」と思いついた | ゲーム開発の経験 |
| 1日で形にできた | フルスタックの実力 |
| 生成AIに正しく指示できた | 何を作るか知ってたから |

**幸運なのは「きっかけの記事に出会った」こと。そこから先は実力。**

### タイミングの幸運

| タイミング | 結果 |
|-----------|------|
| 2年早かったら | 生成AIがなくて1人では無理だった |
| 2年遅かったら | 誰でもできるようになってて差別化できない |
| **2026年** | **生成AIが実用化された直後。経験がある人だけが活かせる時期** |

## 再現可能性

### 再現できる部分（記事で公開できる）

| 要素 | 方法 |
|------|------|
| 1人+生成AIで開発 | 2026年以降は当たり前になる |
| 人を雇わず利益率を高く保つ | 判断の問題 |

### 再現できない部分（kaedevn 固有）

| 要素 | 理由 |
|------|------|
| 3年+5億のアセットライブラリ | 時間と金が必要。先行者優位 |
| 作者コミュニティ | 後から同じものを作っても作者は移らない |
| 15年のキャリアの組み合わせ | ゲームアプリ+ソシャゲ+Quon+フルスタック。同じ人がいない |
| Minto 社長との関係 | 個人の人脈。記事を読んでも手に入らない |

### 結論

**「やり方」は全部公開できる。でも同じ結果にはならない。** 料理のレシピを公開しても同じ味にならないのと同じ。材料（経験）と腕（判断力）が違うから。

別の領域で同じ構造（1人+AI、高利益率、再投資）を作る人は出てくる。でも kaedevn の競合にはならない。領域が違うから。

## このプロジェクトの本質

**「全部無料で始められるゲーム制作プラットフォーム」** がコア。

```
エンジン無料 + 素材無料 + インフラ無料
  → 作者の参入障壁ゼロ
    → 作者が来る → 作品が増える → 読者が来る
      → 容量課金・Switch・運営型手数料で回収

利益率77%（人を雇わない）
  → 利益をアセットに再投資できる
    → 無料素材がさらに増える
      → さらに作者が来る（複利）
```

利益率が高いからアセットに投資でき、アセットが無料だから作者が来て、作者が来るから利益が出る。このループが回る限り、成長が続く。

**そしてこのループを回せるのは「1人+AI」だから。** 人を雇った瞬間に利益率が下がり、再投資に回せる額が減り、ループが弱くなる。
