## 現状

- **アーキテクチャ**: Next.js + Hono API + PostgreSQL（Azure Container Apps）
- **MAU**: ~100（開発・テスト段階）
- **今日対応済み**: Health Dashboard API、API 件数リミット、二段構え認証、execSync 非同期化、パストラバーサル対策

## Phase 0: 今月中（MAU ~100、コスト削減）

| # | 項目 | 種類 | 作業内容 | 工数 |
|---|------|------|---------|------|
| 0-1 | **Front Door CDN 有効化** | Azure設定 | Azure Portal で Front Door プロファイル作成 → Container Apps にバインド → 静的アセット（/uploads/*, /storage/*）のキャッシュルール設定 | 半日 |
| 0-2 | **アセット配信の Cache-Control 見直し** | コード | 画像以外（JSON, HTML）のキャッシュ戦略を確認・調整 | 小 |

**なぜ今**: CDN なしだと全リクエストが Container Apps に到達する。画像配信が増えるとコストが直線的に増加。ユーザーが少ない今のうちに設定しておくのが最もコスト効率が良い。

## Phase 1: ユーザー増加開始時（MAU ~1,000）

| # | 項目 | 種類 | 作業内容 | 工数 |
|---|------|------|---------|------|
| 1-1 | **エラー監視（Sentry / Azure Monitor）** | インフラ+コード | Sentry SDK 導入 or Azure Monitor 設定。未キャッチ例外・API 5xx を自動通知 | 中 |
| 1-2 | **簡易アナリティクス** | コード | 管理画面に日次登録数・アクティブユーザー・作品数の推移グラフ。既存の BigQuery ログから集計 | 中 |
| 1-3 | **DB 接続プール設定最適化** | 設定 | Prisma の `connection_limit` を明示的に設定（現在デフォルト）。ACA インスタンス数 × 接続数が PostgreSQL の max_connections を超えないよう計算 | 小 |

**トリガー**: 初めてのユーザー登録が定期的に来るようになったら。

## Phase 2: 成長期（MAU ~3,000〜5,000）

| # | 項目 | 種類 | 作業内容 | 工数 |
|---|------|------|---------|------|
| 2-1 | **PgBouncer 導入** | インフラ | Azure Database for PostgreSQL の PgBouncer 有効化（マネージド）or サイドカーコンテナ。接続プーリングで DB 接続数を 1/10 に削減 | 中 |
| 2-2 | **動画生成ジョブキュー** | インフラ+コード | Redis + BullMQ 導入。`exec` 直接呼び出し → ジョブ投入 + Worker コンテナで処理。API サーバーの負荷を完全分離 | 大 |
| 2-3 | **統計ダッシュボード** | コード | 管理画面に収益・コンバージョン・リテンション指標を追加。課金機能の前に必要 | 中 |
| 2-4 | **カーソルベースページネーション** | コード | 作品一覧の `skip` → `cursor` 方式移行。1万件超で性能差が出始める | 中 |

**トリガー**: Health Dashboard で DB latency が 100ms を超える or 同時動画生成が3件以上になったら。

## Phase 3: 拡大期（MAU ~10,000）

| # | 項目 | 種類 | 作業内容 | 工数 |
|---|------|------|---------|------|
| 3-1 | **AI モデレーション** | インフラ+コード | Cloud Vision / Gemini Vision でアップロード画像の自動検閲。不適切コンテンツの自動フラグ付け | 大 |
| 3-2 | **管理画面強化** | コード | 一括非公開・BAN 機能、モデレーションキュー、監査ログ | 中 |
| 3-3 | **アセット自動 WebP 変換パイプライン** | コード | アップロード時だけでなく、既存アセットの一括変換ジョブ | 中 |

**トリガー**: UGC（ユーザー生成コンテンツ）が1日10件以上投稿されるようになったら。

## Phase 4: 大規模化（MAU ~50,000〜100,000）

| # | 項目 | 種類 | 作業内容 | 工数 |
|---|------|------|---------|------|
| 4-1 | **DB リードレプリカ** | インフラ | 読み取りクエリを分散。Prisma の `$queryRaw` でリードレプリカにルーティング | 大 |
| 4-2 | **検索エンジン** | インフラ+コード | Meilisearch / Elasticsearch で全文検索。作品タイトル・タグ・説明文のインデックス化 | 大 |
| 4-3 | **マイクロサービス分離** | インフラ | API を機能別に分割（認証・作品管理・アセット・動画生成）。独立スケーリング | 特大 |

**トリガー**: DB の読み取りクエリが秒間100を超えたら。
