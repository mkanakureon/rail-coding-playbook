## 手動デプロイ手順

### 前提

- Azure CLI (`az`) がログイン済みであること
- Docker Desktop が起動していること

### Step 1: ACR ログイン

```bash
az acr login --name acrnextacamin
```

### Step 2: イメージタグの決定

```bash
TAG=$(git rev-parse --short HEAD)
echo "Tag: $TAG"
```

### Step 3: Docker ビルド（並列可）

**API:**

```bash
docker build --platform linux/amd64 -f apps/hono/Dockerfile \
  -t acrnextacamin.azurecr.io/hono-api:$TAG \
  -t acrnextacamin.azurecr.io/hono-api:latest \
  .
```

**Next.js:**

```bash
docker build --platform linux/amd64 -f apps/next/Dockerfile \
  --build-arg NEXT_PUBLIC_API_URL=https://ca-api.icymeadow-82e272bc.japaneast.azurecontainerapps.io \
  --build-arg NEXT_PUBLIC_EDITOR_URL=https://agreeable-river-0bfb78000.4.azurestaticapps.net \
  --build-arg NEXT_PUBLIC_PREVIEW_URL=https://happy-tree-012282700.1.azurestaticapps.net \
  -t acrnextacamin.azurecr.io/ca-nextjs:$TAG \
  -t acrnextacamin.azurecr.io/ca-nextjs:latest \
  .
```

### Step 4: ACR に push

```bash
docker push acrnextacamin.azurecr.io/hono-api:$TAG
docker push acrnextacamin.azurecr.io/hono-api:latest
docker push acrnextacamin.azurecr.io/ca-nextjs:$TAG
docker push acrnextacamin.azurecr.io/ca-nextjs:latest
```

### Step 5: Container Apps を更新

```bash
az containerapp update \
  --name ca-api \
  --resource-group rg-next-aca-min \
  --image acrnextacamin.azurecr.io/hono-api:$TAG

az containerapp update \
  --name ca-nextjs \
  --resource-group rg-next-aca-min \
  --image acrnextacamin.azurecr.io/ca-nextjs:$TAG
```

### Step 6: デプロイ確認

```bash
# ヘルスチェック
./scripts/test/azure/health.sh

# 現在のイメージ確認
az containerapp list \
  --resource-group rg-next-aca-min \
  --query "[].{name:name, image:properties.template.containers[0].image}" \
  -o table
```

## トラブルシューティング

### Docker push 時の I/O エラー

```
write /var/lib/desktop-containerd/daemon/io.containerd.metadata.v1.bolt/meta.db: input/output error
```

**原因:** Docker Desktop の内部 DB が破損。
**対処:** Docker Desktop を再起動（メニューバーのクジラアイコン → Restart、またはコマンドで）。

```bash
# 通常の終了
osascript -e 'quit app "Docker"'
sleep 5
open -a Docker
# 起動完了まで 60〜90 秒かかる

# 上記で復旧しない場合は強制 kill
pkill -9 -f "Docker"
sleep 3
rm -f ~/.docker/run/docker.sock   # 古いソケットを削除
sleep 2
open -a Docker
# 起動完了まで 60〜90 秒かかる
```

**注意:** Docker Desktop を再起動すると、**ローカルのビルド済みイメージが消える場合がある**（Docker Desktop の仮想ディスクの状態による）。再起動後に `docker images | grep acrnextacamin` で確認し、なければ Step 3（ビルド）からやり直すこと。

### core / battle の循環依存ビルドエラー

`packages/core/dist` と `packages/battle/dist` を削除してしまうと、循環依存で両方ビルドできなくなる。

**対処（ブートストラップ手順）:**

```bash
# 1. battle のスタブを作成
mkdir -p packages/battle/dist
echo 'export type StatusId = "poison" | "paralyze" | "stun";' > packages/battle/dist/index.d.ts
echo 'export {};' > packages/battle/dist/index.js

# 2. core をビルド
npx tsc -b packages/core --force

# 3. スタブを消して battle を正式ビルド
rm -rf packages/battle/dist
npx tsc -b packages/battle --force
```

### ディスク容量不足（ENOSPC）

Docker イメージが蓄積すると仮想ディスクが肥大化し、`ENOSPC: no space left on device` でビルドが失敗する。

**確認:**

```bash
df -h /                                                                  # OS 全体
du -sh ~/Library/Containers/com.docker.docker/Data/vms/0/data           # Docker 仮想ディスク
```

**対処（仮想ディスクリセット）:**

```bash
# 1. Docker Desktop を完全終了
osascript -e 'quit app "Docker"'
pkill -9 -f Docker
sleep 3

# 2. 仮想ディスクを削除（全イメージ消失、再ビルドが必要）
rm -rf ~/Library/Containers/com.docker.docker/Data/vms/0/data

# 3. Docker Desktop を再起動（新しい仮想ディスクが自動作成される）
open -a Docker
# 起動完了まで 15〜90 秒
```

Docker Desktop の GUI からは Settings → Resources の「Disk image location」に表示されるが、「Purge data」ボタンは新しいバージョンでは廃止されている。上記コマンドで直接削除するのが確実。

### Next.js コールドスタート

Container Apps のスケールが 0→1 の場合、初回リクエストに **最大 30 秒** かかる。ヘルスチェックのタイムアウト（30秒）に間に合わないことがある。2回目以降は正常。

### ARM/AMD64 アーキテクチャ不一致

```
Invalid value: "acrnextacamin.azurecr.io/hono-api:98ac323":
no child with platform linux/amd64 in index
```

**原因:** Apple Silicon (M1/M2/M3) Mac でビルドすると ARM64 イメージになるが、Azure Container Apps は `linux/amd64` が必要。
**対処:** `docker build` に `--platform linux/amd64` を必ず付ける（上記手順に反映済み）。

### Dockerfile の @kaedevn/core 依存エラー（API）

```
src/lib/editor-schema.ts(7,30): error TS2307: Cannot find module '@kaedevn/core'
```

**原因:** `apps/hono/Dockerfile` に `packages/core`（および循環依存先の `packages/battle`）がコピーされていなかった。`editor-schema.ts` が `@kaedevn/core` の `FILTER_TYPES` を import するようになったため顕在化。

**修正:** Dockerfile に以下を追加済み:
- `packages/core` と `packages/battle` の package.json コピー（deps ステージ）
- ソースコピー（builder ステージ）
- ブートストラップビルド手順（battle stub → core → battle の順）

## 注意事項

- この手動デプロイは **GitHub Actions 停止時の緊急手段**。Actions が復旧したら通常経路に戻すこと
- CLAUDE.md の「ローカルデプロイ禁止」ルールは通常時のもの。Actions 停止時はこの手順で代替する
- Static Web Apps（Editor / Preview）は別ワークフロー `deploy-swa.yml` で管理。手動デプロイは `az staticwebapp` コマンドまたは SWA CLI を使う（今回は対象外）
