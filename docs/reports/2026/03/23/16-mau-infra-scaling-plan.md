## MAU ~5,000: 障害検知 & 基本防御

追加費用: +¥10,500/月

### 1. アラート通知

**目的:** サービスダウンを即座に検知し、人間に通知する

- [ ] 通知先を決定する（Slack / Discord / LINE Notify のいずれか）
- [ ] Webhook URL を取得し、GitHub Secrets に `ALERT_WEBHOOK_URL` として登録
- [ ] 通知送信用のシェル関数を作成（`scripts/alert/notify.sh`）
  ```bash
  # 引数: $1=タイトル $2=本文 $3=severity(info/warning/critical)
  curl -X POST "$ALERT_WEBHOOK_URL" \
    -H "Content-Type: application/json" \
    -d "{\"text\": \"[$3] $1\n$2\"}"
  ```

### 2. GitHub Actions 定期ヘルスチェック

**目的:** 5分おきに全エンドポイントを監視し、異常時にアラートを飛ばす

- [ ] `.github/workflows/health-monitor.yml` を作成
- [ ] `schedule: cron: '*/5 * * * *'` でトリガー
- [ ] チェック対象エンドポイント:
  - `GET https://ca-api.icymeadow-82e272bc.japaneast.azurecontainerapps.io/api/health` → 200
  - `GET https://ca-nextjs.icymeadow-82e272bc.japaneast.azurecontainerapps.io` → 200-399
  - `GET https://agreeable-river-0bfb78000.4.azurestaticapps.net` → 200-399
  - `GET https://happy-tree-012282700.1.azurestaticapps.net/ksc-demo.html` → 200-399
- [ ] タイムアウト: 各30秒
- [ ] 失敗時: `scripts/alert/notify.sh` で通知（severity=critical）
- [ ] レスポンスタイムも記録し、3秒超の場合は warning 通知

### 3. Container Apps liveness / readiness probe

**目的:** コンテナがハングした場合に自動再起動させる

- [ ] API（ca-api）に liveness probe を設定
  ```bash
  az containerapp update -n ca-api -g rg-next-aca-min \
    --set-env-vars "" \
    --container-name ca-api \
    --probe-type liveness \
    --probe-path /api/health \
    --probe-port 8080 \
    --probe-protocol http \
    --probe-initial-delay 10 \
    --probe-period 30 \
    --probe-failure-threshold 3
  ```
- [ ] API（ca-api）に readiness probe を設定
  ```bash
  az containerapp update -n ca-api -g rg-next-aca-min \
    --container-name ca-api \
    --probe-type readiness \
    --probe-path /api/health \
    --probe-port 8080 \
    --probe-protocol http \
    --probe-initial-delay 5 \
    --probe-period 10 \
    --probe-failure-threshold 3
  ```
- [ ] Next.js（ca-nextjs）にも同様に設定（probe-path: `/`, port: 3000）
- [ ] 設定後、`az containerapp show` で probe が反映されていることを確認

### 4. DB バックアップ cron 化

**目的:** 毎日自動でバックアップ状態を検証し、異常時に通知する

- [ ] `.github/workflows/db-backup-check.yml` を作成
- [ ] `schedule: cron: '0 0 * * *'`（UTC 0:00 = JST 9:00）
- [ ] 実行内容:
  1. Azure にログイン（`AZURE_CREDENTIALS`）
  2. `az postgres flexible-server backup list` でバックアップ一覧取得
  3. 最新バックアップが24時間以内であることを確認
  4. バックアップ保持期間が7日以上であることを確認
  5. 失敗時: `scripts/alert/notify.sh` で通知（severity=critical）
- [ ] 月1回: リストアテスト（`scripts/db/backup-restore-test.sh` を手動実行）

### 5. 外形監視サービス（オプション）

**目的:** GitHub Actions 自体が落ちた場合の二重化

- [ ] Better Stack（旧 Uptime.com）または UptimeRobot に登録
- [ ] 4エンドポイントを登録（5分間隔）
- [ ] アラート先: メール + 上記と同じ通知チャネル
- [ ] 費用: 無料枠 or ~¥3,000/月

## MAU ~50,000: パフォーマンス & セキュリティ強化

追加費用: +¥72,000/月（前段階比）

### 1. PostgreSQL リードレプリカ

**目的:** 読み取りクエリを分散し、プライマリの負荷を軽減する

- [ ] Azure PostgreSQL Flexible Server でリードレプリカを作成
  ```bash
  az postgres flexible-server replica create \
    --replica-name pgnextacamin-read \
    --resource-group rg-next-aca-min \
    --source-server pgnextacamin
  ```
- [ ] リードレプリカの接続文字列を取得
- [ ] Prisma の設定を変更（読み書き分離）
  - `apps/hono/src/lib/db.ts` に読み取り専用クライアントを追加

- [ ] 読み取り専用エンドポイント（プロジェクト一覧、アセット一覧、公開作品取得等）で `readDb` を使用
- [ ] Container App に `DATABASE_READ_URL` 環境変数を追加
- [ ] レプリケーションラグの監視アラートを設定（10秒超で warning）

### 2. Redis キャッシュ導入

**目的:** 頻繁にアクセスされるデータをキャッシュし、DB負荷とレスポンスタイムを改善する

- [ ] Azure Cache for Redis を作成（Basic C0 or Standard C0）
  ```bash
  az redis create \
    --name redis-kaedevn \
    --resource-group rg-next-aca-min \
    --location japaneast \
    --sku Basic \
    --vm-size c0
  ```
- [ ] 接続文字列を Container App の環境変数に設定（`REDIS_URL`）
- [ ] Redis クライアントを作成（`apps/hono/src/lib/redis.ts`）
  - パッケージ: `ioredis`
  - 接続エラー時はキャッシュなしで動作（フォールバック）
- [ ] キャッシュ対象:
  | エンドポイント | TTL | キー |
  |---|---|---|
  | `GET /api/editor-schema` | 24時間 | `schema:editor` |
  | `GET /api/official-assets` | 1時間 | `assets:official` |
  | `GET /api/projects/:id` の公開作品 | 5分 | `project:{id}` |
  | セッション/JWT 検証結果 | 15分 | `session:{token_hash}` |
- [ ] キャッシュヒット率の監視を追加

### 3. WAF（Web Application Firewall）

**目的:** SQLi、XSS、Bot 等の攻撃を防御する

- [ ] Azure Front Door の WAF ポリシーを作成
  ```bash
  az network front-door waf-policy create \
    --name waf-kaedevn \
    --resource-group rg-next-aca-min \
    --sku Premium_AzureFrontDoor
  ```
- [ ] マネージドルールセットを有効化:
  - Microsoft_DefaultRuleSet（OWASP Top 10）
  - Microsoft_BotManagerRuleSet（Bot 検知）
- [ ] カスタムルール:
  - レートリミット: 同一 IP から 1000 req/分 を超えたらブロック
  - ジオブロック: 必要に応じて特定国を除外
- [ ] WAF を Front Door エンドポイントに関連付け
- [ ] 最初は「検出モード」で運用し、誤検知を確認後「防止モード」に切り替え
- [ ] WAF ログを Log Analytics に送信

### 4. CDN 最適化

**目的:** 静的アセットの配信を高速化し、オリジンサーバーの負荷を減らす

- [ ] Front Door のキャッシュルールを最適化:
  | パス | キャッシュ TTL | 説明 |
  |---|---|---|
  | `/assets/images/*` | 30日 | 背景・スプライト画像 |
  | `/assets/audio/*` | 30日 | BGM・SE・ボイス |
  | `/assets/fonts/*` | 90日 | フォント |
  | `/api/*` | キャッシュなし | API レスポンス |
- [ ] 圧縮設定: gzip + Brotli を有効化
- [ ] カスタムドメインを Front Door に設定（将来的に）
- [ ] キャッシュヒット率をダッシュボードに追加（目標: 90%以上）

**目的:** DB 障害時の自動フェイルオーバー

  ```bash
  az postgres flexible-server update \
    --name pgnextacamin \
    --resource-group rg-next-aca-min \
  ```

### 3. Connection Pooler（PgBouncer）

**目的:** DB 接続数を効率的に管理し、レプリカ増加時の接続爆発を防ぐ

- [ ] Azure PostgreSQL の組み込み PgBouncer を有効化

  | パラメータ | 値 | 説明 |
  |---|---|---|
  | `pgbouncer.default_pool_size` | 50 | プールあたりの最大接続数 |
  | `pgbouncer.max_client_conn` | 200 | クライアント最大接続数 |
  | `pgbouncer.pool_mode` | transaction | トランザクション単位でプール |
- [ ] Prisma の接続文字列にポート 6432（PgBouncer）を指定
- [ ] `?pgbouncer=true` パラメータを DATABASE_URL に追加
- [ ] 接続数の監視アラートを設定（80%超で warning）
