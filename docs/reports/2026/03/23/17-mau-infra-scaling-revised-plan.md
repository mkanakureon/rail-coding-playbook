### 調査から得られた重要な知見

1. **ChatGPT が8億MAUを単一PostgreSQL（シャーディングなし）で運用している** — リードレプリカ + コネクションプーリング + クエリ最適化のみ
2. **itch.io は1人でVPS 2台 + CDN だけで数百万ユーザーを捌いている** — Kubernetes もマネージドサービスも不使用
3. **GoSnap は `.lean()` 1行でサーバー負荷を90%削減した** — インフラ増強よりコード最適化が先
4. **pixiv が Kubernetes を導入したのは数千万MAU以降** — 1M MAU 時点では基本的な LAMP 構成
5. **note.com が EKS に移行したのは運用改善のためであり、生存のためではない**

## MAU別 実装詳細（修正版）

### MAU ~1,000: 現状維持

初版と同じ。変更なし。

### MAU ~5,000: 障害検知 & 基本防御

初版と同じ。変更なし。詳細は `16-mau-infra-scaling-plan.md` を参照。

### MAU ~20,000: 可観測性 & 冗長化

初版と同じ。変更なし。詳細は `16-mau-infra-scaling-plan.md` を参照。

### MAU ~50,000: パフォーマンス最適化（修正）

追加費用: +¥29,000/月（前段階比）

初版から WAF を削除し、CDN を Cloudflare 無料枠に変更。

#### 1. PostgreSQL リードレプリカ

初版と同じ。詳細は `16-mau-infra-scaling-plan.md` を参照。

#### 2. Redis キャッシュ導入

初版と同じ。詳細は `16-mau-infra-scaling-plan.md` を参照。

#### 3. CDN 最適化（変更）

- [ ] Cloudflare 無料枠の導入を検討（Front Door より安価な可能性）
- [ ] 静的アセット（画像・音声・フォント）を CDN 経由に
- [ ] Cache-Control ヘッダーの最適化
- [ ] 圧縮（gzip + Brotli）有効化

#### 4. コード最適化（追加）

**GoSnap の事例から学ぶ: インフラ増強よりコード最適化が先**

- [ ] Prisma クエリの `select` 指定（必要なカラムだけ取得）
- [ ] N+1 クエリの検出と修正（`include` の適切な使用）
- [ ] スロークエリの特定と改善（`log: ['query']` で計測）
- [ ] 不要なデータのシリアライズ回避
- [ ] API レスポンスの gzip 圧縮

### MAU ~200,000: 堅牢化（大幅修正）

追加費用: +¥40,000/月（前段階比）

初版のマルチリージョン・DB HA・Blob geo冗長を削除。代わりに実用的な強化に集中。

#### 1. PgBouncer 有効化

初版と同じ。詳細は `16-mau-infra-scaling-plan.md` を参照。

#### 2. DB 垂直スケール（4vCPU/16GB）

- [ ] PostgreSQL を General Purpose tier にアップグレード
  ```bash
  az postgres flexible-server update \
    --name pgnextacamin \
    --resource-group rg-next-aca-min \
    --sku-name Standard_D4s_v3 \
    --storage-size 128
  ```
- [ ] スロークエリログを有効化（500ms超）
- [ ] `pg_stat_statements` でクエリパフォーマンスを定期レビュー

#### 3. WAF（Front Door 組み込み）

- [ ] Front Door の組み込み WAF 機能を有効化（追加の DDoS Protection Standard は不要）
- [ ] レートリミット: 同一 IP から 1000 req/分
- [ ] マネージドルールセット（OWASP Top 10）を検出モードで有効化
- [ ] 誤検知確認後に防止モードに切り替え

#### 4. VNet 検討（必要に応じて）

- [ ] DB へのアクセスをプライベートエンドポイント経由に制限するか検討
- [ ] 判断基準: セキュリティ監査で指摘された場合、または金融系データを扱う場合

### MAU ~500,000: スケールアップ（大幅修正）

追加費用: +¥60,000/月（前段階比）

初版の DDoS Protection Standard（¥300,000/月）、専用 VNet を削除。

#### 1. DB 垂直スケール（8vCPU/32GB）

- [ ] PostgreSQL をさらにアップグレード
  ```bash
  az postgres flexible-server update \
    --name pgnextacamin \
    --resource-group rg-next-aca-min \
    --sku-name Standard_D8s_v3 \
    --storage-size 256
  ```

#### 2. API 4レプリカ

- [ ] Container Apps のレプリカ数を更新
  ```bash
  az containerapp update -n ca-api -g rg-next-aca-min \
    --min-replicas 4 \
    --max-replicas 8
  ```

#### 3. クエリ最適化集中

- [ ] `pg_stat_statements` で上位20クエリを分析
- [ ] 不足しているインデックスの追加
- [ ] 重いクエリの Redis キャッシュ化
- [ ] バッチ処理の非同期化（キュー導入検討）

### MAU ~1,000,000: 最適化の深化（全面修正）

追加費用: +¥50,000/月（前段階比）

**初版の Kubernetes 移行・DB シャーディング・3リージョン分散・外注人員をすべて削除。**

#### 1. キャッシュ戦略強化

- [ ] Redis のキャッシュ対象を拡大
- [ ] CDN のキャッシュヒット率を95%以上に最適化
- [ ] API レスポンスの ETag/条件付きリクエスト対応
- [ ] 静的ページの事前レンダリング（ISR）

#### 2. インデックス最適化

- [ ] 全テーブルの `EXPLAIN ANALYZE` でフルスキャンを排除
- [ ] パーティショニング検討（テーブルサイズが100GB超の場合のみ）
- [ ] VACUUM/ANALYZE の自動チューニング

#### 3. リードレプリカ追加（必要に応じて）

- [ ] 読み取り負荷がリードレプリカ1台で賄えない場合のみ追加
- [ ] 判断基準: リードレプリカの CPU 使用率が常時80%超

#### 4. 判断ポイント: 本当にスケール限界に達したら

以下の症状が出た場合のみ、より大きな構成変更を検討:

| 症状 | 対応 | 条件 |
|------|------|------|
| DB CPU が常時90%超（垂直スケール済み） | シャーディング検討 | 8vCPU/32GB でも不足する場合のみ |
| Container Apps の同時リクエスト上限に到達 | AKS 移行検討 | Container Apps の制約に実際にぶつかった場合のみ |
| 海外ユーザーが30%を超えた | マルチリージョン検討 | 日本向け中心なら不要 |
| 夜間障害が月2回以上 | 運用外注検討 | それまでは1人+AI で十分 |

## 最重要の教訓

### 「インフラを増やす前にコードを直せ」

| 事例 | やったこと | 効果 |
|------|-----------|------|
| GoSnap | Mongoose の `.lean()` 1行追加 | サーバー負荷 **90%減** |
| OpenAI | クエリ最適化 + コネクションプーリング | レスポンス 50ms → 5ms |
| itch.io | 適切なキャッシュ + CDN | VPS 2台で数百万ユーザー |

**コード最適化チェックリスト（インフラ増強より先にやること）:**

- [ ] Prisma クエリに `select` を付けて必要カラムだけ取得
- [ ] N+1 クエリを `include` で解消
- [ ] `pg_stat_statements` でスロークエリを特定・修正
- [ ] 頻繁にアクセスされるデータを Redis にキャッシュ
- [ ] 静的アセットを CDN 配信（Cache-Control 最適化）
- [ ] API レスポンスの gzip 圧縮
- [ ] 不要なミドルウェアの削除
- [ ] DB インデックスの見直し
