# Gemini CLI 向け: スキル作成ガイド

> 作成日: 2026-03-15
> 作成者: Claude Opus 4.6
> 目的: Gemini CLI がこのリポジトリでスキルを作成・活用するためのアドバイス

## スキルとは

`.claude/skills/{スキル名}/skill.md` に配置する Markdown ファイル。
ユーザーが特定のキーワード（「テストして」「pushして」等）を言ったときに、
Claude Code がこのファイルを読み込んで手順に従って実行する。

**スキル ≠ シェルスクリプト**。コードを実行するのはスキルを読んだ AI エージェントであり、
スキルファイル自体は「手順書」に過ぎない。

## ファイル構造

```
.claude/skills/
├── test/
│   └── skill.md          # スマートテスト
├── commit/
│   └── skill.md          # git commit
├── push/
│   └── skill.md          # git push
├── devlog/
│   └── skill.md          # 開発日誌
├── deploy-azure/
│   └── skill.md          # Azure デプロイ
├── stream/
│   └── skill.md          # OBS 録画/配信
└── ...（他 20+ スキル）
```

## スキルの書き方

### 必須要素

```markdown
# {スキル名} Skill

{1行の概要}

## トリガー

| ユーザーの表現 | 動作 |
|---|---|
| "テストして" | {何をするか} |
| "test" | 同上 |

## 手順

### Step 1: {最初にやること}
...
### Step 2: {次にやること}
...
```

### オプション: frontmatter

```markdown
---
description: {Claude Code がスキルを選択するための説明文。トリガーワードを含める}
---
```

frontmatter があると Claude Code のスキル一覧に表示される説明文を制御できる。
なくても動作するが、あると発動精度が上がる。

### 良いスキルの特徴

1. **判断ロジックが明確** — 条件分岐を表で示す（例: テストスキルの変更パス→テスト対象マッピング）
2. **具体的なコマンドを書く** — `npm test -w @kaedevn/hono` のように、そのままコピペで実行できる
3. **出力先を指定する** — レポートや成果物の保存先パスを明記する
4. **注意事項を書く** — 「`--no-verify` は使わない」「main への push は確認する」等のガードレール

### 避けるべきパターン

1. **長すぎるスキル** — 400行超えると読み込みコストが増える。分割を検討する
2. **曖昧な手順** — 「適切にテストする」ではなく「`npm test -w @kaedevn/hono` を実行する」
3. **実装詳細の埋め込み** — スキルは手順書。ソースコードを貼り付けない
4. **状態依存** — 「前回の結果を参照」のような、コンテキスト外の情報に依存する手順

## 既存スキルの参考例

### シンプル: push スキル（59行）

```
1. git status で未コミット確認
2. ブランチ確認
3. git push origin {ブランチ}
```

ガードレール: pre-push フック失敗時の対応、main への確認、--force 禁止

### 中規模: test スキル（113行）

```
1. git diff で変更ファイル取得
2. パス→ワークスペースのマッピングで対象判定
3. テスト実行
4. 報告書を docs/09_reports/ に保存
```

ポイント: 判定ロジックを表で明示、特殊ルール（schema 変更時は必ず hono テスト）を列挙

### 大規模: devlog スキル（148行）

```
1. git log で素材収集
2. 過去の日誌を読んでトーン確認
3. 記事構成のルールに従って執筆
4. docs/landing/devlog/ に保存
```

ポイント: 文体・構成・署名まで細かく指定。「誰がやったか明示しない」等のスタイルガイド

## Gemini CLI 向け具体的アドバイス

### 1. 実装とテストを同時にコミットする

今回 assist-* で 36 テストが壊れていたのは、実装を複数回リファクタした後にテストを更新しなかったため。

**推奨パターン:**
```
1. 実装を変更する
2. 関連テストを実行する（npm test -w {workspace}）
3. 失敗したら テスト側を更新する（実装が正しい場合）
4. 全テスト通過を確認してからコミットする
```

### 2. 報告書を書く前にコミットする

報告書の生成は得意だが、報告書を書く時間でテストを回す方が手戻りが減る。

**推奨順序:**
```
実装 → テスト → コミット → 報告書
```

**避ける順序:**
```
実装 → 報告書 → 実装 → 報告書 → ... → テスト（ここで壊れてる）
```

### 3. マイグレーションを忘れない

Prisma schema を変更したら:

```bash
npx prisma migrate dev --name {変更内容}
npm test -w @kaedevn/hono  # schema-sync テストで検証
```

`db push` だけで終わらない。マイグレーションファイルがないと本番デプロイで壊れる。

### 4. ブランチ戦略

- 大きな機能は**ブランチで作業**する（例: `feature/map-system-integration`）
- main へのマージ前に `npm run typecheck && npm run lint` を通す
- マージコミットが大きくなりすぎたら、途中で main を取り込んでコンフリクトを小さく保つ

### 5. スキルを作るタイミング

**スキルにすべきもの:**
- 3回以上繰り返した手順
- 間違えやすい手順（デプロイ、テスト、録画）
- ユーザーが1語で発動したい操作

**スキルにしなくていいもの:**
- 1回限りの作業
- 毎回やり方が変わる作業
- 既存スキルの組み合わせで済む作業

## このリポジトリのルール（CLAUDE.md より抜粋）

Gemini CLI も従うべき重要ルール:

1. **原因不明のまま推測でコードを変更しない** — ログで確認してから修正
2. **テストの目的は「テストを通すこと」ではなく「エラーの発見」** — `expect([200, 500]).toContain()` 禁止
3. **新ファイルを作る前に既存実装を検索する** — 重複を避ける
4. **`docs/01_in_specs/` は書き込み禁止** — 成果物は `docs/09_reports/` に保存
5. **デプロイは GitHub Actions 経由** — `./scripts/deploy-azure.sh` は使わない
