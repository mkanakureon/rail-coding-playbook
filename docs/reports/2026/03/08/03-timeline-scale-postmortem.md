# タイムラインスケール問題の振り返りと再発防止

> Generated by Claude Opus 4.6

## 背景

タイムラインブロックを含むプロジェクトをプレイ画面で実行すると、キャラクターの頭が切れて表示される問題が発生した。エディタのプレビューでは正常に表示されており、プレイ画面のみで再現した。

## 問題の経緯

### 1. 誤ったアプローチ（約1時間の浪費）

原因を特定しないまま、以下の推測ベースの変更を繰り返した。**すべて誤りだった。**

| # | 推測 | 変更内容 | 結果 |
|---|------|---------|------|
| 1 | ビューポートサイズが原因 | `setViewportSize(640, 360)` を追加 | 変化なし |
| 2 | CSS の `object-fit` が原因 | `cover` → `contain` に変更 | 上下左右に黒帯が出現 |
| 3 | `fitCanvas()` 関数が必要 | アスペクト比計算ロジックを追加 | 別の表示崩れが発生 |
| 4 | `from=editor` 判定が必要 | `isFullView` 条件を拡張 | 変化なし |
| 5 | キャッシュが原因 | ハードリロードを指示 | 変化なし |

**根本的な問題は CSS ではなかった。**

### 2. 正しい原因特定

ユーザーが「TLを削除したら正常になった」と報告。この一言で原因がタイムラインの適用処理にあることが判明した。

## 根本原因

`WebOpHandler.ts` の `applyTimelineResult()` で、タイムラインの `scale` 値を PixiJS スプライトに**絶対値として上書き**していた。

```typescript
// 修正前（バグ）
if (props.scale != null) sprite.scale.set(props.scale);
```

キャラクターの初期スケールは `VIRTUAL_HEIGHT / texture.height`（例: 720/1024 ≈ 0.703）で設定される。タイムラインで `scale: 1.0`（変化なし）を指定すると、スケールが 0.703 → 1.0 に上書きされ、キャラクターが拡大されて頭が切れていた。

### エディタプレビューとの差異

| 環境 | scale の扱い | 方式 |
|------|-------------|------|
| エディタプレビュー (`TimelinePreview.tsx`) | CSS `transform: scale()` | **相対的な乗算** |
| プレイ画面 (`WebOpHandler.ts`) | PixiJS `sprite.scale.set()` | **絶対値の上書き**（バグ） |

エディタでは CSS の `transform` が元のサイズに対する乗算なので正しく表示されるが、プレイ画面では PixiJS のスケールが直接上書きされるため表示が崩れていた。

## 修正内容

`spriteBaseScales` Map を追加し、スプライト作成時の初期スケールを記録。タイムライン適用時にベーススケールと乗算する方式に変更した。

```typescript
// 修正後
const baseScale = this.spriteBaseScales.get(spriteKey) ?? 1;
if (props.scale != null) sprite.scale.set(baseScale * props.scale);
```

これは既存の `spriteBasePositions`（位置のオフセット計算）と同じパターン。

### 変更ファイル

| # | ファイル | 変更 |
|---|---------|------|
| 1 | `packages/web/src/renderer/WebOpHandler.ts` | `spriteBaseScales` Map 追加、5箇所で初期スケール記録、`applyTimelineResult` で乗算方式に変更 |
| 2 | `packages/web/src/ksc-demo.ts` | 誤った CSS 変更をすべて元に戻した |

## 反省点

### やってはいけなかったこと

1. **原因を特定しないままソースコードを変更した**
   - 推測で5回以上コードを変更し、すべて間違いだった
   - 変更のたびに新しい問題が発生し、混乱が拡大した

2. **確認していないのに「正常」と報告した**
   - 変更後にスクリーンショットや実行確認をせず、コードの論理だけで「修正完了」と判断した

3. **問題の切り分けをしなかった**
   - 最初に「タイムラインの有無で違いが出るか」を確認していれば、1回目で原因特定できた
   - ユーザーが「TLを削除したら正常になった」と教えてくれるまで気づかなかった

4. **既存の類似パターン（spriteBasePositions）を参照しなかった**
   - 位置（position）は既にベースとの加算方式だった
   - スケール（scale）も同じパターンであるべきだと最初から気づくべきだった

## 再発防止ルール

### 原則: 未確認でソースコードを変更しない

以下のプロセスを厳守する。

```
1. 問題の再現条件を特定する
   - 何をしたら起きるか？何をしなかったら起きないか？
   - 条件を1つずつ変えて切り分ける

2. 原因を特定する
   - ログ出力やデバッグ情報で実際の値を確認する
   - 推測ではなく、データに基づいて判断する

3. 修正方針をユーザーに説明する
   - 「〇〇が原因で、△△を修正します」と明示する
   - 原因不明の場合は「原因不明」と正直に伝える

4. 修正後に動作確認する
   - スクリーンショットまたは実行結果で確認する
   - 確認前に「修正完了」と言わない
```

### 具体的なチェックリスト

- [ ] 原因が特定できているか？ → No なら**コードを変更しない**
- [ ] 変更箇所は原因に直結しているか？ → No なら**別の箇所を調査する**
- [ ] 既存の類似パターンはあるか？ → あるなら**そのパターンに合わせる**
- [ ] 変更後に動作確認したか？ → No なら**「修正完了」と言わない**
- [ ] 「たぶんこれが原因」で変更しようとしていないか？ → Yes なら**まずデバッグで確認する**

### PixiJS スプライト操作の注意点

| 操作 | 注意 |
|------|------|
| `sprite.scale.set(value)` | **絶対値で上書き**される。初期スケールが失われる |
| `sprite.position.set(x, y)` | **絶対値で上書き**される。ベース位置が失われる |
| CSS `transform: scale()` | **相対的な乗算**。元のサイズは保持される |

タイムラインなど外部から値を適用する場合、初期値との**乗算（scale）**または**加算（position）**を使い、上書きしない。

## 教訓

- **「原因不明」は「変更するな」のサイン。** デバッグして原因を特定してから修正する
- **ユーザーの報告（「TLを削除したら直った」）は最も信頼できる情報。** 推測より優先する
- **既存パターンを確認する。** `spriteBasePositions` が既に正しいパターンを示していた
- **CSS とレンダリングエンジンの違いを意識する。** 同じ `scale` でもCSS（相対）とPixiJS（絶対）で挙動が異なる
