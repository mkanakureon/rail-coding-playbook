# 次世代ツクールエディタ 技術仕様策定プロセス

> Generated by Claude Opus 4.6

## 目的

統合検討資料（09-next-gen-tsukuru-editor-consolidated-reference.md）で整理したペインポイントを出発点に、**既存の試作コードやツクールの画面構成に依存しない**技術仕様を策定する。

---

## 原則

1. **ペインから始める** — ツクールの機能一覧からではなく、「ゲーム制作者が苦しんでいること」から設計する
2. **試作に引っ張られない** — 既存コードは試作レベル。製品に使うかは仕様確定後に判断する
3. **ツクールのコピーを作らない** — 結果的に似る部分はあっても、模倣が目的ではない
4. **優先順位は後で決める** — リリース後のユーザーフィードバックが最良の根拠。今は「何を作るか」の定義に集中する

## Step 1: ペインの抽象化

統合資料のペインポイントを「ツクールの不満」ではなく、**ゲーム制作ツールが持つ構造的問題**として再定義する。

### P1: データ間の関係が見えない

> 元のペイン: モーダル地獄、データの断片化

ゲームのデータ（キャラクター、スキル、アイテム、敵）は相互に依存している。しかし既存ツールはデータ種別ごとに画面を分断するため、**関係性が不可視になる**。

- スキルの威力がキャラクターの攻撃力に対して適正かを判断するには、両方の画面を開く必要がある
- ある敵がどのスキルを使い、どのアイテムをドロップするかの全体像が見えない
- 変更の影響範囲が予測できない（スキルの威力を変えたら、どの戦闘バランスが崩れるか）

**本質**: データは「表」ではなく「グラフ」（関係の網）なのに、ツールが「表」としてしか扱えない。

### P2: 状態の意味が失われる

> 元のペイン: 不可視のスイッチ・変数

ゲームの進行状態（フラグ、変数）は、ゲームロジックの骨格である。しかし既存ツールでは：

- 状態が番号（ID）で管理され、人間が意味を読み取れない
- どのイベントがどの状態を読み書きするかの依存関係が不可視
- デバッグ = 全イベントを目視で追跡する人力作業

**本質**: 状態管理が「暗黙知」に依存しており、ツールが構造的に支援していない。

### P3: 見た目の変更に専門知識が要る

> 元のペイン: UI カスタマイズの壁、アセット管理の硬直性

ゲームの「顔」（UI、画面レイアウト）を変えたいのは全作者共通の欲求。しかし：

- メニュー画面やバトル UI の変更にプログラミングが必須
- 素材の取り込みに厳格な規格を要求し、少しのズレで拒否される
- エディタで配置したものが実機でどう見えるか確信が持てない

**本質**: 「見た目」と「ロジック」が分離されておらず、視覚的な変更が技術的な変更を強制する。

### P4: 空間とロジックが断絶している

> 元のペイン: マップイベント系の不足（レビュー指摘）、RPG制作シミュレーションでの「接着剤不足」

RPG では「どこで」「何が起きるか」が本質。しかし既存ツールでは：

- マップ（空間）とイベント（ロジック）が別画面に分かれており、「この場所でこれが起きる」を直感的に把握できない
- マップの「見えない属性」（通行判定、エンカウント率、BGM）が別画面に隠れている
- イベントの起動条件（トリガー）が多層的で、どの条件で何が発火するかの全体像が掴めない

**本質**: ゲームは「空間 × 時間 × 条件」の3次元だが、ツールが1次元（リスト）でしか表現できない。

### P5: 制作環境がクリエイターの生活に合っていない

> 元のペイン: Windows 専用、Git 非対応、チーム開発不可

これは kaedevn が Web ベースである時点で**構造的に解決済み**。技術仕様の対象外とする。ただし、Web ネイティブであることを活かした設計（リアルタイム協調編集、クロスデバイス）は Step 2 で検討する。

## Step 2: 解決原理の定義

各ペインに対し、**どういう性質を持つシステムなら解決するか**を技術に踏み込まずに定義する。

### P1 → R1: データをグラフとして扱えるシステム

- データ種別間の**参照関係を双方向に辿れる**こと
- 任意のデータを**並べて比較**できること（同種でも異種でも）
- ある値を変えたときの**影響範囲が即座に分かる**こと
- 一覧性と詳細性を**同じ画面内で切り替え**られること（モーダル遷移ではなく）

### P2 → R2: 状態を「名前付きの構造」として扱えるシステム

- 状態が**名前と型と用途を持つ**こと（番号ではなく）
- どのイベントがどの状態を**読み/書きするかの依存関係が自動的に可視化**されること
- 実行中に**状態の流れをリアルタイムで追跡**できること
- 矛盾する条件（到達不能なパス）を**静的に検出**できること

### P3 → R3: 見た目をロジックから独立して編集できるシステム

- UI の配置・スタイルが**宣言的なデータ**として定義されること（コードではなく）
- 編集結果が**即座にプレビュー**されること（WYSIWYG）
- エディタでの表示と実機での表示が**保証された対応関係**を持つこと
- 素材の取り込みが**規格に依存しない**こと（自動変換・自動調整）

### P4 → R4: 空間・時間・条件を統合的に表現できるシステム

- マップ上で**その場所に紐づくロジックが透視**できること
- イベントの起動条件（いつ・どの条件で）が**空間上にオーバーレイ**として見えること
- 時間軸（ゲーム進行に伴う変化）を**スクラブ/シーク**できること
- 「この状態のとき、このマップはどう見えるか」を**シミュレーション**できること

## Step 3: 技術仕様の策定（次ステップ）

Step 2 の解決原理（R1〜R4）を満たす具体的な：

- **データモデル**（どういう構造でデータを保持するか）
- **API**（データをどう読み書きするか）
- **コンポーネント設計**（ユーザーにどう見せるか）

を定義する。

このステップでは：
- R1〜R4 を**同時に満たす統一的なアーキテクチャ**があるか検討する
- なければ、各原理ごとに独立した仕様を定義し、接合点を明示する
- 結果的にツクールの「タブ」に似た構成になるかもしれないし、まったく異なる形になるかもしれない — **原理が先、形は後**
