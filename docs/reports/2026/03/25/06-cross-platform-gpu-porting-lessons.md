# クロスプラットフォーム GPU 移植が「サクッとできた」理由の考察

> Generated by Claude Opus 4.6

## 背景

kaedevn の Native Engine に GPU ポストプロセスパイプライン（FBO → フィルターシェーダー → 画面）、45 種フィルター移植、CPU パーティクルシステムを実装した。Web（PixiJS/WebGL2）から Native（SDL2/OpenGL 3.3 Core）への移植で、計画から実装完了まで **1セッション** で完了。

この速度が出た理由を分析し、今後の SDL ベースのクロスプラットフォーム開発（Switch NVN、iOS Metal 等）で再現可能な設計パターンとして記録する。

---

## 原則: 「移植が速い」は設計の結果であり、実装力の問題ではない

移植作業の所要時間は、**移植時ではなく最初の設計時に決まっている**。今回の移植が速かったのは、以下の3層の設計判断がすべて正しい方向を向いていたから。

---

## 第1層: 描画抽象化（IRenderer パターン）

### 何をしたか

```
IRenderer (interface)
  ├── SoftRenderer  (SDL_Renderer — フォールバック)
  └── GLRenderer    (OpenGL 3.3 Core / ES 3.0)
```

SDL2Engine は `IRenderer&` を参照するだけで、描画バックエンドを知らない。

### なぜ効いたか

- FBO パイプラインの追加が **GLRenderer の内部変更だけ** で完了した
- `beginFrame()` / `endFrame()` の契約が明確だったので、FBO バインドを差し込む場所に迷いがなかった
- SoftRenderer が壊れないことが保証されているので、GLRenderer で実験しても安全

### 設計原則

> **抽象化は「いつかの移植のため」ではなく「今日の安全なリファクタのため」に入れる。**
> SoftRenderer というフォールバックがあるから、GLRenderer で冒険できた。

### Switch NVN への示唆

NVNRenderer を IRenderer の3番目の実装として追加するだけ。ゲームロジック・スクリプトエンジン・アセット管理は一切触らない。これが IRenderer を最初に作った最大の価値。

---

## 第2層: シェーダーの「ロジックとバインディングの分離」

### 何をしたか

Web のフィルターシェーダーは PixiJS のフレームワーク依存部分（頂点シェーダー、uniform バインディング、テクスチャ座標名）と、フィルターロジック本体（色変換、パーティクル生成、ブラー演算）に分かれていた。

```
Web フィルター = PixiJS バインディング + GLSL ロジック
                 ↓ 捨てる              ↓ そのまま使う
Native フィルター = 自作バインディング  + GLSL ロジック
```

### 変換ルール（機械的・例外なし）

| Web (GLSL 300 es / PixiJS) | Native (GLSL 330 core) |
|---|---|
| `#version 300 es` | `#version 330 core` |
| `precision mediump float;` | 削除 |
| `in vec2 vTextureCoord;` | `in vec2 vUV;` |
| `out vec4 finalColor;` | `out vec4 fragColor;` |
| `${GLSL_HASH}` テンプレートリテラル | C++ 文字列にインライン展開 |
| `if (color.a > 0.0) color.rgb /= color.a;` | **削除** |
| `color.rgb *= color.a;` | **削除** |

### なぜ効いたか

- 変換ルールが **5項目、例外なし** だったので、20シェーダーを一括で移植できた
- ロジック部分（`smoothstep`, `exp`, `sin`, `hash` 等）は GLSL の標準関数のみで書かれており、プラットフォーム依存がゼロだった

### 設計原則

> **シェーダーを書くとき、フレームワーク固有の名前を最小限にする。**
> PixiJS の `vTextureCoord` や `finalColor` はフレームワークが決めた名前。
> ロジック内部では `uv`, `color`, `result` など汎用名を使っていたから、
> バインディング部分（入出力宣言）だけ書き換えればロジックはそのまま動いた。

### pre-multiplied alpha の教訓

**最も危険だったのはこの2行の削除判断:**

```glsl
if (color.a > 0.0) color.rgb /= color.a;  // unpremultiply
color.rgb *= color.a;                       // re-premultiply
```

PixiJS は内部で pre-multiplied alpha を使う。SDL2/OpenGL は straight alpha。この違いを知らずに移植すると:

- 半透明テクスチャの縁が暗くなる（ダークフリンジ）
- `color.a = 0` のピクセルでゼロ除算 → NaN → 真っ黒

**今回これに事前に気づけた理由**: firefly フィルターが Playwright の MCP ブラウザで真っ黒になった事件（GLSL 予約語 `active` のコンパイルエラー）を調査する過程で、シェーダーのアルファ処理フローを全行読んだ。そのとき「この unpremultiply/premultiply は PixiJS 固有の処理だ」と認識できた。

> **バグ調査は移植の最高の予習になる。**
> 原因不明のバグを追うとき、コードの1行1行の意味を理解せざるを得ない。
> その理解が、移植時に「この行は必要か？」の判断を即座にできる土台になる。

## 第3層: GLSL バージョン差分の最小化

### 何をしたか

GLSL 330 core（macOS）と GLSL 300 es（iOS/Android）の差分は極小:

```
#version 330 core
→
#version 300 es
precision highp float;
```

**それ以外は完全に同一。** `in`, `out`, `texture()`, `layout(location=N)` はどちらでも使える。

### ランタイム変換ヘルパー（7行で全解決）

```cpp
static std::string convertToES(const char* src330) {
    std::string s(src330);
    const std::string from = "#version 330 core";
    const std::string to   = "#version 300 es\nprecision highp float;";
    auto pos = s.find(from);
    if (pos != std::string::npos) {
        s.replace(pos, from.size(), to);
    }
    return s;
}
```

23シェーダー × 2バージョン = 46 ファイルの管理が、**1関数で不要になった**。

### 設計原則

> **「2バージョン管理」を避けられる設計を最初から考える。**
> GLSL 110/120 時代は ES と Desktop で構文が大きく違った（`varying` vs `in` 等）。
> しかし 330 core / 300 es 以降は構文が統一されている。
> この世代を最低ターゲットにすることで、バージョン差分を `#version` 行だけに閉じ込められた。

### Switch NVN への示唆

NVN のシェーダー言語は GLSL 互換（NVN GLSL）。`#version` ディレクティブは異なるが、ロジック部分は同一のはず。同じパターン（ランタイムヘッダー差し替え）が適用できる可能性が高い。

## 第4層: パーティクルの CPU 実装判断

### 何をしたか

パーティクルを GPU compute shader ではなく、CPU で実装した。各粒子を `IRenderer::drawSprite()` で描画。

### なぜ効いたか

- GPU compute の移植性問題を完全に回避（Switch NVN に compute shader 互換がない）
- `drawSprite()` は IRenderer 経由なので、SoftRenderer でも GLRenderer でも動く
- VN 用途の粒子数（100〜500）なら CPU で 60fps 余裕

### 設計原則

> **「GPU でやるべき」という思い込みを捨てる。**
> パーティクルは GPU の仕事、というのはゲームエンジンの常識だが、
> **VN エンジンで粒子 500 個**なら CPU の方が移植性が高く、デバッグも容易。
> 「正しい抽象化レベル」は用途で決まる。

## カラーマトリクスフィルターの統一パターン

### 何をしたか

Web では 25 種の色調フィルター（morning, sunset, sepia, warm, cool 等）がそれぞれ `ColorMatrixFilter` のパラメータとして実装されていた。Native では **1つの GLSL シェーダー** + `mat4 uColorMatrix` + `vec4 uColorOffset` の uniform で全 25 種を処理する。

```glsl
// 1シェーダーで 25 種対応
uniform mat4 uColorMatrix;
uniform vec4 uColorOffset;
void main() {
    vec4 color = texture(uTexture, vUV);
    vec3 result = (uColorMatrix * vec4(color.rgb, 1.0)).rgb + uColorOffset.rgb;
    fragColor = vec4(mix(color.rgb, clamp(result, 0.0, 1.0), uIntensity), color.a);
}
```

### 設計原則

> **「種類が多い」ときはパラメータ化を考える。**
> 25 個のシェーダーを書く代わりに、1 個のシェーダー + 25 セットの行列パラメータ。
> シェーダーのコンパイルコスト、管理コスト、テストコストがすべて 1/25 になる。

## 遅延コンパイルの効果

### 何をしたか

45 種のフィルターシェーダーを起動時にすべてコンパイルするのではなく、初回 `applyFilter()` 呼び出し時にのみコンパイルする。

```cpp
std::unordered_map<std::string, GLuint> filterPrograms;  // lazy-compiled

GLuint getOrCompileFilter(const std::string& type) {
    auto it = filterPrograms.find(type);
    if (it != filterPrograms.end()) return it->second;
    // ... compile and cache
}
```

### 効果

- 起動時間への影響ゼロ
- 1つのゲームで使われるフィルターは通常 3〜5 種（45 種全部は使わない）
- 文字列キーなのでスクリプトの `@overlay type=rain` と直結（enum 管理不要）

## 今回の事件簿と得られた知見

### firefly 真っ黒事件

**症状**: Playwright でスクリーンショットを撮ると、firefly / sparkle / dust フィルターが完全に真っ黒。

**原因**: GLSL の変数名 `active` が予約語。ANGLE バックエンドの Chrome でシェーダーコンパイルが無言で失敗。

**教訓**:
1. GLSL の予約語リストは GPU ドライバ/バックエンドによって異なる（macOS Safari では動いていた）
2. 「動く環境がある」≠「正しいコード」。CI で複数バックエンド（ANGLE, SwiftShader, Metal）でテストすべき
3. **`active` は GLSL ES の予約語**（OpenGL ES 3.0 仕様 §3.8）。Desktop GL では許容されることがある

### Playwright WebGL 真っ黒事件

**症状**: headless Chromium でスクリーンショットが全フィルター同一画像。

**原因**: headless モードでは WebGL が Canvas2D にフォールバック。カスタム GLSL シェーダーは Canvas2D では動かない。

**解決**: `--use-gl=angle --use-angle=swiftshader --enable-unsafe-swiftshader` フラグで SwiftShader WebGL を強制有効化。

**教訓**: WebGL のテスト自動化には GPU バックエンドの明示的な制御が必要。

## チェックリスト: 次の移植でも使える判断基準

新しいプラットフォーム（Switch NVN, iOS Metal, WebGPU 等）に移植するとき:

- [ ] **IRenderer に新しい実装を追加するだけで済むか？** → YES なら正しい抽象化
- [ ] **シェーダーのロジック部分はそのまま使えるか？** → YES ならバインディング分離ができている
- [ ] **バージョン差分は `#version` 行だけか？** → YES ならランタイム変換で解決
- [ ] **pre-multiplied alpha の扱いは合っているか？** → 移植先のレンダリングパイプラインを確認
- [ ] **GPU 固有機能（compute shader 等）に依存していないか？** → CPU フォールバックがあれば安全
- [ ] **全シェーダーを起動時にコンパイルしていないか？** → 遅延コンパイルで起動時間を守る
