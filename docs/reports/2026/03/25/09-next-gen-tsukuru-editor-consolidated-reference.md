## 1. エグゼクティブサマリー

**RPGツクール（MV/MZ）ユーザーが抱える根本的な不満は 3 層に集約される：**

| 層 | ペイン | 本質 |
|----|--------|------|
| **データ管理** | モーダル地獄・スイッチ/変数の不可視性 | 試行錯誤に集中できない |
| **表現の自由度** | UI カスタマイズに JS 必須・素材規格の硬直性 | 非エンジニアが挫折する |
| **開発環境** | Windows 専用・Git 非対応・チーム開発不可 | 現代の制作スタイルに合わない |

kaedevn は **Web ネイティブ + ノベルエンジン由来の演出力** で、これらを構造的に解決する位置にある。44 件の具体的不満のうち **28 件（64%）は既存機能で対応済み**、8 件がツクール型エディタで対応予定、8 件がロードマップ上にある。

## 2. ペインポイント総覧

### 2.1. データの断片化（モーダル地獄）

- アクター・スキル・アイテム・敵キャラが独立モーダルに分断され、ゲームバランスを俯瞰できない
- 「攻撃力に対してスキル威力は適正か？」の確認に何度もウィンドウ開閉が必要
- **求められる解決**: スプレッドシート型一括編集、リレーショナル・リンク（スキル→使用エネミー一覧へのジャンプ）

### 2.2. 不可視のスイッチ・変数

- 「スイッチ 121 番」のような ID 管理で、外部メモなしでは用途不明になる
- イベントページの入れ子が複雑化し、デバッグに大半の時間を奪われる
- **求められる解決**: セマンティック変数名、イベントテンプレート化、ビジュアルデバッガ

### 2.3. アセット管理の硬直性

- 素材インポートに厳格なファイル規格（サイズ・命名規則）を要求
- エラーメッセージが不親切で、少しでも外れると受け付けない
- **求められる解決**: ドラッグ＆ドロップによるシームレスな登録

### 2.4. UI カスタマイズの壁

- メニュー画面やバトル UI の変更に JavaScript/Ruby の記述が必須
- 非エンジニアが挫折する最大の原因
- **求められる解決**: ノーコード UI ビルダー、100% Parity 保証（エディタ配置 = 実機表示）

### 2.5. 環境・エコシステム

| 不満 | 詳細 |
|------|------|
| Windows 専用 | Mac/Linux ユーザーが排除される |
| 動作が重い・起動が遅い | MV の 32bit 制限、高解像度でラグ |
| Git 非対応 | 巨大 JSON がコンフリクトし、チーム開発が困難 |
| 「ツクール臭」 | 標準素材（RTP）の影響が強く、個性を出すのに全素材差替が必要 |
| プラグインのブラックボックス化 | 難読化・有料化が進み、カスタマイズ困難 |
| アップデートで壊れる | バージョン互換性が保証されない |

## 4. エディタ画面設計

### 4.1. 全体構成（4 画面 + ダイアログ群）

| # | 画面名 | ツクールの該当画面 | 役割 |
|---|--------|------------------|------|
| 1 | **マップエディタ** | メインウィンドウ（マップビュー） | タイル配置、レイヤー切替、マップイベント配置 |
| 2 | **シナリオ（イベント）エディタ** | イベントエディタ / コマンド | ブロックで物語進行・イベント構築 |
| 3 | **データベース** | データベースウィンドウ | キャラ・敵・アイテム・スキル等の定義 |
| 4 | **UIレイアウトエディタ** | （ツクールにない独自機能） | プレイ画面 UI の視覚的配置 |

補助: 素材管理ダイアログ、変数・スイッチ管理、テストプレイ＆デバッガ

### 4.2. マップエディタ

- **ツールバー**: ペン、四角形、塗りつぶし、消しゴム、ズーム、レイヤー切替
- **タイルパレット**: tilesets.json から読込、オートタイル対応
- **レイヤー**: 下層（Ground）/ 中層（Deco-Lower）/ 上層（Deco-Upper）/ 特殊（Shadow, Collision, Region）
- **マッププロパティ**: BGM/BGS 自動再生、ランダムエンカウント設定、タイルセット紐付け
- **イベント配置**: イベントレイヤー選択 → ダブルクリック → マップイベントエディタ

### 4.3. データベース（13 タブ）

アクター / 職業 / スキル / アイテム / 武器 / 防具 / 敵キャラ / トループ / ステート / アニメーション / タイルセット / コモンイベント / システム

- 左にリスト、右にプロパティの **2 ペインレイアウト**を厳守
- 成長曲線エディタ（グラフ UI）
- ダメージ計算式のリアルタイムシミュレータ（`a.atk * 4 - b.def * 2` 形式）
- タイルセットタブ: 通行判定（○・×・☆）、四方向通行、カウンター、茂み

### 4.4. シナリオエディタ（コマンドパレット）

ツクール風カテゴリ分け:

| カテゴリ | コマンド |
|---------|---------|
| メッセージ | text, choice, scroll_text |
| ゲーム進行 | set_var, if, call |
| パーティ・アクター | 所持金/アイテム/武器/防具の増減、パーティ入替、全回復 |
| フロー制御・移動 | jump, map_jump, 場所移動, 移動ルート設定 |
| オーディオ・画面 | BGM/SE, effect, screen_filter, overlay, bg, ch |
| システム・バトル | battle, ショップ処理, セーブ画面 |

### 4.5. マップイベント設定

- **イベントページ**: 複数ページ管理（条件によるページ切替）
- **出現条件**: グローバルスイッチ、アイテム所持、**セルフスイッチ（A/B/C/D）**
- **オプション**: 歩行アニメ、足踏みアニメ、向き固定、すり抜け、最前面表示
- **自律移動**: 固定/ランダム/近づく/カスタム、速度・頻度設定
- **トリガー**: 決定ボタン、プレイヤー接触、イベント接触、自動実行、**並列処理**

## 5. アーキテクチャ方針

### 5.1. Visible Logic コンセプト

データ（数式・変数）と空間（マップ・イベント）をすべて視覚化する。

| モード | 主権 | 内容 |
|--------|------|------|
| プロパティ・データベース | データ | 成長曲線エディタ、ダメージ計算式シミュレータ |
| 空間イベントエディタ | マップ | タイル配置 + 位置トリガー、フラグ変化の透視 |
| バトル・シーケンスエディタ | 演出 | タイムライン形式、1/60 秒単位の同期、敵 AI フローチャート |

### 5.2. DX の階層化

| Level | 対象 | 内容 |
|-------|------|------|
| Entry | 初心者 | プリセット配置 + テキスト変更だけで完成 |
| Standard | 中級者 | 変数・スイッチで独自クエスト・成長システム |
| Pro | 上級者 | GLSL シェーダー、ダメージ計算式の完全カスタマイズ |

### 5.3. 設計原則（ツクール互換 UX）

1. **用語の統一**: 内部が `GameDb` / `templates` でも UI 上は「データベース」「コモンイベント」「スイッチ」
2. **2 ペインレイアウト**: 左にリスト、右に詳細（ツクール伝統を厳守）
3. **ダブルクリック編集**: ブロックリスト → 詳細設定ダイアログ
4. **計算式互換**: `a.atk * 4 - b.def * 2` をそのまま使える
5. **暗黙のショートカット**: 右クリックスポイト、Ctrl+C/V でイベント複製

## 6. レビューで判明した不足要素

3 回のレビュー（02: 機能レビュー、03: RPG 制作シミュレーション、19: 完成度評価）で洗い出された要素。

### 6.1. データベースの不足タブ（Must-Have）

| タブ | 理由 |
|------|------|
| タイルセット（通行判定） | 当たり判定なしではマップが機能しない。○×☆のグリッドクリック切替 |
| アニメーション | バトルエフェクト（斬撃・魔法）の管理場所が未定義 |
| 武器・防具の独立 | 装備スロット（右手/盾/頭/体）の概念が必要 |

### 6.2. マップ・イベント系の不足

| 要素 | 理由 |
|------|------|
| マッププロパティ | ランダムエンカウント、BGM/BGS、タイルセット紐付けが未定義 |
| セルフスイッチ（A/B/C/D） | 宝箱の開閉など、イベント固有状態の管理に不可欠 |
| イベントオプション | 歩行アニメ/向き固定/すり抜け etc. がないと静止 NPC すら作れない |
| 並列処理トリガー | 毒ダメージ、天候、カスタム UI 監視などバックグラウンド処理に必須 |
| イベント接触トリガー | シンボルエンカウント敵に必須 |

### 6.3. イベントコマンドの不足

| コマンド | 用途 |
|---------|------|
| 場所移動 | マップ間遷移（町→家の中） |
| 所持品の増減 | 所持金・アイテム・武器・防具の加減 |
| パーティ変更 | 仲間の加入・離脱 |
| 全回復 | 宿屋・回復の泉 |
| ショップ処理 | 売買画面の呼出 |
| セーブ画面を開く | セーブポイント |
| 移動ルート設定 | キャラを N 歩動かすカットシーン制作 |

## 7. kaedevn の差別化ポイント

### 7.1. ツクール作者が「感動する」本家超えの UX

| 機能 | 効果 |
|------|------|
| **16:9 リアルタイムテキストプレビュー** | 文字のはみ出し確認作業が消滅。本家にない「神機能」 |
| **データベース参照ドロップダウン** | ID 暗記不要。タイポによる「存在しない参照」バグを未然防止 |
| **F9 デバッガ** | 変数・スイッチのリアルタイム監視＋即時書換、座標・所持金の一覧 |
| **◆インデント表記のシナリオエディタ** | 条件分岐の入れ子に完全対応、本家と同じ論理で組める |

### 7.2. 既存エンジンの独自強み

| 強み | 詳細 |
|------|------|
| ノベルエンジン由来の演出 | 立ち絵・バックログ・複雑なテキスト演出が標準。「RPG ベースのノベル」という独自ジャンル |
| KSC スクリプト | プラグイン不要で透明性の高いロジック拡張 |
| AI ネイティブ設計 | JSON/API/CLI ベースで AI が直接操作可能 |
| ブラウザ完結 | インストール不要、OS 不問、クラウド保存 |
| 公式アセット 600 点無料 | セマンティック検索で発見可能 |

### 7.3. ツクール作者が「戸惑う」かもしれない点

|----|---------|
| 素材管理がアップロード式（ローカルフォルダ直接配置ではない） | UI でワンステップ感を最小化 |
| JS プラグインの文化がない | UI レイアウトエディタで「コード不要のデザインカスタマイズ」に昇華予定 |

## 8. 本家超越ビジョン

Phase 1-5（ツクール互換 UI 構築）完了後に実現する、Web ネイティブならではの次世代機能。

### 8.1. ビジュアルスクリプト（ノードエディタ）

- コマンドリスト（◆表記）と Blueprint 風ノード接続をワンクリック切替
- 複雑な条件分岐・並列処理の全体像を視覚的フローチャートで把握

### 8.2. リアルタイム共同編集（Figma 化）

- 同一プロジェクトに複数人が同時ログイン・編集
- 誰がどのマップ/イベントを編集中かアイコン表示
- ファイル競合をゼロに

### 8.3. クロスデバイス制作

- タブレット/スマホ向けタッチ UI
- 「思いついた瞬間にエディタを開く」制作習慣の創出

### 8.4. WYSIWYG UI レイアウトエディタ

- メッセージ枠・HP ゲージ・ボタンをドラッグ＆ドロップで自由配置
- コード不要で「画面デザインの差別化」を実現

### 8.5. AI 制作エージェント統合

- エディタ内に LLM を直結
- 「このマップに合う村人を 3 人置いて、魔王の噂を違う口調で話させて」→ イベントコマンド自動生成
- 作者が「企画・演出」に集中できる環境
