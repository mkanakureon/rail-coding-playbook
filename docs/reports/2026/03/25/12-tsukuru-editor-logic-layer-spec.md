# 次世代ツクールエディタ ロジック層仕様書

> Generated by Claude Opus 4.6

本文書は `11-tsukuru-editor-technical-spec.md` の続編。
データ層（Entity Graph）とビュー層（Multiple Projections）の上に載る**ロジック層**を定義する。

---

## 0. この仕様が解決する問題

P4（空間とロジックの断絶）を技術的に解決する。加えて P2（状態の意味が失われる）のうち、「イベントフローの可視化」と「静的解析」の具体的な実現方法を定める。

ゲーム制作ツールにおける「ロジック」とは：
- **何が起きるか**（コマンド: テキスト表示、画面遷移、アイテム増減…）
- **いつ起きるか**（トリガー: プレイヤー接触、フラグ成立、自動実行…）
- **どの順で起きるか**（フロー: 逐次、分岐、ループ、並列…）

これらを Entity Graph 上でどう表現し、どう実行するかを定義する。

---

## 1. イベントの Entity 表現

### 1.1. イベント = Entity + 子ノードのツリー

イベントは単一の Entity ではなく、**コマンドのツリー構造**を持つ。このツリーを Entity Graph 上で表現する方法が2つある：

**案A: ツリーを Entity に分解する**
- 各コマンドが Entity、親子関係が Relation
- 利点: グラフクエリが統一的に使える
- 欠点: コマンド1つに Entity 1つは粒度が細かすぎる。数千〜数万 Entity が生まれる

**案B: イベント Entity の properties にツリーを格納する**
- イベント全体が1つの Entity、コマンドツリーは `properties.commands` に JSON として入る
- 利点: 粒度が自然。1イベント = 1 Entity
- 欠点: ツリー内部はグラフクエリの対象外。別途パーサーが必要

**採用: 案B + インデックス**

イベント Entity の `properties.commands` にコマンドツリーを格納する。ただし、**コマンドツリー内の状態参照（変数の読み書き）を Relation として抽出**することで、R2（依存関係の自動可視化）を実現する。

```typescript
interface EventEntity extends Entity {
  type: "event";
  properties: {
    commands: CommandNode[];  // コマンドツリー
    // 以下はコマンドツリーから自動抽出（キャッシュ）
    readsStates: string[];   // 参照する state_variable の Entity ID 一覧
    writesStates: string[];  // 更新する state_variable の Entity ID 一覧
  };
}
```

コマンドツリーが変更されるたびに、`readsStates` / `writesStates` を再計算し、対応する Relation を自動更新する。これにより：
- イベント内部の構造はコンパクト（案Bの利点）
- 状態依存はグラフクエリで辿れる（案Aの利点）

### 1.2. コマンドノード

```typescript
interface CommandNode {
  id: string;              // ノード内で一意（ULID）
  type: CommandType;       // コマンド種別
  params: Record<string, unknown>;  // コマンド固有のパラメータ
  children?: CommandNode[];         // 分岐・ループの中身
}

// コマンド種別は固定セットではなく、Schema で拡張可能にする
type CommandType = string;
```

### 1.3. コマンドの分類

コマンドを**機能**ではなく**性質**で分類する。これにより、新しいコマンドを追加したときに分類が破綻しない。

| 性質 | 説明 | 例 |
|------|------|-----|
| **表現** | プレイヤーに何かを見せる・聞かせる | テキスト表示、画像変更、SE再生、画面効果 |
| **状態変更** | ゲームの内部状態を変える | 変数操作、アイテム増減、パーティ変更、HP回復 |
| **フロー制御** | 実行の流れを変える | 条件分岐、ループ、ジャンプ、待機 |
| **シーン遷移** | ゲームの「場所」を変える | マップ移動、バトル開始、ショップ呼出、セーブ画面 |
| **外部呼出** | 別のイベントやテンプレートを実行する | コモンイベント呼出、サブルーチン |

この分類はコマンドパレットの表示に使うが、実行時には区別しない。

## 2. トリガーシステム

### 2.1. トリガー = 「いつ・どの条件で実行するか」の宣言

トリガーはイベント Entity の properties として定義する。

```typescript
interface TriggerDef {
  type: TriggerType;
  conditions: TriggerCondition[];  // AND 結合（すべて満たしたとき発火）
}

type TriggerType =
  | "interact"       // プレイヤーが決定ボタンを押した
  | "player_touch"   // プレイヤーがこのイベントに触れた
  | "event_touch"    // このイベントがプレイヤーに触れた（自律移動で接近）
  | "auto"           // 条件を満たしたら自動実行（1回）
  | "parallel";      // 条件を満たしている間、バックグラウンドで繰り返し実行

interface TriggerCondition {
  type: "state_equals" | "item_owned" | "actor_in_party" | "custom";
  stateEntityId?: string;   // 参照する state_variable
  operator?: "==" | "!=" | ">" | "<" | ">=" | "<=";
  value?: unknown;
  // custom の場合
  expression?: string;
}
```

### 2.2. イベントページ = トリガー + コマンドツリーの組

1つのイベント Entity が複数の「ページ」を持つ。ページは優先度の高い方から評価され、条件を満たす最初のページが有効になる。

```typescript
interface EventPage {
  id: string;
  priority: number;           // 大きいほど優先（後ろのページほど優先）
  trigger: TriggerDef;
  commands: CommandNode[];
  // 空間ビューでの見た目（このページが有効なときの表示）
  appearance?: {
    spriteEntityId?: string;  // 表示するスプライト（Asset Entity）
    direction?: number;
    animation?: "walk" | "idle" | "none";
    passthrough?: boolean;    // すり抜け可能か
    abovePlayer?: boolean;    // プレイヤーの上に表示するか
  };
  // 自律移動
  autonomousMovement?: {
    type: "fixed" | "random" | "approach" | "custom";
    speed?: number;
    frequency?: number;
    route?: MoveCommand[];    // custom の場合
  };
}
```

**R4 との接続**: 空間ビューの `stateOverlay` で「フラグX = true のとき」を選ぶと、各イベントのどのページが有効になるかが計算され、`appearance` がそのページのものに切り替わる。これが「透視」機能の実体。

### 2.3. 並列処理の実行モデル

`parallel` トリガーは他のイベントと同時に走る。実行モデル：

```
メインループ（1フレームごと）:
  1. parallel イベントを全て1ステップ実行（コマンド1つ分）
  2. auto イベントがあれば、それを実行（完了まで占有）
  3. プレイヤー入力を受け付ける
  4. interact / touch トリガーをチェック → 該当すれば実行（完了まで占有）
```

parallel は**協調的マルチタスク**（プリエンプティブではない）。1フレームに1コマンドずつ進む。これにより：
- デバッガでステップ実行が可能
- 競合状態が起きにくい
- 実装が単純

## 3. コマンドツリーからの Relation 自動抽出

### 3.1. 抽出ルール

コマンドツリーが保存されるたびに、以下のルールで Relation を生成・更新する。

```typescript
interface RelationExtractor {
  extract(event: EventEntity): Relation[];
}
```

| コマンドの性質 | 抽出される Relation | 例 |
|--------------|-------------------|-----|
| 状態変更（変数書込） | `event --writes--> state_variable` | 「変数Xに1を加算」→ writes |
| フロー制御（条件参照） | `event --reads--> state_variable` | 「変数Xが5以上なら」→ reads |
| 状態変更（アイテム増減） | `event --modifies--> item` | 「薬草を3個追加」→ modifies |
| シーン遷移（マップ移動） | `event --transitions_to--> map` | 「城マップに移動」→ transitions_to |
| シーン遷移（バトル開始） | `event --starts_battle--> troop` | 「スライム群と戦闘」→ starts_battle |
| 外部呼出 | `event --calls--> event` | 「コモンイベント:宿屋を呼出」→ calls |
| トリガー条件 | `event --requires--> state_variable` | ページの出現条件 → requires |

### 3.2. 抽出のタイミング

- **保存時**: コマンドツリーの変更を検知し、差分で Relation を更新
- **バッチ**: プロジェクト全体の Relation を再構築（整合性チェック用）

これは**インデックス**であり、真のデータはコマンドツリー内にある。Relation が壊れてもコマンドツリーから再構築できる。

## 4. 静的解析

### 4.1. 解析の種類

Relation が揃った状態で、以下の解析が可能になる。

```typescript
interface StaticAnalyzer {
  // 書かれるが読まれない変数（不要な可能性）
  findUnreadStates(): Entity[];

  // 読まれるが書かれない変数（初期値のまま＝バグの可能性）
  findUnwrittenStates(): Entity[];

  // 到達不能イベント（条件を満たす手段がない）
  findUnreachableEvents(): { event: Entity; reason: string }[];

  // 循環呼出（A→B→C→A）
  findCircularCalls(): Entity[][];

  // 影響分析（この変数を変えたら何が動くか）
  getDownstream(stateEntityId: string): Entity[];

  // 逆影響分析（このイベントが動くには何が必要か）
  getUpstream(eventEntityId: string): Entity[];
}
```

### 4.2. フロービューとの接続

フロービュー（11-spec セクション 3.6）は、この静的解析の結果を可視化する。

- `findUnreachableEvents()` の結果 → フロービューで赤表示
- `getDownstream()` の結果 → グラフビューでハイライト
- `findUnwrittenStates()` の結果 → 変数一覧に警告アイコン

## 5. コンパイルパイプライン

### 5.1. 設計思想

Entity Graph（設計時データ）からランタイムが実行する命令列（実行時データ）への変換パイプライン。

```
Entity Graph (設計時)
    ↓ [Compiler]
Instruction Stream (実行時)
    ↓ [Runtime]
Game Execution
```

設計時データと実行時データを分離する理由：
- エディタは「人間が理解しやすい構造」を使う（ツリー、名前、関係）
- ランタイムは「機械が実行しやすい構造」を使う（フラットな命令列、アドレス）
- 分離することで、エディタとランタイムを独立に進化できる

### 5.2. コンパイラの入出力

```typescript
interface Compiler {
  compile(project: {
    entities: Entity[];
    relations: Relation[];
  }): CompilationResult;
}

interface CompilationResult {
  // マップごとの命令バンドル
  maps: Map<string, MapBundle>;
  // グローバルデータ（データベース的なもの）
  data: DataBundle;
  // エラー・警告
  diagnostics: Diagnostic[];
}

interface MapBundle {
  mapId: string;
  events: CompiledEvent[];
  tileData: unknown;       // タイル情報
  properties: unknown;     // マッププロパティ（BGM、エンカウント等）
}

interface CompiledEvent {
  eventId: string;
  pages: CompiledPage[];
}

interface CompiledPage {
  priority: number;
  trigger: TriggerDef;             // トリガーはそのまま渡す
  instructions: Instruction[];     // コマンドツリーをフラット化した命令列
  appearance?: unknown;
  autonomousMovement?: unknown;
}

// 命令 = ランタイムが直接実行する最小単位
interface Instruction {
  op: string;                       // オペコード
  args: unknown[];                  // 引数（ID は解決済みのアドレスや値）
}
```

### 5.3. コンパイル時の処理

| 処理 | 内容 |
|------|------|
| **名前解決** | Entity の名前参照を ID に変換 |
| **フラット化** | コマンドツリーの分岐・ループを条件ジャンプ命令に変換 |
| **最適化** | 不要な命令の除去、定数畳み込み（将来） |
| **検証** | 静的解析（セクション4）を実行し、diagnostics に追加 |
| **バンドル** | マップごとに必要なデータをまとめる |

### 5.4. ランタイムの責務

ランタイムは Instruction を逐次実行する。ランタイムの設計はプラットフォーム（Web / Switch）ごとに異なるが、Instruction のフォーマットは共通。

```
同じ CompilationResult
    ├── Web Runtime (PixiJS + Web Audio)
    └── Switch Runtime (SDL2 + Native Audio)
```

ランタイムの詳細仕様は本文書のスコープ外（プラットフォーム固有）。
