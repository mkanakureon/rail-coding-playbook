## 0. 統一的な洞察

R1〜R4 を並べると、共通する構造的問題が浮かぶ：

| 原理 | 要求 | 共通点 |
|------|------|--------|
| R1 | データ間の関係を双方向に辿れる | **関係の可視化** |
| R2 | 状態の依存関係が自動で分かる | **関係の可視化** |
| R3 | 見た目がロジックから独立している | **関心の分離** |
| R4 | 空間×時間×条件を統合的に見える | **多次元ビュー** |

**根本解**: ゲームデータを「種別ごとの表」ではなく **「型付きエンティティのグラフ」** として扱い、そのグラフに対する **複数のビュー（投影）** を提供する。

## 1. データモデル: Entity Graph

### 1.1. 設計思想

既存ツールの問題は、データが種別ごとに閉じた配列（`actors[]`, `skills[]`, `items[]`）に格納され、関係性が暗黙の ID 参照に埋もれることにある。

本設計では：
- すべてのゲームデータが **Entity**（ノード）
- Entity 間の依存が **Relation**（エッジ）として明示される
- Entity の種別と構造は **Schema** が定義する
- エディタ UI は Schema から動的に構成される（ハードコードしない）

### 1.2. Entity

```typescript
interface Entity {
  id: string;                          // ULID
  type: EntityType;                    // Schema で定義された型名
  name: string;                        // 人間可読な識別名
  properties: Record<string, unknown>; // 型ごとに異なるプロパティ
  spatial?: SpatialInfo;               // 空間情報（マップ上の位置等）
  createdAt: number;
  updatedAt: number;
}

// 空間情報（R4 対応）
interface SpatialInfo {
  mapId: string;       // 所属マップ
  x: number;
  y: number;
  layer?: string;      // 表示レイヤー
  direction?: number;  // 向き（0-3）
}
```

**Entity はフラットな一枚のテーブルに入る。** 種別ごとにテーブルを分けない。これにより、異なる種別の Entity を横断して検索・比較できる。

### 1.3. Relation

```typescript
interface Relation {
  id: string;
  sourceId: string;     // Entity ID
  targetId: string;     // Entity ID
  type: RelationType;   // Schema で定義された関係種別
  metadata?: Record<string, unknown>; // 関係に付随するデータ
}
```

**関係の例:**

| source | relation | target | metadata |
|--------|----------|--------|----------|
| 勇者 | `learns` | ファイアボール | `{ level: 5 }` |
| ファイアボール | `costs` | MP | `{ amount: 10 }` |
| スライム | `drops` | 薬草 | `{ rate: 0.3 }` |
| イベント:宝箱 | `writes` | フラグ:宝箱A開封 | |
| イベント:門番 | `reads` | フラグ:宝箱A開封 | |
| フィールドマップ | `encounters` | スライム群 | `{ steps: 30 }` |

**R1 への回答**: Relation テーブルを `sourceId` でも `targetId` でも引けるので、「このスキルを使うキャラ一覧」も「このキャラが使うスキル一覧」も同じクエリで取れる。

**R2 への回答**: イベントが変数を `reads` / `writes` する関係が明示されるため、「この変数に依存するイベント一覧」が自動的に分かる。

### 1.4. Schema（型定義）

```typescript
interface EntitySchema {
  type: string;                    // "actor", "skill", "map_event", etc.
  label: string;                   // UI表示名（「アクター」「スキル」等）
  icon?: string;
  properties: PropertyDef[];       // プロパティ定義
  allowedRelations: RelationDef[]; // この型が持てる関係
  spatialCapable: boolean;         // マップ上に配置可能か
  category?: string;               // グルーピング（UI用）
}

interface PropertyDef {
  key: string;
  label: string;
  type: "string" | "number" | "boolean" | "enum" | "formula" | "rich_text" | "asset_ref" | "color";
  required?: boolean;
  default?: unknown;
  constraints?: {
    min?: number;
    max?: number;
    options?: { value: string; label: string }[];  // enum用
  };
  description?: string;  // AI・ヘルプ向け説明
}

interface RelationDef {
  type: string;            // "learns", "drops", "reads", "writes", etc.
  label: string;           // UI表示名
  targetType: string;      // 接続先の EntityType
  cardinality: "one" | "many";
  metadata?: PropertyDef[];  // 関係に付随するデータの定義
}
```

**ポイント**: Schema が「何タブ作るか」を決める。エディタは Schema を読んで UI を動的に生成する。Schema を変えればエディタが変わる。コードを変える必要がない。

### 1.5. 永続化

```
Project
├── schema.json          ← EntitySchema[] （プロジェクト種別で異なる）
├── entities.json        ← Entity[]
├── relations.json       ← Relation[]
├── views/               ← ビュー設定（後述）
└── assets/              ← 素材ファイル
```

製品はゼロから構築する前提。試作コードのデータ構造との互換性は考慮しない。

## 2. クエリ層: Graph Queries

### 2.1. 基本クエリ

```typescript
interface EntityStore {
  // 単体取得
  get(id: string): Entity | null;

  // 型で絞り込み
  findByType(type: EntityType): Entity[];

  // プロパティで検索
  find(query: EntityQuery): Entity[];

  // 関係を辿る（R1 の核）
  getRelated(entityId: string, relationType?: string, direction?: "outgoing" | "incoming" | "both"): RelatedEntity[];

  // 影響分析（R1: 変更の影響範囲）
  getImpactGraph(entityId: string, depth?: number): ImpactNode[];

  // 空間クエリ（R4: マップ上の検索）
  findInArea(mapId: string, rect: { x: number; y: number; w: number; h: number }): Entity[];
}

interface RelatedEntity {
  entity: Entity;
  relation: Relation;
  direction: "outgoing" | "incoming";
}

interface ImpactNode {
  entity: Entity;
  path: Relation[];  // ルートからの経路
  depth: number;
}
```

### 2.2. 状態依存分析（R2 の核）

```typescript
interface StateAnalyzer {
  // この変数/フラグを読むイベント一覧
  getReaders(stateEntityId: string): Entity[];

  // この変数/フラグを書くイベント一覧
  getWriters(stateEntityId: string): Entity[];

  // 到達不能パスの検出（静的解析）
  findUnreachablePaths(): UnreachablePath[];

  // 循環依存の検出
  findCircularDependencies(): Relation[][];
}

interface UnreachablePath {
  event: Entity;
  condition: string;     // 到達不能と判定された条件
  reason: string;        // 理由（「変数Xを書くイベントが存在しない」等）
}
```

### 2.3. バランス分析（R1 の応用）

```typescript
interface BalanceAnalyzer {
  // ダメージ計算式のシミュレーション
  simulateDamage(skillId: string, attackerId: string, defenderId: string): DamageResult;

  // レベル帯ごとのバランス概観
  getBalanceOverview(levelRange: [number, number]): BalanceSnapshot;

  // 「このスキルの威力を変えたら何が変わるか」
  whatIf(entityId: string, propertyKey: string, newValue: unknown): ImpactNode[];
}
```

## 3. ビューシステム: Multiple Projections

### 3.1. 設計思想

同じ Entity Graph に対して、複数の「見方」を提供する。ビューは投影（Projection）であり、データそのものではない。

**R3 への回答**: ビューの定義（どう見せるか）は Entity のプロパティ（何であるか）から完全に分離される。

**R4 への回答**: 空間ビュー、テーブルビュー、グラフビューなど、同じデータを異なる次元で表示できる。

### 3.2. ビュー種別

```typescript
type ViewType =
  | "table"       // スプレッドシート型（R1: 一括比較）
  | "detail"      // 単一 Entity の詳細編集
  | "graph"       // 関係グラフの可視化（R1, R2）
  | "spatial"     // マップ上の空間表示（R4）
  | "timeline"    // 時間軸表示（R4: シーケンス編集）
  | "canvas"      // 自由配置（R3: UIレイアウト）
  | "flow"        // フローチャート（R2: イベント条件の可視化）
  | "dashboard";  // 複数ビューの組み合わせ

interface ViewConfig {
  id: string;
  type: ViewType;
  label: string;
  // どの Entity をこのビューに含めるか
  filter: {
    types?: EntityType[];
    spatial?: { mapId: string };
    relatedTo?: string;        // 特定 Entity に関連するもの全て
    custom?: string;           // 将来の拡張用クエリ式
  };
  // ビュー固有の設定
  options: Record<string, unknown>;
}
```

### 3.3. テーブルビュー（R1: スプレッドシート型一括編集）

```typescript
interface TableViewOptions {
  columns: {
    key: string;        // Entity の property key
    label: string;
    width?: number;
    sortable?: boolean;
    editable?: boolean;
  }[];
  // 異なる型の Entity を横に並べて比較（例: スキルと敵の耐性）
  compareWith?: {
    type: EntityType;
    joinRelation: string;  // どの Relation で紐付けるか
  };
  groupBy?: string;  // プロパティでグルーピング
}
```

**ユースケース**: 全スキルの威力を一覧で見ながら、横に「そのスキルを使う敵」のHPを表示する。ツクールでは不可能だった**横断的なバランス調整**が 1 画面で完結する。

### 3.4. グラフビュー（R1, R2: 関係の可視化）

```typescript
interface GraphViewOptions {
  rootEntityId?: string;  // 起点 Entity（省略で全体）
  depth: number;          // 何ホップ先まで表示するか
  relationFilter?: string[];  // 表示する Relation type
  layout: "force" | "hierarchical" | "radial";
  // ハイライト
  highlight?: {
    entityIds?: string[];
    impactOf?: string;    // この Entity を変更した場合の影響範囲をハイライト
  };
}
```

**ユースケース**: 「勇者」を起点にグラフを展開すると、習得スキル→スキルの消費MP→MPの成長曲線→装備武器→武器の攻撃力→ダメージ計算式が**一つの絵**で見える。

### 3.5. 空間ビュー（R4: マップ上の統合表示）

```typescript
interface SpatialViewOptions {
  mapId: string;
  layers: {
    tiles: boolean;
    collision: boolean;
    events: boolean;
    regions: boolean;
    // R4: 条件による「透視」
    stateOverlay?: {
      stateEntityId: string;  // このフラグが ON のとき
      value: unknown;
    };
  };
  // R4: 見えない属性のオーバーレイ
  overlays: {
    encounterRate?: boolean;  // エンカウント率のヒートマップ
    bgm?: boolean;            // BGM 境界線
    eventTriggers?: boolean;  // トリガー種別のアイコン表示
    movementRoutes?: boolean; // NPC移動ルートの軌跡表示
  };
}
```

**ユースケース**: マップエディタで「フラグ:ボス撃破 = true のとき」を選択すると、その状態で消えるイベント（門番）がゴースト表示され、出現するイベント（新しいNPC）がハイライトされる。

### 3.6. フロービュー（R2: イベントロジックの可視化）

```typescript
interface FlowViewOptions {
  eventEntityId: string;
  showConditions: boolean;       // 条件分岐をノードとして展開
  showStateAccess: boolean;      // 読み書きする変数を接続表示
  highlightUnreachable: boolean; // 到達不能パスを赤表示
  simulationState?: Record<string, unknown>;  // 「この変数値のとき」のパスをハイライト
}
```

### 3.7. キャンバスビュー（R3: UIレイアウト編集）

```typescript
interface CanvasViewOptions {
  resolution: { width: number; height: number };  // 論理解像度
  // キャンバス上の要素は Entity（type: "ui_element"）
  // 位置・サイズ・スタイルは Entity の properties
  snap: { grid?: number; guides?: boolean };
  preview: {
    mode: "editor" | "runtime";  // エディタ表示 / 実機プレビュー切替
    device?: "switch" | "web" | "mobile";  // R3: 100% Parity 確認
  };
}
```

### 3.8. ダッシュボード（複数ビューの合成）

```typescript
interface DashboardViewOptions {
  layout: "split-h" | "split-v" | "quad" | "custom";
  panels: {
    viewId: string;
    position: { x: number; y: number; w: number; h: number };
  }[];
  // パネル間の連動
  sync?: {
    selection: boolean;  // 一方で Entity を選択すると他方も追従
    filter: boolean;     // 一方のフィルタが他方に伝播
  };
}
```

**ユースケース**: 左にマップ（空間ビュー）、右上にそのマップのイベント一覧（テーブルビュー）、右下に選択中イベントのフロー（フロービュー）。マップ上でイベントをクリックすると3つが連動する。

## 4. 状態システム: Named State

### 4.1. 変数/フラグの Entity 化（R2 の核）

変数・フラグは特別扱いではなく、Entity の一種として統一管理する。

```typescript
// Schema での定義例
const stateSchema: EntitySchema = {
  type: "state_variable",
  label: "変数",
  properties: [
    { key: "dataType", label: "型", type: "enum",
      constraints: { options: [
        { value: "boolean", label: "スイッチ（ON/OFF）" },
        { value: "number",  label: "数値" },
        { value: "string",  label: "文字列" },
      ]}
    },
    { key: "scope", label: "スコープ", type: "enum",
      constraints: { options: [
        { value: "global",     label: "グローバル" },
        { value: "self",       label: "セルフ（イベント固有）" },
        { value: "map",        label: "マップローカル" },
      ]}
    },
    { key: "initialValue", label: "初期値", type: "string" },
    { key: "description",  label: "用途メモ", type: "string" },
  ],
  allowedRelations: [
    { type: "read_by",    label: "参照元", targetType: "map_event", cardinality: "many" },
    { type: "written_by", label: "更新元", targetType: "map_event", cardinality: "many" },
  ],
  spatialCapable: false,
};
```

**R2 への回答**:
- 番号ではなく名前で管理（`Entity.name = "ボス撃破フラグ"`）
- 型がある（boolean / number / string）
- 依存関係が Relation で自動追跡
- スコープが明示（global / self / map）→ セルフスイッチ問題も解決

### 4.2. ランタイムデバッガ（R2: リアルタイム追跡）

```typescript
interface RuntimeDebugger {
  // 全状態変数の現在値
  getStateSnapshot(): Map<string, { entity: Entity; currentValue: unknown }>;

  // 状態変更のリアルタイムストリーム
  onStateChange(callback: (change: StateChange) => void): () => void;

  // 値の書き換え（デバッグ用）
  setState(entityId: string, value: unknown): void;

  // ブレークポイント（特定の変数が変わったら停止）
  setBreakpoint(entityId: string, condition?: string): void;
}

interface StateChange {
  stateEntityId: string;
  oldValue: unknown;
  newValue: unknown;
  causedBy: Entity;     // どのイベントが変更したか
  timestamp: number;
}
```

## 5. プレゼンテーション層: Declarative Visuals（R3）

### 5.1. 設計思想

「見た目」は Entity のプロパティとして宣言的に定義し、エディタとランタイムで同じデータを読む。

```
Entity (type: "ui_element")
├── properties.rect: { x, y, w, h }
├── properties.style: { bgColor, borderRadius, opacity, ... }
├── properties.content: { type: "text" | "image" | "gauge", ... }
└── properties.binding: { source: "state_variable_id", transform?: "..." }
```

### 5.2. スタイルシステム

```typescript
interface VisualStyle {
  // 基本
  backgroundColor?: string;
  borderColor?: string;
  borderWidth?: number;
  borderRadius?: number;
  opacity?: number;

  // テキスト
  fontFamily?: string;
  fontSize?: number;
  fontColor?: string;
  textAlign?: "left" | "center" | "right";
  lineHeight?: number;

  // 背景画像
  backgroundImage?: string;  // Asset Entity の ID
  backgroundFit?: "cover" | "contain" | "stretch" | "tile";

  // アニメーション
  transition?: {
    property: string;
    duration: number;
    easing: string;
  }[];
}
```

### 5.3. データバインディング

UI 要素が動的な値（HP, 所持金, 変数）を表示する場合、プログラミングではなく**宣言的なバインディング**で接続する。

```typescript
interface DataBinding {
  source: string;           // Entity ID（state_variable, actor 等）
  property: string;         // 参照するプロパティ（"properties.hp" 等）
  transform?: string;       // 表示変換式（"value / maxValue * 100" → ゲージ幅%）
  format?: string;          // 表示フォーマット（"{value} / {max}" 等）
}
```

**R3 への回答**: HP ゲージを作るのに必要なのは、キャンバス上にゲージ要素を置き、`source: actor_entity_id, property: "properties.hp"` をバインドするだけ。コード不要。

### 5.4. Parity 保証

```typescript
interface ParityCheck {
  // エディタ上の表示とランタイムの表示を比較
  compare(viewConfig: CanvasViewOptions): ParityResult;
}

interface ParityResult {
  match: boolean;
  differences: {
    elementId: string;
    property: string;
    editorValue: unknown;
    runtimeValue: unknown;
  }[];
}
```

エディタとランタイムが**同じレンダリングエンジン（PixiJS）で同じ宣言データを読む**ことで、構造的に Parity を保証する。「同じエンジンなので同じ表示になる」が原理。

## 6. Schema プリセット

プロジェクト種別（ノベル、RPG、その他）ごとに Schema のプリセットを用意する。プリセットに含まれる EntityType の種類と数がエディタの UI 構造を決定する。

具体的な EntityType の定義は、製品要件（どのジャンルのゲームを作れるようにするか）から導出する。既存ツールの画面構成を模倣して決めない。

## 8. R1〜R4 との対応まとめ

| 原理 | 本仕様での実現 |
|------|--------------|
| **R1**: データをグラフとして双方向に辿れる | Entity + Relation モデル、`getRelated()` 双方向クエリ、`getImpactGraph()` 影響分析 |
| **R2**: 状態が名前付き構造で依存関係が自動可視 | state_variable Entity、read_by/written_by Relation、StateAnalyzer（静的解析）、RuntimeDebugger（動的追跡） |
| **R3**: 見た目がロジックから独立して宣言的 | VisualStyle + DataBinding による宣言的 UI、Canvas ビュー、同一エンジンによる Parity 保証 |
| **R4**: 空間×時間×条件の統合表示 | SpatialInfo 付き Entity、Spatial ビューの stateOverlay、フロービューのシミュレーション |
