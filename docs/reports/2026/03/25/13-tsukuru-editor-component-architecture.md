> Generated by Claude Opus 4.6

本文書は 11（データ・ビュー仕様）と 12（ロジック層仕様）を React コンポーネントとしてどう実装するかを定義する。

---

## 1. レイヤー構成

```
React Components（UI）
    ↓ 読み書き
EntityStore（状態管理）
    ↓ 永続化
Storage Adapter（保存先抽象化）
```

### 1.1. EntityStore

Entity Graph のインメモリ表現。全コンポーネントがここを通してデータにアクセスする。

```typescript
// Zustand ストア
interface EntityStoreState {
  entities: Map<string, Entity>;
  relations: Map<string, Relation>;
  schema: EntitySchema[];

  // 読み取り
  get(id: string): Entity | undefined;
  findByType(type: string): Entity[];
  getRelated(id: string, relationType?: string, direction?: "out" | "in" | "both"): RelatedEntity[];
  findInArea(mapId: string, rect: Rect): Entity[];

  // 書き込み
  upsertEntity(entity: Entity): void;
  removeEntity(id: string): void;
  upsertRelation(relation: Relation): void;
  removeRelation(id: string): void;

  // 一括操作
  importProject(data: { entities: Entity[]; relations: Relation[]; schema: EntitySchema[] }): void;
  exportProject(): { entities: Entity[]; relations: Relation[]; schema: EntitySchema[] };
}
```

Zustand を使う理由: React の外（コンパイラ、解析器）からも `getState()` でアクセスできる。Context に閉じない。

### 1.2. Storage Adapter

EntityStore とバックエンドの間の抽象化層。

```typescript
interface StorageAdapter {
  save(projectId: string, data: ProjectData): Promise<void>;
  load(projectId: string): Promise<ProjectData>;
  // 将来: リアルタイム同期用
  subscribe?(projectId: string, callback: (patch: Patch) => void): () => void;
}
```

最初は `LocalStorageAdapter`（JSON ファイル）で十分。後からサーバー同期に差し替え可能。

## 2. ビューコンポーネント

11-spec のビュー種別ごとに React コンポーネントを対応させる。

### 2.1. 共通構造

すべてのビューは同じインターフェースに従う。

```typescript
interface ViewProps {
  config: ViewConfig;          // どの Entity をどう表示するか
  onSelectEntity?: (id: string) => void;  // 選択の伝播（ダッシュボード連動用）
  selectedEntityId?: string;              // 外部から選択を受け取る
}
```

### 2.2. TableView — スプレッドシート型

```
┌─────────────────────────────────────────┐
│ [フィルタ] [カラム設定]        [検索]   │
├──────┬────────┬────────┬────────┬───────┤
│ 名前 │ HP     │ 攻撃力 │ 使用スキル│ ... │
├──────┼────────┼────────┼────────┼───────┤
│ スラ │ 50     │ 12     │ 体当たり  │     │
│ ゴブ │ 80     │ 18     │ 毒攻撃   │     │
│ ドラ │ 500    │ 45     │ 火炎     │     │
└──────┴────────┴────────┴────────┴───────┘
```

- セルをクリック → インライン編集
- 「使用スキル」列はRelation 経由で取得（Entity 間のリンク）
- カラムの追加/削除/並替はユーザーがカスタマイズ可能
- ソート・フィルタは EntityStore のクエリで処理

### 2.3. GraphView — 関係グラフ

```
         ┌──────┐
    ┌───→│ 薬草 │
    │    └──────┘
┌───┴──┐         ┌──────────┐
│スライム│────────→│ 体当たり │
└───┬──┘ drops   └──────────┘
    │    learns
    │    ┌────────┐
    └───→│ 逃げる │
         └────────┘
```

- Entity をノード、Relation をエッジとして描画
- ノードをドラッグで移動
- ノードをクリック → 他のビューの選択が連動
- 力学モデルレイアウト（d3-force 等）

### 2.4. SpatialView — マップエディタ

```
┌──────────────┬──────────────────────┐
│ タイル       │                      │
│ パレット     │    マップキャンバス    │
│              │    (PixiJS)          │
│ [地面]       │    ┌──┐              │
│ [装飾]       │    │NPC│← イベント   │
│ [壁]        │    └──┘              │
│              │         ┌──┐        │
│              │         │宝箱│       │
│              │         └──┘        │
└──────────────┴──────────────────────┘
```

- タイル描画は PixiJS（Canvas ではなく WebGL）
- イベント Entity は `spatial` プロパティを持つものをマップ上に表示
- オーバーレイ切替: 通行判定 / エンカウント率 / トリガー種別 / 状態透視

### 2.5. FlowView — イベントフロー

```
┌─────────┐     ┌──────────┐
│ 開始    │────→│ テキスト │
└─────────┘     │「話を聞い│
                │ てくれ」 │
                └────┬─────┘
                     │
                ┌────▼─────┐
                │ 条件分岐 │
                │ フラグX?  │
                └──┬────┬──┘
              true │    │ false
           ┌───────▼┐ ┌▼───────┐
           │「ありが│ │「まだだ│
           │ とう」 │ │ ね」  │
           └────────┘ └────────┘
```

- CommandNode ツリーをフローチャートに変換して描画
- 状態参照は state_variable Entity へのリンクとして表示
- 到達不能パスは赤表示（StaticAnalyzer の結果）
- シミュレーション: 変数値を仮設定してパスをハイライト

### 2.6. CanvasView — UIレイアウト

```
┌──────────────────────────────────────┐
│              ゲーム画面              │
│                                      │
│                                      │
│  ┌──────────────────────────────┐   │
│  │  メッセージウィンドウ         │   │  ← ドラッグで移動・リサイズ
│  │  「ここにテキストが入る」     │   │
│  └──────────────────────────────┘   │
│  [Auto] [Skip] [Log] [Save]         │  ← ボタン配置
└──────────────────────────────────────┘
```

- UI 要素は Entity（type: "ui_element"）
- ドラッグ＆ドロップで配置
- プロパティパネルでスタイル編集
- プレビュー切替: エディタモード / ランタイムモード

### 2.7. Dashboard — 複数ビューの合成

```
┌────────────────────┬───────────────────┐
│                    │  TableView        │
│  SpatialView       │  (イベント一覧)    │
│  (マップ)          ├───────────────────┤
│                    │  FlowView         │
│                    │  (選択中イベント)  │
└────────────────────┴───────────────────┘
```

- 分割レイアウト（水平/垂直/4分割/自由）
- パネル間の選択連動: マップでイベントをクリック → テーブルとフローが追従
- ユーザーがパネル構成をカスタマイズ・保存可能

## 3. コマンドエディタ

イベントのコマンドツリーを編集する UI。FlowView（可視化）とは別に、リスト型の編集 UI を用意する。

### 3.1. コマンドリスト

```
◆ 文章の表示: 「魔王が復活したらしい...」
◆ 選択肢の表示: [はい, いいえ]
  ◆ [はい] の場合
    ◆ 変数操作: クエスト受諾 = true
    ◆ 文章の表示: 「頼んだぞ！」
  ◆ [いいえ] の場合
    ◆ 文章の表示: 「そうか...」
◆ 分岐終了
```

- コマンドをクリック → プロパティパネルで引数を編集
- ドラッグで順序変更
- コマンドパレットから追加（性質別カテゴリ: 表現/状態変更/フロー制御/シーン遷移/外部呼出）

### 3.2. コマンドプロパティパネル

各コマンドの params を編集するフォーム。Schema 駆動で自動生成する。

```typescript
interface CommandSchema {
  type: string;              // "show_text", "set_variable", etc.
  label: string;             // UI表示名
  category: "expression" | "state_mutation" | "flow_control" | "scene_transition" | "external_call";
  params: PropertyDef[];     // Entity の PropertyDef と同じ形式
}
```

例: 「文章の表示」コマンドのスキーマ

```typescript
const showTextSchema: CommandSchema = {
  type: "show_text",
  label: "文章の表示",
  category: "expression",
  params: [
    { key: "speaker", label: "話者", type: "string" },
    { key: "text", label: "本文", type: "rich_text", required: true },
    { key: "faceEntityId", label: "顔グラフィック", type: "asset_ref" },
    { key: "position", label: "ウィンドウ位置", type: "enum",
      constraints: { options: [
        { value: "bottom", label: "下" },
        { value: "top", label: "上" },
        { value: "center", label: "中央" },
      ]}
    },
  ],
};
```

プロパティパネルはこの Schema から動的にフォームを生成する。コマンドを追加するとき、コードを書くのは Schema 定義だけ。UIコンポーネントは共通。

## 4. エディタ全体のルーティング

```
/projects/:projectId/
  ├── map/:mapId          → Dashboard (SpatialView + TableView + FlowView)
  ├── data/:entityType    → Dashboard (TableView + DetailView)
  ├── ui-layout           → CanvasView
  ├── graph               → GraphView (プロジェクト全体の関係図)
  └── test-play           → RuntimeView + DebuggerPanel
```

「画面」はルーティングで切り替わるが、各画面の中身は ViewConfig で定義された Dashboard。ユーザーがパネル構成を変えれば、画面の役割も変わる。

## 5. 実装順序

依存関係から逆算した実装順序。

### Phase A: 基盤

| # | コンポーネント | 依存 | 内容 |
|---|--------------|------|------|
| A1 | EntityStore | なし | Zustand ストア。CRUD + 基本クエリ |
| A2 | Schema 定義 | A1 | EntitySchema / CommandSchema の型と最小プリセット |
| A3 | StorageAdapter | A1 | LocalStorage 版。JSON 読み書き |
| A4 | PropertyEditor | A2 | Schema 駆動のフォーム自動生成 |

### Phase B: コアビュー

| # | コンポーネント | 依存 | 内容 |
|---|--------------|------|------|
| B1 | TableView | A1, A4 | Entity 一覧 + インライン編集 |
| B2 | DetailView | A1, A4 | 単一 Entity のプロパティ編集 |
| B3 | CommandEditor | A2, A4 | コマンドリスト + プロパティパネル |
| B4 | RelationExtractor | A1, B3 | コマンドツリーからの Relation 自動抽出 |

### Phase C: 空間・可視化

| # | コンポーネント | 依存 | 内容 |
|---|--------------|------|------|
| C1 | SpatialView | A1, PixiJS | マップエディタ（タイル描画 + イベント配置） |
| C2 | FlowView | A1, B4 | コマンドツリーのフローチャート表示 |
| C3 | GraphView | A1, B4 | Entity 間関係のグラフ表示 |
| C4 | Dashboard | B1-C3 | 複数ビューの合成 + 選択連動 |

### Phase D: ランタイム

| # | コンポーネント | 依存 | 内容 |
|---|--------------|------|------|
| D1 | Compiler | A1, A2 | Entity Graph → Instruction Stream |
| D2 | WebRuntime | D1, PixiJS | ブラウザ上での実行エンジン |
| D3 | DebuggerPanel | D2 | 状態監視 + ブレークポイント |
| D4 | TestPlayView | D2, D3 | エディタ内テストプレイ |

### Phase E: 高度な機能

| # | コンポーネント | 依存 | 内容 |
|---|--------------|------|------|
| E1 | StaticAnalyzer | B4 | 到達不能パス検出、未使用変数警告 |
| E2 | CanvasView | A1, A4, PixiJS | UIレイアウトエディタ |
| E3 | BalanceAnalyzer | A1 | ダメージシミュレーション、what-if 分析 |
| E4 | StateOverlay | C1, E1 | 空間ビューの状態透視機能 |
