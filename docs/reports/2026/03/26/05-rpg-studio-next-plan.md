## 計画（優先順）

### Phase 1: EditorAPI 層

**目的**: GUI と CLI の操作コードを1箇所にまとめる

**場所**: `packages/entity-graph/src/api/editor-api.ts`

```
GUI (React)   ──→ EditorAPI ──→ EntityStore
CLI (Node.js) ──→ EditorAPI ──→ EntityStore
AI スクリプト  ──→ EditorAPI ──→ EntityStore
```

**含める操作**:

| グループ | メソッド |
|---------|---------|
| Entity | createEntity, deleteEntity, setProperty, renameEntity, cloneEntity |
| Relation | addRelation, removeRelation |
| Map | createMap, setTile, fillRect, setStartPosition |
| Event | placeEvent, addCommand, clearCommands |
| Query | search, findByType, getRelated, getImpactGraph |
| Analysis | analyze, simulateBalance, compile, validate |
| Project | exportProject, importProject, getStats |

**効果**:
- 操作の定義が1箇所。GUI/CLI どちらを変えても整合性が保たれる
- EditorAPI のユニットテストで GUI/CLI 両方をカバー
- 新機能追加 = API にメソッド追加 → 両方で即使える

### Phase 2: CLI 拡張

**目的**: 実用的なコマンドを追加。EditorAPI の上に薄く乗せる

| コマンド | ファイル | 内容 |
|---------|---------|------|
| `clone <id> [name]` | `commands/entity.ts` | Entity を Relation ごと複製 |
| `search <keyword>` | `commands/entity.ts` | 全 Entity の名前・プロパティを横断検索 |
| `graph <id>` | `commands/entity.ts` | 関係をテキストツリーで表示 |
| `stats` | `commands/project.ts` | プロジェクト統計（Entity 数、マップ数、イベント数、コマンド数） |
| `validate` | `commands/analysis.ts` | Schema バリデーション + 整合性チェック全実行 |
| `diff <file1> <file2>` | `commands/project.ts` | 2つのプロジェクト JSON の差分表示 |
| `test-play <event>` | `commands/run.ts` | 特定イベントだけテキスト実行 |
| `script <file.ts>` | `commands/script.ts` | EditorAPI を使ったバッチスクリプト実行 |
| `template <genre>` | `commands/project.ts` | ジャンル別テンプレートで一括生成 |
| `--json` フラグ | 全コマンド共通 | 出力を JSON にしてパイプライン対応 |

### Phase 3: バッチスクリプト実行

**目的**: AI やユーザーが EditorAPI を使ったスクリプトを書いて一括操作

```bash
rpg-cli script scripts/batch/buff-all-enemies.ts
```

```typescript
// scripts/batch/buff-all-enemies.ts
export default function(api: EditorAPI) {
  for (const enemy of api.findByType("enemy")) {
    const hp = api.getProperty(enemy.id, "hp") as number;
    api.setProperty(enemy.id, "hp", Math.floor(hp * 1.5));
  }
}
```

**ユースケース**:
- 全敵の HP を 1.5 倍にする
- 全マップの外周を壁で囲む
- AI がシナリオを生成して `placeEvent` + `addCommand` を連続呼び出し
- テストデータの一括生成
- CI でプロジェクトの整合性検証

### Phase 4: GUI 改善（EditorAPI 適用）

**目的**: GUI のコンポーネントが EditorAPI を使うように書き換え

| 対象 | 変更 |
|------|------|
| SpatialView | `placeEvent()` → `api.placeEvent()` |
| TableView | `createEntity()` → `api.createEntity()` |
| DetailView | `upsertEntity()` → `api.setProperty()` / `api.renameEntity()` |
| CommandEditor → App.tsx | `handleCommandsChange` → `api.addCommand()` |

コンポーネントが store を直接触らなくなり、ロジックが API に集約される。

## 実装順序と依存関係

```
Phase 1: EditorAPI
    ↓
Phase 2: CLI 拡張          Phase 4: GUI 書き換え
    ↓                          ↓
Phase 3: バッチスクリプト
```

Phase 1 が全ての基盤。Phase 2 と 4 は並行可能。Phase 3 は Phase 1 完了後いつでも。

---

## やらないこと

| 項目 | 理由 |
|------|------|
| ノベルエディタのリファクタ | 動いている。統合判断が先 |
| ヘッドレスエディタ | CLI が担っている |
| リアルタイム同期（GUI ↔ CLI） | Import/Export で十分。複雑性に見合わない |
| プラグインシステム | EditorAPI + script コマンドで十分。形式化は後 |

## 検証方法

| Phase | 検証 |
|-------|------|
| 1 | EditorAPI のユニットテスト。全メソッドの入出力を検証 |
| 2 | 各新コマンドの実行テスト。`rpg-cli validate` を CI に組み込み |
| 3 | サンプルスクリプトの実行 → `rpg-cli analyze` で結果確認 |
| 4 | エディタの全タブで操作 → Play Game タブで動作確認 |
