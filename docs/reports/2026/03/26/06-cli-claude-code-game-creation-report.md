# CLI + Claude Code によるゲーム制作可否の検討

> Generated by Claude Opus 4.6

---

## 1. 背景

kaedevn RPG Studio は GUI（ブラウザエディタ）と CLI（rpg-cli + scripts/cli）の両方でゲームデータを操作する。[04-gui-cli-architecture-reference.md](./04-gui-cli-architecture-reference.md) で Godot Engine の GUI/CLI 統合パターンを調査し、EditorAPI 層の導入が提案された。

本文書では、**Claude Code（AI コーディングエージェント）と現行 CLI を組み合わせた場合にゲーム制作が実用的に可能かどうか** を検討し、ギャップと対応の優先順位を整理する。

## 2. 現行 CLI の全体像

### 2.1. CLI の構成

| CLI | パッケージ | 実行方法 | 役割 |

| **RPG Studio CLI** | `@kaedevn/entity-graph` | `rpg-cli <command>` | Entity CRUD, マップ, イベント, 分析, 実行 |

## 3. Claude Code + CLI でゲームを作るワークフロー

Claude Code は CLI コマンドを直接実行できる。以下の制作フロー全体が CLI のみで完結する。

```
┌─────────────────────────────────────────────────────┐
│                  Claude Code                         │
│                                                     │
│  1. 企画・設計                                        │
│     └→ ユーザーとの対話で決定                           │
│                                                     │
│  2. プロジェクト作成                                    │
│     └→ init-project --config fantasy                 │
│                                                     │
│  3. アセット準備                                       │
│     └→ asset-cli import-official / upload             │
│     └→ kaede-tools convert (動画→WebP)               │
│                                                     │
│  4. キャラクター作成                                    │
│     └→ character-cli create-from-official             │
│                                                     │
│  5. シナリオ作成                                       │
│     ├→ import-story.ts（台本一括変換）                  │
│     └→ editor-cli add/update（ブロック単位編集）         │
│                                                     │
│  6. マップ制作（RPG）                                   │
│     └→ rpg-cli create-map / fill-rect / place-event  │
│                                                     │
│  7. コンパイル・検証                                    │
│     └→ rpg-cli compile / analyze / balance            │
│     └→ kaedevn-compile（KSC コンパイル）               │
│                                                     │
│  8. プレビュー確認                                      │
│     └→ ブラウザで localhost:5175                       │
│                                                     │
│  9. 修正ループ → 5〜8 を繰り返し                        │
└─────────────────────────────────────────────────────┘
```

**結論: 現行 CLI だけでゲーム制作の全工程をカバーできる。**

## 4. 前回文書（04）が指摘した3つのギャップの再評価

### 4.1. EditorAPI 層 → 今は不要

| 観点 | 評価 |
|------|------|
| 前回の指摘 | GUI と CLI で操作コードが重複しており、EditorAPI で統一すべき |
| Claude Code での影響 | **なし。** CLI が Entity Graph Store を直接操作する現状で、ゲーム制作に支障はない |
| いつ必要か | GUI との操作コード共有が保守コストになったとき。チーム開発・長期保守の問題 |

EditorAPI は「GUI とのコード共有」が目的であり、CLI 単体の能力は変わらない。

### 4.2. `script` コマンド（バッチ実行） → Claude Code が代替

| 観点 | 評価 |
|------|------|
| 前回の指摘 | Godot の `@tool` に相当するバッチスクリプト実行機能がない |
| Claude Code での影響 | **低。** Claude Code 自身が「全敵の HP を1.5倍にする」のような操作を CLI コマンドの連続呼び出しで実行できる |
| いつ必要か | Claude Code を使わない人間ユーザーがバッチ操作したいとき |

```
# Claude Code が実行するイメージ（script コマンド不要）
rpg-cli list enemy          # 敵一覧取得
rpg-cli set ent-abc hp 150  # HP を個別に更新
rpg-cli set ent-def hp 225  # （繰り返し）
```

### 4.3. Composable CLI（パイプライン） → 優先度低

| 観点 | 評価 |
|------|------|
| 前回の指摘 | `compile | run` のようなパイプライン連携ができない |
| Claude Code での影響 | **なし。** Claude Code が順番にコマンドを実行すれば同じ結果を得られる |
| いつ必要か | CI/CD パイプラインで CLI を自動化するとき |

## 5. 前回文書が指摘していない実用上のボトルネック

ゲーム制作を実際に行う場合、以下が真のボトルネックになる。

### 5.1. `--json` 出力がない（最重要）

**現状:** CLI 出力は人間向けのテキスト形式。Claude Code はテキストをパースして結果を判定する必要がある。

**問題:**
- エラーと成功の判別が不安定（出力テキストのパターンに依存）
- 作成されたエンティティの ID を取得するのにテキスト解析が必要
- 一覧取得の結果を構造的に処理できない

**対策:** 全 CLI コマンドに `--json` フラグを追加する。

```bash
# Before（人間向け）
$ rpg-cli create enemy スライム
Created entity: ent-abc123 (enemy) "スライム"

# After（JSON モード）
$ rpg-cli create enemy スライム --json
{"id":"ent-abc123","type":"enemy","name":"スライム","status":"created"}
```

**工数:** 各コマンドの出力部分を `if (json) console.log(JSON.stringify(...))` で分岐するだけ。1日で全コマンド対応可能。

### 5.2. プレビュー確認がブラウザ依存

**現状:** ブロックを追加しても、見た目の確認は `localhost:5175` のブラウザのみ。

**問題:** Claude Code 単体ではビジュアルの確認ができない。ユーザーがブラウザを開いて目視確認する必要がある。

**対策（段階的）:**
1. **Playwright MCP を活用** — スクリーンショット取得で Claude Code がビジュアル確認可能（既に MCP 接続済み）
2. **`rpg-cli preview`** — ヘッドレスブラウザでスクリーンショットを撮るコマンド（将来）

### 5.3. ノベル側と RPG 側の CLI が別系統

| | ノベル CLI | RPG Studio CLI |
|---|---|---|
| データアクセス | Hono API（HTTP）経由 | ファイル直接読み書き |
| 認証 | 必要 | 不要 |
| データ保存先 | PostgreSQL | `.rpg-studio/default.json` |

**問題:** 1つのゲームでノベルパートと RPG パートの両方を使う場合、2つの CLI を使い分ける必要がある。

**対策:** 統合は EditorAPI 層の導入タイミングで検討する。現時点では Claude Code が使い分ければ十分。

## 6. 推奨アクション（優先順）

| # | 内容 | 理由 | 工数 |
|---|------|------|------|
| **1** | **`--json` フラグを全 CLI コマンドに追加** | Claude Code が結果を正確にパースできる。最小工数で最大効果 | 小 |
| **2** | **Playwright MCP でプレビュー確認のスキル化** | Claude Code がビジュアル確認まで完結できる | 小 |
| **3** | **EditorAPI 層の導入** | GUI との統合が必要になったタイミングで | 中 |
| **4** | **`script` コマンドの追加** | Claude Code 以外のユーザーがバッチ操作したいとき | 中 |
| **5** | **Composable CLI（パイプライン対応）** | CI/CD 連携が必要になったとき | 小 |

## 7. まとめ

| 問い | 回答 |
|------|------|
| 今の CLI + Claude Code でゲームは作れるか？ | **作れる。** 全工程をカバーする CLI コマンドが揃っている |
| EditorAPI は今必要か？ | **不要。** GUI/CLI コード共有の問題であり、CLI の能力には影響しない |
| 何を先にやるべきか？ | **`--json` フラグ。** Claude Code との連携精度が大幅に向上する |
| 前回文書（04）の提案は間違いか？ | **間違いではない。** ただし前提が「チーム開発・長期保守」であり、「Claude Code + CLI でゲームを作る」という文脈では優先度が変わる |

**現行 CLI は Claude Code と組み合わせる前提で十分な機能を持っている。`--json` 出力の追加が、最小コストで最大の改善をもたらす唯一の優先事項である。**
