**バックエンド（entity-graph）は堅牢で製品レベル**。フロントエンド（rpg-studio）は各機能が**バラバラに動くデモ**の状態で、エディタのタブ間とゲーム実行が繋がっていない。

## 1. 作れるか？

### できること

| 機能 | 状態 |
|------|------|
| Entity のプロパティ編集（HP, ATK 等の数値変更） | ✅ 動く |
| マップにタイルを描く | ✅ 動く |
| マップにイベントを配置する | ✅ 動く（唯一の Entity 作成手段） |
| コマンドツリーを組む（テキスト表示、条件分岐等） | ✅ 動く |
| UI レイアウトを配置する | ✅ 動く |

### できないこと（致命的）

| 不足 | 影響 |
|------|------|
| **Entity の新規作成ができない** | Actor, Skill, Enemy, Item, Variable を追加するボタンがない |
| **Entity の削除ができない** | 不要なデータを消せない |
| **Relation の管理ができない** | 「Hero が Fire を覚える」を UI から設定できない |
| **コマンドエディタがイベントと未接続** | コマンドを組んでも、マップ上のイベントに紐付かない |
| **タイルデータが保存されない** | マップを描いてもタブを切り替えると消える |
| **UI レイアウトが保存されない** | 同上 |
| **マップが1枚固定** | 複数マップの作成・切替ができない |
| **Entity 名の編集ができない** | テーブルの Name 列は表示のみ |

**つまり**: 既存のサンプルデータの数値をいじることはできるが、**自分のゲームを一から作る手段がない**。

## 2. 遊べるか？

### 動くもの

| 機能 | 状態 |
|------|------|
| マップ上を歩く（矢印キー、壁衝突判定） | ✅ |
| NPC に話しかけてメッセージ表示 | ✅ |
| 変数セット + アイテム取得 | ✅ |
| マップ遷移（町 → ダンジョン → 町） | ✅ |
| ターン制バトル（Attack, 敵 AI, ダメージ計算, 勝利/敗北） | ✅ |
| 勝利後に EXP/Gold/ドロップ獲得 | ✅ |
| バトル後マップに復帰 | ✅ |

### 動かないもの

| 不足 | 影響 |
|------|------|
| **バトルの Item コマンドが未実装** | メニューに表示されるが選んでも何もしない |
| **Skill 選択が常に1番目** | スキル選択 UI がない |
| **HP 回復手段がない** | 宿屋やポーション使用が未実装 |
| **EXP がレベルアップに反映されない** | 数値は計算されるが意味がない |
| **Game Over 後の復帰がない** | 全滅すると詰む |
| **イベントが何度でも発火する** | 宝箱を無限に開ける、NPC が同じ話を繰り返す |
| **ゲーム中のセーブ/ロードがない** | ランタイム状態は揮発性 |

### 最大の問題: エディタとゲームが断絶している

PlayableGameView の `handleStart()` で**マップとイベントを JavaScript のコードとして直接構築**している（約130行のハードコード）。エディタのマップタブで描いたタイルデータは**完全に無視**される。

つまり**「エディタで作って、ゲームで遊ぶ」という体験が成立していない**。

## 3. 何が良いか

### バックエンド（entity-graph）は製品レベル

| モジュール | 評価 |
|-----------|------|
| Entity + Relation データモデル | 設計が正しい。双方向クエリ、空間クエリ、影響分析が動く |
| Schema バリデーション | 型・範囲・enum・必須チェックが完備 |
| RelationExtractor | コマンドツリーから reads/writes/transitions_to 等を自動抽出 |
| StaticAnalyzer | 未使用変数、未書込変数、循環呼出を検出 |
| BalanceAnalyzer | ダメージ計算式を実際にパースしてワンショット/0ダメージを検出 |
| Compiler | Entity Graph → Instruction Stream の変換が正しい |
| Runtime | Promise ベースの WAIT_CLICK/CHOICE が正しく動く |
| テスト | 49テスト全通過、カバレッジが良い |

**この entity-graph パッケージは、フロントエンドを作り直しても再利用できる**。

### アーキテクチャの判断が正しい

- 「Schema からUIを自動生成する」という設計 → PropertyEditor, TableView が Schema 変更だけで対応可能
- 「設計時データと実行時データを分離する」→ Compiler が間に入ることでエディタとランタイムが独立
- 「変数を Entity として統一管理する」→ StaticAnalyzer が自然に動く

## 5. 総合評価

| 観点 | 評価 |
|------|------|
| アーキテクチャ | 優秀。Entity Graph + Schema 駆動 + Compiler/Runtime 分離は正しい判断 |
| バックエンド（entity-graph） | 製品レベル。49テスト、再利用可能 |
| エディタ UI | **デモ品質**。各タブは個別に動くが、互いに繋がっていない |
| ゲームプレイ | **デモ品質**。ハードコードされたサンプルゲームが動くが、ユーザーが作ったゲームは動かない |
| 「作って遊べる」まで | Phase 1-3（上記11項目）の完了が必要 |
