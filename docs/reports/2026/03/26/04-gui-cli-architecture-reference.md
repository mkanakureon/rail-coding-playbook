# GUI / CLI 統合アーキテクチャ — Godot に学ぶ kaedevn の設計指針

> Generated by Claude Opus 4.6

---

## 1. なぜこの文書が必要か

kaedevn RPG Studio は GUI（ブラウザエディタ）と CLI（rpg-cli）の両方でゲームデータを操作する。現状、両者は同じ Entity Graph JSON を読み書きするが、操作の抽象度にばらつきがある。Godot Engine の GUI/CLI 統合パターンを調査し、kaedevn が取り入れるべき設計指針を定義する。

---

## 2. Godot Engine の GUI/CLI 統合パターン

### 2.1. テキストファーストの共有データ形式

Godot のプロジェクトデータは `.tscn`（シーン）/ `.tres`（リソース）というテキスト形式で保存される。GUI エディタも CLI も、外部ツールも、同じテキストファイルを読み書きする。

```
.tscn ファイル = 唯一の真実
  ↑ 読み書き          ↑ 読み書き
GUI エディタ          CLI / 外部ツール / AI
```

**データベースは存在しない。** ファイルがそのまま API。

### 2.2. 共有コア + 分離されたアプリ層

Godot のコードベースは5層に分かれる：

| 層 | 内容 | GUI/CLI 共有 |
|---|------|:---:|
| Platform | OS 抽象化 | ✅ |
| Core | オブジェクトモデル、型、I/O | ✅ |
| Servers | レンダリング、物理、オーディオ | ✅ |
| Scene | ノード階層、UI コントロール | ✅ |
| Application | エディタ UI **or** ゲームループ | ❌（ここだけ違う） |

層1〜4はエディタとランタイムで完全に共有。層5だけが `EditorNode`（エディタ）か `SceneTree`（ゲーム）かで分かれる。

### 2.3. EditorInterface — 安定した公開 API

Godot には `EditorInterface` というシングルトンがあり、エディタの全操作の公開 API を提供する。

- GUI の各パネル（インスペクタ、シーンドック等）はこの API を通してデータを操作
- プラグイン（`EditorPlugin`）も同じ API を使う
- ヘッドレスモード（`--headless --editor`）でもこの API が使える

**GUI コンポーネントが内部実装を直接触らない。** 全て `EditorInterface` を経由する。

### 2.4. `@tool` スクリプト — エディタ内マクロ

GDScript に `@tool` を付けるとエディタプロセス内で実行される。`EditorInterface` のフル API にアクセスでき、バッチ操作が可能。

```gdscript
@tool
extends EditorScript
func _run():
    for light in get_scene().find_children("*", "OmniLight3D"):
        light.omni_range *= 2.0
```

「全シーンの照明を2倍にする」のような操作が、GUI を開くことなくスクリプト1つで完結する。

### 2.5. Composable CLI フラグ

Godot の CLI はフラグを組み合わせて使う：

```bash
godot --headless --export-release "Linux/X11" /var/builds/project
godot --headless --script res://tools/batch_import.gd
godot --editor --headless res://SomeScene.tscn
```

`--headless` が他のフラグと組み合わさることで、CI/CD パイプラインやバッチ処理に対応する。

## 3. kaedevn の現状との対比

### 3.1. 既にできていること

| Godot パターン | kaedevn の実装 |
|---------------|---------------|
| テキストファースト | ✅ Entity Graph JSON。GUI（LocalStorage）と CLI（.rpg-studio/default.json）が同じ形式 |
| 共有コア + 別アプリ層 | ✅ `packages/entity-graph` = 共有コア。`apps/rpg-studio` = GUI、`src/cli/` = CLI |
| CLI でプロジェクト操作 | ✅ rpg-cli 24 コマンド（CRUD, マップ, イベント, 分析, 実行） |
| Git フレンドリー | ✅ JSON なので差分が読める |

### 3.2. 足りていないこと

| Godot パターン | kaedevn の現状 | ギャップ |
|---------------|--------------|---------|
| **EditorInterface** | GUI は `store.getState().upsertEntity()` を直接呼ぶ。CLI も同じ。共通の操作関数がない | 同じ操作が GUI と CLI で別々のコードになっている |
| **`@tool` スクリプト** | なし | バッチ操作やAI連携のスクリプト実行機能がない |
| **Composable CLI** | コマンドは独立。パイプライン連携なし | `compile | run` のような組み合わせができない |

## 4. 設計指針：EditorAPI 層の導入

### 4.1. 問題

現状、「マップにイベントを配置する」操作は：

**GUI（SpatialView.tsx）:**
```typescript
const newEvent: Entity = {
  id: `evt-${Date.now().toString(36)}`,
  type: "map_event",
  name: `Event (${x},${y})`,
  properties: { trigger: "interact", commands: [] },
  spatial: { mapId, x, y },
  createdAt: Date.now(),
  updatedAt: Date.now(),
};
state.upsertEntity(newEvent);
```

**CLI（commands/event.ts）:**
```typescript
const entity = createEntity("map_event", args[3], s.schema, {
  properties: { trigger, commands: [] },
  spatial: { mapId: map.id, x: parseInt(args[1]), y: parseInt(args[2]) },
});
s.upsertEntity(entity);
```

**同じ操作が2箇所に書かれている。** 片方を変えるともう片方との整合性が壊れる。

### 4.2. 解決：EditorAPI

```
packages/entity-graph/src/api/
├── editor-api.ts       ← 全操作の公開 API（Godot の EditorInterface に相当）
```

```typescript
// editor-api.ts
export class EditorAPI {
  constructor(private store: EntityStoreState) {}

  // ── Entity 操作 ────────────────
  createEntity(type: string, name: string): Entity
  deleteEntity(id: string): void
  setProperty(id: string, key: string, value: unknown): void
  renameEntity(id: string, name: string): void

  // ── Relation 操作 ──────────────
  addRelation(sourceId: string, targetId: string, type: string): Relation
  removeRelation(id: string): void

  // ── マップ操作 ─────────────────
  createMap(name: string, width: number, height: number): Entity
  setTile(mapId: string, x: number, y: number, tileId: number): void
  fillRect(mapId: string, x: number, y: number, w: number, h: number, tileId: number): void
  setStartPosition(mapId: string, x: number, y: number): void

  // ── イベント操作 ────────────────
  placeEvent(mapId: string, x: number, y: number, name: string, trigger?: string): Entity
  addCommand(eventId: string, type: string, params: Record<string, unknown>): void
  clearCommands(eventId: string): void

  // ── 分析 ────────────────────────
  analyze(): AnalysisResult[]
  simulateBalance(): { results: DamageResult[]; warnings: BalanceWarning[] }
  compile(): CompilationResult
}
```

**GUI も CLI もこの API を呼ぶ:**

```
GUI (React)  ──→ EditorAPI ──→ EntityStore
CLI (Node.js) ──→ EditorAPI ──→ EntityStore
AI スクリプト ──→ EditorAPI ──→ EntityStore
```

### 4.3. 導入の効果

| 観点 | Before | After |
|------|--------|-------|
| 操作の一貫性 | GUI と CLI で別実装 | 1箇所の定義 |
| バグの表面積 | GUI を変えると CLI が壊れる可能性 | API を変えれば両方変わる |
| テスト | GUI はブラウザテスト、CLI は別テスト | API のユニットテストで両方カバー |
| AI 連携 | Claude Code が store を直接触る | EditorAPI を呼ぶだけ |
| 新機能追加 | GUI と CLI の両方に実装 | API に追加 → 両方で使える |

## 5. 設計指針：バッチスクリプト実行

### 5.1. Godot の @tool に相当する機能

ユーザーや AI が書く小さなスクリプトで、EditorAPI を通じてプロジェクトを操作する。

```typescript
// scripts/batch/add-fade-to-all-events.ts
import { EditorAPI } from "@kaedevn/entity-graph";

export default function(api: EditorAPI) {
  const events = api.findByType("map_event");
  for (const event of events) {
    api.addCommand(event.id, "effect", { type: "fade_in", duration: 500 });
  }
}
```

CLI から実行：
```bash
rpg-cli script scripts/batch/add-fade-to-all-events.ts
```

### 5.2. ユースケース

| スクリプト | 内容 |
|-----------|------|
| 全敵のHPを1.5倍にする | `api.findByType("enemy")` → `api.setProperty(id, "hp", hp * 1.5)` |
| 全マップの外周を壁で囲む | `api.findByType("game_map")` → `api.fillRect(id, 0, 0, w, 1, 4)` ... |
| AI がシナリオを生成 | LLM → `api.placeEvent()` + `api.addCommand()` の連続呼び出し |
| テストデータ一括生成 | ループで `api.createEntity()` → `api.addRelation()` |

## 6. ノベル側 CLI との比較

| | ノベル CLI | RPG Studio CLI |
|---|---|---|
| 共通層 | Hono API（HTTP） | Entity Graph Store（直接） |
| EditorAPI 相当 | `lib/api-client.ts`（API ラッパー） | 未実装 → 今後作るべき |
| バッチスクリプト | `ai/assist-cli.ts`（AI 生成） | 未実装 |
| データ形式 | JSON（API レスポンス） | JSON（ファイル） |

ノベル側は API サーバー経由で CLI が動く。RPG Studio は Store 直接。どちらも JSON が共通形式だが、抽象化の場所が違う。

**将来的にノベルと RPG を統合するなら、EditorAPI 層がその接合点になる。**

## 7. やること・やらないこと

### やること（優先順）

| # | 内容 | 理由 |
|---|------|------|
| 1 | **EditorAPI 層を作る** | GUI/CLI の操作コードを1箇所にまとめる。テスト容易性が上がる |
| 2 | **CLI の `script` コマンドを追加** | バッチ操作と AI 連携の入口 |
| 3 | **CLI 出力の JSON モード** (`--json` フラグ) | パイプラインでの利用に対応 |

### やらないこと（今は）

| 内容 | 理由 |
|------|------|
| ノベルエディタのリファクタ | 動いているものを壊すリスク。統合判断が先 |
| ヘッドレスエディタ | CLI が既にヘッドレス操作を担っている |
| `#ifdef` 的コンパイル時分離 | Web アプリなのでツリーシェイキングで十分 |
