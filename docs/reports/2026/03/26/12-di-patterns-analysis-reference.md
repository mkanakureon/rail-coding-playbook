# kaedevn-monorepo DI パターン分析

> Generated by Claude Opus 4.6

## 背景

本リポジトリで使われている依存性注入（DI）パターンを整理する。外部 DI コンテナ（tsyringe, inversify 等）は使用しておらず、インターフェース + ファクトリ + コンストラクタ注入 を組み合わせた自前のパターンで構成されている。

## DI パターン一覧

| # | パターン | 使用箇所 | 概要 |
|---|---------|---------|------|
| 1 | インターフェース抽象化 | core, web, hono | 契約を定義し、実装を差し替え可能に |
| 2 | ファクトリ関数 | hono | 環境変数に応じて実装を選択 |
| 3 | コンストラクタ注入 | web | 依存をコンストラクタ引数で渡す |
| 4 | Hono コンテキスト注入 | hono | `c.set()` / `c.get()` でリクエストスコープの値を伝搬 |
| 5 | モジュールシングルトン | hono | `export const` で唯一のインスタンスを共有 |
| 6 | アダプタパターン | web | 同期 VM と非同期ハンドラの橋渡し |
| 7 | テスト時モック差し替え | web, hono | `vi.mock()` で依存をテスト用に置換 |

## 8. テストにおける DI パターン

このリポジトリのテストでは外部 DI コンテナを使わず、Vitest の `vi.mock()` / `vi.hoisted()` とコンストラクタ注入を組み合わせて依存を差し替えている。テスト対象が「何に依存しているか」を意識し、その依存だけをモックに置き換えることで、テスト対象のロジックだけを検証できるようにしている。

### 8-1. vi.mock() によるモジュール差し替え

**何をしているか:** テスト対象のコードが `import` している外部モジュールを、テスト用の偽物に丸ごと差し替える。テスト対象のコード自体は一切変更せずに、その裏側で動くライブラリだけを入れ替えるイメージ。

**なぜ必要か:** たとえば `WebOpHandler` は PixiJS（Canvas 描画ライブラリ）に依存しているが、テスト環境にはブラウザも Canvas もない。実物の PixiJS を使うとテストが実行できないので、「Container を new したら MockContainer が返る」ように差し替える。同様に、Prisma（DB アクセス）や `fs`（ファイル操作）も実際の DB やディスクに触らせたくないので偽物にする。

**どこで使っているか:**

| 差し替え対象 | テストファイル | 理由 |
|------------|-------------|------|
| PixiJS（Container, Sprite, Text 等） | `packages/web/test/WebOpHandler.bgch.test.ts` | ブラウザ/Canvas なしで描画ロジックをテスト |
| fadeTo, lerpPosition, shake 等 | 同上 | アニメーション関数を即座に完了させて待ち時間をなくす |
| Prisma クライアント | `apps/hono/test/assist-api.test.ts` | 実 DB に接続せずに API ロジックをテスト |
| AI Gateway（createClient） | 同上 | 外部 AI API を呼ばずにレスポンスを制御 |
| fs/promises, child_process | `apps/hono/test/export.test.ts` | ファイル書き込みやプロセス実行を防止 |
| rate-limit ミドルウェア | `apps/hono/test/assist-api.test.ts` | テスト中にレート制限に引っかからないようにする |

```typescript
// 例: PixiJS を丸ごと偽物に差し替え
vi.mock("pixi.js", () => ({
  Container: MockContainer,   // new Container() → new MockContainer()
  Sprite: MockSprite,
  Text: MockText,
  Assets: { load: vi.fn().mockResolvedValue(mockTexture) },
}));

// 例: アニメーション関数を「即座に完了する」偽物に
vi.mock("../src/renderer/fadeTo.js", () => ({
  fadeTo: vi.fn().mockImplementation((target, toAlpha) => {
    target.alpha = toAlpha;       // 値だけ即座に反映
    return Promise.resolve();     // アニメーション待ちなし
  }),
}));
```

```typescript
// 例: DB アクセスを偽物に（実際の PostgreSQL に接続しない）
vi.mock('../src/lib/db.js', () => ({ prisma: mockPrisma }));

// 例: レート制限を無効化（テスト中に 429 が返ると困る）
vi.mock('../src/middleware/rate-limit.js', () => {
  const passthrough = async (_c: any, next: any) => next();
  return { apiLimiter: passthrough, loginLimiter: passthrough };
});
```

### 8-2. vi.hoisted() でモックを先に定義

**何をしているか:** `vi.mock()` の中で使いたいモック変数を、先に宣言しておく仕組み。

**なぜ必要か:** Vitest は `vi.mock()` をファイルの一番上に「巻き上げ（hoist）」して実行する。つまり `const mockPrisma = {...}` を `vi.mock()` の前に書いても、実行順序は `vi.mock()` が先になる。そのため、通常の変数宣言では `vi.mock()` の中から参照できない。`vi.hoisted()` を使うと「巻き上げても大丈夫な変数」として宣言でき、`vi.mock()` 内で参照可能になる。

**具体的な流れ:**

```
コードの記述順:                     実際の実行順:
1. vi.hoisted(mockPrisma)        1. vi.hoisted(mockPrisma) ← 先に実行
2. vi.mock(db → mockPrisma)      2. vi.mock(db → mockPrisma) ← mockPrisma を参照可能
3. import { prisma }             3. import { prisma } ← mock が適用済み
4. describe(...)                 4. describe(...)
```

```typescript
// apps/hono/test/assist-api.test.ts
// ① hoisted で先に宣言 → vi.mock() より先に実行される
const mockPrisma = vi.hoisted(() => ({
  user: { findUnique: vi.fn() },
  project: { findUnique: vi.fn() },
  asset: { findMany: vi.fn() },
}));

// ② vi.mock() 内で ① を参照できる
vi.mock('../src/lib/db.js', () => ({ prisma: mockPrisma }));
```

```typescript
// apps/hono/test/rag-hybrid.test.ts
// モッククラス全体を hoisted で定義
const mockSearch = vi.hoisted(() => vi.fn().mockResolvedValue([...]));
vi.mock('../src/lib/assist/hybrid-rag.js', () => {
  const MockHybridRAGService = vi.fn(function (this: any) {
    this.search = mockSearch;   // ← hoisted 変数を参照
  });
  return { HybridRAGService: MockHybridRAGService };
});
```

### 8-3. コンストラクタ注入でモックハンドラを渡す

**何をしているか:** テスト対象クラスのコンストラクタや引数に、本物の代わりにモック（偽物）を渡す。これが最も「正統派の DI」に近いパターン。

**なぜ必要か:** `OpRunner` はゲームの命令列を順番に実行するクラスで、各命令の実行を `IOpHandler` に委譲する。テストでは「OpRunner が正しい順序で正しい引数を handler に渡しているか」を検証したいだけなので、handler は呼び出しを記録するだけの偽物でよい。本物の `WebOpHandler`（PixiJS に依存）を使うとテスト環境では動かない。

**どこで使っているか:**

| テスト対象 | 注入するモック | 検証内容 |
|-----------|-------------|---------|
| `OpRunner` | `IOpHandler` のモック | 命令ごとに正しいメソッドが正しい引数で呼ばれるか |
| `KscHostAdapter` | `IOpHandler` のモック + 本物の FlagSystem/InventorySystem | KSC スクリプトの命令が handler に正しく変換されるか |

```typescript
// packages/core/test/OpRunner.test.ts

// IOpHandler の全メソッドをモック関数で埋めた偽ハンドラを作る
function createMockHandler(): IOpHandler {
  return {
    textAppend: vi.fn().mockResolvedValue(undefined),
    waitClick: vi.fn().mockResolvedValue(undefined),
    bgSet: vi.fn().mockResolvedValue(undefined),
    chSet: vi.fn().mockResolvedValue(undefined),
    bgmPlay: vi.fn(),
    choice: vi.fn().mockResolvedValue(0),
    // ... 30+ メソッド全てを vi.fn() で定義
  };
}

beforeEach(() => {
  runner = new OpRunner();
  handler = createMockHandler();  // ← テストごとに新しいモックを注入
});

// テスト: TEXT_APPEND 命令を実行すると handler.textAppend が呼ばれるか？
it("TEXT_APPEND → handler.textAppend が正しい引数で呼ばれる", async () => {
  await runner.start(
    scenario([{ op: "TEXT_APPEND", who: "太郎", text: "こんにちは" }]),
    handler,  // ← ここで偽ハンドラを渡している（DI）
  );
  // handler.textAppend が ("太郎", "こんにちは") で呼ばれたことを検証
  expect(handler.textAppend).toHaveBeenCalledWith("太郎", "こんにちは");
});
```

```typescript
// packages/web/test/KscHostAdapter.test.ts
// KscHostAdapter にはモック handler + 本物の FlagSystem/InventorySystem を混在注入
beforeEach(() => {
  handler = createMockHandler();
  adapter = new KscHostAdapter(handler, new FlagSystem(), new InventorySystem());
  //                           ↑ 偽物      ↑ 本物            ↑ 本物
});
```

### 8-4. モッククラス階層（PixiJS）

**何をしているか:** PixiJS の `Container`、`Sprite`、`Graphics`、`Text` といったクラスを、テスト用の軽量な偽クラスで再現している。偽クラスは本物と同じプロパティ（`position`, `alpha`, `children` など）とメソッド（`addChild`, `removeChild` など）を持つが、Canvas への描画は一切行わない。

**なぜ必要か:** `WebOpHandler` のテストでは「背景を設定したら backgroundLayer に Sprite が追加されるか」「キャラクターを非表示にしたら alpha が 0 になるか」といった状態変更を検証する。描画自体は不要だが、PixiJS の親子関係（`addChild`/`removeChild`）や座標（`position`）は正しく動く必要がある。そのため、状態管理だけ本物と同じ挙動をするモッククラスを作っている。

**クラス階層の対応:**

```
PixiJS 本物                  テスト用モック
Container                  MockContainer（children, position, alpha を管理）
  └─ Sprite                MockSprite extends MockContainer（texture, anchor 追加）
  └─ Graphics              MockGraphics extends MockContainer（rect, fill 等がチェーン可能な空実装）
  └─ Text                  MockText extends MockContainer（style プロパティ追加）
```

```typescript
// packages/web/test/WebOpHandler.bgch.test.ts
const { MockContainer, MockSprite, MockGraphics, MockText } = vi.hoisted(() => {
  class MockContainer {
    children: MockContainer[] = [];
    position = { x: 0, y: 0, set(x: number, y: number) { this.x = x; this.y = y; } };
    alpha = 1;
    visible = true;
    // addChild: 本物と同じく親子関係を管理する
    addChild(child: MockContainer) { child.parent = this; this.children.push(child); return child; }
    removeChild(child: MockContainer) { this.children = this.children.filter(c => c !== child); }
    destroy() {}
  }
  class MockSprite extends MockContainer {
    texture: any;
    anchor = { set() {} };
    constructor(texture?: any) { super(); this.texture = texture ?? { width: 1280, height: 720 }; }
  }
  // MockGraphics: rect().fill() のようなメソッドチェーンが動くよう this を返す
  // MockText: style プロパティを持つだけ
  return { MockContainer, MockSprite, MockGraphics, MockText, mockTexture };
});
```

### 8-5. WebOpHandler の統合テスト用ファクトリ

**何をしているか:** `WebOpHandler` が必要とする 6 つの依存（LayerManager, TextWindow, ChoiceOverlay, InputManager, AudioManager, Ticker）を全てモックで組み立てて、テスト可能な WebOpHandler インスタンスを返すヘルパー関数。

**なぜ必要か:** WebOpHandler は最も依存が多いクラスで、本物を使おうとするとブラウザ環境が必要になる。しかしテストしたいのは「bgSet が呼ばれたら backgroundLayer に Sprite が追加されるか」のようなロジックなので、各依存は最低限の API だけ持つモックで十分。このファクトリがないと、テストごとに 6 つのモックを毎回手書きすることになり、コードが膨大になる。

**ファクトリの構造:**

```
createHandler()
  ├─ createMockLayers()        → 5つの MockContainer で構成された擬似 LayerManager
  ├─ createMockTextWindow()    → show/hide/clear が vi.fn() のオブジェクト
  ├─ createMockChoiceOverlay() → show が vi.fn() のオブジェクト
  ├─ createMockInput()         → on が vi.fn() のオブジェクト
  ├─ createMockAudio()         → play/stop/setVolume が vi.fn() のオブジェクト
  └─ createMockTicker()        → add/remove が vi.fn()、deltaMS = 16
       ↓
  new WebOpHandler(上記全てを注入) → テスト可能なインスタンス
```

```typescript
// packages/web/test/WebOpHandler.bgch.test.ts
function createHandler() {
  const layers = createMockLayers();
  const ticker = createMockTicker();
  const handler = new WebOpHandler(
    layers,
    createMockTextWindow() as any,
    createMockChoiceOverlay() as any,
    createMockInput() as any,
    createMockAudio() as any,
    ticker as any,
  );
  return { handler, layers, ticker };
}

// テスト側: 毎回 createHandler() で新鮮な環境を作る
beforeEach(() => {
  vi.clearAllMocks();
  ({ handler, layers } = createHandler());
});
```

### 8-6. 環境変数による挙動切り替え

**何をしているか:** `process.env` を書き換えて、ファクトリ関数が返す実装を制御する。たとえば `AZURE_COMMUNICATION_CONNECTION_STRING` を削除すると、メール送信がコンソール出力に切り替わる。

**なぜ必要か:** 本番コードのファクトリ（セクション 2 参照）は環境変数を見て実装を選ぶ設計になっている。テストでは環境変数を操作するだけで「Azure ありの挙動」と「Azure なしのフォールバック挙動」の両方をテストできる。DI コンテナなしで実装を切り替えられるのは、このファクトリ設計のおかげ。

**どこで使っているか:**

| テストファイル | 操作する環境変数 | テスト内容 |
|-------------|---------------|-----------|
| `apps/hono/test/setup.ts` | `DATABASE_URL`, `JWT_SECRET`, `NODE_ENV` | 全テスト共通のテスト用接続先・シークレット |
| `apps/hono/test/lib-email.test.ts` | `AZURE_COMMUNICATION_CONNECTION_STRING` を削除 | Azure 未設定時にコンソール出力にフォールバックするか |
| `apps/hono/test/rag-hybrid.test.ts` | `RAG_DATABASE_URL`, `OPENAI_API_KEY` | RAG サービスにテスト用接続情報を渡す |

```typescript
// apps/hono/test/setup.ts — 全テストで共通のテスト環境を設定
process.env.DATABASE_URL = 'postgresql://kaedevn:<YOUR_DB_PASSWORD>@localhost:5433/kaedevn_test';
process.env.JWT_SECRET = 'test-secret-key';
process.env.NODE_ENV = 'test';

// apps/hono/test/lib-email.test.ts — Azure 未設定のケースをテスト
beforeEach(() => {
  delete process.env.AZURE_COMMUNICATION_CONNECTION_STRING;
  // → sendEmail() 内の hasAzure が false になり、コンソール出力パスに入る
});
```

### 8-7. vi.spyOn() で既存関数を監視

**何をしているか:** 既存の関数（`console.log` など）を差し替えずに「呼び出されたかどうか」「何を引数に呼ばれたか」を記録する。差し替えと違い、元の関数は動かしつつ監視だけすることもできるし、`.mockImplementation(() => {})` で出力を黙らせることもできる。

**なぜ必要か:** メール送信のフォールバック（Azure 未設定時）は `console.log` にプレビューを出力する仕様。この挙動をテストするには「console.log が特定の文字列付きで呼ばれたこと」を検証する必要がある。`vi.spyOn()` で console.log を監視すれば、出力内容を後から `expect` で検証できる。

```typescript
// apps/hono/test/lib-email.test.ts
// console.log を監視しつつ、テスト出力に混ざらないよう黙らせる
consoleSpy = vi.spyOn(console, 'log').mockImplementation(() => {});

// テスト: Azure 未設定時に "[Email Preview]" がコンソールに出るか
expect(consoleSpy).toHaveBeenCalledWith(expect.stringContaining('[Email Preview]'));
```

## パターン選定の判断基準

| 状況 | このリポジトリで使うパターン |
|------|--------------------------|
| プラットフォーム差し替え（Web ↔ Switch） | インターフェース + コンストラクタ注入 |
| 環境差し替え（Azure ↔ ローカル） | ファクトリ関数 + 環境変数 |
| リクエストスコープの値伝搬 | Hono コンテキスト `c.set()` / `c.get()` |
| アプリ全体で1つのインスタンス | モジュールシングルトン `export const` |
| 同期/非同期の橋渡し | アダプタパターン |
| テスト時の依存差し替え | `vi.mock()` |

## 教訓

- **外部 DI コンテナは不要**: インターフェース + ファクトリ + コンストラクタ注入で十分。プロジェクト規模に対して過剰な抽象化を避けている
- **ファクトリの判定は環境変数に一元化**: `config.ts` に集約されており、判定ロジックが散らばらない
- **テストのモックは `vi.mock()` で完結**: DI コンテナ経由でなくモジュールレベルで差し替えるため、テストコードがシンプル
- **Switch 移植の準備は整っている**: `IInput`, `IAudio`, `IStorage`, `IOpHandler` の4インターフェースを実装すれば、ゲームロジックは変更不要
