# kaedevn スタンドアロン化の検討 — Godot 型ゲームエンジンへの道

> Generated by Claude Opus 4.6

---

## 1. 問い

> kaedevn を Godot のようなスタンドアロンのゲームエンジンにすることは可能か？

---

## 2. 結論

**可能。ただし「ランタイムは既にスタンドアロン」「エディタはクラウド依存」という非対称がある。**

```
                    今の状態              スタンドアロン化後
                 ┌──────────┐          ┌──────────────────┐
  ランタイム     │ ✅ 独立   │    →     │ ✅ そのまま       │
  (ゲーム実行)   │ サーバー不要│          │                  │
                 └──────────┘          │                  │
                 ┌──────────┐          │  1つのアプリ      │
  エディタ       │ ❌ クラウド │    →     │  (Godot 型)      │
  (ゲーム制作)   │ API+DB必須 │          │                  │
                 └──────────┘          └──────────────────┘
```

核となるエンジン（コンパイラ、レンダラ、セーブ/ロード）はサーバー依存がゼロ。エディタのクラウド依存を剥がせば Godot 型になる。

---

## 3. 現状のアーキテクチャ分析

### 3.1. 依存関係マップ

```
スタンドアロン（サーバー不要）
├─ packages/core          → 型定義のみ。依存ゼロ
├─ packages/compiler      → CLI コンパイラ。Node.js のみ
├─ packages/web           → PixiJS + Web Audio + IndexedDB
├─ packages/native-engine → SDL2 + OpenGL + C++17
├─ packages/entity-graph  → Zustand（メモリ内状態管理）
└─ packages/map           → マップレンダリング（PixiJS）

サーバー必須（現状）
├─ apps/editor            → Hono API に全 CRUD を依存
├─ apps/rpg-studio        → 一部 API 依存
├─ apps/next              → 認証ゲートウェイ
└─ apps/hono              → PostgreSQL + Prisma
     ├─ ユーザー認証
     ├─ プロジェクト CRUD
     ├─ アセット管理（Azure Blob Storage）
     └─ キャラクター管理

クラウド（なくても動く）
├─ Azure Blob Storage     → アセット配信（代替: ローカルファイル）
├─ Azure Communication    → メール認証（代替: 不要）
├─ BigQuery               → 分析ログ（代替: 無効化）
└─ GitHub Actions         → CI/CD（代替: ローカルビルド）
```

### 3.2. ランタイム：既にスタンドアロン

| エンジン | 技術 | サーバー依存 | プラットフォーム |
|---------|------|:---:|------------|
| **Web エンジン** | PixiJS + Web Audio + IndexedDB | なし | ブラウザ全般 |
| **ネイティブエンジン** | SDL2 + OpenGL + C++17 | なし | macOS, iOS, Android |

Web エンジンは3つのモードを持つ：

```typescript
// packages/web/src/ksc-demo.ts
if (params.work)  → API 経由でシナリオ取得（サーバー必要）
if (params.file)  → ローカル JSON ファイル読み込み（スタンドアロン）
else              → 内蔵デモ（スタンドアロン）
```

**ゲームを遊ぶだけならサーバーは不要。** ローカルの JSON ファイルを読み込んで動作する。

### 3.3. エディタ：クラウド依存

エディタ（`apps/editor`）は全操作を Hono API 経由で行う：

| 操作 | API エンドポイント | DB テーブル |
|------|-----------------|-----------|
| プロジェクト一覧 | `GET /api/projects` | `projects` |
| プロジェクト読み込み | `GET /api/projects/:id` | `projects` + `works` |
| プロジェクト保存 | `PUT /api/projects/:id` | `projects` |
| アセットアップロード | `POST /api/assets/:id/upload` | `assets` + Azure Blob |
| キャラクター管理 | `POST /api/projects/:id/character-class` | `character_classes` |
| ログイン | `POST /api/auth/login` | `users` |

**エディタを開くだけで PostgreSQL + Hono API + Next.js 認証が必要。** これが Godot 型との最大の差。

### 3.4. データ形式：ポータブル

| データ | 形式 | 保存先 |
|-------|------|-------|
| シナリオ | JSON（CompiledScenario） | ファイル or DB |
| セーブデータ | JSON（SaveData v1） | IndexedDB or ファイル |
| エンティティグラフ | JSON（Entity + Relation） | `.rpg-studio/default.json` or DB |
| アセット | PNG, WebP, OGG, MP3 | ローカル or Azure Blob |
| アセットパック | `.pak`（独自バイナリ） | ローカルファイル |

データ形式自体はストレージに依存しない。ファイルでも DB でも同じ JSON。

## 4. Godot との構造比較

### 4.1. Godot のアーキテクチャ

```
Godot（1つの実行ファイル）
├─ Platform    → OS 抽象化
├─ Core        → オブジェクトモデル、型、I/O
├─ Servers     → レンダリング、物理、オーディオ
├─ Scene       → ノード階層、UI
└─ Application → エディタ UI or ゲームループ（ここだけ分岐）

データ形式: .tscn / .tres（テキストファイル）
プロジェクト: project.godot（ファイル）
保存先: ファイルシステム
認証: なし
DB: なし
```

### 4.2. kaedevn の対応関係

| Godot の層 | kaedevn の対応 | スタンドアロン可否 |
|-----------|--------------|:---:|
| Platform | `packages/core` (型) + SDL2/PixiJS | ✅ |
| Core | `packages/entity-graph` (エンティティモデル) | ✅ |
| Servers | PixiJS (レンダリング) + Web Audio | ✅ |
| Scene | `packages/compiler` (シナリオ→Op) | ✅ |
| Application (ランタイム) | `packages/web` + `packages/native-engine` | ✅ |
| Application (エディタ) | `apps/editor` | ❌ API/DB 依存 |
| データ形式 | JSON（.tscn に相当） | ✅ ポータブル |
| プロジェクト管理 | PostgreSQL テーブル | ❌ → ファイル化が必要 |

**層1〜4はスタンドアロン。層5のエディタだけがクラウド依存。** Godot と同じ構造にするには、エディタのデータアクセスをファイルベースに切り替えるだけでよい。

## 5. スタンドアロン化のロードマップ

### Phase 1: エディタのオフライン化（最小構成）

**目標:** エディタが PostgreSQL/Hono なしで動く

| タスク | 内容 | 工数 |
|-------|------|------|
| **ファイルベース Storage Adapter** | API 呼び出しをファイル読み書きに差し替え。`apps/editor/src/config/api.ts` のエンドポイントを抽象化 | 1〜2週間 |
| **認証スキップモード** | `VITE_AUTH_MODE=local` を拡張し、ログイン画面をバイパス | 2日 |
| **アセットのローカル参照** | Azure Blob URL → ローカルファイルパス。`/assets/` ディレクトリから直接読み込み | 3日 |
| **プロジェクトファイル形式の定義** | Godot の `project.godot` に相当する `kaedevn.project.json` | 1日 |

**成果物:** ブラウザで `index.html` を開くだけでエディタが動く。サーバー不要。

### Phase 2: デスクトップアプリ化

**目標:** ダウンロード → 起動 → 使える

| タスク | 内容 | 工数 |
|-------|------|------|
| **Tauri でラップ** | Phase 1 のエディタを Tauri（Rust ベースの軽量デスクトップシェル）で包む | 1〜2週間 |
| **ファイルダイアログ統合** | プロジェクトを開く / 保存 / エクスポート | 3日 |
| **ビルトインプレビュー** | エディタ内でゲーム実行（現在は別タブ） | 1週間 |

Electron ではなく **Tauri を推奨**する理由：

| | Electron | Tauri |
|---|---|---|
| バイナリサイズ | 150MB+ | 5〜10MB |
| メモリ使用量 | 200MB+ | 30〜50MB |
| バックエンド | Node.js | Rust |
| Web ビュー | Chromium 同梱 | OS 標準 WebView |
| ゲームエンジンとの親和性 | 低（重い） | 高（軽量） |

### Phase 3: 統合パッケージ

**目標:** Godot のように「エディタ + ランタイム + コンパイラ」が1つ

```
kaedevn-studio(.app / .exe)
├─ エディタ（Tauri + React）
├─ プレビューランタイム（PixiJS、エディタ内埋め込み）
├─ コンパイラ（@kaedevn/compiler）
├─ CLI（rpg-cli、ターミナルから利用可能）
└─ エクスポーター
   ├─ Web ビルド（HTML + JS + アセット）
   ├─ Switch ビルド（ネイティブエンジン + .pak）
   └─ モバイルビルド（Android APK / iOS IPA）
```

| タスク | 内容 | 工数 |
|-------|------|------|
| **エクスポート機能** | プロジェクト → 配布用パッケージ生成 | 2〜3週間 |
| **Switch エクスポート** | ネイティブエンジン + アセット `.pak` をバンドル | 4〜6週間 |
| **テンプレートシステム** | 「新規プロジェクト」からジャンル別テンプレートを選択 | 1週間 |

## 6. 技術的に何が活きるか・何を捨てるか

### 活きるもの（そのまま使える）

| コンポーネント | 理由 |
|-------------|------|
| `packages/core` | 型定義のみ。依存ゼロ |
| `packages/compiler` | ファイル入力 → JSON 出力。サーバー不要 |
| `packages/web` | PixiJS ランタイム。ローカルファイルモード実装済み |
| `packages/native-engine` | SDL2 + C++。完全スタンドアロン |
| `packages/entity-graph` | Zustand ベース。ファイル保存アダプタ実装済み |
| `apps/editor` の UI コンポーネント | React コンポーネントは API 呼び出しを差し替えれば再利用可能 |
| KSC スクリプト言語 | コンパイラがスタンドアロンなので完全に独立 |
| セーブデータ形式 | JSON、ストレージ非依存 |

### 捨てるもの（スタンドアロンでは不要）

| コンポーネント | 理由 |
|-------------|------|
| `apps/hono`（API サーバー） | ファイルベースに置き換え |
| `apps/next`（認証ゲートウェイ） | シングルユーザーなので認証不要 |
| PostgreSQL / Prisma | ファイルシステムに置き換え |
| Azure Blob Storage | ローカルファイルに置き換え |
| Azure Communication / BigQuery | 不要 |

**「捨てる」のは SaaS プラットフォーム部分だけ。エンジンのコアは全て残る。**

### 両方残す（SaaS + スタンドアロンの共存）

```
                ┌─────────────────────────┐
                │   packages/*（共通コア）   │
                │   compiler, core, web,   │
                │   entity-graph, map      │
                └──────────┬──────────────┘
                           │
              ┌────────────┴────────────┐
              │                         │
    ┌─────────▼─────────┐   ┌──────────▼──────────┐
    │  SaaS モード       │   │  スタンドアロンモード  │
    │  (apps/hono+next)  │   │  (Tauri + ファイル)   │
    │  マルチユーザー      │   │  シングルユーザー      │
    │  クラウド保存        │   │  ローカル保存          │
    │  Web 配信           │   │  デスクトップアプリ     │
    └───────────────────┘   └─────────────────────┘
```

SaaS を捨てる必要はない。**Storage Adapter パターン**で抽象化すれば両方のモードを共存できる。

## 7. リスクと判断ポイント

| リスク | 影響 | 対策 |
|-------|------|------|
| **エディタの API 呼び出し箇所が多い** | ファイルベース化の工数が膨らむ | RPG Studio エディタ（`apps/rpg-studio`）は API 依存が少ないので、こちらをベースにする選択肢 |
| **Tauri の WebView 差異** | OS によって描画が微妙に異なる | PixiJS は WebView 互換性が高いのでリスク低 |
| **Switch SDK の NDA** | スタンドアロンツールとして配布する場合、Switch エクスポート機能を一般公開できない | Switch エクスポートはプラグインとして分離し、認可ユーザーのみに配布 |
| **SaaS との保守コスト二重化** | 2モードのメンテナンスが必要 | Storage Adapter を1つの抽象にまとめれば、ビジネスロジックは共通 |

## 8. まとめ

| 問い | 回答 |
|------|------|
| スタンドアロン化は可能か？ | **可能。** ランタイムは既にスタンドアロン。エディタのクラウド依存を剥がすだけ |
| どのくらいかかるか？ | Phase 1（オフラインエディタ）: 2〜3週間。Phase 2（デスクトップアプリ）: さらに2〜3週間 |
| 何を捨てるか？ | SaaS インフラ部分（API サーバー、DB、認証）。エンジンのコアは全て残る |
| SaaS と共存できるか？ | **できる。** Storage Adapter パターンで抽象化すれば両モード共存 |
| Godot との最大の差は？ | Godot は C++ で全て1バイナリ。kaedevn は Web 技術ベースなので Tauri でラップする形になる |
| なぜ Tauri か？ | Electron より10倍以上軽量。ゲームエンジンに必要な軽さを確保できる |
| 今すぐやるべきか？ | Claude Code + CLI で制作フローが成立している現状では急務ではない。ユーザー配布・チーム利用のタイミングで着手 |
