## 各原則の詳細

### 1. 自己治癒のための設計 — 優先度: 高

**できていること:**
- `/api/health` ヘルスエンドポイント（status, timestamp, uptime）
- デプロイ後の自動ヘルスチェック（4サービス全て）
- 起動時の環境変数バリデーション
- グローバルエラーハンドラ（構造化JSONログ）
- レートリミット

**ギャップ:**
- Container Apps のライブネス/レディネスプローブが未設定（ヘルスエンドポイントはあるのに使われていない）
- ヘルスエンドポイントが DB 接続を確認していない（DB が落ちても `ok` を返す）
- 外部依存（AI API, Blob Storage）へのサーキットブレーカーがない
- DB 操作のリトライロジックがない

**対応:**
1. Container Apps にライブネス/レディネスプローブを設定（`/api/health` を指定）
2. ヘルスエンドポイントに DB ping (`SELECT 1`) を追加
3. AI API 呼び出しにリトライ（指数バックオフ）を追加

---

### 2. 冗長性 — 優先度: 中

**できていること:**
- Static Web Apps はグローバル CDN で冗長
- Container Apps + Container Registry は Azure マネージド

**ギャップ:**
- Container Apps のレプリカ数がデフォルト（0-1）。コールドスタートでダウンタイムが発生
- PostgreSQL が単一インスタンス（リードレプリカなし）
- Blob Storage の geo-replication が未設定

**対応:**
1. Container Apps の最小レプリカ数を 1 に設定（即時）
2. Blob Storage の GRS（geo 冗長）を有効化（低コスト）
3. DB のゾーン冗長はユーザー増加後に検討

---

### 3. 調整の最小化 — 優先度: 低

**できていること:** ✅
- ステートレス API（Hono）、ステートレス SSR（Next.js）、静的 SPA（Editor, Preview）
- JWT ベースの認証（サーバーサイドセッション不要）
- レートリミットはプロセス内メモリ（外部ストア不要）

**残課題:** スケール時にレートリミットを Redis に移行。現状は問題なし。

---

### 4. スケールアウト設計 — 優先度: 中

**できていること:** ✅
- API がステートレスで水平スケール可能
- SWA は自動スケール
- JWT でセッション共有不要

**ギャップ:**
- アセット配信が API サーバー経由（`/uploads/*`, `/storage/*`）。API がボトルネック
- API の前に CDN がない

**対応:**
- アセットを Azure Blob Storage から直接配信（SAS トークン or 公開 + CDN）。API は URL を返すだけに

---

### 5. パーティション分割 — 優先度: 低

**できていること:** ✅
- `projectId` / `userId` による自然なパーティショニング
- ホットクエリパターンに対応するコンポジットインデックス
- データの関心分離（Novel JSON, RPG JSON, Analytics）

**残課題:** `Project.data` の JSON blob が大きくなった場合、ページ単位でテーブル分割。pgvector へのマイグレーション（TODO 済み）。

---

### 6. 運用設計 — 優先度: 高

**できていること:**
- 構造化 JSON エラーログ
- OpenTelemetry + Jaeger 基盤（コードあり）
- アナリティクスミドルウェア（リクエスト毎のログ）
- BigQuery ログシンク
- デプロイ後の包括的検証（ヘルスチェック、アセット URL、CORS、SPA フォールバック）
- CI: typecheck + lint + ユニットテスト（実 PostgreSQL）

**ギャップ:**
- **OpenTelemetry が本番で無効**（`OTEL_ENABLED !== 'true'`）
- **アラート設定なし**（Azure Monitor アラート、Slack/PagerDuty 連携なし）
- デプロイが手動トリガーのみ（push トリガーがコメントアウト）
- Azure Log Analytics へのルーティングなし

**対応:**
1. OpenTelemetry を本番で有効化 → Azure Monitor Application Insights に送信
2. 基本アラート設定: 「API ヘルスチェック 5 分間失敗」
3. push-to-main 自動デプロイを再有効化
4. コンテナ stdout/stderr を Log Analytics にルーティング

---

### 7. マネージドサービス — 優先度: 低

**できていること:** ✅
- Azure Container Apps（マネージドコンテナ）
- Azure Static Web Apps（マネージド静的ホスティング + CDN）
- Azure Container Registry
- Azure Blob Storage
- Azure PostgreSQL Flexible Server

現状のスケールに対して十分。Redis / API Management はスケール後に検討。

---

### 8. ID サービス — 優先度: 低

**できていること:** ✅
- JWT 認証（パスワードハッシュ、メール認証、ロールベースアクセス）
- CORS オリジン許可リスト
- セキュリティヘッダー

**残課題:** ソーシャルログイン（Google/GitHub OAuth）はセキュリティではなく機能要件として検討。Azure AD B2C は過剰。

---

### 9. 進化の設計 — 優先度: 低

**できていること:** ✅ **（最も強い領域）**
- `IInput`, `IAudio`, `IStorage` 抽象化（Switch 移植対応）
- `save_schema_version` による後方互換性
- 凍結されたスクリプトコマンドセット（core vs web-only の分離）
- `commandRegistry.ts` による単一定義源
- モノレポ内のクリーンなパッケージ境界
- RPG Studio は Novel Editor から完全独立
- `_ai_context` パターン（AI 統合のスキーマ非依存）
- Prisma マイグレーション

**残課題:** API バージョニング（`/api/v1/`）は外部公開時に。

---

### 10. ビジネスニーズに合わせて構築 — 優先度: 中

**できていること:** ✅
- 収益化基盤: Wallet, PointTransaction, WorkPurchase, paywall
- エンゲージメント追跡: PlaySession, WorkScore, PageReaction
- AI 支援制作: WorkSetting, GenerationResult, RagChunk
- 公開フロー: Project → Work（draft → published）

**ギャップ:**
- デプロイが手動（push トリガーがコメントアウト）
- フィーチャーフラグ / A/B テスト基盤なし

**対応:** 自動デプロイの再有効化が即時アクション。

---

### 11. 障害モード分析 — 優先度: 中

**できていること:**
- デプロイパイプラインが複数の障害モードを検証
- レートリミットによる濫用防止
- CI でリグレッション検出

**ギャップ:**
- **障害モード分析が文書化されていない**（DB 停止時、Blob Storage 停止時、AI API レート制限時に何が起きるか）
- グレースフルデグラデーション戦略なし（DB が落ちると API 全体が落ちる）
- DB クエリのタイムアウト設定なし（長時間クエリがハング）
- 非同期操作の失敗時リカバリなし

**対応:**
1. ヘルスエンドポイントに DB 接続チェック追加（タイムアウト付き）
2. Prisma クエリにタイムアウト設定
3. 障害モード一覧表を文書化（依存サービス × 障害パターン × ユーザー影響 × 対応）

## kaedevn の強みと弱み

### 強い領域
- **進化の設計**: Switch 移植を見据えた抽象化、凍結スキーマ、コマンドレジストリ。設計思想が一貫
- **マネージドサービス活用**: Container Apps, SWA, Blob Storage, PostgreSQL。適切な選択
- **調整の最小化**: ステートレス設計、JWT 認証、静的 SPA

### 弱い領域
- **運用設計**: 基盤コードは書かれているが本番で有効化されていない。「計器はあるが電源が入っていない」状態
- **自己治癒**: ヘルスエンドポイントはあるがプローブ未設定。DB 障害を検知できない
- **障害モード分析**: 文書化も対策もない。「何が壊れたら何が起きるか」が定義されていない
