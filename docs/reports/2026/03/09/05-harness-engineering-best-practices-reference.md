## 核心メッセージ

**モデルよりハーネス（環境設計）が重要。** 同じモデルでもハーネス次第で SWE-bench スコアが 22 ポイント変動するが、モデル交換では 1 ポイント程度。

## 7 つの柱

### 1. リポジトリ衛生: 腐敗を前提に設計

- 実行可能なアーティファクト（コード、テスト、型定義）をリポジトリに置く
- 説明的ドキュメント（システム概要、アーキテクチャ解説）は避ける — エージェントは古い情報と新しい情報を区別できない
- Architecture Decision Records（ADR）で決定の履歴を保全
- ガベージコレクションエージェント（定期リファクタリング Codex タスク）を導入し、決定論的ルール（リンター、型チェック）に依拠させる

### 2. 決定論的ツールとアーキテクチャガードレール

- LLM によるスタイルチェックではなくリンター（ESLint、Ruff 等）で強制
- **PostToolUse Hook** でファイル編集後に自動リント → エラーを additionalContext として注入 → エージェントが自己修正
- **PreToolUse Hook** でリンター設定ファイルの編集をブロック

**言語別推奨スタック（2026 年 3 月時点）:**

| 言語 | ツール | 理由 |
|------|--------|------|
| TypeScript | Oxlint（リント）+ Biome（フォーマット） | ESLint 比 50-100x 高速。PostToolUse 向き |
| Python | Ruff | 900 以上のルール。CLI ベースで高速 |
| Go | golangci-lint | 50 以上リンター。35 個自動修正対応 |
| Rust | Clippy（pedantic） | `allow_attributes="deny"` でルール回避防止 |

**カスタムリンターのエラーメッセージ設計:**

```
ERROR: [違反内容]
WHY: [ルール存在理由、ADR リンク]
FIX: [具体的修正手順とコード例]
```

**フィードバック速度の階層:**

最速(ms) ← PostToolUse Hook → プリコミット(s) → CI(min) → 人間レビュー(h) → 最遅

### 3. AGENTS.md / CLAUDE.md をポインタとして設計

**書くべきもの:**
- ルーティング指示（`npm test` で走る、ADR は `/docs/adr/` にある等）
- 禁止事項一覧（各項目が ADR / リンタールール参照付き）
- ビルド・テスト・デプロイのコマンド最小限

**書くべきでないもの:**
- システム現状説明（コードとテストが真実の源）
- 技術スタック解説（エージェントは package.json で読める）
- コーディングスタイルガイド（リンターが強制）

**サイズ目安:** 理想は 50 行以下。150 行でも primacy bias（先頭指示への偏り）が顕著。

### 4. 計画と実行を分離

- 計画フェーズ → 人間レビュー・承認 → 実行フェーズ → テストで検証
- 明示的に「一度に 1 機能」と指示（エージェントは全部やろうとする）
- E2E テストを通すまで「完了」としない

### 5. E2E テスト戦略

**普遍的原則:** アクセシビリティツリー（構造化テキスト）経由で UI と対話。スクリーンショットより決定論的・効率的。

| カテゴリ | 推奨ツール |
|---------|-----------|
| Web | Playwright CLI / agent-browser（Playwright 比 5.7 倍トークン効率） |
| モバイル（iOS） | XcodeBuildMCP |
| モバイル（Android） | mobile-mcp / Appium MCP |
| React Native | Detox + mobile-mcp |
| CLI/TUI | bats-core / pexpect |
| API | Hurl（エージェント親和性最高）/ Pact |
| Electron | circuit-mcp / Playwright MCP フォーク |
| Tauri | tauri-plugin-mcp / tauri-webdriver |
| Terraform | terraform test + Conftest |
| Docker | container-structure-test |
| K8s | kubeconform + Conftest |

**AI/ML パイプライン 6 レイヤー:**

1. データ品質: GX Core、dbt Tests、Soda Core
2. モデル評価: lm-evaluation-harness、LightEval
3. アプリケーション品質: DeepEval、promptfoo、RAGAS
4. エージェント評価: Langfuse、Arize Phoenix
5. 安全性: PyRIT、Guardrails AI、Constitutional Classifiers
6. 可観測性: Evidently AI、WhyLabs

### 6. セッション間の状態管理

- **Git log が最も信頼性高い引き継ぎ手段** — 各セッション終了時に記述的コミットメッセージでコミット
- 進捗記録は Markdown より JSON 推奨（不適切な編集の確率が低い）
- 起動ルーチンの標準化: 作業ディレクトリ確認 → Git ログ読み取り → 次タスク選択 → dev server 起動

### 7. Claude Code vs Codex

| 項目 | Claude Code | Codex |
|------|-----------|--------|
| 実行モデル | 作業場型（開発者環境で直接作業） | 密室型（クラウドサンドボックス） |
| Hooks | PreToolUse / PostToolUse / Stop / PreCompact | notify（ターン完了時のみ） |
| 並列実行 | 不可 | 可（async sandbox queue） |
| MCP 統合 | Tool Search（コンテキスト 85% 削減） | Agents SDK（プログラマティック呼び出し） |

- **品質重視**: Claude Code 主軸（Hooks による決定論的ゲート）
- **スループット重視**: Codex 主軸（サンドボックス並列実行）
- **両立**: Claude Code でハーネス構築 → Codex でスケール実行

## アンチパターン

1. **プロンプトだけに頼る** — 「テストを書いてからコミット」は指示ではなくプリコミットフックで強制
2. **説明文書の蓄積** — 型定義やスキーマ + 構造テストで代替
3. **巨大な CLAUDE.md** — 50 行以下を目指す。1,000 行超はコンテキスト消費が指数関数的に増加
4. **エージェント専用インフラ** — 優れた開発者インフラを作ればエージェントも恩恵を受ける（Stripe）
5. **ハーネスなしでスケール** — 複利的なレバレッジではなく複利的な技術的負債

## 導入ロードマップ

### Week 1
- CLAUDE.md 作成（50 行以下）
- Lefthook でプリコミットフック設定
- PostToolUse Hook で自動フォーマット
- ADR 第 1 号を執筆

### Week 2-4
- エージェント失敗時にテスト / リンタールール追加
- 計画 → 承認 → 実行ワークフロー確立
- E2E テストツール導入（Playwright CLI or agent-browser）
- Stop Hook でテスト通過を完了条件に
- セッション起動ルーチン標準化

### Month 2-3
- カスタムリンター構築（修正指示含むエラーメッセージ）
- ADR ↔ リンタールール紐づけ（archgate パターン）
- ドキュメント → テスト・ADR への段階的移行
- PreToolUse Hook で安全性ゲート（機密ファイル保護等）

### Month 3+
- 高度なフィードバックループ（Plankton パターン）
- ガベージコレクション導入
- 複数エージェント同時実行試行
- ハーネス効果の定量測定（PR/日、手戻り率等）

## PostToolUse Hook 実装例（TypeScript 自動リント）

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'FILE=$(jq -r .tool_input.file_path <<< \"$(cat)\"); case \"$FILE\" in *.ts|*.tsx|*.js|*.jsx) npx biome format --write \"$FILE\" 2>/dev/null; npx oxlint --fix \"$FILE\" 2>&1 | head -20;; esac'"
          }
        ]
      }
    ]
  }
}
```
