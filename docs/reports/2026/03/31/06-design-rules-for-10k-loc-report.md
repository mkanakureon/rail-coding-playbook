### 第2層: 設計違反を「検出可能な形」に変換する仕組み

ここが一番重要で、一番見落とされる部分です。

型チェックやリントが守れるのは**機械的な壊れ方**だけ。設計の腐敗は**意味的な壊れ方**なので、これらでは止まりません。

例えば:
- `localStorage.setItem("token", value)` は TypeScript 的には完全に正しい。でも Switch 移植時に全書き換えが必要になる
- `sprite.position.set(320, 240)` は動く。でも解像度変更時に40箇所修正が必要になる
- `waitForTimeout(2000)` はテストが通る。でもCIが不安定になり、品質ゲート全体が無効化される

**これらは「動くけど腐っている」コード。** 型チェックでは見つからない。

対策は、**設計ルールを「文書」から「壊したら落ちるコード」に変換する**ことです。

```
文書ルール「localStorage 直接禁止」
  ↓ 変換
architecture-check.sh が git diff で新規追加を検出 → push ブロック

文書ルール「app 間の直接 import 禁止」
  ↓ 変換
grep で apps/A → apps/B の import を検出 → push ブロック

文書ルール「waitForTimeout を使わない」
  ↓ 変換
新規 waitForTimeout を git diff で検出 → push ブロック
```

**ルールが文書にしかない状態は、ルールがないのと同じです。** AI はドキュメントを「参考にする」だけで「守る」わけではない。コードレベルで落ちる仕組みにして初めて、AI の出力品質が担保されます。
