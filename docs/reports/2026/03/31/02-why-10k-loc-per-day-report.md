## 2. 「1万行/日」を可能にしている7つの構造的要因

### 要因1: CLAUDE.md による「AIが迷わない設計」

CLAUDE.md は280行超の開発ルールブック。AI に対して以下を事前に定義している:

- **アーキテクチャの全体像**（IInput, IAudio, IStorage の抽象化、解像度、セーブスキーマ）
- **コマンド追加時の8ステップ手順**（どのファイルを、どの順で、何を変更するか）
- **バグ修正の4段階プロトコル**（推測禁止、原因特定必須）
- **テストの書き方ルール**（expect は1つだけ、if でスキップしない）
- **DB 接続情報、ポート番号、管理者ログイン情報**

これにより AI は「何をすべきか」を質問せずに実装に入れる。**指示のオーバーヘッドがほぼゼロ。**

> 一般的なプロジェクトでは、AI への指示出しと確認のやりとりが作業時間の30-50%を占める。CLAUDE.md はこれを構造的に排除している。

### 要因2: 32のカスタムスキルによる定型作業の完全自動化

| カテゴリ | スキル例 | 効果 |
|---------|---------|------|
| Git ワークフロー | commit, push, pr | コミットメッセージ自動生成、Co-Authored-By 自動付与 |
| テスト | test, playwright-e2e-test, browser-verify | 変更ファイルから影響テスト自動選択 |
| コンテンツ | edit-blocks, create-project, character, asset, map | API 経由でブラウザ不要の CRUD |
| ドキュメント | save-report, devlog, rag-search | レポート自動保存、RAG 検索 |
| デプロイ | deploy-azure, test-azure | ワンコマンドデプロイ + ヘルスチェック |
| 配信 | stream, youtube-upload, narrate | OBS 操作、YouTube アップロード |

**定型作業のたびに手順を思い出す必要がない。** スキルが手順を内包しているため、AI は `/commit` と言われれば適切なフォーマットで、`/test` と言われれば影響範囲を自動判定してテストを実行する。

### 要因3: モノレポ × TypeScript × 型安全の相乗効果

```
packages/
  core/        ← 型定義の源泉
  compiler/    ← core の型を参照
  interpreter/ ← core の型を参照
  web/         ← core + compiler の型を参照
apps/
  editor/      ← packages/* を全て参照
  hono/        ← core を参照
  next/        ← core を参照
```

- AI がファイル A を変更すると、TypeScript が即座にファイル B, C の型エラーを報告
- モノレポなので `import` を追うだけで影響範囲が確定
- **AI にとって「型エラー → 修正 → 型チェック通過」のループが最速のフィードバック**
- `npm run typecheck` が pre-push フックに組み込まれ、壊れたコードが push されない

### 要因4: 75+の自動化スクリプトが「手作業」を消滅させた

| スクリプト | 手作業だったもの |

**CLIで完結するため、AI がブラウザを操作する必要がない。** これが速度に直結する。ブラウザ操作は AI の最大のボトルネック（待機時間、要素の特定、状態の把握）だが、API + CLI に寄せることでこれを回避。

### 要因5: テスト基盤が「書いた直後に検証」を可能にしている

- **282のテストファイル**が既に存在 → 新機能追加時にリグレッションを即検出

- **GitHub Actions CI** で typecheck, lint, unit tests, migration check が並列実行

テストを書くコストが低い（テンプレートがある）＋ テストを走らせるコストが低い（自動選択）＝ **「テスト込みで1万行」が可能になる。**

### 要因6: ドキュメント駆動の文脈保持（RAG + 2,755 md ファイル）

- `docs/` に 2,755 の Markdown ファイル → 設計意図、障害履歴、仕様変更がすべて記録

- CLAUDE.md で「作業前の RAG 検索」がルール化されている

AI にとって「なぜこう設計したか」が即座にわかる → **同じ議論を二度しない。** 文脈の再構築コストがゼロに近い。

### 要因7: 人間の判断が「設計・レビュー・方針決定」に集中している

git log を見ると、コミットの中身は以下のパターン:

人間がやっているのは:

**実装・テスト・ドキュメントという「作業量の大部分」を AI が担当し、人間は判断に専念。** これが1人で1万行/日を可能にする本質。

## 4. よくある誤解への回答

### 「LOC は生産性の指標にならない」

正しい。だが、このリポジトリの場合:
- 203,732行のソースコード（.ts + .tsx）は実動するプロダクション品質
- 282のテストファイルがリグレッションを検出
- Azure にデプロイされ稼働中
- pre-push フックで typecheck + lint が強制

**「動くコード」が1万行/日であり、コメントやボイラープレートの水増しではない。**

### 「AI が書いたコードは品質が低い」

一般論としては正しいリスクがある（63%がデバッグ問題を経験）。しかし:
- TypeScript strict モードが型レベルのバグを排除
- 282のテストが振る舞いレベルのバグを検出
- CLAUDE.md のバグ修正プロトコルが「推測修正」を禁止
- pre-push フック + CI が品質ゲートとして機能
- oxlint が PostToolUse フックで自動実行（編集直後にリント）

**品質管理を構造に埋め込むことで、「AI の速度」と「人間の品質」を両立している。**

### 「テスト込みだから水増しでは？」

テストは:
- バグの早期発見（本番障害の防止）
- リファクタリングの安全ネット
- AI が安心してコードを変更できる基盤

**テストがなければ AI は怖くてコードを変更できない。** テスト込みの1万行は、テストなしの3,000行より価値が高い。
