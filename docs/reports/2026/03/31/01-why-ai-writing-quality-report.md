# なぜ AI が書く文書の品質が高いのか — 一次情報・制約・事実の三角構造

> Generated by Claude Opus 4.6

## 背景

kaedevn プロジェクトでは、開発日誌・Qiita 記事・操作説明書・テスト報告書など多種多様な文書を AI（Claude Code）が作成している。これらの文書は「AI が書いたとは思えない」レベルの品質を維持しているが、なぜそうなるのかを分析し、再現可能な知見として残す。

---

## 品質を生む3つの構造的要因

### 1. 一次情報で書いている

最大の要因は、**コードを読み、テストを実行し、バグを発見した本人が文書を書いている**こと。

開発日誌に「DustFilter の alpha が演算子優先順位ミスで 1.0 を超えていた」と書けるのは、GLSL シェーダーの式を解析して `0.9 + 0.3 * sin(...)` の値域が `[0.6, 1.2]` に達することを計算で確認したから。テスト報告書で「107本全パス」と書けるのは、`npx vitest run` の出力を直接見ているから。

一次情報で書くことの利点:

| 観点 | 一次情報 | 伝聞・推測 |
|------|---------|-----------|
| 具体性 | 「`pixelate` が `FamicomFilter` を返していた」 | 「フィルターにバグがあった」 |
| 感情 | 「44.8秒で全パスした時は達成感があった」 | 「テストが通って良かった」 |
| 数字 | 「28ファイル全部読んで6件発見」 | 「いくつかバグが見つかった」 |
| 背景 | 「コピペで case 文を書き換え忘れたのだろう」 | 「原因は不明」 |

**逆に言えば、コードに触れていない話題の文書は品質が落ちる。** 一次情報がないまま書くと一般論に逃げざるを得ない。フィルター関連の文書が特に良いのは、シェーダー28ファイルを全部読み、バグ6件を発見し、テスト107本を書き、ブラウザで21フィルターを目視確認した後に書いているから。

### 2. スキルファイルの制約が文体を安定させる

kaedevn では文書作成に「スキル」と呼ばれる指示テンプレートを使っている。自由に書かせると散漫になるが、**具体的な制約があることで品質が安定する。**

#### 開発日誌スキル（devlog）の制約例

```
- 主語を省略し、誰がやったか明示しない
- 変更行数やファイル数を織り交ぜて具体性を出す
- 技術用語は使うが、理解してもらう必要はない（雰囲気が伝わればいい）
- 感想は3〜5段落、各段落2〜3行（計10行以上）
- 毎回その日固有の内容を反映して書く（使い回さない）
```

#### レポート保存スキル（save-report）の制約例

```
- ヘッダーに「Generated by Claude {モデル名}」を必ず付与
- セクション構成: 背景 → 方針 → 変更ファイル一覧 → 実装の詳細 → 教訓
- コード例は実際の変更から抜粋
- 本文は日本語、コード・識別子は英語
```

#### コミットメッセージスキル（commit）の制約例

```
- プレフィックス: feat / fix / docs / refactor / chore
- 説明は日本語、複数変更は「・」で区切る
- 末尾に Claude の感想（1〜3行、ゆるいトーン）
- Co-Authored-By を必ず付与
```

これらの制約がもたらす効果:

| 制約 | 効果 |
|------|------|
| 主語省略 | 「AI が書いた」感が消え、開発者日記のトーンになる |
| 行数・ファイル数の明記 | 「何をどれくらいやったか」が一目でわかる |
| 感想10行以上 | 毎回異なる内容を書く必要があり、テンプレ化を防ぐ |
| 過去の日誌を読んでから書く | トーンの一貫性が保たれる |

**制約はクリエイティブの敵ではなく、品質の味方。** 「なんでも自由に書いていい」より「この構成で書け」の方が良い文書が生まれる。

### 3. git log という事実のバックボーン

文書の骨格を `git log --stat`、`git log --numstat` の出力から構成している。

```
17 files changed, 365 insertions(+), 60 deletions(-)
```

この数字は事実。ここに嘘や誇張が入る余地がない。事実を骨格にして、その周りに文脈と所感を肉付けする構成になっている。

#### git log から文書への変換プロセス

```
git log の事実           → 文書への変換
────────────────────────────────────────────────────
17 files, +365/-60       → 「17ファイル、+365行の変更」
commit msg: "PC-98..."  → セクション見出しの素材
--stat の diff           → 変更内容の具体的な記述
--numstat の集計         → 「md が +1,312行、ソースが +426行」
```

開発日誌の冒頭にある `> 10コミット / +9,576行 / -281,448行` は `git log --numstat` の集計そのまま。この「嘘のつけない数字」が文書全体の信頼性を担保している。

## 品質が落ちるケースの分析

品質が下がるのは、上記3要因のいずれかが欠ける場合。

| 欠ける要因 | 結果 | 例 |
|-----------|------|------|
| 一次情報がない | 一般論・抽象的な記述になる | 触ったことのないライブラリの説明 |
| 制約がない | 冗長・トーン不統一 | スキルを経由せず自由に書いた文書 |
| 事実データがない | 根拠のない主張 | 数字なしで「大幅に改善」と書く |

### 逆パターン: 品質が最も高くなる条件

```
一次情報: シェーダー28ファイル全読み + バグ6件発見 + テスト107本実行
制約:     devlog スキル + save-report スキル
事実:     git log 22コミット + vitest 110 passed + Playwright 10 passed
```

この3つが揃った時、文書は自然と具体的で、構造的で、信頼できるものになる。

## 再現のための実践ガイド

### 1. 「書く前にやる」を徹底する

文書を書く前に、必ず対象のコードを読む・テストを実行する・ブラウザで確認する。書く行為を先にしない。

### 2. スキルファイルで制約を定義する

新しい種類の文書を書く前に、以下を決める:
- 構成テンプレート（必須セクション）
- トーンと人称のルール
- 数字の入れ方（行数・ファイル数・テスト数）
- 感想・所感の分量目安

### 3. git log を素材にする

文書の骨格を git log から自動生成し、事実ベースで肉付けする。
- `git log --since --until --stat` で変更の全体像
- `git log --numstat` で行数集計
- 個別コミットの diff で詳細な変更内容

### 4. 過去の文書を読んでから書く

トーンの一貫性を保つため、直近の同種文書を必ず読む。devlog スキルの「過去の日誌を参照する」指示はこのため。

---

## 結論

AI が書く文書の品質は、AI の言語能力だけでは決まらない。**一次情報の密度 × 制約の具体性 × 事実データの裏付け** の三角構造で決まる。この3つのうち1つでも欠けると品質は落ちるし、3つが揃えば「AI が書いた」ことが気にならないレベルの文書が生まれる。

これは AI に限った話ではなく、人間の文書作成にも当てはまる。実際にコードを書いたエンジニアが、テンプレートに沿って、git log の数字を使って書くレポートは品質が高い。AI はその「当たり前のプロセス」を高速に実行しているだけ。

品質の秘訣は、能力ではなくプロセスにある。
