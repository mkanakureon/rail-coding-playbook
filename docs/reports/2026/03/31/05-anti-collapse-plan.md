### 1-2. 段階的 Error 化

```
Week 1:  新規追加分 → Error / 既存分 → Warning
Week 2-4: 既存 Warning を週10件ずつ Error 化
Month 2:  全て Error
```

**既存分は一括修正しない。** 週10件のペースで新機能開発と並行して消化する。

## Layer 3: 変更境界の定義

### 3分類

| ゾーン | AI の権限 | 対象 | 変更時の要件 |
|--------|----------|------|-------------|
| **A. 自由変更** | 制限なし | UI の見た目、文言、小コンポーネント、テスト追加、docs | なし |
| **B. 条件付き変更** | 型チェック + テスト通過必須 | store, api client, shared utilities, route handler, command parser | テスト追加/更新必須 |
| **C. 設計境界** | 設計メモ必須。AI 単独変更禁止 | 下記参照 | Design Change Note 必須 |

### C ゾーン（設計境界）の対象

| 対象 | ファイル/ディレクトリ | 理由 |
|------|---------------------|------|
| Storage 抽象 | `packages/core/src/interfaces/IStorage.ts` | Switch 移植の要 |
| Input 抽象 | `packages/core/src/interfaces/IInput.ts` | 全入力の唯一の入口 |
| Audio 抽象 | `packages/core/src/interfaces/IAudio.ts` | 全音声の唯一の入口 |
| Renderer 抽象 | `packages/web/src/renderer/` のインターフェース | 描画パイプライン |
| Interpreter core | `packages/interpreter/src/core/` | スクリプト実行の心臓 |
| Save Schema | `packages/core/src/types/SaveData.ts` | frozen schema（互換性必須） |
| Op 型 | `packages/core/src/types/Op.ts` | 全コマンドの型定義 |
| Auth/Session | `apps/hono/src/routes/auth.ts` のスキーマ部分 | セキュリティ境界 |
| DB Schema | `prisma/schema.prisma` | マイグレーション必須 |
| Public API | `apps/hono/src/routes/` のエンドポイント定義 | 契約変更 |

### Design Change Note テンプレート

C ゾーン変更時、コミット前に記述:

```markdown
## Design Change Note
- **何を変えるか:**
- **なぜ必要か:**
- **依存方向は変わるか:** Yes / No
- **代替案は何か:**
- **既存抽象を再利用できない理由:**
- **破壊的変更か:** Yes / No
```

commit スキルに組み込み、C ゾーンファイルが staged にあれば自動でテンプレートを要求する。

### 5-2. 新規抽象の追加条件

以下の **全て** を満たす場合のみ新規 abstraction / helper / service / hook / util を許可:

1. 同じ処理が **3箇所以上** にある
2. ドメイン概念として **名前がある**
3. 今後も **再利用が見込める**
4. 依存方向を **改善する**
5. テストしやすく **なる**

満たさなければ既存に寄せる。CLAUDE.md に追記。

### 5-3. 作業開始チェックリスト

CLAUDE.md の「Adding New Script Commands」と同様に、設計境界に触れる作業の開始テンプレートを定義:
