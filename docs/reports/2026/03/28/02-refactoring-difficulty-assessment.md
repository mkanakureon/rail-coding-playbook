# GameScreenState リファクタリング 難易度・タイミング評価

## 難易度評価: 中〜高

### 難しい部分

#### 1. ブロッキング→ノンブロッキングは設計変更

単純な「名前変え」や「ファイル移動」ではない。Native 側の `Interpreter.step()` は「呼んだら `waitForClick` まで一気に実行して止まる」前提で書かれている。これをステートマシンに変えると:

- **呼び出し側**（`main.cpp`）の制御フローが変わる
- **呼ばれる側**（`SDL2Engine`）のインターフェースが変わる
- **部分的に変えると動かない** — 両方を同時に書き換える必要がある

```
Before: main → interpreter.step() → engine.waitForClick() [ブロック] → 返ってくる
After:  main → interpreter.step() → engine.requestWaitClick() → main に戻る → click → step() 再開
```

制御の流れが根本的に逆転する。

#### 2. WebOpHandler の密結合

2,100 行の中で、状態管理と描画が同じメソッド内で混在している。

```typescript
// bgSet() の中で状態更新と描画が一体化
async bgSet(id: string, fadeMs?: number) {
  this.currentBgId = id;                          // ← 状態管理
  const sprite = await this.loadSprite(id);        // ← リソース読み込み
  await fadeTo(sprite, 1, fadeMs, this.ticker);    // ← アニメーション
  this.sprites.set("bg", sprite);                  // ← 描画オブジェクト管理
}
```

「ここからここまでが状態管理」と切り出すのに 1 行ずつ判断が必要。機械的な分割ができない。

### 簡単な部分

#### 1. 状態の数が少ない

ゲーム画面の状態は BG、キャラ数体、テキスト、選択肢、BGM 程度。RPG のインベントリやスキルツリーのような複雑な状態木ではない。`GameScreenState` の型定義が 30 行で収まる規模。

#### 2. Op IR が既に共通

スクリプトの命令セット（`Op.ts`）は Web も Native も同じ。「何をするか」は統一済みで、「どう管理するか」だけ変えればいい。ゼロからの設計ではなく、既存の共通基盤の上に乗せる作業。

#### 3. 段階的に進められる

Step 1（Native ブロッキング廃止）は独立して着手可能。既存の Web 側を壊さずに Native だけ先に変更できる。Step 2（GameScreenState 共通化）も新規ファイル追加から始められるため、既存コードを段階的に移行できる。

## タイミング評価: 今気づいてよかった

### 理由 1: Native エンジンがまだ小さい

| ファイル | 現在の行数 | 将来（機能追加後）の予測 |
|---------|-----------|---------------------|
| SDL2Engine.cpp | 800行 | 2,000〜3,000行 |
| Interpreter.cpp | 200行 | 500行 |
| main.cpp | 500行 | 800行 |

今 800 行のブロッキング廃止と、3,000 行になってからのブロッキング廃止では、影響範囲が 3 倍になる。

さらに重要なのは、**idle アニメーション・パーティクル・Live2D 的な機能を足す前に**ステートマシン化しておくと、それらの機能が自然にメインループに乗る点。ブロッキングのまま idle アニメーションを入れると、`waitForClick()` の while ループ内にもアニメーション更新を入れる必要があり、コードが 2 重化する。

### 理由 2: WebOpHandler の肥大化パターンに気づけた

現在 2,100 行の WebOpHandler は、新機能のたびに太る構造になっている:

```
02月: テキスト + BG + キャラ                  → 800行
03月: フィルター + パーティクル + マップ + タイムライン → 2,100行
04月: Live2D + RPGバトル + セーブUI（予想）     → 3,500行?
```

`GameScreenState` に分離する方針が立ったことで、今後の新機能を「状態定義 + update ロジック + 描画」に分けて書ける。肥大化を止められる。

### 理由 3: Switch 移植前でないと遅い

Switch の描画バックエンドを書くとき、WebOpHandler を参考にする。状態管理と描画が混在したまま参考にすると、**同じ密結合が Switch 側にもコピーされる**。

```
今リファクタしてから Switch 移植:
  GameScreenState (共通) → Switch Renderer (新規、描画のみ)

Switch 移植後にリファクタ:
  WebOpHandler (密結合) → コピーして Switch 版作成 → 3つとも密結合
  → 後から 3 つ同時にリファクタ = 作業量 3 倍
```
