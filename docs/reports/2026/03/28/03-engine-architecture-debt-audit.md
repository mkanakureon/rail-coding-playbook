## 概要

Web（PixiJS）と Native（SDL2）の両ゲーム画面を横断的に監査し、「今やるべきこと」と「後でいいこと」を分類した。判断基準は **「後からやると何倍コストがかかるか」**。

## 発見した問題一覧

| # | 問題 | 深刻度 | 後回しコスト | 推奨 |
|---|------|--------|------------|------|
| 1 | Native ブロッキングループ | 高 | 3倍 | **今やる** |
| 2 | スクリプト実行モデルの不一致 | 高 | 5倍 | **今やる** |
| 3 | IInput が Native にない | 高 | 3倍 | **今やる** |

| 7 | Native のテキスト文字送りがない | 低 | 1倍 | 後でいい |
| 8 | RPG/マップ/バトルが Native にない | 低 | 1倍 | 後でいい |

## 問題 1: Native ブロッキングループ（既出）

**現状:** `waitForClick()` / `showChoices()` がスレッドをブロックし、自前で render ループを回す。

**後回しにすると:** idle アニメーション、パーティクル、フィルター時間更新など、メインループで回すべき機能を追加するたびに、`waitForClick()` 内のループにも同じコードをコピーする必要がある。機能が増えるほどコピー箇所が増え、修正コストが線形に増加。

**対策:** ステートマシン化（詳細は 01-game-screen-state-unification-design.md を参照）

## 問題 2: スクリプト実行モデルの不一致（最重要の発見）

### 現状

```
Web:    KSC テキスト → Compiler → Op IR (JSON) → OpRunner → IOpHandler
Native: KSC テキスト → Tokenizer → Parser → Evaluator → IEngineAPI
```

**Web と Native で全く別のスクリプト実行パイプラインを持っている。**

- Web は `packages/compiler` で KSC を **Op IR（中間表現）** にコンパイルし、`OpRunner` が実行する
- Native は独自の `Tokenizer → Parser → Evaluator` で **KSC テキストを直接解釈**する
- Op IR を経由しないため、**コンパイラが追加した最適化・検証が Native では効かない**

### なぜこれが最重要か

1. **同じ KSC スクリプトで異なる動作になりうる**
   - Web のコンパイラが `@if` の評価順序を最適化しても、Native のパーサーは独自ルールで評価する
   - 新コマンドを追加したとき、Web（コンパイラ + OpRunner）と Native（Parser + Evaluator）の **2 箇所に実装が必要**

2. **Op IR への統一を後回しにすると**
   - Native の Parser/Evaluator に機能を足すたびに、Web のコンパイラとの動作差異テストが必要
   - 最終的に Switch 移植時に「Native のパーサーで動くが Web のコンパイラで動かない」問題が出る
   - **修正コストは Op 数に比例して増加**（現在 40+、今後さらに増える）

3. **Native を Op IR ベースに統一すれば**
   - コンパイラは 1 つ（Web の `packages/compiler`）
   - 実行エンジンのインターフェースが `IOpHandler` に統一
   - 新コマンド追加時の実装箇所が半減
   - **「コンパイル済み Op JSON を読んで実行する」だけの C++ ランナーを書けばいい**

### 対策

```
統一後:
  KSC テキスト → Compiler (TypeScript) → Op IR (JSON ファイル)
                                              ↓
                              ┌────────────────┼────────────────┐
                              ↓                ↓                ↓
                         Web OpRunner    Native OpRunner    Switch OpRunner
                              ↓                ↓                ↓
                        WebOpHandler     SDL2Handler      SwitchHandler
```

- Native の Tokenizer/Parser/Evaluator を **廃止**
- 代わりに C++ の OpRunner を新規作成（Op JSON を読んで実行するだけ）
- IEngineAPI を IOpHandler 相当に拡張

**実装量:**
- 廃止: Tokenizer.cpp + Parser.cpp + Evaluator.cpp（約 600 行削除）
- 新規: C++ OpRunner（300〜400 行、JSON パース + switch 文）
- 変更: SDL2Engine を IOpHandler メソッドに合わせてリネーム

## 問題 3: IInput が Native にない

### 現状

```typescript
// packages/core — 共通インターフェース（定義済み）
interface IInput {
  dispatch(action: Action): void;
  on(action: Action, handler: () => void): void;
  off(action: Action, handler: () => void): void;
}

enum Action { OK, Back, Menu, SkipToggle, AutoToggle, Log, QuickSave, QuickLoad, ... }
```

- **Web**: `InputManager implements IInput` — 完全実装、キーマッピング・タッチ・スワイプ対応
- **Native**: `main.cpp` で `SDL_PollEvent` を直接 if 分岐。**IInput 相当の抽象化なし**

### なぜ今やるべきか

1. **Switch のコントローラー対応で必ず必要になる**
   - Switch は Joy-Con（ボタン + スティック + ジャイロ）。SDL_PollEvent の if 分岐では対応不能
   - IInput を通さないと、入力ソースごとに分岐が爆発する

2. **ブロッキング廃止（問題 1）と同時にやると効率的**
   - main.cpp の入力処理を書き換えるなら、IInput 導入も同時にやれる
   - 別々にやると main.cpp を 2 回リファクタすることになる

3. **Action enum は既に core に定義済み**
   - C++ 側に同じ enum を定義して、`InputManager::dispatch(Action)` を実装するだけ

### 対策

```cpp
// packages/native-engine/src/engine/InputManager.hpp

enum class Action {
    OK, Back, Menu, SkipToggle, AutoToggle, Log,
    QuickSave, QuickLoad, Screenshot, HideWindow,
    MapUp, MapDown, MapLeft, MapRight, Interact
};

class InputManager {
public:
    using Handler = std::function<void()>;

    void dispatch(Action action);
    void on(Action action, Handler handler);
    void off(Action action, Handler handler);

    // SDL イベントを Action に変換
    void processEvent(const SDL_Event& event);

private:
    std::unordered_map<Action, std::vector<Handler>> handlers_;

    // キーマッピング（設定変更可能）
    std::unordered_map<SDL_Keycode, Action> keyMap_ = {
        {SDLK_RETURN, Action::OK},
        {SDLK_SPACE, Action::OK},
        {SDLK_ESCAPE, Action::Back},
        // ...
    };
};
```

**main.cpp の変更:**

```cpp
// Before: SDL イベントを直接解釈
while (SDL_PollEvent(&event)) {
    if (event.type == SDL_KEYDOWN && event.key.keysym.sym == SDLK_RETURN) {
        // ... 直接処理
    }
}

// After: InputManager 経由
while (SDL_PollEvent(&event)) {
    inputManager.processEvent(event);  // Action に変換して dispatch
}
```

## 問題 5: IAudio / IStorage シグネチャ不一致

### 現状

**IAudio:**

| メソッド | Web (IAudio) | Native (AudioManager) |
|---------|-------------|----------------------|
| play | `play(category, asset, loop?)` | `playBgm(path, vol, loop, fadeMs)` / `playSe(path, vol)` |
| stop | `stop(category)` | `stopBgm(fadeMs)` |
| setVolume | `setVolume(category, vol)` | なし（play 時のみ指定） |
| getVolume | `getVolume(category)` | なし |
| fadeTo | `fadeTo(category, vol, ms)` | fadeMs パラメータで代用 |

**IStorage:**

| メソッド | Web (IStorage) | Native (FileStorage) |
|---------|---------------|---------------------|
| save | `save(string slotId, SaveData)` → `Promise<void>` | `save(int slotId, json)` → `bool` |
| load | `load(string slotId)` → `Promise<SaveData\|null>` | `load(int slotId)` → `json` |
| delete | `delete(string slotId)` → `Promise<void>` | なし |
| list | `list()` → `Promise<string[]>` | なし |
| exists | なし | `exists(int slotId)` → `bool` |

### なぜ「近いうちに」か

- 現時点で Native のセーブ/ロードは基本動作する。**壊れてはいない**
- ただし Switch 移植時に IStorage を実装する必要があり、そのとき Native のシグネチャと合わないと混乱する
- 問題 1〜4 より優先度は低いが、問題 2（Op IR 統一）のついでに合わせると効率的

### 対策

- Native の AudioManager を IAudio インターフェースに合わせてリネーム
- Native の FileStorage を IStorage インターフェースに合わせてリネーム
- `int slotId` → `string slotId` に変更
- `bool` 返り値 → エラーハンドリング方式を統一（例外 or Result 型）

## 問題 6〜8: 後でいい理由

### 問題 6: フィルター/パーティクルの Op 未接続

Native の GLRenderer には 24 種のフィルターシェーダーと CPU パーティクルシステムが**実装済み**。ただし Op ハンドラーと接続されていないため、スクリプトから呼び出せない。

**後でいい理由:** 問題 2（Op IR 統一）を先にやれば、Op ハンドラーの追加は機械的作業になる。接続コードは 1 メソッド数行で済む。今接続しても、Op IR 統一時に書き直しになる。

### 問題 7: テキスト文字送り

Native はテキストを一括表示している。文字送りアニメーションがない。

**後でいい理由:** 問題 4（GameScreenState）で `text.displayedChars` を共通化すれば、Native の render() で `text.content.substr(0, displayedChars)` を描画するだけ。update() の文字送りロジックは core に書くので、Native 側の実装は数行。

### 問題 8: RPG/マップ/バトル

Native には RPG 機能がまったくない。

**後でいい理由:** Web 側もまだ開発中。仕様が固まっていない段階で Native に移植しても、Web 側の変更に追従するコストが発生する。**Web で仕様が安定してから** Native に移植すればよい。

## 実装ロードマップ

### フェーズ A: 基盤統一（今やる）

問題 1〜4 を一括で対処する。順番が重要。

```
A-1: Native OpRunner 新規作成（問題 2）
  └── Tokenizer/Parser/Evaluator を廃止し、Op JSON を読む C++ ランナーを作る
  └── IEngineAPI を IOpHandler 相当にリネーム・拡張

A-2: Native ブロッキング廃止 + IInput 導入（問題 1 + 3）
  └── waitForClick/showChoices をステートマシン化
  └── InputManager を C++ で実装
  └── main.cpp をシングルループ化

A-3: GameScreenState 共通化（問題 4）
  └── core に型定義 + update() ロジック
  └── Web/Native 両方の Handler を移行
```

**A-1 が最初の理由:** Op IR 統一がないと、A-2 で書き換える Interpreter.step() がすぐ廃止になる。先に OpRunner を作ってから main.cpp を整理する。

### フェーズ B: インターフェース整合（近いうちに）

```
B-1: IAudio シグネチャ統一（問題 5）
B-2: IStorage シグネチャ統一（問題 5）
```

フェーズ A で Native のコードが整理された後に着手。影響範囲が小さいので独立して進められる。

### フェーズ C: 機能接続（後でいい）

```
C-1: フィルター/パーティクルの Op 接続（問題 6）
C-2: テキスト文字送り（問題 7）
C-3: RPG 機能の Native 移植（問題 8）
```

Web 側の仕様が安定してから。フェーズ A の基盤があれば、各機能の追加は独立して進められる。

```
┌──────────────────────────────────────────────────────────┐
│ packages/core                                             │
│                                                           │
│  KSC → Compiler → Op IR (JSON)                           │
│                      ↓                                    │
│               GameScreenState ←── GameScreenUpdater        │
│                      ↓                                    │
│             IOpHandler interface                          │
│             IInput interface                              │
│             IAudio interface                              │
│             IStorage interface                            │
│             Action enum                                   │
├────────────────────┬─────────────────────────────────────┤
│ packages/web       │ packages/native-engine               │
│                    │                                      │
│ TS OpRunner        │ C++ OpRunner (新規)                   │
│ WebOpHandler       │ SDL2Handler (リネーム)                │
│ InputManager       │ InputManager (新規)                   │
│ AudioManager       │ AudioManager (シグネチャ修正)          │
│ StorageManager     │ FileStorage (シグネチャ修正)           │
│ PixiJS Renderer    │ IRenderer (GL/Soft)                  │
└────────────────────┴─────────────────────────────────────┘
```

### 差分の変化

| 層 | Before | After |
|---|---|---|
| スクリプト実行 | Web: Compiler+OpRunner, **Native: 独自 Parser** | **両方 Op IR** |
| Handler I/F | Web: IOpHandler (40+), **Native: IEngineAPI (10)** | **両方 IOpHandler** |
| 入力 | Web: IInput, **Native: 直接 SDL_PollEvent** | **両方 IInput** |
| 状態管理 | Web: WebOpHandler 内, **Native: SDL2Engine 内** | **GameScreenState 共通** |
| ブロッキング | Web: ノンブロッキング, **Native: ブロッキング** | **両方ノンブロッキング** |

### 新コマンド追加時のフロー

**Before:**
1. `Op.ts` に型追加
2. `commandRegistry.ts` にパーサー追加
3. `IOpHandler.ts` にメソッド追加
4. `OpRunner.ts` に dispatch 追加
5. `WebOpHandler.ts` に実装
6. **`Parser.cpp` に解析ルール追加（Native 独自）**
7. **`Evaluator.cpp` に実行ルール追加（Native 独自）**
8. **`SDL2Engine.cpp` に実装（IEngineAPI の別メソッド名）**

**After:**
1. `Op.ts` に型追加
2. `commandRegistry.ts` にパーサー追加
3. `IOpHandler.ts` にメソッド追加
4. `OpRunner.ts` に dispatch 追加（TS 版）
5. `WebOpHandler.ts` に Web 実装
6. C++ OpRunner に dispatch 追加（switch case 1 行）
7. `SDL2Handler.cpp` に Native 実装

**手順 6〜8 が 6〜7 に減り、かつ手順 6 は機械的な 1 行追加になる。**

## コスト見積もり

### フェーズ A の作業量

| タスク | 新規 | 変更 | 削除 | 難易度 |
|--------|------|------|------|--------|
| A-1: C++ OpRunner | +400行 | SDL2Engine リネーム | -600行 (Parser/Evaluator) | 中 |
| A-2: ブロッキング廃止 + IInput | +200行 | main.cpp 全面書換 | - | 高 |
| A-3: GameScreenState | +350行 (core) | WebOpHandler 分割 | - | 中 |
| **合計** | **+950行** | **大規模変更 3 ファイル** | **-600行** | |

ネットで +350 行程度。コード量はほぼ変わらないが、構造が根本的に改善される。

### やらなかった場合のコスト

Switch 移植時に:
- Native の独自 Parser を Switch にもコピー → 3 プラットフォームで 3 つの実行エンジン
- IEngineAPI を Switch 用にも作成 → 3 プラットフォームで 3 つの Handler I/F
- 各プラットフォームの動作差異テスト → Op 数 × プラットフォーム数 = **120+ テストケース**

**今やる: 950 行の変更 vs 後でやる: 3 倍の変更 + テスト爆発**
