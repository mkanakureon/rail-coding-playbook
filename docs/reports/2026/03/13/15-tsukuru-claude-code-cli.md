# Claude Code を前提としたシナリオ制作 CLI 検討

- **作成日**: 2026-03-13
- **ステータス**: 検討段階
- **前提**: Claude Code（CLI版 Claude）をシナリオ制作のメインインターフェースとする

---

## 1. 現状の課題

### 既存パイプライン（assist-cli.ts）

```
設定ファイル(.md) → Gemini API × 4段階 → editor-json → APIインポート
```

| 課題 | 詳細 |
|------|------|
| 対話がない | 4段階を一気通貫で回す。途中で「ここ違う」が言えない |
| フィードバック反映が遅い | 出力を見て → 設定ファイルを修正 → 全段階やり直し |
| LLMが文脈を持たない | 各ステージが独立。Stage 3 で Stage 0 の意図を忘れる |
| 作者の好みを学習しない | 毎回同じプロンプトテンプレート |
| セットアップが重い | API キー設定、設定ファイル4本作成、config.json 準備 |

### ツクールユーザーが本当に欲しいもの

```
「学園ものを作りたい。主人公は転校生の女の子。幼馴染と再会する話。」
↓
すぐにプレビューで動くものが出てくる
↓
「もっと切なくして」「選択肢を2箇所入れて」
↓
即座に反映される
```

---

## 2. 根本的な発想転換

### assist-cli 方式（現行）

```
Claude Code → assist-cli.ts → Gemini API → JSON → API → Editor
        ↑                                                    |
        └──── 修正が必要なら設定ファイルを書き直す ────────────┘
```

- Claude Code は「スクリプトを実行する係」
- 実際の生成は Gemini（文脈を保持しない）
- 修正ループが長い

### Claude Code 直接方式（提案）

```
作者 → Claude Code → [直接 JSON 生成] → API → Editor/Preview
  ↑                                              |
  └──── 「ここ変えて」→ Claude Code が即修正 ──────┘
```

- **Claude Code 自身が LLM であり、生成者であり、ツールの操作者**
- 会話の中で文脈を保持したまま修正できる
- 設定ファイル不要（会話が設定）
- Gemini API 不要（Claude 自身が生成）

---

## 3. Claude Code が「そのまま使える」理由

| 能力 | assist-cli | Claude Code |
|------|-----------|-------------|
| LLM として文章を生成 | Gemini を呼ぶ | **自分自身が生成** |
| JSON を構築 | テンプレート変換 | **直接 JSON を書く** |
| ファイルを読み書き | fs モジュール | **Read/Write ツール** |
| API を呼ぶ | fetch | **Bash で curl** |
| コードベースを理解 | 理解しない | **CLAUDE.md + 型定義を読む** |
| ユーザーと対話 | できない | **会話で即座にフィードバック** |
| エラーを修正 | 手動 | **自分でデバッグ** |

**つまり**: assist-cli が4段階のパイプラインで実現していたことを、Claude Code は1つの会話の中で全部できる。

## 4. 具体的な CLI スキル設計

### 4-1. `/create-story` — 会話からプロジェクト作成

```
ユーザー: 「学園もの作って。転校生の女の子が幼馴染と再会する話」

Claude Code の動作:
1. プロジェクトを API で作成（POST /api/projects）
2. 公式アセットから学園背景・キャラを選択
3. ページ構成を設計（プロローグ → 出会い → 展開 → 選択肢 → 結末）
4. 各ページにブロックを生成（bg → ch → text → choice → jump）
5. PUT /api/projects/:id でまとめて保存
6. プレビュー URL を返す
```

**生成する JSON の例**:

```json
{
  "pages": [
    {
      "id": "page-prologue",
      "name": "プロローグ",
      "blocks": [
        { "id": "start-1", "type": "start" },
        { "id": "bg-1", "type": "bg", "assetId": "school-gate" },
        { "id": "text-1", "type": "text", "body": "四月。桜が風に舞う朝。\\n私は新しい制服に袖を通し、校門をくぐった。" },
        { "id": "ch-1", "type": "ch", "characterId": "protagonist", "expressionId": "neutral", "pos": "C" },
        { "id": "text-2", "type": "text", "body": "――あれ？", "speaker": "主人公" },
        { "id": "text-3", "type": "text", "body": "校庭の隅に見覚えのある横顔。\\nまさか、そんなはずは……。" }
      ]
    }
  ]
}
```

Claude Code は **型定義を読んでいるので**、14種のブロック型を正しく使える。

### 4-2. `/edit-story` — 対話でシナリオ修正

```
ユーザー: 「2ページ目のセリフ、もっと切なくして」

Claude Code の動作:
1. GET /api/projects/:id で現在のデータを取得
2. 2ページ目の text ブロックを特定
3. セリフを書き換え
4. PUT /api/projects/:id で保存
```

```
ユーザー: 「3ページ目の後に選択肢を入れて。『素直に話しかける』と『遠くから見守る』」

Claude Code の動作:
1. choice ブロックを生成
2. 分岐先の2ページを作成（それぞれ別展開）
3. jump ブロックで合流させる
4. 保存
```

### 4-3. `/add-scene` — シーン追加

```
ユーザー: 「放課後の図書室で二人きりになるシーンを追加して」

Claude Code の動作:
1. 利用可能なアセットを _ai_context から確認
2. 図書室の背景がなければ近いものを提案
3. ページを新規作成
4. bg → ch → text の順でブロックを生成
5. 既存ページからの導線（jump）を追加
6. 保存
```

### 4-4. `/preview` — プレビュー確認

```
ユーザー: 「プレビューして」

Claude Code の動作:
1. プレビュー URL を構築（http://localhost:5175/preview.html?id={projectId}）
2. open コマンドでブラウザを開く
3. 必要なら Playwright でスクリーンショットを撮って見せる
```

### 4-5. `/layout` — UI レイアウト編集

```
ユーザー: 「メッセージウィンドウをもう少し上に」

Claude Code の動作:
1. プロジェクトの playLayout を取得
2. message-window の y を調整
3. 保存
4. レイアウトプレビュー HTML を開いて確認
```

## 5. 実装に必要なもの

### 5-1. Claude Code スキル（.claude/skills/）

| スキル名 | トリガー | 機能 |
|---------|---------|------|
| `create-story` | 「物語を作って」「シナリオ作成」 | 会話 → プロジェクト作成 |
| `edit-story` | 「セリフ変えて」「シーン修正」 | 対話でブロック編集 |
| `add-scene` | 「シーン追加」「ページ追加」 | 新ページ・ブロック生成 |
| `preview` | 「プレビュー」「確認」 | ブラウザでプレビュー表示 |
| `list-assets` | 「使える背景は？」「キャラ一覧」 | 利用可能アセット表示 |
| `export-ksc` | 「KSCに変換」「スクリプト出力」 | editor JSON → .ksc 変換 |

### 5-2. CLAUDE.md への追記

```markdown
## AI シナリオ制作（Claude Code CLI）

### ブロック生成ルール
- ブロック ID は `{type}-{Date.now()}` で生成
- start ブロックは各ページに1つ、先頭固定
- assetId / characterId / expressionId は _ai_context の実在 ID を使う
- text ブロックの body は40文字前後で改行（\\n）

### API 操作
- 認証: POST /api/auth/login → token 取得
- プロジェクト取得: GET /api/projects/:id（_ai_context 付き）
- 保存: PUT /api/projects/:id（data.pages を送信）
- _ai_context は読み取り専用（PUT 時に含めても自動除去）

### テキスト生成ガイドライン
- ノベルゲームの文体で書く（地の文は三人称 or 一人称）
- セリフは speaker フィールドにキャラ名
- 1テキストブロック = 1〜3行（長すぎない）
- 選択肢は2〜4個が適切
```

### 5-3. ヘルパースクリプト（あると便利）

| スクリプト | 用途 |
|-----------|------|
| `scripts/cli/kaedevn-auth.ts` | ログイン → トークン取得 → 環境変数保存 |
| `scripts/cli/kaedevn-project.ts` | プロジェクト CRUD ラッパー |
| `scripts/cli/kaedevn-assets.ts` | アセット一覧・検索 |

ただしこれらは **必須ではない**。Claude Code は curl で直接 API を叩ける。
スクリプトがあると「毎回トークン取得」の手間が省けるという程度。

## 6. assist-cli（Gemini）との使い分け

| 用途 | Claude Code CLI | assist-cli（Gemini） |
|------|----------------|---------------------|
| 短編（1〜5ページ） | ◎ 会話で即座に完成 | △ セットアップが重い |
| 中編（10〜30ページ） | ○ 対話しながら制作 | ○ パイプラインが効率的 |
| 長編（50ページ以上） | △ コンテキスト圧縮で文脈ロス | ◎ 段階的生成が向く |
| 修正・微調整 | ◎ 「ここ変えて」で即反映 | × 設定変更 → 再生成 |
| プロット設計 | ◎ 会話で練り上げ | △ 自動生成のみ |
| 量産（同ジャンル複数本） | △ 手動繰り返し | ◎ 自動バッチ処理 |
| UI レイアウト | ◎ JSON 直接編集 | × 対応なし |

**結論**: 両方を用途に応じて使い分ける。短〜中編の制作と修正は Claude Code、長編の量産は assist-cli。

## 7. ユーザー体験のイメージ

### シナリオ A: 初めての作品制作（10分）

```
$ claude

ユーザー: 学園ラブコメを作りたい。主人公は天然ボケの女子高生。
         クールな幼馴染の男子と、転校生のイケメンの三角関係。

Claude: プロジェクトを作成します。
        利用可能な学園背景: 教室、廊下、屋上、校門、図書室
        キャラクター3人を設定しました。

        5ページ構成で作ります：
        1. プロローグ（校門で幼馴染と登校）
        2. 出会い（教室に転校生）
        3. 放課後（屋上で幼馴染と）
        4. 選択肢（どちらと帰る？）
        5. エンディング（選択に応じて分岐）

        作成中...
        完了しました。プレビュー: http://localhost:5175/preview.html?id=01ABC...

ユーザー: プロローグのセリフ、もっとボケてる感じにして

Claude: text ブロックを修正しました。
        「おはよー！今日も地球は回ってるね！」
        → 「おはよー！……あれ、今日って月曜？火曜？
           ていうか、何月だっけ？」
        保存しました。

ユーザー: いいね。選択肢のところ、3つ目の選択肢も入れて。
         「一人で帰る」

Claude: choice ブロックに3つ目の選択肢を追加しました。
        分岐先のページ「一人帰り」も作成し、
        独り言で幼馴染のことを考えるシーンにしました。
        保存しました。
```

### シナリオ B: 既存プロジェクトの修正（3分）

```
ユーザー: プロジェクト 01KKDX2W のページ3、テキストが長すぎる。
         分割して

Claude: GET /api/projects/01KKDX2W でデータを取得しました。
        ページ3「放課後の告白」の text ブロック:
        - text-3: 「夕日が教室を…」（127文字）← 長い

        3つのブロックに分割します：
        1. 「夕日が教室を橙色に染めていた。」
        2. 「窓際に立つ彼女の横顔が、やけに眩しく見えた。」
        3. 「言葉が出ない。胸の奥で何かが、ぎゅっと締まる。」

        保存しました。
```

### シナリオ C: レイアウトと合わせて制作

```
ユーザー: ホラーっぽい雰囲気にしたい。
         メッセージウィンドウを暗くして、フォントを小さくして

Claude: playLayout を更新します。
        message-window:
          backgroundColor: rgba(0,0,0,0.9)  ← より暗く
          opacity: 0.95
          options.fontSize: 20  ← 24 → 20 に縮小
          options.textColor: "#cccccc"  ← 白 → やや暗い白

        シナリオ側にも screen_filter を追加：
          filterType: "vignette", intensity: 0.6

        保存しました。プレビューで確認しますか？
```

## 8. 技術的な実現可能性

### 今すぐできること（実装不要）

Claude Code は **今この瞬間から** 以下ができる：

| 操作 | 方法 |
|------|------|
| プロジェクト作成 | `curl -X POST localhost:8080/api/projects` |
| データ取得 | `curl localhost:8080/api/projects/:id` |
| ブロック追加・編集 | JSON を構築して `curl -X PUT` |
| アセット確認 | `_ai_context` を読む |
| プレビュー | `open http://localhost:5175/preview.html?id=...` |
| KSC 変換 | `ks-generator.ts` の関数を呼ぶ |
| スクリーンショット | Playwright を実行 |

**つまり、スキルファイル（.claude/skills/）を書くだけで動く。**
新しいコードの実装はほぼ不要。

### あると効率が上がるもの

| 実装 | 効果 | 工数 |
|------|------|------|
| 認証トークン永続化スクリプト | 毎回ログインしなくて済む | 小 |
| `GET /api/editor-schema` のキャッシュ読み | 型情報を毎回取得しなくて済む | 小 |
| プロジェクト一覧ビュー（CLI） | 「どのプロジェクトを編集する？」がスムーズ | 小 |
| ブロック差分表示 | 「何が変わったか」を見やすく | 中 |
| undo 機能（API 側） | 「戻して」に対応 | 中 |

## 9. assist-cli との統合案

既存の assist-cli を「捨てる」のではなく、Claude Code から呼び出す形で統合する。

```
Claude Code（対話・短編・修正）
    │
    ├── 直接 API 操作（短編・修正）
    │
    └── assist-cli 呼び出し（長編の量産）
        ├── Stage 0-1: Claude Code が対話でプロットを練る
        ├── Stage 2-3: assist-cli（Gemini）でバッチ生成
        └── 修正: Claude Code が対話で微調整
```

長編でも「プロットを練る」フェーズと「修正する」フェーズは Claude Code が向いている。
「大量のテキストを一気に生成する」フェーズだけ assist-cli（Gemini）に任せる。

## 10. 競合との差別化

| 競合 | AI 制作方式 | kaedevn + Claude Code |
|------|-----------|----------------------|
| RPGツクール | AIプラグインなし | **会話で即座に動くゲームが完成** |
| ティラノスクリプト | AIスクリプト生成は手動コピペ | **APIで直接プロジェクトに反映** |
| Ren'Py | GitHub Copilot で補完程度 | **ブロック単位で構造的に生成** |
| NovelAI | テキスト生成のみ | **背景・キャラ・選択肢・分岐も含めて完成品を出力** |
| ChatGPT + 手動組み立て | テキストを生成→手動でゲームに入れる | **生成→保存→プレビューが1コマンド** |

**最大の差別化**: 「AIが生成したものを手動でゲームに組み込む」工程がゼロ。
Claude Code が API を叩いて直接プロジェクトに書き込み、プレビューで即確認できる。

## 11. リスクと対策

| リスク | 対策 |
|--------|------|
| Claude Code の利用料金 | 短編なら1作品あたり数百円程度。assist-cli（Gemini）より安い可能性 |
| コンテキスト圧縮で文脈ロス | 長編は段階保存（ページ単位で API に保存しながら進める） |
| 作者が Claude Code を使えない | スキルのトリガーを日本語で設定（「物語を作って」で起動） |
| API が落ちている | ローカル JSON ファイルに一時保存 → 後で API に反映 |
| 生成テキストの品質 | Claude Opus/Sonnet は日本語小説の品質が高い。ジャンルルール YAML も適用可 |

2. プロジェクト一覧表示

## 13. まとめ

**結論: Claude Code 前提の CLI は「作れる」というより「もうほぼ動く」。**

Claude Code は LLM + ファイル操作 + API 呼び出し + 対話を1つのインターフェースで持っている。
kaedevn の API は JSON ベースで AI フレンドリーに設計されている。
この2つを繋ぐのに必要なのは **スキルファイル（.md）だけ**。新しいコードはほぼ不要。

既存の assist-cli（Gemini パイプライン）は長編量産用として残し、
短〜中編の対話的制作・修正・レイアウト調整は Claude Code に任せる。

これは「ツクールの不満」の中でも最も刺さる差別化ポイントになる：
**「会話するだけでゲームができる」は、他のどのツールにもない。**
