## A 候補

### 1. CLI 完備 — Claude Code だけでゲームが作れる

**他にあるか: ない**

32 スキル + 専用 CLI で、ターミナルからプロジェクト作成→キャラ追加→シナリオ編集→プレビュー→デプロイまで完結する。ビジュアルノベルエンジンで CLI が完備されているものは存在しない。

| ステップ | スキル |
|---------|--------|
| プロジェクト作成 | `/create-project`（7ジャンルテンプレート） |
| テキスト台本投入 | `/create-project` + import-story |
| キャラクター追加 | `/character`（公式アセットから） |
| シナリオ編集 | `/edit-blocks`（ブロック CRUD・検証） |
| アセット管理 | `/asset`（アップロード・公式連携） |
| マップ作成 | `/map`（タイル配置・衝突判定） |
| プレビュー | API + Web エンジン |
| デプロイ | `/deploy-azure` + git push |

**刺さる層:** AI コーディングツールのヘビーユーザー、開発者、CLI 好き

---

### 2. 公式アセット — 開いた瞬間にゲームが動く

**他にあるか: ない（この粒度では）**

プロジェクト作成時に背景 5 枚 + キャラ 5 体 + 初期ブロックが自動セットアップされる。素材ゼロの人がすぐゲームを作り始められる。Canva が「テンプレートが最初からある」で勝ったのと同じ構造。

- DB 3 層（OfficialAsset / Asset / UserAsset）
- セマンティック検索（Gemini Embedding、「夕暮れの教室」で背景が出る）
- カテゴリ + タグ + AI ビジョンによるメタデータ自動付与

**刺さる層:** 素材を持っていない初心者、「とりあえず触ってみたい」層

---

### 3. 4 プラットフォーム完全互換 — 書いたら全部で動く

**他にあるか: ない（この範囲では）**

1 つのシナリオデータ（Op IR / scenario.json）が Web・macOS・Android・Switch で動く。53 コマンドが TypeScript と C++ で完全一致。

| プラットフォーム | 技術 | 状態 |
|----------------|------|------|
| Web (PixiJS) | TypeScript OpRunner + WebOpHandler | 本番稼働 |
| macOS/Linux | SDL2 + C++ OpRunner | ビルド済み |
| Android | SDL2 + NDK | ビルド済み |
| Nintendo Switch | SDL2 + DevkitPro | SDK 待ち |

**刺さる層:** マルチプラットフォーム展開したいクリエイター、Switch を目指す個人開発者

---

### 4. ゲームアセット完全互換 — slug 一つで全環境で解決

**他にあるか: ない**

アセットを slug（例: `forest_evening`）で参照し、プラットフォームごとに解決方法が異なるが、シナリオ側は一切変更不要。

```
slug: "forest_evening"
  → Web:    GET /api/preview/:id → CDN URL → fetch
  → Editor: AssetSelectModal → プレビュー表示
  → Native: assets.json → SDL_Texture 読み込み
  → Switch: 同上（pak ファイルから）
```

セーブデータにも slug（参照 ID）のみ格納。画像・音声の実体は含まない。

**刺さる層:** 4PF 候補と同じ層 + アセット管理に苦しんだ経験がある開発者

### 6. GUI ⇔ KSC 双方向変換 — ビジュアルとテキストを自由に行き来

**他にあるか: ない**

エディタで GUI ブロックを並べると KSC スクリプトが生成され、KSC を手書きすると GUI に反映される。ksConverter.ts（360+ 行）で双方向変換。

- GUI で大まかに作る → KSC で微調整
- KSC で書いたものを GUI で確認
- `/edit-blocks` CLI でも同じデータを操作可能

**刺さる層:** GUI と テキストの両方を使いたい中級者、スクリプト派とビジュアル派のチーム

---

### 7. _ai_context API — AI エージェントがゲームを理解して編集できる

**他にあるか: ない**

`GET /api/projects/:id` で `_ai_context`（利用可能アセット・キャラ・ページ・変数）が返る。`GET /api/editor-schema` で全 14 ブロック型の定義が返る。AI が「このプロジェクトで何が使えるか」を理解した上でブロックを編集できる。

- strip-on-write: PUT 時に `_ai_context` は自動削除（DB 汚染防止）
- 認証不要のスキーマ API（24 時間キャッシュ）

**刺さる層:** AI エージェント開発者、Claude Code / MCP 連携に関心がある技術者

個人が作ったビジュアルノベルを Switch に直接出せるプラットフォームは存在しない。技術基盤は 85-90% 完成（SDK 待ち）。

### 9. 7 ジャンルテンプレート — 「何を作るか」を選ぶだけで始まる

**他にあるか: 部分的（ティラノビルダーにテンプレートはあるが、API + 公式アセット連動はない）**

fantasy, horror, school, romance, mystery, comedy, longstory の 7 ジャンル。テンプレート選択 → 公式アセット自動配置 → 初期ブロック生成が 1 コマンドで完了。

**刺さる層:** 「何から始めればいいかわからない」初心者

---

### 10. Op IR 設計 — コンパイラとランタイムの完全分離

**他にあるか: この粒度ではない**

スクリプト（.ks / .ksc）→ コンパイラ → Op IR（JSON）→ ランタイム（Web / Native）の設計。53 コマンドが frozen で、コンパイラとランタイムが独立して進化できる。

- `packages/compiler`: パーサー + 変換
- `packages/core`: Op 型定義（TypeScript）
- `packages/native-engine`: Op 型定義（C++、完全一致）
- `commandRegistry.ts` が単一定義源、同期テストで検証

**刺さる層:** エンジン設計に関心がある技術者、OSS コントリビューター

## 所感

「競合にあるか: なし」が 10 個中 7 個。A が多いこと自体は問題ではない。

**これらの A は Switch 実現のための準備。** Web で収益と実績を立てながら、Switch が動いた瞬間に「CLI も公式アセットもエディタも 4PF 互換も全部 Switch で使える」状態になる。今の A は Switch の前座ではなく、**Switch と一緒に爆発する弾の仕込み**。

ターゲット層ごとに前面に出す A を使い分ける:

| ターゲット | 前面に出す A |
|-----------|------------|
| 初心者・素材なしの人 | 2. 公式アセット + 9. テンプレート |
| Claude Code / AI ツールユーザー | 1. CLI 完備 + 7. _ai_context |
| 演出にこだわるクリエイター | 5. エディタ 22 種 + 6. 双方向変換 |
| マルチプラットフォーム志向 | 3. 4PF 互換 + 4. アセット互換 |
| Switch を目指す個人 | 8. Switch（動画 1 本で証明） |

**全部を 1 ページで均等に並べると BBBBB になる。** だが、ターゲットごとにランディングページや訴求を分ければ、それぞれが A として刺さる。
