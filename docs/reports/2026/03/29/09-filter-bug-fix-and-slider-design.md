### BUG 1: pixelate → FamicomFilter マッピングミス

```typescript
// Before
case "pixelate":
  return new FamicomFilter(intensity != null ? 2.0 + intensity * 10.0 : undefined);

// After
case "pixelate":
  return new PixelateFilter(intensity != null ? 2.0 + intensity * 10.0 : undefined);
```

単純なピクセル化を期待するユーザーに NES 52色パレット変換が適用されていた。

### BUG 2: DustFilter alpha overflow

```glsl
// Before — 演算子優先順位で 0.9 + 0.3*sin(...) → [0.6, 1.2]
float alpha = 0.4 + 0.6 * sin(t * 0.5 + alphaPhase) * 0.5 + 0.5;

// After — 括弧追加で [0.4, 1.0]
float alpha = 0.4 + 0.6 * (sin(t * 0.5 + alphaPhase) * 0.5 + 0.5);
```

### BUG 3: ChromaticAberration premultiply alpha

```glsl
// Before — center の alpha で全チャンネルを premultiply
color.rgb = vec3(r, g, b) * color.a;

// After — 3サンプルの最大 alpha で統一
float maxAlpha = max(sR.a, max(color.a, sB.a));
color.rgb = vec3(r, g, b) * maxAlpha;
color.a = maxAlpha;
```

## Part 2: パラメータスライダー設計

### 目的

各フィルターには `intensity` 以外にも調整可能なパラメータがある（radius, speed, angle 等）。
ColorAdjustBlock のスライダー UI パターンを踏襲し、フィルター選択時に該当パラメータのスライダーを動的に表示する。

### データフロー

```
Editor UI (sliders) → ScreenFilterBlock.params → KSC script → Engine ScreenFilter.createFilter(options)
```

### ScreenFilterBlock 型拡張

```typescript
type ScreenFilterBlock = {
  id: string;
  type: 'screen_filter';
  filterType: string;
  intensity?: number;
  pc98Tiling?: boolean;            // 既存互換
  params?: Record<string, number>; // NEW: フィルター別パラメータ
};
```

### KSC 出力フォーマット

```
@filter vignette 0.6 radius=0.8 softness=0.3
@filter rain 0.4 speed=2.0 angle=-0.1
@filter pc98 0.5 tiling ditherStrength=0.2
```

### 後方互換性

- `params` は optional → 既存プロジェクト移行不要
- `@filter sepia 0.8` はそのまま動作
- 品質 low フォールバック時は options 無視

### BUG 4（解像度ハードコード）について

6ファイルで 1280x720 がハードコードされている。論理解像度固定の現状では実害なしだが、
スライダーで `resolution` を公開するのではなく、将来の解像度可変対応時にエンジン側で自動設定する方針。

## 教訓

- GLSL の演算子優先順位バグはデバッガがないため発見が遅れやすい。レビュー時に括弧の有無を意識する
- `createFilter` の switch 文が 500 行超 — フィルター数が増えるほどメンテコストが上がる。パラメータ定義をメタデータ化することで管理を改善できる
- PixiJS の premultiply alpha はフィルター間でサンプル位置が異なる場合に不整合が起きやすい
