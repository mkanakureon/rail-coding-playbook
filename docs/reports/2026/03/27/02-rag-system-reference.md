# kaedevn RAG システム解説

> Generated by Claude Opus 4.6

## 何のための RAG？

**1,396 個の Markdown ファイル、約 29 万行** のドキュメントの中から、必要な情報を瞬時に見つけるための仕組み。

| フォルダ | ファイル数 | 行数 | 内容 |
|---------|-----------|------|------|
| `docs/01_in_specs/` | 109 | 64,627 | 仕様書（ChatGPT との会話で作成） |
| `docs/09_reports/` | 471 | 109,812 | レポート・設計書 |
| `docs/10_ai_docs/` | 441 | 22,822 | AI 生成ドキュメント |
| `docs/landing/` | 120 | 10,202 | ランディングページ・日誌 |

普通の grep だとキーワード完全一致しか引っかからない。RAG なら「バトルシステムの設計意図」で検索すると、「戦闘」「combat」「ターン制」など意味的に近い文書もヒットする。

## 仕組み（ハイブリッド検索）

### インデックス時

```
ドキュメント（.md）
    ↓
見出し（H1/H2/H3）ごとにチャンク分割
    ↓
OpenAI text-embedding-3-small で 1536 次元ベクトル化
    ↓
PostgreSQL（pgvector）に保存
```

### 検索時

```
検索クエリ
    ↓ 同時に2つの検索を実行
┌────────────────────┬───────────────────┐
│ ベクトル検索         │ キーワード検索      │
│ 意味的な類似度       │ 全文検索（tsvector）│
│ コサイン距離         │ ts_rank            │
│ 重み: 60%           │ 重み: 40%          │
└────────────────────┴───────────────────┘
    ↓ スコアを合成
パス・見出しにクエリが含まれていたら 1.2 倍ブースト
    ↓
上位 5 件を返す
```

**ベクトル検索だけだと** 意味は近いが関係ない文書が混じる。**キーワード検索だけだと** 表記揺れに弱い。**両方を 6:4 で混ぜる** ことで精度が上がる。

## 技術スタック

| コンポーネント | 技術 |
|-------------|------|
| ベクトル DB | PostgreSQL + pgvector 拡張 |
| 埋め込みモデル | OpenAI `text-embedding-3-small`（1536 次元） |
| 全文検索 | PostgreSQL `tsvector` + `plainto_tsquery` |
| ベクトルインデックス | HNSW（vector_cosine_ops） |
| API | Hono（`/api/rag-hybrid/search`） |
| MCP | `search_docs` ツール |

## 2 つの用途

| 用途 | 対象 | 使う場面 |
|------|------|---------|
| **ドキュメント検索** | `docs/` の 1,396 ファイル | 設計意図・過去の決定を調べる |
| **プロジェクト固有 RAG** | 作品の設定（世界観・キャラ設定） | AI がシナリオ生成するときの文脈 |

### ドキュメント検索

`scripts/cli/rag/rag-index.ts` で `docs/**/*.md` を見出しごとにチャンク分割 → 埋め込み → PostgreSQL に保存。SHA-256 ハッシュで変更検出し、変更がないファイルはスキップ。

### プロジェクト固有 RAG

作品の設定（世界観、キャラクター、あらすじ）を以下のチャンクに分割:
- `overview`: 概要・ジャンル・時代・トーン
- `world`: 世界設定
- `character`: キャラクターごと（名前・性格・口調・背景）
- `history`: 歴史的事実
- `plot`: あらすじ
- `episode_summary`: 各話の要約

## DB スキーマ

```sql
CREATE TABLE rag_chunks (
  id BIGSERIAL PRIMARY KEY,
  doc_path TEXT NOT NULL,
  heading_path TEXT NOT NULL,       -- "H1 > H2 > H3"
  chunk_index INT NOT NULL,
  content TEXT NOT NULL,
  content_tsv TSVECTOR,             -- 全文検索用
  embedding VECTOR(1536),           -- pgvector
  content_hash VARCHAR(64) NOT NULL,
  updated_at TIMESTAMPTZ
);
```

インデックス: `doc_path`, `content_hash`, tsvector GIN, HNSW（ベクトル検索用）

## RAG は便利か？

**結論: このプロジェクトでは非常に便利。**

### 便利な理由

- **ドキュメントが 29 万行ある**。grep で探すのは現実的でない
- **設計意図は本文に書いてある**。ファイル名や見出しだけでは見つからない
- **Claude Code が作業前に自動で過去文書を調べる** ワークフローに組み込まれている（CLAUDE.md に「作業前の RAG 検索」がルール化されている）
- **同じ議論を繰り返さない**。「これ前にも検討した？」→ RAG で即確認

### 注意点

- **小規模プロジェクト（ドキュメント数十個）なら grep で十分**。RAG のセットアップコスト（PostgreSQL + pgvector + OpenAI API）に見合わない
- **OpenAI の API 料金がかかる**。embedding は安いが、頻繁に再インデックスすると積もる
- **インデックスが古いと嘘をつく**。更新を忘れると「この関数は存在する」と言いつつ実際は削除されている、みたいなことが起きる

### 導入の目安

**ドキュメントが 100 ファイル / 1 万行を超えたら RAG を検討する価値がある。**

## 過去文書が残っていることの価値

### 実例（2026-03-27）

- **2.5D 検討文書**（Gemini CLI が作成）を読んで、既存コードとの整合性を確認した上で「Phase 1+2 だけ先にやるのがコスパ最高」と判断できた
- **Azure Communication Services セットアップガイド**（2月作成）が残っていたから、リソース再構築の手順がすぐわかった

### RAG がなかったら

- 2.5D の過去検討を知らずにゼロから設計していた
- Azure のリソース名やコマンドを思い出すのに時間がかかっていた

### コードだけでは残らないもの

コードを読めば「何をした」はわかる。しかし以下は文書にしか残らない:

- **なぜその設計にしたか**（設計意図）
- **他にどんな案を検討して棄却したか**（検討経緯）
- **どんな障害があって、どう対処したか**（インシデント記録）

RAG はこれらを検索可能にする。同じ議論を繰り返さず、過去の判断を踏まえた上で次の判断ができる。
