# シェルスクリプト運用ガイドライン

## sh で手順を作るべきケース

| 条件 | 理由 |
|------|------|
| 毎回同じ手順を繰り返す | 再現性が高い。人によるブレがなくなる |
| デプロイ・テスト・ビルドなど定型作業 | 手順書と違ってコードとして管理され腐りにくい |
| CI/CD に組み込む予定がある | sh ならそのまま workflow に移行できる |
| 失敗箇所の切り分けが必要 | 段階的に実行すれば原因特定が早い |
| `--repeat N` など手動では面倒なオプション | 自動化の効果が高い |

## sh にしない方がいいケース

| 条件 | 代替手段 |
|------|---------|
| 1回きりの作業 | 手動実行で十分。スクリプト化のコストに見合わない |
| 判断が毎回変わる作業 | 結果を見て次の手順が変わるなら手動 or Claude に任せる |
| 複雑なロジックが必要 | TypeScript の方がテスト・デバッグしやすい |
| 環境依存が多い処理 | Docker や Node.js スクリプトの方がポータブル |

## 注意点

### macOS vs Linux の互換性

実際にハマった例:

| コマンド | macOS (BSD) | Linux (GNU) | 対策 |
|---------|-------------|-------------|------|
| `head -n -1` | 非対応 | 対応 | `sed '$d'` を使う |
| `date +%s%N` | ナノ秒非対応 | 対応 | `python3 -c 'import time; ...'` を使う |
| `sed -i` | `-i ''` が必要 | `-i` のみ | `sed -i '' 's/...'` or perl を使う |
| `grep -P` | 非対応 | 対応 | `grep -E` (拡張正規表現) を使う |

### スクリプトが増えすぎた場合の整理

- `scripts/README.md` で一覧と用途を管理する
- 用途別にサブディレクトリを切る（`scripts/azure/`, `scripts/dev/`）
- 使われなくなったスクリプトは削除する（git に残るので恐れない）

### sh のデバッグ

- `set -eo pipefail` を必ず先頭に書く（エラーで即停止）
- `set -x` を一時的に追加してトレース出力
- 複雑なロジックは関数に分けて個別テスト可能にする
