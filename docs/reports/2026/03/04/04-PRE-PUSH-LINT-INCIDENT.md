# pre-push フック導入で発覚した Editor ESLint 技術的負債

**日付**: 2026-03-04

## 概要

Gap Analysis 対策として `.husky/pre-push` フック（typecheck + lint）を導入したところ、`git push` が Editor の既存 ESLint エラー（約50件）でブロックされた。

## 経緯

1. `30dbe9b` で `.husky/pre-push` を新規追加（typecheck + lint を push 前に実行）
2. root `package.json` の `lint` スクリプトを `npm run lint -w apps/editor && npm run lint -w apps/next` に設定
3. `git push origin main` 実行 → Editor の `eslint .` が大量のエラーを報告 → push 失敗

## 根本原因

**Editor (`apps/editor`) は ESLint 設定 (`eslint.config.js`) を持っていたが、これまで CI でも pre-push でも実行されていなかった。** 開発中に `vite dev` は ESLint を実行しないため、エラーが蓄積していた。

### 検出されたエラーの種類

| エラー | 件数(概算) | 重要度 |
|--------|-----------|--------|
| `@typescript-eslint/no-unused-vars` | 15+ | 低（動作に影響なし） |
| `react-hooks/set-state-in-effect` | 10+ | 中（パフォーマンス影響あり） |
| `@typescript-eslint/ban-ts-comment` (`@ts-ignore`) | 5+ | 低 |
| `react-hooks/rules-of-hooks` (条件付きフック) | 1 | **高（バグの可能性）** |
| `react-refresh/only-export-components` | 5+ | 低 |
| `prefer-const` | 3+ | 低 |

## 対応

### 即時対応（`8983bb6`）

pre-push フックの lint 対象を **Next.js のみ** に変更:

```diff
- npm run lint || {
+ npm run lint -w apps/next || {
```

理由: Editor の既存エラーをすべて修正するには時間がかかり、デプロイをブロックすべきではない。

### 残タスク

1. **Editor ESLint エラーの段階的修正**
   - `react-hooks/rules-of-hooks` (BlockList.tsx:146) — 条件付きフック呼び出しはバグの可能性が高いため最優先
   - `react-hooks/set-state-in-effect` — パフォーマンス改善（中優先）
   - `no-unused-vars`, `ban-ts-comment`, `prefer-const` — 低優先だが量が多い

2. **CI での Editor lint 実行**
   - `ci.yml` の lint ジョブは root `npm run lint`（Editor + Next.js）を実行する
   - Editor のエラーが修正されるまで CI が失敗する可能性がある
   - **対策**: CI の lint も Next.js のみにするか、Editor lint エラーを修正するか決める

3. **root `lint` スクリプトの整理**
   - 現在: `npm run lint -w apps/editor && npm run lint -w apps/next`
   - Editor を含めるかどうかは残タスクの進捗次第

## 教訓

- **新しいガード（pre-push, CI lint）を導入する前に、既存コードベースでの実行結果を確認する**
- `vite dev` は ESLint を実行しないため、Editor のようなViteプロジェクトでは lint が長期間放置されやすい
- ガードの導入は段階的に：まず warning で実行 → エラーを修正 → blocking に昇格
