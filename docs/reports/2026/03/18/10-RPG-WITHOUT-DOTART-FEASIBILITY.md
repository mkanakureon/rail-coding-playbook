# ドット絵なし RPG 実現可能性調査

> Generated by Claude Opus 4.6

## 結論: 作れる

このリポジトリには**ドット絵なしで RPG を動かすシステムがすべて揃っている**。プレースホルダータイルセットが自動生成され、ゲームロジックはアセットの品質に依存しない設計になっている。

---

## 1. 既存システムの棚卸し

### 揃っているもの

| レイヤー | 実装 | ドット絵依存 |
|---------|------|------------|
| **マップ移動** | `MapSystem` + `MapTestPlay` — タイルベース移動・衝突判定・カメラ追従 | **なし** — プレイヤーは青い円、イベントは青いボックスで動く |
| **イベント発火** | `MapEvent` — action/touch/auto/parallel の4トリガー + scenarioId 連携 | **なし** — 座標ベース |
| **シナリオ実行** | KSC 47コマンド — テキスト・選択肢・変数・条件分岐・ジャンプ | **なし** — テキスト表示はフォント描画 |
| **戦闘** | `packages/battle/` — ターン制、ダメージ計算、状態異常、経験値 | **なし** — UIベース |
| **インベントリ** | `InventorySystem` — gold, items, equipment | **なし** |
| **セーブ/ロード** | `SaveData v2` — vars, party, inventory, mapState を IndexedDB に保存 | **なし** |
| **タイルセット生成** | `gen-placeholder-tilesets.mjs` — tilesets.json から自動生成 | **ここがドット絵の代わり** |

### 足りないもの・弱いもの

| 項目 | 状態 | 影響 |
|------|------|------|
| **戦闘 UI** | BattleScene の描画は簡易実装 | テキストベースなら問題なし |
| **NPC 歩行アニメ** | moveRoute 型定義済み、簡易実装 | プレースホルダーで可 |
| **メニュー画面** | UIレイアウトエディタで配置可能だがRPGメニュープリセットは未完 | テキストメニューで代替可 |
| **ワールドマップ遷移** | `@map_load` で切替可能だが、ワールドマップ UI はない | マップ間ジャンプで代替 |

## 2. なぜドット絵なしで成立するか

### プレースホルダータイルセットの仕組み

`scripts/cli/map/gen-placeholder-tilesets.mjs` が `maps/tilesets.json` から**幾何学パターンの PNG を自動生成**する。

- 壁: レンガ模様の横線
- 床: 木目 or 石目パターン
- 草: 緑ベタ + ランダムドット
- 水: 波紋パターン
- 木: 円形の葉 + 幹
- 家具: 矩形の組み合わせ

```bash
node scripts/cli/map/gen-placeholder-tilesets.mjs
# → assets/tilesets/outdoor.png (3.3KB)
# → assets/tilesets/interior.png (2.5KB)
```

これだけで **MapTestPlay が完全に動作する**。見た目はシンプルだが、通行判定・イベント・カメラ追従すべて機能する。

### ゲームロジックがアセットに依存しない設計

```
tilesets.json (メタデータ)
    ↓ passable, terrain, damage, slip
MapSystem (ロジック)
    ↓ 衝突判定、イベント発火
WebOpHandler (描画)
    ↓ PixiJS Sprite
画面表示
```

ロジック層は「タイル ID → メタデータ参照」だけで動く。PNG が何であるかは関係ない。

## 3. ドット絵なし RPG の3つのアプローチ

### A. プレースホルダーそのまま（最速）

**工数: 0** — 今すぐ使える。

```
outdoor.png (幾何学パターン)
  → マップエディタで配置
  → テストプレイで動作確認
  → シナリオ・戦闘を KSC で記述
```

見た目は「開発中のプロトタイプ」だが、ゲームプレイは完全に成立する。ゲームジャムや企画検証に最適。

### B. カラーブロック + ラベル付きタイル（低工数）

**工数: 半日** — `gen-placeholder-tilesets.mjs` のパターンを改良。

各タイルに **色 + 1文字テキスト** を入れる方式:

```
[緑G] [緑花] [青W] [茶D]   ← 色で地形、文字で識別
[灰壁] [茶ドア] [赤屋根]    ← 建物
```

ツクールの「デバッグモード」に近い見た目。プレイヤーには不向きだが、作者（自分）がマップを組むには十分。

### C. AI 生成タイル（現行パイプライン）

**工数: 数時間** — `gen-official-asset.ts` で Imagen 4.0 生成 → DB32 減色。

今日作ったパイプラインがこれ。API キーと時間が必要だが、品質は高い。テストで品質保証済み。

## 4. 推奨: アプローチ A で RPG を先に作り、絵は後から差し替える

### 理由

1. **ドット絵は差し替え可能** — タイルセット PNG を入れ替えるだけ。マップデータ・シナリオ・戦闘ロジックは一切変わらない
2. **ゲームデザインが先** — 面白いマップ・シナリオ・戦闘バランスを先に確認すべき。絵が良くてもゲームがつまらなければ意味がない
3. **プレースホルダーが完成している** — `outdoor.png` / `interior.png` が 48x48 タイルセットとして完全に動作する

### 具体的なワークフロー

```
Step 1: プレースホルダーでマップ作成
  └── maps/map-village.json, map-dungeon.json, map-field.json

Step 2: KSC でシナリオ・イベント記述
  └── NPC 会話、宝箱、ボス戦、村の入り口

Step 3: 戦闘データ定義
  └── troops.json, skills.json, actors.json

Step 4: テストプレイで動作確認
  └── MapTestPlay でプレイ → バランス調整

Step 5: （任意）ドット絵を AI 生成 or 手描きで差し替え
  └── gen-official-asset.ts で再生成 → テストで品質検証
```

## 5. 既存の RPG ゲームフロー（実装済み）

KSC スクリプトで記述できる完全な RPG フロー:

```
*start
@map_load map-village 5 8 down          # 村にスポーン

*after_map
# マップ内でプレイヤーが自由移動
# イベント発火 → scenarioId で以下が実行される

*npc_elder
村長「勇者よ、北の洞窟に魔物が出たのじゃ」
@choice
  "行きます" → *accept_quest
  "まだ準備が…" → *decline_quest

*accept_quest
@set quest_cave = 1
村長「頼んだぞ！」
@item_add potion 3
@map_exit                              # マップに戻る

*cave_boss
@fade_black 500
@battle goblin_king onWin=*boss_win onLose=*boss_lose

*boss_win
@set quest_cave = 2
@item_add treasure_key 1
「ボスを倒した！鍵を手に入れた」
@map_exit

*boss_lose
@fade_black 1000
@map_load map-village 5 8 down          # 村に戻される
村長「無理をするでない…」
```

## 6. まとめ

| 観点 | 評価 |
|------|------|
| ドット絵なしで RPG が作れるか | **YES** — プレースホルダーで完全動作 |
| 必要なシステムは揃っているか | **YES** — マップ・戦闘・セーブ・イベント全実装 |
| 推奨アプローチ | **プレースホルダーで先にゲームを作り、絵は後から差し替え** |
| ドット絵生成が難しい理由 | AI 生成は品質のバラつき + API コスト。手描きは時間がかかる |
| 絵の差し替えコスト | **PNG 入れ替えのみ** — ロジック変更ゼロ |
